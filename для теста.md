### Зачем нужен lock-файл (poetry.lock, pip-tools)?

-«Lock-файл фиксирует точные версии всех пакетов, включая транзитивные зависимости. Это гарантирует детерминированную сборку: окружение на локальной машине, в CI и на проде будет идентичным. Без lock-файла возможна ситуация "у меня работает, а на проде нет" из-за автообновления зависимостей.»

### Опишите типичный Dockerfile
### Опишите типичный Dockerfile для Python-приложения: базовый образ, рабочая директория, копирование зависимостей, установка пакетов, копирование кода, команда запуска.
2/5
Слабо
Как уменьшить размер итогового образа?

FROM python:3.11-slim  # ✅ Базовый образ

WORKDIR /app  # ✅ Рабочая директория

# ✅ Копируем только зависимости (для кэширования слоя)
COPY requirements.txt .

# ✅ Установка пакетов
RUN pip install --no-cache-dir -r requirements.txt

# ✅ Копируем код (после установки, чтобы реже пересобирать)
COPY . .

# ✅ Запуск
CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]

### Как уменьшить размер итогового образа?
1. Выбрать минимальный базовый образ (`alpine`/`slim`).
2. Использовать multi-stage для компилируемых языков/сборки.
3. Очистить кеш пакетов в том же `RUN`, где установка.
4. Настроить `.dockerignore`.
5. Запускать от имени non-root пользователя (безопасность).

### Alpine vs Slim
| Образ      | Размер | libc  | Когда использовать                            |
| ---------- | ------ | ----- | --------------------------------------------- |
| **alpine** | ~5 MB  | musl  | Микросервисы, где важна минимальность         |
| **slim**   | ~40 MB | glibc | Если нужны нативные бинарники (numpy, pandas) |
| **full**   | ~1 GB  | glibc | Отладка, редкие кейсы                         |

> ⚠️ **Alpine caveat:** musl libc может ломать C-расширения. Тестируй перед использованием.

#### Где хранится конфигурация пайплайна?
**Конфигурация пайплайна (CI/CD) хранится в разных местах в зависимости от инструмента**. Ниже приведены примеры для GitHub Actions, GitLab CI и Jenkins. [habr.com](https://habr.com/ru/companies/slurm/articles/723608/)[pressanybutton.ru](https://pressanybutton.ru/post/devops/gitlab-osnovy-napisaniya-pipeline-13/)[tproger.ru](https://tproger.ru/articles/kak-nastroit-i-ispolzovat-jenkins-dlya-avtomatizacii-processov)

## GitHub Actions

Конфигурация пайплайна (workflow) хранится **в директории .github/workflows** в корне проекта. Там размещается YAML-файл с описанием пайплайна. В файле описываются, например: [adminvps.ru](https://adminvps.ru/blog/kak-nastroit-ci-cd-s-github-actions/)[sky.pro](https://sky.pro/wiki/profession/primery-cicd-pajplajnov-prakticheskoe-rukovodstvo/)

- что должно происходить при конкретных событиях (например, запуск пайплайна при пуше в ветку main);
- последовательности задач.

 [adminvps.ru](https://adminvps.ru/blog/kak-nastroit-ci-cd-s-github-actions/)

Actions активируются автоматически после добавления и коммита конфигурационного файла в ветку. [adminvps.ru](https://adminvps.ru/blog/kak-nastroit-ci-cd-s-github-actions/)

## GitLab CI

Конфигурация пайплайна (pipeline) по умолчанию хранится **в файле .gitlab-ci.yml** в корне репозитория. Этот файл определяет: [habr.com](https://habr.com/ru/companies/slurm/articles/723608/)[pressanybutton.ru](https://pressanybutton.ru/post/devops/gitlab-osnovy-napisaniya-pipeline-13/)[slurm.io](https://slurm.io/blog/nastrojka-cicd-v-gitlab)

- этапы (stages): сборка, тестирование, деплой;
- задачи (jobs): конкретные действия на каждом этапе;
- условия запуска: когда и при каких событиях выполнять ту или иную задачу;
- переменные окружения: безопасная работа с ключами, логинами и паролями.

 [slurm.io](https://slurm.io/blog/nastrojka-cicd-v-gitlab)

GitLab автоматически отслеживает наличие и изменения этого файла: как только он появляется в репозитории или в него вносятся правки, GitLab сам применяет конфигурацию и запускает пайплайн при следующем коммите. Для крупных проектов есть механизм include, который позволяет выносить части конфигурации в отдельные файлы и подключать их при необходимости. [pressanybutton.ru](https://pressanybutton.ru/post/devops/gitlab-osnovy-napisaniya-pipeline-13/)

## Jenkins

Конфигурация пайплайна (pipeline) хранится **в файле Jenkinsfile** в репозитории с кодом проекта. В одном репозитории можно хранить множество Jenkinsfile, создавая по одному такому файлу для каждой задачи. Можно размещать такие файлы в одном каталоге, давая им разные имена, или в разных каталогах, оставив традиционное имя Jenkinsfile для всех файлов. [tproger.ru](https://tproger.ru/articles/kak-nastroit-i-ispolzovat-jenkins-dlya-avtomatizacii-processov)[infostart.ru](https://infostart.ru/1c/articles/1210995/)

Путь к репозиторию указывается в настройках задачи, а вместе с ним — из какого файла внутри репозитория нужно извлекать исполняемый код пайплайна. Перед выполнением задачи Jenkins извлекает указанный файл из репозитория и читает исполняемый код пайплайна из этого файла, вместо того чтобы брать его из xml-файла с настройками самой задачи. [infostart.ru](https://infostart.ru/1c/articles/1210995/)

### Какие этапы обычно входят в пайплайн, пайтон докер
**Пайплайн (конвейер CI/CD) с использованием Python и Docker** — это автоматизированная последовательность действий для интеграции, тестирования и доставки обновлений программного обеспечения. Docker позволяет упаковывать приложение вместе со всеми зависимостями в контейнер, что упрощает работу с тестами и сборкой: можно быстро поднять среду, прогнать автоматические тесты и убедиться, что код одинаково работает везде. [dzen.ru](https://dzen.ru/a/Z4kdBqzxeVqu96hO)[cio-navigator.ru](https://cio-navigator.ru/ci-cd-full-guide/)

## Этапы

Некоторые этапы пайплайна с использованием Python и Docker:

- **Извлечение и интеграция кода** (Source/Integration) — изменения кода загружаются в систему контроля версий (например, Git). Сразу запускается проверка синтаксиса и сборка проекта для обнаружения конфликтов и ошибок на ранних стадиях.
- **Компиляция и сборка** (Build) — исходный код компилируется или собирается в пакеты, создаются контейнеры или артефакты, необходимые для тестирования и развёртывания.
- **Автоматическое тестирование** (Test) — выполняются модульные, интеграционные и функциональные тесты, иногда подключаются статический анализ кода, проверки безопасности и соответствия стандартам.
- **Проверка качества и контроль** (Validation/QA) — анализируются результаты тестирования и проверяются соответствия требований. Иногда добавляются контрольные точки для утверждения релиза ответственными лицами.
- **Развёртывание** (Deploy/Delivery) — приложение автоматически разворачивается в промежуточной или тестовой среде. При необходимости применяется ручное подтверждение.

 [cio-navigator.ru](https://cio-navigator.ru/ci-cd-full-guide/)

Дополнительно конвейер может включать анализ кода, проверку безопасности, настройку переменных окружения и уведомления — всё зависит от пожеланий разработчика. [cio-navigator.ru](https://cio-navigator.ru/ci-cd-full-guide/)

## что такое артефакты и кэш в CI
**Артефакты** — это файлы, которые создаются во время выполнения задачи и сохраняются для передачи на следующие этапы. Некоторые примеры: [hmarketing.ru](https://hmarketing.ru/blog/git/fayl-gitlab-ci-yml/)[fastfox.pro](https://fastfox.pro/blog/tutorials/gitlab-ci-cache-artifacts-disk-control/)

- собранные архивы;
- отчёты тестов;
- результаты сканирования.
**Кэш** — это одно или несколько файлов, которые задача загружает и сохраняет. **Цель кэша** — ускорить выполнение задач, позволяя повторно использовать данные между запусками

#### Как тестировать асинхронный код в pytest? Какая библиотека и пометка нужны?
- **Библиотека:** `pytest-asyncio`.
- **Настройка:** В `pytest.ini` можно включить `asyncio_mode = auto`, чтобы не ставить маркер на каждый тест.
```python
# conftest.py или pytest.ini
# asyncio_mode = auto

# test_async.py
import pytest

@pytest.mark.asyncio  # Обязательно, если не включен auto
async def test_fetch_data():
    result = await fetch_data()  # Вызов асинхронной функции
    assert result == {"status": "ok"}
```


- **Нюанс:** Фикстуры тоже должны быть асинхронными (`async def fixture`), если используются в async тестах.
