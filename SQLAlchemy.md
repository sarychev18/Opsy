### Модели
1. **КРАТКИЙ ОТВЕТ (для интервьюера):**
   - «**Модели в SQLAlchemy** — это Python-классы, которые мапятся на таблицы базы данных.
   - Атрибуты класса соответствуют колонкам таблицы, а экземпляры класса — строкам (rows).
   - Они наследуются от `DeclarativeBase` (в SQLAlchemy 2.0) и позволяют работать с БД через объекты, а не сырой SQL 🚀».

2. **ПОЛНЫЙ ОТВЕТ (для глубокого понимания):**

   **👵 Как для бабушки:**
   - **Модель:** Это как анкета для сотрудника.
     - **Класс (Модель):** Сама бланк анкеты с графами «Имя», «Возраст», «Должность» (структура таблицы).
     - **Экземпляр:** Заполненная анкета конкретного человека «Иван, 30 лет, Инженер» (строка в таблице).
     - **ORM:** Секретарь, который берет заполненную анкету и переписывает данные в большую книгу (Базу Данных), чтобы ты не делал это вручную ✍️.

   **👨‍💻 Для глубокого понимания (Technical Deep Dive):**
   Для Middle разработчика важно знать синтаксис SQLAlchemy 2.0 и нюансы маппинга.

   **1. Declarative System (Объявление моделей):**
   В современной SQLAlchemy (2.0+) используется `DeclarativeBase`.
   ```python
   from sqlalchemy import Column, Integer, String
   from sqlalchemy.orm import DeclarativeBase
   
   class Base(DeclarativeBase):
       pass
   
   class User(Base):
       __tablename__ = "users"  # Имя таблицы в БД
       
       id = Column(Integer, primary_key=True)
       name = Column(String(50), nullable=False)
       email = Column(String(100), unique=True)
   ```
   - **Важно:** Без `__tablename__` SQLAlchemy не знает, как создать таблицу.

   **2. Типизация (SQLAlchemy 2.0 Style):**
   Новый стиль использует аннотации типов Python.
   ```python
   from sqlalchemy.orm import Mapped, mapped_column
   
   class User(Base):
       __tablename__ = "users"
       
       id: Mapped[int] = mapped_column(primary_key=True)
       name: Mapped[str] = mapped_column(String(50))
   ```
   - **Плюс:** Лучшая поддержка IDE и проверка типов (mypy).

   **3. Связи (Relationships):**
   Модели могут ссылаться друг на друга (Foreign Keys).
   ```python
   class Post(Base):
       __tablename__ = "posts"
       id = Column(Integer, primary_key=True)
       user_id = Column(Integer, ForeignKey("users.id"))  # Связь
       author = relationship("User")  # Объект пользователя
   ```

   **4. Жизненный цикл (Session):**
   Модель сама по себе не сохраняет данные. Нужна `Session`.
   ```python
   user = User(name="Alex", email="alex@test.com")
   session.add(user)      # Добавляем в сессию
   session.commit()       # Сохраняем в БД (INSERT)
   ```

   **🎯 Совет для Middle:**
   На собеседовании скажи: «Я предпочитаю стиль SQLAlchemy 2.0 с аннотацией типов (`Mapped`), так как это улучшает читаемость и поддержку кода в больших проектах».

### Сессии
1. **КРАТКИЙ ОТВЕТ (для интервьюера):**
   - «**Session** в SQLAlchemy — это рабочая область (workspace) для объектов, а не просто соединение с БД.
   - Она реализует паттерн **Unit of Work**: отслеживает изменения объектов и отправляет их в БД при `commit()`.
   - Также реализует **Identity Map**: гарантирует, что в рамках сессии один объект БД будет представлен одним Python-объектом 🚀».

2. **ПОЛНЫЙ ОТВЕТ (для глубокого понимания):**

   **👵 Как для бабушки:**
   - **Сессия:** Представь корзину в супермаркете 🛒.
     - Ты кладешь товары (объекты) в корзину. Они еще не оплачены (не в БД).
     - Ты можешь убрать товар, заменить его, добавить новый.
     - **Commit:** Ты идешь на кассу и оплачиваешь. Теперь покупка совершена (записано в БД).
     - **Rollback:** Ты передумал и выложил всё обратно. Магазин ничего не потерял.
     - **Соединение (Connection):** Это тележка для перевозки денег между банком и кассой. Корзина (сессия) важнее для покупателя.

   **👨‍💻 Для глубокого понимания (Technical Deep Dive):**
   Для Middle разработчика важно различать `Connection` и `Session`, а также понимать этапы жизненного цикла.

   **1. Session vs Connection:**
   - **Connection:** Низкоуровневое соединение с БД (пул соединений в Engine).
   - **Session:** Высокоуровневый менеджер транзакций. Использует соединение внутри.
   - **Правило:** Одна сессия на один запрос (request) в веб-приложении.

   **2. Жизненный цикл объекта (Object States):**
   - **Transient:** Объект создан, но не в сессии (`user = User()`).
   - **Pending:** Объект добавлен в сессию (`session.add(user)`), но не сохранен.
   - **Persistent:** Объект в сессии и есть в БД (после `commit` или `flush`).
   - **Detached:** Объект был в БД, но сессия закрыта. Доступ к данным есть, но сохранить изменения нельзя.

   **3. Commit vs Flush:**
   - **`flush()`:** Отправляет SQL-запросы в БД, но не завершает транзакцию. ID становится доступен.
   - **`commit()`:** Фиксирует транзакцию. После этого `flush` происходит автоматически.
   ```python
   session.add(obj)
   session.flush()  # obj.id уже доступен, но можно сделать rollback
   session.commit() # Всё сохранено окончательно
   ```

   **4. AsyncSession (для FastAPI):**
   В асинхронном коде используем `AsyncSession`, чтобы не блокировать поток.
   ```python
   from sqlalchemy.ext.asyncio import AsyncSession
   
   async def get_user(session: AsyncSession, user_id: int):
       result = await session.get(User, user_id)
       return result
   ```

   **5. Best Practices (Middle Level):**
   - **Dependency Injection:** Передавай сессию через `Depends(get_session)` в FastAPI.
   - **Close Session:** Всегда закрывай сессию после запроса (через `finally` или контекстный менеджер), чтобы не было утечек соединений.
   - **Expiration:** После `commit()` атрибуты объекта могут стать "истекшими". При доступе к ним сессия попробует подгрузить данные снова (если не закрыта).

   **🎯 Совет для Middle:**
   На собеседовании скажи: «Я понимаю, что сессия — это Unit of Work. Я использую `async_session` для FastAPI, обязательно закрываю её после запроса и стараюсь не держать транзакции открытыми дольше нужного, чтобы не блокировать таблицы».

### Подключение к Базе дынных
1. **КРАТКИЙ ОТВЕТ (для интервьюера):**
   - «Подключение через `create_engine(connection_url)`.
   - **Engine** — это фабрика соединений и пул (pool), он потокобезопасен и создается один раз на приложение.
   - **URL:** Содержит диалект, драйвер, пользователя, пароль, хост и имя БД.
   - Для FastAPI использую `create_async_engine` и `AsyncSession`, чтобы не блокировать event loop 🚀».

2. **ПОЛНЫЙ ОТВЕТ (для глубокого понимания):**

   **👵 Как для бабушки:**
   - **Engine:** Это электростанция 🏭. Она не светит лампочки сама, но вырабатывает электричество (соединения), которое идет по проводам в дома.
   - **Connection String:** Это адрес электростанции («Газопровод, улица Ленина, дом 5»). Если адрес неверный — света не будет.
   - **Pool (Пул):** Это очередь рабочих у станции. Когда тебе нужно электричество, ты берешь свободного рабочего. Когда закончил — он возвращается в очередь, а не увольняется. Это экономит время на найме новых ⚡.

   **👨‍💻 Для глубокого понимания (Technical Deep Dive):**
   Для Middle разработчика важно понимать пулинг и безопасность учетных данных.

   **1. Создание Engine (Sync vs Async):**
   ```python
   from sqlalchemy import create_engine
   from sqlalchemy.ext.asyncio import create_async_engine
   
   # Синхронный (для скриптов, celery)
   engine = create_engine("postgresql+psycopg2://user:pass@localhost/db")
   
   # Асинхронный (для FastAPI)
   async_engine = create_async_engine("postgresql+asyncpg://user:pass@localhost/db")
   ```
   - **Важно:** Engine создается **один раз** при старте приложения (singleton).

   **2. Connection URL Structure:**
   `dialect+driver://username:password@host:port/database`
   - **Dialect:** `postgresql`, `mysql`, `sqlite`.
   - **Driver:** `psycopg2` (sync), `asyncpg` (async), `pymysql`.
   - **Security:** Никогда не хардкодь пароль! Используй `os.getenv("DB_URL")`.

   **3. Connection Pooling (Пул соединений):**
   - **Зачем:** Создание нового TCP-соединения с БД — дорогая операция. Пул держит готовые соединения открытыми.
   - **Параметры:**
     - `pool_size`: Количество постоянных соединений (по умолчанию 5).
     - `max_overflow`: Сколько можно создать сверх лимита (по умолчанию 10).
     - `pool_pre_ping`: Проверка соединения перед использованием (защита от разрывов).
   ```python
   engine = create_engine(
       DATABASE_URL,
       pool_size=10,
       max_overflow=20,
       pool_pre_ping=True  # Важно для продакшена!
   )
   ```

   **4. Echo и Логирование:**
   - `echo=True`: Выводит все SQL-запросы в консоль. Удобно для разработки, **запрещено на проде** (замусоривает логи).

   **🎯 Совет для Middle:**
   На собеседовании скажи: «Я настраиваю `pool_pre_ping=True`, чтобы избегать ошибок "MySQL has gone away". Пароли храню только в ENV, а Engine выношу в отдельный модуль `database.py`».
