## Основы HTTP и сетевые протоколы

### 1. Что происходит при вводе URL в браузере
**Вопрос:** Опиши, что происходит, когда пользователь вводит URL в браузере.

**Ответ:** Браузер сначала разбирает URL на составляющие (схема, домен, путь) и проверяет кэш DNS, чтобы получить IP-адрес сервера по доменному имени, при отсутствии в кэше — обращается к DNS-резолверу. Затем устанавливается TCP-соединение с сервером (three-way handshake), поверх которого при HTTPS выполняется TLS-рукопожатие для согласования шифрования. После этого браузер отправляет HTTP-запрос, сервер обрабатывает его (маршрутизация, бизнес-логика, обращение к БД) и возвращает HTTP-ответ с телом (HTML/JSON) и заголовками. Наконец браузер парсит ответ, строит DOM/CSSOM, при необходимости запрашивает дополнительные ресурсы (CSS, JS, изображения) и рендерит страницу.

| Термин | Определение |
|---|---|
| DNS-резолюция | Преобразование доменного имени в IP-адрес |
| TCP handshake | Установка соединения через обмен SYN/SYN-ACK/ACK |
| TLS handshake | Согласование шифрования для HTTPS-соединения |
| DOM | Объектная модель документа, построенная браузером из HTML |

**Ответ для интервьюера:** Сначала браузер резолвит домен в IP через DNS, затем устанавливает TCP-соединение (и TLS для HTTPS), отправляет HTTP-запрос и получает ответ от сервера. После этого браузер парсит HTML, строит DOM/CSSOM, подгружает дополнительные ресурсы и рендерит страницу на экране.

### 2. Этапы загрузки веб-страницы
**Вопрос:** Объясни все этапы загрузки веб-страницы: DNS-запрос, TCP-подключение, HTTP-запрос, обработка на сервере, HTTP-ответ, рендеринг.

**Ответ:** DNS-запрос переводит доменное имя в IP-адрес через иерархию серверов (локальный резолвер → корневые → TLD → авторитативный сервер), с кэшированием на каждом уровне. TCP-подключение устанавливается через трёхстороннее рукопожатие (SYN, SYN-ACK, ACK), гарантируя надёжный канал передачи данных; для HTTPS сверху добавляется TLS-рукопожатие. HTTP-запрос отправляется с методом, путём, заголовками и опциональным телом; сервер обрабатывает его — маршрутизирует к нужному обработчику, выполняет бизнес-логику, обращается к БД/кэшу — и формирует HTTP-ответ со статус-кодом, заголовками и телом. Финальный этап — рендеринг: браузер парсит HTML в DOM, CSS в CSSOM, строит render tree, вычисляет layout и отрисовывает пиксели (paint), а при наличии JS может дополнительно перестраивать DOM после загрузки.

```
Client                                   Server
  |-- DNS query -------------------------->|
  |<- IP address --------------------------|
  |-- TCP SYN ------------------------------>|
  |<- SYN-ACK -------------------------------|
  |-- ACK ------------------------------------>|
  |-- HTTP GET /page ------------------------->|
  |<- HTTP 200 OK + body -----------------------|
```

| Термин | Определение |
|---|---|
| Авторитативный DNS-сервер | Сервер, хранящий актуальную запись домена |
| Render tree | Дерево видимых элементов страницы для отрисовки |
| Layout / Paint | Вычисление позиций элементов и отрисовка пикселей |

**Ответ для интервьюера:** Последовательность: DNS резолвит домен в IP, TCP (плюс TLS для HTTPS) устанавливает соединение, браузер отправляет HTTP-запрос, сервер его обрабатывает и возвращает ответ, а браузер парсит HTML/CSS в DOM/CSSOM и рендерит страницу через построение render tree, layout и paint.

### 3. HTTP-методы GET, POST, PUT, PATCH, DELETE
**Вопрос:** В чем разница между HTTP-методами GET, POST, PUT, PATCH, DELETE?

**Ответ:** GET запрашивает представление ресурса, не должен иметь тела запроса и является безопасным (не изменяет состояние) и идемпотентным. POST создаёт новый ресурс или выполняет неидемпотентное действие — повторный вызов может создать дубликат. PUT полностью заменяет ресурс по конкретному URL и идемпотентен — повторный вызов с теми же данными даёт тот же результат. PATCH частично обновляет ресурс, применяя только переданные изменения, и в общем случае не гарантированно идемпотентен. DELETE удаляет ресурс и идемпотентен — повторное удаление уже удалённого ресурса не меняет состояние сервера повторно.

|Метод|Идемпотентен|Безопасен|Тело запроса|Назначение|
|---|---|---|---|---|
|**GET**|✅|✅|Нет|Получить ресурс|
|**POST**|❌|❌|Есть|Создать ресурс|
|**PUT**|✅|❌|Есть|Полностью заменить ресурс|
|**PATCH**|❌ (не гарантирован)|❌|Есть|Частично обновить ресурс|
|**DELETE**|✅|❌|Обычно нет|Удалить ресурс|

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/items/{item_id}")
async def read_item(item_id: int):
    ...

@app.post("/items")
async def create_item(item: dict):
    ...

@app.put("/items/{item_id}")
async def replace_item(item_id: int, item: dict):
    ...

@app.patch("/items/{item_id}")
async def update_item(item_id: int, fields: dict):
    ...

@app.delete("/items/{item_id}")
async def delete_item(item_id: int):
    ...
```

| Термин | Определение |
|---|---|
| Идемпотентность | Повторный вызов даёт тот же результат, что и один |
| Безопасный метод | Метод, не изменяющий состояние сервера (GET) |
| Частичное обновление | Изменение только переданных полей ресурса (PATCH) |

**Ответ для интервьюера:** GET читает ресурс безопасно и идемпотентно, POST создаёт ресурс и неидемпотентен, PUT полностью заменяет ресурс идемпотентно, PATCH частично обновляет и не гарантирует идемпотентность, DELETE удаляет ресурс идемпотентно.

### 4. HTTP статусы (2xx, 3xx, 4xx, 5xx)
**Вопрос:** Что такое HTTP статусы? Приведи примеры групп 2xx, 3xx, 4xx, 5xx.

**Ответ:** HTTP-статус — трёхзначный код в ответе сервера, сообщающий клиенту результат обработки запроса; первая цифра определяет класс ответа. 1xx — информационные (запрос принят, обработка продолжается). 2xx — успех: 200 OK (успешный запрос), 201 Created (ресурс создан), 204 No Content (успех без тела ответа). 3xx — редиректы: 301 Moved Permanently, 304 Not Modified. 4xx — ошибка клиента: 400 Bad Request (некорректные данные), 401 Unauthorized (не аутентифицирован), 403 Forbidden (нет прав), 404 Not Found, 422 Unprocessable Entity (ошибка валидации, характерна для FastAPI). 5xx — ошибка сервера: 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable.

```python
from fastapi import FastAPI, HTTPException

app = FastAPI()

@app.get("/items/{item_id}")
async def read_item(item_id: int):
    if item_id not in db:
        raise HTTPException(status_code=404, detail="Item not found")
    return db[item_id]
```

| Термин | Определение                                                 |
| ------ | ----------------------------------------------------------- |
| 2xx    | Класс успешных ответов                                      |
| 3xx    | Класс редиректов                                            |
| 4xx    | Класс ошибок на стороне клиента                             |
| 5xx    | Класс ошибок на стороне сервера                             |
| 422    | Ошибка валидации тела/параметров запроса (FastAPI/Pydantic) |

**Ответ для интервьюера:** Первая цифра статуса определяет класс ответа: 2xx — успех (200, 201, 204), 3xx — редирект (301, 304), 4xx — ошибка клиента (400, 401, 403, 404, 422), 5xx — ошибка сервера (500, 502, 503). В FastAPI 422 автоматически возвращается при провале валидации Pydantic.

### 5. TCP vs UDP
**Вопрос:** Объясни разницу между протоколами TCP и UDP.

**Ответ:** TCP (Transmission Control Protocol) — соединение-ориентированный протокол: устанавливает соединение через three-way handshake, гарантирует доставку, порядок пакетов и контроль перегрузки за счёт подтверждений (ACK) и повторной отправки, что делает его надёжным, но более медленным и с бОльшими накладными расходами. UDP (User Datagram Protocol) — протокол без установления соединения, отправляет датаграммы без гарантии доставки, порядка или повторной отправки, зато с минимальной задержкой и накладными расходами. HTTP исторически работает поверх TCP (надёжность важнее задержки для веб-страниц), а UDP используют там, где важна скорость и допустимы потери — стриминг видео, VoIP, DNS-запросы, а также HTTP/3 (QUIC), который построен поверх UDP.

|Характеристика|TCP|UDP|
|---|---|---|
|**Соединение**|✅ Устанавливается (handshake)|❌ Без соединения|
|**Надежность**|✅ Гарантированная доставка, порядок пакетов|❌ Нет гарантий, пакеты могут потеряться|
|**Скорость**|Медленнее (оверхед на надежность)|**Быстрее** (минимальный оверхед)|
|**Use-cases**|HTTP, FTP, email|DNS, VoIP, видео, онлайн-игры, real-time данные|

| Термин    | Определение                                              |
| --------- | -------------------------------------------------------- |
| TCP       | Надёжный протокол с гарантией доставки и порядка         |
| UDP       | Быстрый протокол без гарантий доставки и порядка         |
| Handshake | Установка соединения перед передачей данных (только TCP) |
| QUIC      | Транспортный протокол HTTP/3, построенный поверх UDP     |

**Ответ для интервьюера:** TCP гарантирует надёжную доставку и порядок пакетов через handshake и подтверждения ценой задержки, UDP отправляет датаграммы без гарантий, зато быстро и с минимальными накладными расходами. TCP используют там, где важна целостность данных (HTTP/1.1, HTTP/2), UDP — где важна скорость (видео, VoIP, DNS, HTTP/3 через QUIC).

### 6. Модель OSI и уровни HTTP/TCP
**Вопрос:** Что такое модель OSI и на каких уровнях работают HTTP и TCP?

**Ответ:** Модель OSI — концептуальная модель сетевого взаимодействия из семи уровней: физический, канальный, сетевой, транспортный, сеансовый, представления и прикладной, каждый из которых решает свою задачу и общается только с соседними уровнями. TCP работает на транспортном уровне (4) — отвечает за доставку данных между процессами через порты, надёжность и контроль потока. HTTP работает на прикладном уровне (7) — определяет формат запроса/ответа для обмена данными между клиентом и сервером, используя TCP (или QUIC/UDP в HTTP/3) как транспорт ниже.

|№|Уровень|Что делает|Примеры протоколов|
|---|---|---|---|
|7|Прикладной (Application)|Формат данных для приложений|HTTP, FTP, DNS|
|6|Представления (Presentation)|Кодирование, сжатие, шифрование|TLS/SSL, JPEG|
|5|Сеансовый (Session)|Установка и поддержка сеанса связи|NetBIOS, RPC|
|4|Транспортный (Transport)|Доставка между процессами, надёжность|TCP, UDP|
|3|Сетевой (Network)|Маршрутизация между сетями|IP, ICMP|
|2|Канальный (Data Link)|Передача кадров в локальной сети|Ethernet, Wi-Fi|
|1|Физический (Physical)|Передача битов по физической среде|Кабели, радиосигнал|

![[Pasted image 20260717094845.png]]

| Термин | Определение |
|---|---|
| Транспортный уровень (L4) | Уровень OSI, отвечающий за доставку между процессами (TCP/UDP) |
| Прикладной уровень (L7) | Уровень OSI с протоколами приложений (HTTP, FTP, DNS) |
| Инкапсуляция | Обёртывание данных заголовками каждого нижележащего уровня |

**Ответ для интервьюера:** OSI — модель из семи уровней абстракции сетевого взаимодействия. TCP работает на транспортном уровне (L4), обеспечивая доставку данных между процессами через порты, а HTTP — на прикладном уровне (L7), определяя формат запроса и ответа поверх транспорта.

### 7. WebSockets vs HTTP/2
**Вопрос:** В чем разница между WebSockets и HTTP 2.0?

**Ответ:** WebSocket — отдельный протокол, устанавливающий постоянное двунаправленное (full-duplex) соединение поверх TCP после начального HTTP-рукопожатия (Upgrade-заголовок), после чего сервер и клиент могут независимо отправлять сообщения друг другу в любой момент — используется для чатов, live-обновлений, торгов в реальном времени. HTTP/2 остаётся request-response протоколом, но добавляет мультиплексирование нескольких запросов в одном TCP-соединении, бинарный формат фреймов, сжатие заголовков (HPACK) и server push — это ускоряет обычную загрузку страниц с множеством ресурсов, но не даёт серверу самостоятельно и произвольно инициировать сообщения клиенту вне ответа на запрос.

|Характеристика|WebSocket|HTTP/2|
|---|---|---|
|**Модель**|Постоянное двунаправленное соединение|Запрос-ответ (пусть и мультиплексированный)|
|**Инициатива сервера**|✅ Может слать данные в любой момент|❌ Только в ответ на запрос (server push — не то же самое)|
|**Установка**|HTTP Upgrade-рукопожатие, затем отдельный протокол|Остаётся HTTP-семантикой поверх бинарных фреймов|
|**Типичное применение**|Чаты, live-обновления, realtime-торги|Быстрая загрузка страниц с множеством ресурсов|

```python
from fastapi import FastAPI, WebSocket

app = FastAPI()

@app.websocket("/ws")
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept()
    while True:
        data = await websocket.receive_text()
        await websocket.send_text(f"echo: {data}")
```

| Термин | Определение |
|---|---|
| Full-duplex | Одновременная двусторонняя передача данных |
| Мультиплексирование | Параллельная передача нескольких запросов в одном соединении |
| Upgrade-заголовок | HTTP-заголовок для перехода с HTTP на WebSocket-протокол |

**Ответ для интервьюера:** WebSocket — отдельный протокол с постоянным двунаправленным соединением, где сервер может сам инициировать отправку данных в любой момент. HTTP/2 остаётся запрос-ответной моделью, но ускоряет её мультиплексированием запросов в одном TCP-соединении и сжатием заголовков — для realtime-коммуникации всё равно нужен WebSocket.

## Архитектурный стиль REST

### 8. REST API и его принципы
**Вопрос:** Что такое REST API и какие основные принципы вы знаете?

**Ответ:** REST (Representational State Transfer) — архитектурный стиль для проектирования сетевых API, описанный Роем Филдингом, основанный на работе с ресурсами через стандартные HTTP-методы. Ключевые принципы: клиент-серверная архитектура (разделение ответственности UI и хранения данных), stateless (отсутствие состояния сессии на сервере между запросами), кэшируемость ответов, единообразный интерфейс (uniform interface — ресурсы идентифицируются URI, манипулируются через представления), многослойность системы (client не знает, общается ли он напрямую с сервером или через прокси) и опционально code-on-demand. API, следующее этим принципам, называется RESTful.

| Термин | Определение |
|---|---|
| Ресурс | Сущность, доступная по URI (пользователь, заказ) |
| Uniform interface | Единообразный способ взаимодействия с ресурсами |
| Stateless | Отсутствие хранения состояния клиента на сервере между запросами |
| HATEOAS | Гипермедиа как двигатель состояния приложения (опциональный принцип REST) |

**Ответ для интервьюера:** REST — архитектурный стиль API поверх HTTP, где всё представлено как ресурсы с уникальным URI, а взаимодействие идёт через стандартные методы. Ключевые принципы: клиент-серверность, statelessness, кэшируемость, единообразный интерфейс и многослойность.

---

### 9. Клиент-серверная архитектура
**Вопрос:** Объясни концепцию клиент-серверной архитектуры.

**Ответ:** Клиент-серверная архитектура разделяет систему на две роли: клиент инициирует запросы и отвечает за представление данных пользователю (UI/UX), а сервер хранит данные, обрабатывает бизнес-логику и отвечает на запросы. Это разделение ответственности позволяет независимо развивать и масштабировать клиент и сервер (например, менять фронтенд, не трогая бэкенд, или горизонтально масштабировать сервер под нагрузку), а также поддерживать разные типы клиентов (веб, мобильные приложения) для одного и того же API.

| Термин | Определение |
|---|---|
| Клиент | Инициатор запроса, отвечающий за представление данных |
| Сервер | Обработчик запросов, хранитель данных и бизнес-логики |
| Разделение ответственности | Независимое развитие клиентской и серверной части |

**Ответ для интервьюера:** Клиент отвечает за представление и пользовательский опыт, сервер — за хранение данных и бизнес-логику, они взаимодействуют через сеть по чётко определённому протоколу. Такое разделение позволяет независимо масштабировать и развивать обе части и поддерживать разные типы клиентов для одного API.

### 10. Statelessness в REST
**Вопрос:** Что означает отсутствие состояния (Statelessness) в REST?

**Ответ:** Statelessness означает, что сервер не хранит контекст клиента (сессию) между запросами — каждый запрос должен содержать всю информацию, необходимую для его обработки (например, токен аутентификации), и сервер не полагается на то, что было в предыдущих запросах. Это упрощает горизонтальное масштабирование — любой инстанс сервера может обработать любой запрос, не нужна привязка клиента к конкретному серверу (sticky session) или синхронизация состояния сессии между инстансами. Плата за это — клиент вынужден каждый раз пересылать больше данных (например, токен вместо ID сессии), а долгоживущее состояние (аутентификация) хранится вне сервера — в токене на стороне клиента или во внешнем хранилище (Redis).

```python
from fastapi import Depends, Header, HTTPException

async def verify_token(authorization: str = Header(...)):
    if not is_valid(authorization):
        raise HTTPException(status_code=401, detail="Invalid token")
    return authorization

@app.get("/profile")
async def profile(token: str = Depends(verify_token)):
    ...
```

| Термин | Определение |
|---|---|
| Stateless | Отсутствие хранения состояния клиента на сервере |
| Sticky session | Привязка клиента к конкретному серверному инстансу |
| Токен аутентификации | Самодостаточное подтверждение личности в каждом запросе |

**Ответ для интервьюера:** Statelessness означает, что сервер не хранит контекст между запросами — каждый запрос самодостаточен и несёт всё необходимое, включая аутентификацию. Это позволяет свободно масштабировать сервер горизонтально, так как любой инстанс может обработать любой запрос без синхронизации сессий.

### 11. Проектирование RESTful endpoints
**Вопрос:** Как правильно проектировать RESTful endpoints?

**Ответ:** URI должен именовать ресурс существительным во множественном числе (`/users`, `/orders`), а не действие — действие выражается HTTP-методом (`POST /orders` вместо `/createOrder`). Вложенность отражает иерархию отношений (`/users/{id}/orders`), но не должна уходить глубже 2-3 уровней. Фильтрация, сортировка и пагинация выносятся в query-параметры (`/orders?status=paid&limit=20`), а версионирование API — либо в путь (`/v1/orders`), либо в заголовок. Ответы должны использовать корректные статус-коды (201 для создания, 204 для удаления без тела) и единообразный формат JSON, включая структуру ошибок.

```python
@app.get("/users/{user_id}/orders")
async def list_user_orders(user_id: int, status: str | None = None, limit: int = 20):
    ...

@app.post("/users/{user_id}/orders", status_code=201)
async def create_order(user_id: int, order: OrderCreate):
    ...
```

| Термин | Определение |
|---|---|
| Ресурс во множественном числе | Соглашение именования URI (/users, не /user) |
| Query-параметры | Параметры фильтрации/пагинации в строке запроса |
| Версионирование API | Явное указание версии контракта API (/v1/...) |

**Ответ для интервьюера:** URI именуют ресурсы существительными во множественном числе, а действие выражается HTTP-методом, а не глаголом в пути. Фильтрацию, сортировку и пагинацию выношу в query-параметры, вложенность ограничиваю 2-3 уровнями и явно версионирую API, чтобы не ломать старых клиентов.

### 12. Endpoint в веб-приложении
**Вопрос:** Что такое endpoint в веб-приложении?

**Ответ:** Endpoint — это конкретная комбинация URL-пути и HTTP-метода, за которой закреплён обработчик на сервере, выполняющий определённое действие и возвращающий ответ. Например, `GET /users/{id}` и `POST /users` — это два разных эндпоинта одного ресурса `users`. В FastAPI каждый endpoint объявляется декоратором маршрута над функцией-обработчиком (path operation function), которая принимает параметры пути/запроса/тела и возвращает данные, автоматически сериализуемые в JSON.

```python
@app.get("/users/{user_id}")
async def get_user(user_id: int):
    return {"id": user_id, "name": "Иван"}
```

| Термин                  | Определение                                              |
| ----------------------- | -------------------------------------------------------- |
| Endpoint                | Комбинация пути и HTTP-метода с привязанным обработчиком |
| Path operation function | Функция-обработчик эндпоинта в FastAPI                   |
| Роутер                  | Компонент, сопоставляющий запрос конкретному обработчику |

**Ответ для интервьюера:** Endpoint — это пара «путь + HTTP-метод», за которой закреплён конкретный обработчик на сервере. В FastAPI это функция, помеченная декоратором вроде `@app.get(...)`, принимающая параметры запроса и возвращающая ответ, который фреймворк сериализует в JSON.

### 13. Запуск FastAPI и роль Uvicorn
**Вопрос:** Как запустить FastAPI приложение и какую роль играет Uvicorn?

**Ответ:** FastAPI сам по себе — это только фреймворк для описания маршрутов, валидации и бизнес-логики, он не умеет принимать сетевые соединения; для этого нужен ASGI-сервер. Uvicorn — быстрый ASGI-сервер на основе uvloop и httptools, который принимает HTTP-соединения, парсит запросы и передаёт их приложению FastAPI по протоколу ASGI, а ответ передаёт обратно клиенту. В production Uvicorn часто запускают под управлением Gunicorn с воркерами класса `UvicornWorker` для использования нескольких процессов на 
многоядерном сервере.

**FastAPI** — это современный веб-фреймворк для создания API. Он отвечает за:

- Маршрутизацию (декораторы `@app.get()`)
- Валидацию данных (Pydantic)
- Автоматическую документацию (Swagger/OpenAPI)
- Асинхронную поддержку (`async def`)

**Uvicorn** — это ASGI-сервер (Asynchronous Server Gateway Interface). Он отвечает за:

- Прием HTTP-запросов от клиентов
- Передачу запросов в FastAPI
- Асинхронное выполнение (event loop)
- Управление воркерами (несколько процессов)

```bash

# ✅ Базовый запуск
uvicorn main:app --reload

# ✅ Production: несколько воркеров
uvicorn main:app --host 0.0.0.0 --port 8000 --workers 4
```

```python
# main.py
from fastapi import FastAPI
app = FastAPI()


@app.get("/")
async def root():
    return {"status": "ok"}
```

| Термин  | Определение                                                   |
| ------- | ------------------------------------------------------------- |
| ASGI    | Асинхронный интерфейс между веб-сервером и Python-приложением |
| Uvicorn | ASGI-сервер, принимающий и обрабатывающий сетевые соединения  |
| uvloop  | Быстрая реализация event loop на основе libuv                 |

**Ответ для интервьюера:** FastAPI описывает маршруты и логику, но сам не слушает сокеты — это делает ASGI-сервер Uvicorn, который принимает TCP-соединения, парсит HTTP и передаёт запрос приложению по протоколу ASGI. В продакшене Uvicorn обычно запускают через Gunicorn с несколькими воркерами для использования всех ядер CPU.

### 14. Path parameters vs Query parameters
**Вопрос:** Объясни разницу между path parameters и query parameters в FastAPI.

**Ответ:** Path parameter — часть самого URL-пути (`/items/{item_id}`), обязателен и обычно идентифицирует конкретный ресурс; в FastAPI объявляется как обычный аргумент функции с именем, совпадающим с плейсхолдером в пути. Query parameter передаётся после `?` в виде `ключ=значение` (`/items?skip=0&limit=10`), по умолчанию необязателен и используется для фильтрации, сортировки, пагинации; в FastAPI это любой аргумент функции, не встречающийся в пути и не являющийся Pydantic-моделью тела запроса — тип и значение по умолчанию определяют, обязателен ли параметр.

| Характеристика           | Path parameter                       | Query parameter                      |
| ------------------------ | ------------------------------------ | ------------------------------------ |
| **Расположение**         | Часть URL-пути (`/items/{id}`)       | После `?` (`/items?limit=10`)        |
| **Обязательность**       | Всегда обязателен                    | По умолчанию опционален              |
| **Назначение**           | Идентификация конкретного ресурса    | Фильтрация, сортировка, пагинация    |
| **Объявление в FastAPI** | Имя совпадает с плейсхолдером в пути | Любой аргумент, не упомянутый в пути |

```python
@app.get("/items/{item_id}")
async def read_item(item_id: int, q: str | None = None, limit: int = 10):
    # item_id — path parameter (обязателен)
    # q, limit — query parameters (необязательны, есть значения по умолчанию)
    return {"item_id": item_id, "q": q, "limit": limit}
```

| Термин                | Определение                                       |
| --------------------- | ------------------------------------------------- |
| Path parameter        | Часть URL-пути, идентифицирующая ресурс           |
| Query parameter       | Параметр после `?` в URL для фильтрации/пагинации |
| Значение по умолчанию | Делает query-параметр необязательным в FastAPI    |

**Ответ для интервьюера:** Path parameter — часть URL-пути, обязателен и идентифицирует конкретный ресурс, объявляется как аргумент функции с тем же именем, что и плейсхолдер в декораторе. Query parameter передаётся после `?`, по умолчанию опционален и используется для фильтрации, сортировки и пагинации — в FastAPI это просто аргумент функции, не упомянутый в пути.

### 15. GET и POST эндпоинты в FastAPI
**Вопрос:** Как в FastAPI объявить GET и POST эндпоинты?

**Ответ:** GET-эндпоинт объявляется декоратором `@app.get(path)` над асинхронной или синхронной функцией; параметры запроса передаются через path/query параметры, тело запроса для GET не предусмотрено семантикой HTTP. POST-эндпоинт объявляется декоратором `@app.post(path)`, а тело запроса описывается Pydantic-моделью в качестве аргумента функции — FastAPI автоматически парсит JSON из тела, валидирует его по схеме модели и возвращает 422 при ошибке валидации, ещё до вызова тела функции.

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class ItemCreate(BaseModel):
    name: str
    price: float

@app.get("/items")
async def list_items(limit: int = 10):
    return {"limit": limit}

@app.post("/items", status_code=201)
async def create_item(item: ItemCreate):
    return {"created": item}
```

| Термин               | Определение                                                 |
| -------------------- | ----------------------------------------------------------- |
| @app.get / @app.post | Декораторы регистрации маршрута под HTTP-метод              |
| Pydantic-модель тела | Схема, по которой парсится и валидируется JSON тела запроса |
| status_code          | Статус-код успешного ответа, задаваемый в декораторе        |

**Ответ для интервьюера:** GET объявляется декоратором `@app.get`, а данные передаются через path/query параметры без тела запроса. POST объявляется через `@app.post`, а тело запроса типизируется Pydantic-моделью — FastAPI сам парсит JSON, валидирует его по схеме и возвращает 422 при несоответствии.

### 16. Жизненный цикл приложения (lifespan)
**Вопрос:** Как работать с жизненным циклом приложения (события startup и shutdown / lifespan)?

**Ответ:** События жизненного цикла позволяют выполнить инициализацию ресурсов при старте приложения (подключение к БД, прогрев кэша, создание пула соединений) и корректно освободить их при остановке (закрытие соединений, дожидание фоновых задач). Раньше это делалось через отдельные декораторы `@app.on_event("startup")` / `@app.on_event("shutdown")` (устарели), а современный подход в FastAPI — асинхронный контекстный менеджер `lifespan`, передаваемый в конструктор `FastAPI()`: код до `yield` выполняется при старте, код после `yield` — при остановке, что гарантирует выполнение shutdown-логики даже при исключении на старте благодаря семантике `with`.

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.db_pool = await create_pool()  # startup
    yield
    await app.state.db_pool.close()  # shutdown

app = FastAPI(lifespan=lifespan)
```

| Термин             | Определение                                                                                                                          |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------ |
| startup            | Событие при запуске приложения; используется для инициализации ресурсов (подключение к БД, прогрев кэша, создание пула соединений)   |
| shutdown           | Событие при остановке приложения; используется для корректного освобождения ресурсов (закрытие соединений, завершение фоновых задач) |
| lifespan           | Асинхронный контекстный менеджер для startup/shutdown логики                                                                         |
| app.state          | Объект для хранения общих ресурсов приложения                                                                                        |
| on_event (устарел) | Старый способ регистрации startup/shutdown обработчиков                                                                              |

**Ответ для интервьюера:** Современный способ — асинхронный контекстный менеджер `lifespan`, передаваемый в `FastAPI(lifespan=...)`: код до `yield` выполняется при запуске (например, создание пула соединений с БД), а код после `yield` — при остановке (закрытие ресурсов). Старые декораторы `on_event("startup"/"shutdown")` считаются устаревшими.

### 17. Пагинация на бэкенде
**Вопрос:** Как реализовать пагинацию на бэкенде?

**Ответ:** Простейший вариант — offset-пагинация через query-параметры `limit` и `offset`/`skip`, которые транслируются в SQL `LIMIT`/`OFFSET`; проста в реализации, но деградирует по производительности на больших смещениях. Для больших наборов данных используют keyset (cursor-based) пагинацию — вместо смещения передаётся значение ключа последней увиденной записи, и следующая страница выбирается условием `id > last_id`, что работает за постоянное время независимо от глубины страницы. В FastAPI параметры пагинации объявляются как обычные query-параметры функции, часто с ограничениями через `Query(le=100)`, чтобы клиент не мог запросить слишком большую страницу.

| Характеристика                               | Offset-пагинация          | Keyset (cursor) пагинация             |
| -------------------------------------------- | ------------------------- | ------------------------------------- |
| **Параметры**                                | `limit` + `offset`/`skip` | `limit` + `cursor` (значение ключа)   |
| **SQL**                                      | `LIMIT/OFFSET`            | `WHERE id > last_id LIMIT ...`        |
| **Производительность на глубоких страницах** | ❌ Деградирует             | ✅ Постоянное время                    |
| **Произвольный переход на страницу N**       | ✅ Просто                  | ❌ Нужен только последовательный обход |
| **Простота реализации**                      | ✅ Проще                   | Сложнее                               |

```python
from fastapi import Query

@app.get("/items")
async def list_items(limit: int = Query(20, le=100), offset: int = Query(0, ge=0)):
    return db_query(limit=limit, offset=offset)

@app.get("/items/cursor")
async def list_items_cursor(cursor: int | None = None, limit: int = Query(20, le=100)):
    return db_query_keyset(after_id=cursor, limit=limit)
```

| Термин                    | Определение                                                                                                                                   |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Offset-пагинация          | Пагинация через LIMIT/OFFSET                                                                                                                  |
| Keyset (cursor) пагинация | Пагинация по значению ключа вместо смещения                                                                                                   |
| Query(le=100)             | Ограничение допустимого диапазона query-параметра в FastAPI                                                                                   |
| Пагинация в FastAPI       | Механизм, который позволяет разбивать большой набор данных на небольшие порции (страницы) и возвращать их клиенту по очереди, а не все сразу. |

**Ответ для интервьюера:** Для небольших наборов использую простую offset-пагинацию через `limit`/`offset` как query-параметры с ограничением максимума через `Query(le=100)`. Для больших таблиц перехожу на keyset-пагинацию по значению ключа последней записи, потому что OFFSET деградирует на глубоких страницах.

### 18. Структура проекта FastAPI
**Вопрос:** Как организовать структуру проекта для крупного веб-приложения?

**Ответ:** Для небольших проектов подходит группировка по типу файла (routers/, models/, schemas/), но для крупных приложений практичнее модульная структура по бизнес-доменам ("package by feature") — отдельная папка на каждый домен (users/, orders/) со своими роутерами, моделями, схемами и сервисным слоем внутри. Общие компоненты (конфигурация, подключение к БД, общие зависимости, middleware) выносятся в core/ или common/. Роутеры отдельных модулей подключаются к главному приложению через `APIRouter` с префиксом, что позволяет командам работать над разными доменами независимо и облегчает будущее выделение домена в отдельный микросервис.

```
app/
├── main.py
├── core/
│   ├── config.py
│   └── database.py
├── users/
│   ├── router.py
│   ├── models.py
│   ├── schemas.py
│   └── service.py
└── orders/
    ├── router.py
    ├── models.py
    ├── schemas.py
    └── service.py
```

```python
# users/router.py
from fastapi import APIRouter
router = APIRouter(prefix="/users", tags=["users"])

# main.py
from users.router import router as users_router
app.include_router(users_router)
```

| Термин             | Определение                                           |
| ------------------ | ----------------------------------------------------- |
| Package by feature | Группировка кода по бизнес-домену, а не по типу файла |
| APIRouter          | Модульный набор маршрутов, подключаемый к приложению  |
| Сервисный слой     | Слой бизнес-логики, отделённый от роутеров и моделей  |

**Ответ для интервьюера:** Для крупных проектов группирую код по бизнес-доменам (feature-based), а не по типу файла — каждый домен получает свой роутер, модели, схемы и сервисный слой. Роутеры подключаются к приложению через `APIRouter` с префиксом, а общая инфраструктура (конфиг, БД, зависимости) выносится в отдельный core-модуль.

## Валидация данных и Pydantic

### 19. Роль Pydantic в FastAPI
**Вопрос:** Какую роль играет Pydantic в FastAPI?

**Ответ:** Pydantic — библиотека для валидации и сериализации данных на основе аннотаций типов Python, и она лежит в основе всей типобезопасности FastAPI. Модели Pydantic описывают ожидаемую форму входных данных (тело запроса, query-параметры) и автоматически валидируют, парсят и приводят типы входящего JSON, возвращая понятную ошибку 422 при несоответствии, а также сериализуют объекты обратно в JSON для ответа. Помимо этого, FastAPI использует Pydantic-схемы для автогенерации интерактивной документации (OpenAPI/Swagger) — описание полей, типов и ограничений в модели напрямую превращается в схему API.

```python
from pydantic import BaseModel

class UserCreate(BaseModel):
    name: str
    age: int

@app.post("/users")
async def create_user(user: UserCreate):
    return user
```

| Термин | Определение |
|---|---|
| Pydantic | Библиотека валидации и сериализации данных по аннотациям типов |
| OpenAPI | Спецификация, автогенерируемая FastAPI из Pydantic-моделей |
| Приведение типов (coercion) | Автоматическое преобразование входных данных к нужному типу |

**Ответ для интервьюера:** Pydantic валидирует и парсит входящие данные по объявленной схеме, приводит типы и сериализует ответы обратно в JSON, а при несоответствии данных схеме FastAPI автоматически возвращает 422 с деталями ошибки. Дополнительно Pydantic-схемы используются для автогенерации OpenAPI-документации.

### 20. Валидация данных через Pydantic
**Вопрос:** Объясни, как использовать Pydantic для валидации данных.

**Ответ:** Модель Pydantic описывается как класс, наследующий `BaseModel`, где каждый атрибут аннотирован типом Python (`str`, `int`, `list[str]`, вложенная модель и т.д.) — Pydantic на основании этих аннотаций автоматически валидирует входные данные при создании экземпляра, приводит совместимые типы (например, строку `"123"` к `int`, если это разрешено) и выбрасывает `ValidationError` с точным указанием невалидного поля при несоответствии. Опциональные поля задаются через `| None` со значением по умолчанию, а обязательные — без значения по умолчанию; вложенные модели валидируются рекурсивно.

```python
from pydantic import BaseModel

class Address(BaseModel):
    city: str
    zip_code: str

class User(BaseModel):
    name: str
    age: int
    address: Address
    nickname: str | None = None

try:
    user = User(name="Иван", age="25", address={"city": "Москва", "zip_code": "101000"})
except ValueError as e:
    print(e)
```

| Термин | Определение |
|---|---|
| BaseModel | Базовый класс для описания схемы данных в Pydantic |
| ValidationError | Исключение с детальным описанием ошибок валидации |
| Вложенная модель | Pydantic-модель как тип поля другой модели, валидируется рекурсивно |

**Ответ для интервьюера:** Модель наследуется от `BaseModel`, а поля аннотируются типами Python — Pydantic валидирует и приводит типы входных данных автоматически при создании экземпляра. При несоответствии выбрасывается `ValidationError` с точным указанием, какое поле и почему невалидно, а вложенные модели валидируются рекурсивно.

### 21. Ограничения полей и кастомные валидаторы
**Вопрос:** Как добавить ограничения на поля и создать кастомные валидаторы?

**Ответ:** Простые ограничения (диапазон, длина строки, regex) задаются через `Field()` — например, `Field(gt=0)` для положительного числа или `Field(min_length=3, max_length=50)` для строки. Для логики, которую нельзя выразить декларативно (проверка формата email, сравнение двух полей между собой), используется декоратор `@field_validator` (Pydantic v2) на методе класса — он получает значение поля и может преобразовать его или выбросить `ValueError`, которая автоматически превращается в понятную ошибку валидации. Для проверок, зависящих от нескольких полей сразу, применяется `@model_validator`.

```python
from pydantic import BaseModel, Field, field_validator

class UserCreate(BaseModel):
    name: str = Field(min_length=2, max_length=50)
    age: int = Field(gt=0, le=120)
    email: str

    @field_validator("email")
    @classmethod
    def validate_email(cls, v: str) -> str:
        if "@" not in v:
            raise ValueError("некорректный email")
        return v.lower()
```

| Термин | Определение |
|---|---|
| Field() | Декларативные ограничения и метаданные поля |
| field_validator | Декоратор кастомного валидатора одного поля (Pydantic v2) |
| model_validator | Декоратор валидатора, проверяющего несколько полей сразу |

**Ответ для интервьюера:** Простые ограничения задаю декларативно через `Field(gt=0, min_length=3, ...)`, а сложную логику — через `@field_validator`, который получает значение поля, может его преобразовать и выбрасывает `ValueError` при невалидности. Для проверок, зависящих сразу от нескольких полей, использую `@model_validator`.

## Зависимости и Middleware

### 22. Dependency Injection и Depends
**Вопрос:** Объясни, что такое Dependency Injection в FastAPI и как использовать Depends.

**Ответ:** Dependency Injection — паттерн, при котором функция или объект не создаёт свои зависимости сама, а получает их извне, что упрощает переиспользование логики и тестирование (зависимость легко подменить моком). В FastAPI зависимость — это обычная функция (или класс с `__call__`), а `Depends(func)` в качестве значения по умолчанию аргумента говорит FastAPI вызвать эту функцию перед основным обработчиком и передать результат как значение параметра. Зависимости могут быть вложенными (одна зависит от другой), кэшируются в рамках одного запроса по умолчанию и часто используются для аутентификации, получения сессии БД, проверки прав доступа и общей логики, повторяющейся между эндпоинтами.

```python
from fastapi import Depends

async def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

async def get_current_user(token: str = Depends(oauth2_scheme), db=Depends(get_db)):
    user = decode_token_and_fetch_user(token, db)
    return user

@app.get("/me")
async def read_me(user=Depends(get_current_user)):
    return user
```

| Термин               | Определение                                                      |
| -------------------- | ---------------------------------------------------------------- |
| Dependency Injection | Передача зависимостей извне вместо их создания внутри функции    |
| Depends()            | Механизм FastAPI для объявления и внедрения зависимости          |
| yield-зависимость    | Зависимость с логикой очистки после ответа (finally после yield) |

**Ответ для интервьюера:** Dependency Injection — паттерн, при котором зависимости (сессия БД, текущий пользователь) передаются функции извне, а не создаются ей самой, что упрощает тестирование и переиспользование. В FastAPI это реализовано через `Depends(func)`: фреймворк вызывает функцию-зависимость перед обработчиком, кэширует результат в рамках запроса и поддерживает yield-зависимости с логикой очистки после ответа.

### 23. Middleware в FastAPI
**Вопрос:** Что такое Middleware в FastAPI?

**Ответ:** Middleware — это слой кода, оборачивающий обработку каждого запроса: он выполняется до того, как запрос дойдёт до конкретного эндпоинта, и после того, как обработчик сформирует ответ, имея возможность модифицировать и запрос, и ответ либо вовсе прервать цепочку раньше. В отличие от зависимостей (Depends), middleware применяется глобально ко всем запросам приложения, а не декларативно к конкретным эндпоинтам, и работает на более низком уровне ASGI. Типичные применения — логирование, замер времени обработки, добавление заголовков (CORS, security headers), сжатие ответа, обработка исключений верхнего уровня.

```python
from fastapi import FastAPI, Request
import time

app = FastAPI()

@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    start = time.time()
    response = await call_next(request)
    response.headers["X-Process-Time"] = str(time.time() - start)
    return response
```

| Термин       | Определение                                                                |
| ------------ | -------------------------------------------------------------------------- |
| Middleware   | Слой, оборачивающий обработку каждого запроса глобально                    |
| call_next    | Функция вызова следующего слоя/обработчика в цепочке                       |
| ASGI-уровень | Более низкий уровень, на котором работает middleware, в отличие от Depends |

**Ответ для интервьюера:** Middleware — глобальный перехватчик, который оборачивает каждый запрос до и после обработчика, в отличие от `Depends`, применяемого точечно к конкретным эндпоинтам. Используется для сквозной логики: логирования, замера времени, добавления заголовков, CORS, сжатия ответов.

---

### 24. Middleware для логирования и метрик
**Вопрос:** Как создать Middleware для логирования запросов или сбора метрик?

**Ответ:** Middleware для логирования оборачивает вызов `call_next`, фиксируя время начала запроса, а после получения ответа — считает длительность обработки, логирует метод, путь, статус-код и время выполнения, а для метрик дополнительно инкрементирует счётчики (например, в Prometheus) по этим же измерениям. Такой middleware можно реализовать как функцию с декоратором `@app.middleware("http")` для простых случаев или как класс, наследующий `BaseHTTPMiddleware`, для более сложной логики с состоянием или переиспользуемой конфигурацией.

```python
import time
import logging
from starlette.middleware.base import BaseHTTPMiddleware

logger = logging.getLogger("app")

class LoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        start = time.perf_counter()
        response = await call_next(request)
        duration = time.perf_counter() - start
        logger.info(
            "%s %s -> %s (%.3fs)",
            request.method, request.url.path, response.status_code, duration,
        )
        return response

app.add_middleware(LoggingMiddleware)
```

| Термин | Определение |
|---|---|
| BaseHTTPMiddleware | Базовый класс Starlette для middleware со сложной логикой |
| dispatch | Метод, реализующий обработку запроса/ответа в классовом middleware |
| Метрика | Числовое измерение (длительность, счётчик) для мониторинга |

**Ответ для интервьюера:** Реализую middleware как класс, наследующий `BaseHTTPMiddleware`, с методом `dispatch`, который засекает время до вызова `call_next`, а после получения ответа логирует метод, путь, статус и длительность, либо отправляет эти данные в систему метрик вроде Prometheus. Регистрируется через `app.add_middleware(LoggingMiddleware)`.

### 25. CORS и настройка Middleware
**Вопрос:** Что такое CORS и как настроить соответствующую Middleware?

**Ответ:** CORS (Cross-Origin Resource Sharing) — механизм браузерной безопасности, по умолчанию блокирующий JavaScript-код одного origin (домен+схема+порт) от выполнения запросов к другому origin, если сервер явно не разрешил это через специальные заголовки ответа. Для «небезопасных» запросов (не GET/HEAD с простыми заголовками) браузер сначала отправляет preflight-запрос методом OPTIONS, и только если сервер ответил разрешающими заголовками (`Access-Control-Allow-Origin` и др.), выполняется реальный запрос. В FastAPI это настраивается стандартным `CORSMiddleware` из Starlette, где явно перечисляются разрешённые origins, методы и заголовки.

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://example.com"],
    allow_credentials=True,
    allow_methods=["GET", "POST"],
    allow_headers=["*"],
)
```

| Термин | Определение |
|---|---|
| CORS | Механизм разрешения кросс-доменных запросов из браузера |
| Origin | Комбинация схемы, домена и порта |
| Preflight-запрос | Предварительный OPTIONS-запрос для проверки разрешений |

**Ответ для интервьюера:** CORS — браузерный механизм, блокирующий кросс-доменные запросы, пока сервер явно их не разрешит заголовками ответа; для нестандартных запросов браузер сначала шлёт preflight OPTIONS-запрос. В FastAPI настраивается через `CORSMiddleware` с явным списком разрешённых origins, методов и заголовков.

## Безопасность и Аутентификация

### 26. Аутентификация vs авторизация
**Вопрос:** Объясни разницу между аутентификацией и авторизацией.

**Ответ:** Аутентификация (authentication) — процесс подтверждения личности пользователя, ответ на вопрос «кто ты?»: проверка логина/пароля, валидация JWT-токена, проверка API-ключа. Авторизация (authorization) — процесс проверки прав уже аутентифицированного пользователя на конкретное действие, ответ на вопрос «что тебе разрешено?»: имеет ли пользователь роль admin, владеет ли он этим ресурсом. Аутентификация всегда выполняется первой — без подтверждённой личности проверять права бессмысленно; в FastAPI обе стадии обычно реализуются как последовательные зависимости (`Depends`).

|Характеристика|Аутентификация|Авторизация|
|---|---|---|
|**Вопрос**|Кто ты?|Что тебе разрешено?|
|**Проверяет**|Личность (пароль, токен)|Права на действие/ресурс|
|**Порядок**|Выполняется первой|Выполняется после аутентификации|
|**Код ошибки**|401 Unauthorized|403 Forbidden|

```python
async def get_current_user(token: str = Depends(oauth2_scheme)) -> User:
    return decode_and_fetch_user(token)  # аутентификация

async def require_admin(user: User = Depends(get_current_user)) -> User:
    if user.role != "admin":
        raise HTTPException(status_code=403, detail="Forbidden")  # авторизация
    return user
```

| Термин | Определение |
|---|---|
| Аутентификация | Подтверждение личности пользователя |
| Авторизация | Проверка прав пользователя на действие |
| 401 vs 403 | Не аутентифицирован vs аутентифицирован, но нет прав |

**Ответ для интервьюера:** Аутентификация отвечает на вопрос «кто ты» и проверяет личность (пароль, токен), авторизация — на вопрос «что тебе разрешено» и проверяет права уже опознанного пользователя. Соответственно и статусы разные: 401 — не прошёл аутентификацию, 403 — прошёл, но нет прав на конкретное действие.

### 27. JWT аутентификация в FastAPI
**Вопрос:** Как реализовать JWT аутентификацию в FastAPI?

**Ответ:** При логине сервер проверяет учётные данные и генерирует JWT-токен, подписанный секретным ключом, с payload, содержащим идентификатор пользователя и срок жизни (exp), и возвращает его клиенту. На последующих запросах клиент передаёт токен в заголовке `Authorization: Bearer <token>`, а зависимость на сервере (обычно через `OAuth2PasswordBearer`) извлекает токен, проверяет подпись и срок действия библиотекой вроде `python-jose` или `PyJWT`, и при успехе достаёт пользователя из payload или БД. Пароли при этом никогда не хранятся в открытом виде — используется хэширование (bcrypt/argon2) через `passlib`.

```python
from datetime import datetime, timedelta, timezone
from jose import jwt, JWTError
from fastapi import Depends, HTTPException
from fastapi.security import OAuth2PasswordBearer

SECRET_KEY = "secret"
ALGORITHM = "HS256"
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")

def create_access_token(data: dict, expires_delta: timedelta):
    payload = data.copy()
    payload["exp"] = datetime.now(timezone.utc) + expires_delta
    return jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)

async def get_current_user(token: str = Depends(oauth2_scheme)):
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    except JWTError:
        raise HTTPException(status_code=401, detail="Invalid token")
    return payload["sub"]
```

| Термин | Определение |
|---|---|
| Access token | Короткоживущий JWT для доступа к защищённым эндпоинтам |
| OAuth2PasswordBearer | Схема FastAPI для извлечения Bearer-токена из заголовка |
| passlib / bcrypt | Библиотека и алгоритм хэширования паролей |

**Ответ для интервьюера:** При логине сервер выдаёт подписанный JWT с payload и сроком жизни, клиент передаёт его в заголовке `Authorization: Bearer`, а зависимость `get_current_user` через `OAuth2PasswordBearer` извлекает токен, проверяет подпись и exp через `jose`/`PyJWT`. Пароли никогда не хранятся в открытом виде — только хэш через bcrypt/argon2.

### 28. Части JWT токена
**Вопрос:** Из каких частей состоит JWT токен?

**Ответ:** JWT (JSON Web Token) состоит из трёх частей, разделённых точками и закодированных в Base64URL: header, payload и signature. Header содержит тип токена и алгоритм подписи (например, `{"alg": "HS256", "typ": "JWT"}`). Payload содержит claims — произвольные данные о пользователе и служебные поля (`sub` — субъект, `exp` — время истечения, `iat` — время выпуска). Signature — результат криптографического подписания строки `base64(header).base64(payload)` секретным ключом (HMAC) или приватным ключом (RSA/ECDSA), гарантирующий, что header и payload не были изменены после выпуска.

```

└──────── header ────────┘└──────────── payload ────────────┘└──────── signature ────────┘
```

| Термин | Определение |
|---|---|
| Header | Метаданные токена: алгоритм и тип |
| Payload (claims) | Полезные данные токена (sub, exp, iat и др.) |
| Signature | Криптографическая подпись header+payload |

**Ответ для интервьюера:** JWT состоит из трёх Base64URL-частей через точку: header с алгоритмом подписи, payload с claims (sub, exp и произвольные данные), и signature — подпись первых двух частей секретным или приватным ключом, гарантирующая целостность токена.

### 29. Base64 vs шифрование подписи JWT
**Вопрос:** В чем разница между кодированием (Base64) и шифрованием подписи в JWT?

**Ответ:** Header и payload JWT лишь кодируются в Base64URL — это обратимое преобразование формата без секретного ключа, а значит любой может декодировать и прочитать содержимое токена, поэтому в payload нельзя класть чувствительные данные (пароли, номера карт) в открытом виде. Signature, напротив, формируется криптографической операцией с использованием секрета (HMAC-SHA256 в HS256) или пары ключей (RSA/ECDSA в RS256/ES256) — она не шифрует payload, а лишь подтверждает его подлинность: без знания секретного/приватного ключа невозможно подделать подпись под изменённым payload, хотя сам payload всё ещё читаем всеми.

|Характеристика|Base64 (header/payload)|Подпись (signature)|
|---|---|---|
|**Цель**|Формат передачи данных|Подтверждение подлинности|
|**Обратимость**|✅ Любой может декодировать без ключа|Не предназначена для чтения данных|
|**Требует секрет/ключ**|❌ Нет|✅ Да (HMAC-секрет или приватный ключ)|
|**Конфиденциальность payload**|❌ Не обеспечивает|❌ Не обеспечивает (только целостность)|

```python
import base64, json
payload_b64 = "eyJzdWIiOiIxMjMifQ"
print(json.loads(base64.urlsafe_b64decode(payload_b64 + "==")))  # {'sub': '123'} — читается без ключа
```

| Термин | Определение |
|---|---|
| Base64URL | Обратимое кодирование без секрета, не обеспечивает конфиденциальность |
| HMAC | Симметричная подпись общим секретным ключом (HS256) |
| RSA/ECDSA подпись | Асимметричная подпись приватным ключом, проверяется публичным |

**Ответ для интервьюера:** Header и payload JWT только кодируются в Base64URL — это не шифрование, содержимое читает любой без ключа. Signature — это криптографическая подпись (HMAC или RSA/ECDSA), которая гарантирует целостность и подлинность токена, но не скрывает payload, поэтому чувствительные данные в JWT класть нельзя.

### 30. OAuth2 и вход через внешних провайдеров
**Вопрос:** Что такое OAuth2 и как работает вход через внешних провайдеров?

**Ответ:** OAuth2 — открытый протокол авторизации, позволяющий приложению получить ограниченный доступ к ресурсам пользователя на стороне провайдера (Google, GitHub) без передачи приложению пароля пользователя. В самом распространённом сценарии (Authorization Code Flow) пользователь перенаправляется на страницу провайдера, где логинится и подтверждает доступ, провайдер возвращает приложению временный authorization code через redirect, который бэкенд обменивает на access token напрямую (server-to-server), а затем использует этот токен для запроса данных профиля пользователя у провайдера. FastAPI предоставляет строительные блоки (`OAuth2AuthorizationCodeBearer`, `OAuth2PasswordBearer`), а полноценную интеграцию с внешними провайдерами обычно реализуют через готовые библиотеки (`Authlib`).

```python
from authlib.integrations.starlette_client import OAuth

oauth = OAuth()
oauth.register(
    name="google",
    client_id="...",
    client_secret="...",
    server_metadata_url="https://accounts.google.com/.well-known/openid-configuration",
    client_kwargs={"scope": "openid email profile"},
)
```

| Термин | Определение |
|---|---|
| Authorization Code Flow | Основной сценарий OAuth2 с обменом кода на токен |
| Access token | Токен доступа к ресурсам пользователя у провайдера |
| Scope | Запрашиваемый набор разрешений при авторизации |

**Ответ для интервьюера:** OAuth2 позволяет приложению получить ограниченный доступ к данным пользователя у внешнего провайдера без передачи пароля. В Authorization Code Flow пользователь логинится на стороне провайдера, тот возвращает временный код через redirect, а бэкенд обменивает его на access token напрямую и запрашивает профиль пользователя — на практике это реализуют через готовые библиотеки вроде Authlib.

### 31. FastAPI + PostgreSQL через SQLAlchemy
**Вопрос:** Как связать FastAPI с базой данных PostgreSQL через SQLAlchemy?

**Ответ:** Связь строится через асинхронный `Engine` SQLAlchemy с драйвером `asyncpg`, поверх которого создаётся фабрика асинхронных сессий (`async_sessionmaker`). Сессия БД оформляется как зависимость с `yield`, которую FastAPI внедряет в обработчик через `Depends` — это гарантирует, что сессия открывается на время запроса и корректно закрывается после ответа (или отката при исключении) независимо от результата обработки.

```python
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession
from fastapi import Depends

engine = create_async_engine("postgresql+asyncpg://user:pass@localhost/db")
SessionLocal = async_sessionmaker(engine, expire_on_commit=False)

async def get_db() -> AsyncSession:
    async with SessionLocal() as session:
        yield session

@app.get("/users/{user_id}")
async def get_user(user_id: int, db: AsyncSession = Depends(get_db)):
    return await db.get(User, user_id)
```

| Термин | Определение |
|---|---|
| create_async_engine | Асинхронный движок SQLAlchemy для подключения к БД |
| async_sessionmaker | Фабрика асинхронных сессий SQLAlchemy |
| yield-зависимость | Зависимость FastAPI, гарантирующая закрытие сессии после запроса |

**Ответ для интервьюера:** Создаю асинхронный `Engine` с драйвером `asyncpg` и фабрику `async_sessionmaker`, а сессию БД оформляю как yield-зависимость (`Depends(get_db)`), которую FastAPI внедряет в обработчик и автоматически закрывает после ответа, даже при исключении.

### 32. Pydantic-модели в эндпоинтах
**Вопрос:** Как использовать классы Python и модели Pydantic в эндпоинтах для преобразования JSON в объекты?

**Ответ:** Pydantic-модель, указанная как тип аргумента функции-обработчика, автоматически парсится FastAPI из тела JSON-запроса в типизированный объект Python с валидацией на входе. Для отдачи данных наружу принято разделять схему записи (input, например `UserCreate`) и схему чтения (output, например `UserRead`, без пароля), а на границе с ORM-моделью используется `model_config = ConfigDict(from_attributes=True)`, чтобы Pydantic мог строить схему прямо из атрибутов ORM-объекта SQLAlchemy, а не только из словаря.

```python
from pydantic import BaseModel, ConfigDict

class UserCreate(BaseModel):
    name: str
    password: str

class UserRead(BaseModel):
    model_config = ConfigDict(from_attributes=True)
    id: int
    name: str

@app.post("/users", response_model=UserRead)
async def create_user(data: UserCreate, db: AsyncSession = Depends(get_db)):
    user = User(name=data.name, hashed_password=hash_password(data.password))
    db.add(user)
    await db.commit()
    return user  # ORM-объект -> UserRead благодаря from_attributes
```

| Термин | Определение |
|---|---|
| response_model | Схема, по которой FastAPI сериализует и фильтрует ответ |
| from_attributes | Позволяет строить Pydantic-модель из атрибутов ORM-объекта |
| Разделение input/output схем | UserCreate для записи, UserRead для чтения без чувствительных полей |

**Ответ для интервьюера:** Pydantic-модель в аргументе функции автоматически парсит и валидирует тело запроса в объект Python, а `response_model` в декораторе фильтрует и сериализует то, что возвращает обработчик. Схемы записи и чтения разделяю (UserCreate/UserRead), а для конвертации ORM-объекта в схему ответа использую `ConfigDict(from_attributes=True)`.

### 33. requests vs aiohttp/httpx
**Вопрос:** Как работать с внешними API в Python и в чем разница между requests и aiohttp/httpx?

**Ответ:** `requests` — синхронная блокирующая библиотека: вызов `requests.get()` останавливает поток до получения ответа, что несовместимо с асинхронным event loop FastAPI — вызов внутри `async def` заблокирует обработку всех остальных запросов на этом воркере. `aiohttp` и `httpx` поддерживают `async/await` — запрос не блокирует event loop, позволяя обрабатывать другие запросы, пока ждём ответа от внешнего API; `httpx` дополнительно даёт единый API и для синхронного, и для асинхронного клиента и ближе по интерфейсу к `requests`, поэтому в новых FastAPI-проектах обычно выбирают именно его.

|Характеристика|requests|aiohttp|httpx|
|---|---|---|---|
|**Модель**|Синхронная (блокирующая)|Асинхронная|Синхронная и асинхронная|
|**Совместим с event loop FastAPI**|❌ Блокирует|✅|✅|
|**API близок к requests**|—|Отличается|✅ Похож|
|**Типичный выбор в новых FastAPI-проектах**|❌|Возможен|✅|

```python
import httpx

async def fetch_user(user_id: int):
    async with httpx.AsyncClient() as client:
        response = await client.get(f"https://api.example.com/users/{user_id}")
        response.raise_for_status()
        return response.json()
```

| Термин | Определение |
|---|---|
| Блокирующий вызов | Вызов, останавливающий поток/event loop до завершения |
| httpx.AsyncClient | Асинхронный HTTP-клиент, совместимый с event loop FastAPI |
| Event loop | Цикл событий, на котором выполняются все async-задачи |

**Ответ для интервьюера:** `requests` синхронный и блокирует event loop, поэтому его нельзя безопасно вызывать внутри `async def` в FastAPI. `httpx.AsyncClient` (или `aiohttp`) выполняет запрос неблокирующе через `await`, позволяя обрабатывать другие запросы, пока ждём ответа — это стандартный выбор для обращения к внешним API из асинхронного приложения.

### 34. Retry-логика для внешних сервисов
**Вопрос:** Как реализовать retry-логику при обращении к внешним сервисам?

**Ответ:** Retry-логика повторяет неудачный запрос к нестабильному внешнему сервису вместо немедленного отказа, но должна применяться осторожно: повторять стоит только на идемпотентных операциях и на определённых классах ошибок (сетевые сбои, 5xx, таймауты), но не на 4xx (ошибка клиента не исчезнет от повтора). Между попытками используют экспоненциальную задержку (exponential backoff) с джиттером (случайной добавкой), чтобы избежать одновременного повторного удара множества клиентов по восстанавливающемуся сервису (thundering herd), и ограничивают общее число попыток. В Python это обычно реализуют через библиотеку `tenacity`.

```python
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
import httpx

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=1, max=10),
    retry=retry_if_exception_type(httpx.HTTPStatusError),
)
async def fetch_with_retry(client: httpx.AsyncClient, url: str):
    response = await client.get(url)
    response.raise_for_status()
    return response.json()
```

| Термин | Определение |
|---|---|
| Exponential backoff | Экспоненциально растущая задержка между повторными попытками |
| Jitter | Случайная добавка к задержке для распределения нагрузки |
| Idempotent-операция | Операция, безопасная для повтора без побочных эффектов |

**Ответ для интервьюера:** Повторяю только идемпотентные операции и только на «повторяемых» ошибках (5xx, таймауты, сетевые сбои), не на 4xx. Между попытками использую экспоненциальный backoff с джиттером и ограниченным числом попыток, чтобы не создавать thundering herd на восстанавливающийся сервис — на практике это реализую через `tenacity`.

### 35. FastAPI vs Django
**Вопрос:** Когда стоит использовать FastAPI, а когда Django?

**Ответ:** FastAPI — легковесный асинхронный микрофреймворк, ориентированный на построение API: даёт высокую производительность (сопоставимую с Node.js/Go благодаря ASGI и asyncio), автоматическую валидацию через Pydantic и автогенерацию OpenAPI-документации, но почти не даёт «батареек» из коробки (нет встроенной админки, ORM, системы аутентификации) — их нужно выбирать и подключать самостоятельно. Django — полнофункциональный синхронный (с частичной async-поддержкой) фреймворк «батарейки в комплекте»: встроенная admin-панель, ORM, система аутентификации, форм и шаблонизатор, что ускоряет разработку классических серверных веб-приложений и CRUD-порталов с админкой. FastAPI выбирают для высоконагруженных API, микросервисов и I/O-bound систем с большим числом внешних вызовов; Django — для проектов, где нужна готовая административная часть и меньше самостоятельной сборки инфраструктуры.

|Характеристика|FastAPI|Django|
|---|---|---|
|**Тип**|Асинхронный микрофреймворк|Полнофункциональный (batteries included)|
|**Производительность**|✅ Высокая (ASGI, asyncio)|Ниже (в основном синхронный)|
|**Встроенная админка/ORM/auth**|❌ Нет, подключаются отдельно|✅ Есть из коробки|
|**Валидация и документация**|✅ Автоматически через Pydantic/OpenAPI|Ручная / через DRF|
|**Когда выбирать**|Высоконагруженные API, микросервисы, I/O-bound|CRUD-порталы с админкой, меньше кастомной сборки|

| Термин | Определение |
|---|---|
| ASGI | Асинхронный стандарт, на котором построен FastAPI |
| Batteries included | Философия Django — готовые компоненты из коробки |
| I/O-bound нагрузка | Нагрузка, где узкое место — ожидание внешних вызовов (сеть, БД) |

**Ответ для интервьюера:** FastAPI — легковесный async-фреймворк для высокопроизводительных API с автовалидацией и автодокументацией, но без встроенных «батареек». Django — фреймворк «всё включено» с готовой админкой, ORM и аутентификацией, что ускоряет разработку классических веб-порталов; для микросервисов и I/O-bound нагрузки выбираю FastAPI, для проектов с админкой и меньшей потребностью в кастомизации инфраструктуры — Django.

### 36. Балансировщик нагрузки и API Gateway
**Вопрос:** Какую роль в инфраструктуре играет балансировщик нагрузки (Nginx) и API Gateway?

**Ответ:** Балансировщик нагрузки (Nginx, HAProxy) распределяет входящие запросы между несколькими инстансами приложения по алгоритму (round-robin, least connections, по хешу IP), обеспечивая горизонтальное масштабирование, отказоустойчивость (исключение упавших инстансов через health check) и часто терминацию TLS. API Gateway — более высокоуровневый компонент, который выступает единой точкой входа для клиентов в микросервисную систему: маршрутизирует запросы к нужному сервису, централизованно занимается аутентификацией, rate limiting, логированием и агрегацией ответов от нескольких сервисов, снимая эту ответственность с самих сервисов.

|Характеристика|Балансировщик нагрузки|API Gateway|
|---|---|---|
|**Уровень**|Сетевой/транспортный (распределение трафика)|Прикладной (бизнес-логика маршрутизации)|
|**Основная задача**|Распределить запросы между инстансами одного сервиса|Единая точка входа в набор разных сервисов|
|**Доп. функции**|Health check, TLS-терминация|Аутентификация, rate limiting, агрегация ответов|

```nginx
upstream fastapi_app {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
}

server {
    listen 80;
    location / {
        proxy_pass http://fastapi_app;
    }
}
```

| Термин | Определение |
|---|---|
| Балансировщик нагрузки | Распределяет запросы между инстансами приложения |
| API Gateway | Единая точка входа с маршрутизацией и сквозной логикой |
| Health check | Проверка доступности инстанса перед направлением трафика |

**Ответ для интервьюера:** Балансировщик нагрузки распределяет трафик между инстансами приложения по алгоритму (round-robin, least connections), исключая упавшие через health check, и часто терминирует TLS. API Gateway — единая точка входа для клиентов в микросервисную архитектуру, которая маршрутизирует запросы к нужному сервису и централизует аутентификацию, rate limiting и логирование.

### 37. Вертикальное vs горизонтальное масштабирование
**Вопрос:** В чем разница между вертикальным и горизонтальным масштабированием?

**Ответ:** Вертикальное масштабирование (scale up) — увеличение мощности одного сервера (больше CPU, RAM, быстрее диск); просто реализовать без изменения архитектуры приложения, но имеет физический потолок мощности одной машины и создаёт единую точку отказа. Горизонтальное масштабирование (scale out) — добавление новых серверов/инстансов приложения, между которыми распределяется нагрузка через балансировщик; практически неограниченно по масштабу и повышает отказоустойчивость, но требует, чтобы приложение было stateless (или состояние хранилось вовне — в БД/Redis), и добавляет сложность координации между инстансами.

|Характеристика|Вертикальное (scale up)|Горизонтальное (scale out)|
|---|---|---|
|**Способ**|Увеличить мощность одного сервера|Добавить новые серверы/инстансы|
|**Потолок масштаба**|❌ Ограничен мощностью одной машины|✅ Практически неограничен|
|**Отказоустойчивость**|❌ Единая точка отказа|✅ Выше|
|**Требования к приложению**|Без изменения архитектуры|Нужен stateless (или внешнее хранилище состояния)|
|**Сложность**|Проще|Требует балансировщика и координации|

| Термин | Определение |
|---|---|
| Scale up (вертикальное) | Увеличение мощности одного сервера |
| Scale out (горизонтальное) | Добавление новых серверов/инстансов |
| Единая точка отказа | Риск полной остановки системы при отказе одного узла |

**Ответ для интервьюера:** Вертикальное масштабирование увеличивает мощность одного сервера — просто, но упирается в физический потолок и создаёт единую точку отказа. Горизонтальное масштабирование добавляет инстансы приложения за балансировщиком — почти безгранично и отказоустойчиво, но требует stateless-архитектуры, чтобы любой инстанс мог обработать любой запрос.

### 38. Кэширование (Redis) для производительности
**Вопрос:** Как использовать кэширование (например, Redis) для оптимизации производительности?

**Ответ:** Кэш хранит результат дорогой операции (запрос к БД, вызов внешнего API, тяжёлое вычисление) в быстром in-memory хранилище вроде Redis, чтобы повторные обращения за теми же данными не повторяли эту работу. Типичный паттерн — cache-aside: перед обращением к БД приложение проверяет кэш по ключу, при попадании (hit) возвращает закэшированное значение, при промахе (miss) читает из БД и кладёт результат в кэш с TTL, чтобы данные не устаревали бесконтрольно. Важно продумать инвалидацию — при изменении данных явно удалять или обновлять соответствующий ключ в кэше, иначе клиенты будут получать устаревшие данные до истечения TTL.

```python
import json
from redis.asyncio import Redis

redis = Redis(host="localhost", port=6379)

async def get_user(user_id: int, db) -> dict:
    key = f"user:{user_id}"
    cached = await redis.get(key)
    if cached:
        return json.loads(cached)
    user = await db.get(User, user_id)
    data = {"id": user.id, "name": user.name}
    await redis.set(key, json.dumps(data), ex=300)
    return data
```

| Термин | Определение |
|---|---|
| Cache-aside | Паттерн: приложение само проверяет кэш, затем БД |
| TTL | Время жизни ключа в кэше до автоматического истечения |
| Инвалидация кэша | Явное удаление/обновление устаревшего значения в кэше |

**Ответ для интервьюера:** Использую паттерн cache-aside: перед запросом к БД проверяю Redis по ключу, при промахе читаю из БД и сохраняю результат в Redis с TTL. Ключевой момент — инвалидация: при изменении данных в БД явно удаляю или обновляю соответствующий ключ, иначе клиенты будут получать устаревшие данные до истечения TTL.

### 39. Принципы микросервисной архитектуры
**Вопрос:** Объясни принципы микросервисной архитектуры и способы взаимодействия между сервисами.

**Ответ:** Микросервисная архитектура разбивает приложение на набор независимо развёртываемых сервисов, каждый из которых отвечает за свой ограниченный бизнес-домен (bounded context), имеет собственную БД (database per service) и может быть написан на своём стеке технологий, масштабируется и деплоится отдельно от остальных. Синхронное взаимодействие между сервисами обычно идёт через REST/HTTP или gRPC (быстрее за счёт бинарного протокола Protobuf), что просто в реализации, но создаёт временную связанность — вызывающий сервис ждёт ответа и деградирует при недоступности вызываемого. Асинхронное взаимодействие идёт через брокер сообщений (RabbitMQ, Kafka) — сервис публикует событие, не дожидаясь обработки, что снижает связанность и повышает отказоустойчивость ценой eventual consistency и большей сложности отладки распределённой системы.

|Характеристика|Синхронное (REST/gRPC)|Асинхронное (брокер сообщений)|
|---|---|---|
|**Ожидание ответа**|✅ Вызывающий сервис ждёт|❌ Не ждёт, публикует событие|
|**Связанность**|Выше (temporal coupling)|Ниже|
|**Отказоустойчивость**|Деградирует при недоступности вызываемого|Выше — сообщение подождёт в очереди|
|**Согласованность данных**|Немедленная|Eventual consistency|
|**Сложность отладки**|Проще|Сложнее (распределённая система)|

| Термин | Определение |
|---|---|
| Bounded context | Ограниченная область ответственности одного сервиса |
| Database per service | Каждый сервис владеет своей БД, не лезет в чужую напрямую |
| Message broker | Посредник для асинхронного обмена событиями (Kafka, RabbitMQ) |
| Eventual consistency | Согласованность данных между сервисами с задержкой |

**Ответ для интервьюера:** Каждый микросервис отвечает за свой bounded context, владеет собственной БД и деплоится независимо. Синхронно сервисы общаются через REST/gRPC, что просто, но создаёт временную связанность; асинхронно — через брокер сообщений (Kafka, RabbitMQ), что снижает связанность и повышает отказоустойчивость ценой eventual consistency и более сложной отладки.
