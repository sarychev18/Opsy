#flashcards/Тестирование 
что такое TDD и в чем его преимущества и недостатки ? :: Методология, где тесты пишутся **до** кода реализации.<br>Преимущества - Код сразу тестируется, Не страшно менять код (тесты ловят регрессию), Тесты работают как документация - показывают, как работает код, <br>Недостатки - +30-50% к времени разработки на старте, ложная безопасность (Тесты есть, но не покрывают реальные кейсы).<br>Применять стоит когда (критичная логика(финансы, безопасность), Сложные алгоритмы,)<br>Когда TDD можно пропустить: (Прототипы, Одноразовые скрипты)

| Плюс             | Описание                                      |
| ---------------- | --------------------------------------------- |
| **Покрытие**     | Код сразу тестируется (90%+ coverage)         |
| **Рефакторинг**  | Не страшно менять код (тесты ловят регрессию) |
| **Документация** | Тесты показывают, как использовать API        |
| **Дизайн**       | Лучшая архитектура (код проще, модульнее)     |
| **Фокус**        | Пишем только нужный код (под тест)            |

### 🔹 Недостатки

| Минус                   | Описание                                    |
| ----------------------- | ------------------------------------------- |
| **Время**               | +30-50% к времени разработки на старте      |
| **Сложность**           | Трудно тестировать UI, внешние API, БД      |
| **Ложная безопасность** | Тесты есть, но не покрывают реальные кейсы  |
| **Дисциплина**          | Требует привычки (команды часто пропускают) |
```python
1. 🔴 RED: Пишем тест → он падает (функции ещё нет)
2. 🟢 GREEN: Пишем минимальный код → тест проходит
3. 🟡 REFACTOR: Улучшаем код → тест остаётся зелёным

# 1. 🔴 RED: Тест падает
def test_add():
    assert add(2, 3) == 5  # ❌ NameError: add не существует

# 2. 🟢 GREEN: Минимальный код
def add(a, b):
    return 5  # ✅ Тест проходит (хардкод)

# 3. 🟡 REFACTOR: Реальная логика
def add(a, b):
    return a + b  # ✅ Тест проходит, код правильный
```
