**Для Middle разработчика Git — это инструмент безопасности и координации.**

   **1. Основные команды (Lifecycle):**
   ```bash
   git init                # Создание репозитория
   git add .               # Добавление всех изменений в staging area
   git commit -m "msg"     # Фиксация снимка (snapshot)
   git push origin main    # Отправка на удаленный сервер
   git pull origin main    # Получение изменений + слияние (fetch + merge)
   ```

   **2. Ветвление и Слияние (Branching & Merging):**
   - **Fast-Forward:** Если ветка `feature` ответвилась от `main` и `main` не менялся, Git просто передвигает указатель (история линейна).
   - **Recursive Merge:** Если были изменения в обеих ветках, Git создает новый «commit merge» (история ветвится).
   ```bash
   git checkout -b feature
   # ... работа ...
   git checkout main
   git merge feature       # Слияние
   ```

   **3. Решение конфликтов (Conflict Resolution):**
   Возникает, когда изменены одни и те же строки в разных ветках.
   ```text
   # В файле появится маркер:
   <<<<<<< HEAD
   print("Hello World")    # Твои изменения
   =======
   print("Hello Python")   # Изменения в ветке
   >>>>>>> feature
   ```
   - **Алгоритм:**
     1. Открыть файл.
     2. Выбрать нужный код (или объединить).
     3. Удалить маркеры (`<<<<`, `====`, `>>>>`).
     4. `git add <file>` → `git commit`.

   **4. Нюансы для Middle (Best Practices):**
   - **Rebase vs Merge:** `rebase` переписывает историю (делает линейной), `merge` сохраняет историю слияний. В общих ветках (`main`) `rebase` запрещен!
   - **Atomic Commits:** Один коммит — одна логическая задача. Не мешай рефакторинг и фичи.
   - **.gitignore:** Игнорируй `.env`, `__pycache__`, `venv`.
   - **SSH Keys:** Для безопасности используй SSH вместо HTTPS.

   **🎯 Совет для Middle:**
   На собеседовании скажи: «Я предпочитаю `merge --no-ff` для фич-веток, чтобы сохранять историю разработки, но использую `rebase` локально для очистки истории перед пушем».
