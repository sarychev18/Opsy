## Основы работы и базовые команды

### 1. Что такое Git
**Вопрос:** Что такое Git и зачем он нужен?

**Ответ:** Git — это распределённая система контроля версий (DVCS), которая отслеживает изменения в файлах проекта и хранит их историю в виде цепочки коммитов. В отличие от централизованных VCS (например, SVN), у каждого разработчика есть полная локальная копия репозитория со всей историей, поэтому большинство операций (просмотр истории, создание веток, коммиты) выполняются без обращения к серверу. Git нужен, чтобы безопасно вести параллельную разработку несколькими людьми, откатываться к любой точке истории, сравнивать версии кода и объединять изменения из разных источников без потери данных.

| Термин | Определение |
|---|---|
| DVCS | Distributed Version Control System — распределённая система контроля версий |
| Коммит | Зафиксированный снимок изменений с уникальным хэшем |
| Репозиторий | Хранилище проекта со всей историей изменений |
| Рабочая копия | Локальные файлы проекта на диске разработчика |

**Ответ для интервьюера:** Git — распределённая система контроля версий: у каждого разработчика есть полная копия истории репозитория локально, поэтому большинство операций выполняется офлайн. Она нужна для параллельной командной разработки, отката к любой версии и слияния изменений без потери данных.

### 2. git add, commit, push
**Вопрос:** Объясни разницу между git add, git commit и git push.

**Ответ:** `git add` переносит изменения из рабочей директории в staging area (индекс) — это подготовка к коммиту, где можно выбрать, какие именно изменения войдут в следующий снимок. `git commit` фиксирует содержимое staging area как новый коммит в локальной истории репозитория с сообщением, автором и хэшем — это чисто локальная операция. `git push` отправляет локальные коммиты, которых ещё нет в удалённом репозитории, на удалённый сервер (origin), синхронизируя ветку на сервере с локальной.

```bash
git add file.py           # добавить файл в staging area
git add .                 # добавить все изменённые файлы
git commit -m "fix: исправлена валидация"   # зафиксировать локально
git push origin main      # отправить коммиты на удалённый репозиторий
```

| Термин | Определение |
|---|---|
| Staging area (индекс) | Промежуточная зона между рабочей копией и коммитом |
| Рабочая директория | Файлы на диске в текущем состоянии |
| origin | Стандартное имя удалённого репозитория по умолчанию |

**Ответ для интервьюера:** git add помещает изменения в staging area, git commit фиксирует их как снимок в локальной истории, git push отправляет локальные коммиты на удалённый сервер. Первые два — локальные операции, push — единственный из трёх, требующий сети и синхронизации с сервером.

### 3. .gitignore
**Вопрос:** Что такое .gitignore и зачем он нужен?

**Ответ:** `.gitignore` — это файл с шаблонами путей, которые Git должен игнорировать и не отслеживать: не показывать их в `git status`, не предлагать добавить в staging и не включать в коммиты. Он нужен, чтобы не засорять репозиторий артефактами сборки, зависимостями (node_modules, venv), файлами окружения с секретами (.env), кэшами IDE и локальными конфигами — то есть всем, что генерируется автоматически или уникально для машины разработчика. Если файл уже отслеживается Git, добавление его в .gitignore не уберёт его из истории — нужно явно выполнить `git rm --cached`.

```gitconfig
# .gitignore
__pycache__/
*.pyc
.env
node_modules/
venv/
.idea/
dist/
```

| Термин | Определение |
|---|---|
| Шаблон (pattern) | Правило, по которому путь исключается из отслеживания |
| Отслеживаемый файл (tracked) | Файл, уже добавленный в индекс Git хотя бы раз |
| git rm --cached | Убрать файл из индекса, не удаляя с диска |

**Ответ для интервьюера:** .gitignore задаёт список путей и шаблонов, которые Git не должен отслеживать — сборочные артефакты, зависимости, секреты, файлы IDE. Важный нюанс: он не действует на файлы, которые уже попали в историю, их нужно предварительно убрать через git rm --cached.

### 4. История коммитов
**Вопрос:** Как посмотреть историю коммитов и какие команды для этого используются?

**Ответ:** Основная команда — `git log`, которая показывает список коммитов от последнего к первому с хэшем, автором, датой и сообщением; с флагами она становится гибче: `--oneline` сжимает вывод до одной строки на коммит, `--graph` рисует ветвление ASCII-графом, `-p` показывает diff каждого коммита, `--author` и `--since` фильтруют по автору и дате. Для истории изменений конкретного файла удобен `git log -- <file>` или `git blame <file>`, который показывает, кто и в каком коммите последний раз менял каждую строку. `git show <hash>` показывает содержимое конкретного коммита целиком.

```bash
git log --oneline --graph --all
git log -p -- src/app.py
git blame src/app.py
git show a1b2c3d
```

| Термин | Определение |
|---|---|
| git log | Основная команда просмотра истории коммитов |
| git blame | Показывает автора последнего изменения каждой строки файла |
| git show | Показывает содержимое конкретного коммита |

**Ответ для интервьюера:** git log — базовая команда для просмотра истории, обычно использую с --oneline --graph для компактного визуального представления ветвления. Для истории конкретного файла или строки — git log -- file и git blame, для содержимого конкретного коммита — git show.

### 5. Откат изменений
**Вопрос:** Как откатить изменения в Git?

**Ответ:** Способ отката зависит от того, где находятся изменения и нужно ли сохранить историю. `git checkout -- file` (или `git restore file`) отменяет незакоммиченные изменения в рабочей директории. `git reset` отменяет коммиты, перемещая указатель ветки назад: `--soft` оставляет изменения в staging area, `--mixed` (по умолчанию) — в рабочей директории, `--hard` удаляет их безвозвратно. `git revert` создаёт новый коммит, отменяющий изменения указанного коммита, не переписывая историю — это безопасный способ для уже запушенных и публичных коммитов, в отличие от reset, который переписывает историю и опасен после push.

|Команда|Что происходит с коммитами|Что происходит с изменениями|Безопасно после push|
|---|---|---|---|
|**git restore**|Не трогает коммиты|Отменяет незакоммиченные изменения в рабочей директории|✅|
|**git reset --soft**|Указатель ветки сдвигается назад|Изменения остаются в staging area|❌ (переписывает историю)|
|**git reset --mixed**|Указатель ветки сдвигается назад|Изменения остаются в рабочей директории|❌ (переписывает историю)|
|**git reset --hard**|Указатель ветки сдвигается назад|Изменения удаляются безвозвратно|❌ (переписывает историю)|
|**git revert**|Создаёт новый коммит, отменяющий старый|Изменения отменяются новым коммитом|✅|

```bash
git restore file.py               # отменить незакоммиченные изменения файла
git reset --soft HEAD~1           # отменить коммит, изменения остаются в staging
git reset --hard HEAD~1           # отменить коммит и удалить изменения полностью
git revert a1b2c3d                # создать новый коммит, отменяющий старый
```

| Термин | Определение |
|---|---|
| git reset | Перемещение указателя ветки назад, с опциями soft/mixed/hard |
| git revert | Отмена коммита новым коммитом без переписывания истории |
| HEAD | Указатель на текущий коммит |

**Ответ для интервьюера:** Для локальных незакоммиченных изменений использую git restore, для отмены локальных коммитов — git reset с нужным режимом (soft/mixed/hard). Для уже запушенных публичных коммитов использую git revert, потому что он не переписывает историю, а reset --hard на опубликованной ветке ломает историю у всех, кто её уже забрал.

### 6. git pull vs git fetch
**Вопрос:** В чем разница между git pull и git fetch?

**Ответ:** `git fetch` скачивает новые коммиты и ветки с удалённого репозитория в локальный кэш (`origin/main` и т.д.), но не трогает текущую рабочую ветку и не делает merge — это безопасный способ узнать, что изменилось на сервере, перед тем как решить, что с этим делать. `git pull` — это `git fetch` плюс автоматическое слияние (`merge` или, с флагом `--rebase`, `rebase`) полученных изменений в текущую локальную ветку, то есть сразу применяет изменения и может привести к конфликтам или неожиданному merge-коммиту.

|Характеристика|git fetch|git pull|
|---|---|---|
|**Скачивает изменения**|✅|✅|
|**Применяет к текущей ветке**|❌|✅ (merge или rebase)|
|**Риск конфликтов сразу**|❌|✅|
|**Когда использовать**|Посмотреть, что нового, не трогая рабочую ветку|Сразу синхронизировать локальную ветку с удалённой|

```bash
git fetch origin              # скачать изменения, не применяя их
git diff main origin/main     # посмотреть, что изменилось на сервере
git pull origin main          # fetch + merge за одну команду
git pull --rebase origin main # fetch + rebase вместо merge
```

| Термин | Определение |
|---|---|
| git fetch | Скачивание изменений без применения к локальной ветке |
| git pull | fetch + merge (или rebase) в одну команду |
| origin/main | Локальная ссылка на состояние ветки main на удалённом сервере |

**Ответ для интервьюера:** git fetch только скачивает изменения с сервера, не трогая рабочую ветку — безопасно посмотреть, что нового. git pull — это fetch плюс немедленный merge (или rebase с флагом) в текущую ветку, поэтому может сразу породить конфликты; я предпочитаю fetch и осознанный merge/rebase, когда важно контролировать процесс.

## Ветвление и слияние

### 7. Ветки (branch)
**Вопрос:** Что такое ветка (branch) в Git и зачем они нужны?

**Ответ:** Ветка (branch) — это независимый указатель на конкретный коммит, который движется вперёд при каждом новом коммите; технически это просто файл с хэшем коммита, поэтому создание ветки в Git — мгновенная и дешёвая операция. Ветки нужны, чтобы разрабатывать новую функциональность, исправлять баги или экспериментировать изолированно от стабильного кода в main, не мешая другим разработчикам, а затем интегрировать изменения обратно через merge или rebase, когда работа готова.

```bash
git branch feature/login          # создать ветку
git checkout feature/login        # переключиться на неё
git switch -c feature/login       # создать и сразу переключиться (современный синтаксис)
```

| Термин | Определение |
|---|---|
| Branch | Подвижный указатель на коммит |
| HEAD | Указатель на текущую активную ветку/коммит |
| Checkout / switch | Переключение между ветками |

**Ответ для интервьюера:** Ветка — это лёгкий подвижный указатель на коммит, поэтому создавать их в Git дёшево и быстро. Они нужны для изолированной разработки фич, багфиксов и экспериментов без риска сломать стабильный код в основной ветке, с последующей интеграцией через merge или rebase.

### 8. git merge vs git rebase
**Вопрос:** Объясни разницу между git merge и git rebase.

**Ответ:** `git merge` объединяет две ветки, создавая новый merge-коммит с двумя родителями, который сохраняет фактическую историю ветвления как она была — это безопасно (не переписывает существующие коммиты), но история со временем становится «дерево с развилками» и менее читаемой. `git rebase` переносит коммиты текущей ветки поверх последнего коммита целевой ветки, переписывая их хэши, в результате получается линейная история без merge-коммитов, но это опасно для уже опубликованных (запушенных и используемых другими) веток — переписанная история конфликтует с чужими копиями. Правило: rebase используют для локальной, ещё не опубликованной ветки перед тем, как влить её в общую, merge — для интеграции завершённой работы в общую ветку.

|Характеристика|git merge|git rebase|
|---|---|---|
|**История**|Нелинейная, с развилками|Линейная|
|**Переписывает хэши коммитов**|❌|✅|
|**Создаёт merge-коммит**|✅|❌|
|**Безопасность на опубликованной ветке**|✅|❌ (золотое правило: не rebase-ить публичную ветку)|
|**Когда использовать**|Интеграция завершённой работы в общую ветку|Локальная, ещё не опубликованная ветка|

```bash
git checkout feature
git merge main              # создаёт merge-коммит, история нелинейная

git checkout feature
git rebase main             # переносит коммиты feature поверх main, история линейная
```

| Термин | Определение |
|---|---|
| Merge-коммит | Коммит с двумя родителями, объединяющий ветки |
| Rebase | Перенос коммитов на новую базу с изменением их хэшей |
| Линейная история | История без развилок, коммит за коммитом |

**Ответ для интервьюера:** merge создаёт merge-коммит и сохраняет реальную историю ветвления, ничего не переписывая — безопасно всегда. rebase переносит коммиты на новую базу, давая линейную историю, но переписывает хэши, поэтому его нельзя применять к уже опубликованной ветке, которую используют другие — это золотое правило rebase.

### 9. Конфликт слияния
**Вопрос:** Что такое конфликт слияния, когда он возникает и как его разрешить?

**Ответ:** Конфликт слияния возникает, когда Git не может автоматически объединить изменения, потому что одни и те же строки одного файла были изменены по-разному в обеих сливаемых ветках (или файл удалён в одной ветке и изменён в другой). Git останавливает merge/rebase, помечает конфликтующие места в файле специальными маркерами и оставляет разрешение конфликта человеку. Разработчик открывает файл, вручную выбирает или комбинирует нужный вариант, удаляет маркеры конфликта, добавляет файл через `git add` и завершает операцию (`git commit` для merge или `git rebase --continue` для rebase).

```bash
<<<<<<< HEAD
return calculate_total(items, discount=0.1)
=======
return calculate_total(items, apply_tax=True)
>>>>>>> feature/tax

# после ручного разрешения:
git add pricing.py
git commit          # для merge
git rebase --continue  # для rebase
```

| Термин | Определение |
|---|---|
| Конфликт слияния | Ситуация, когда Git не может автоматически объединить изменения |
| Маркеры конфликта | `<<<<<<<`, `=======`, `>>>>>>>` — границы конфликтующих версий |
| git mergetool | Утилита для визуального разрешения конфликтов |

**Ответ для интервьюера:** Конфликт возникает, когда одни и те же строки файла изменены по-разному в обеих ветках, и Git не может выбрать версию сам. Разрешаю вручную: открываю файл, смотрю на маркеры `<<<<<<<`/`=======`/`>>>>>>>`, оставляю нужный код, убираю маркеры, добавляю файл через git add и завершаю операцию commit или rebase --continue.

### 10. main, develop, feature
**Вопрос:** В чем разница между ветками main (master), develop и feature?

**Ответ:** `main` (или `master`) — основная стабильная ветка, отражающая production-состояние кода, в неё обычно попадают только протестированные и готовые изменения. `develop` — интеграционная ветка, куда мержатся завершённые фичи перед тем, как накопленные изменения пойдут в релиз на main; она менее стабильна, чем main, но стабильнее отдельных feature-веток. `feature/*` — короткоживущие ветки для разработки конкретной задачи или функциональности, создаются от develop (или main в упрощённых схемах) и удаляются после слияния — в них может быть нестабильный, ещё не готовый код.

|Ветка|Стабильность|Время жизни|Создаётся от|
|---|---|---|---|
|**main/master**|Максимальная (production)|Долгоживущая|—|
|**develop**|Средняя (интеграционная)|Долгоживущая|main|
|**feature/\***|Низкая (в разработке)|Короткоживущая, удаляется после мержа|develop (или main)|

```bash
git checkout develop
git switch -c feature/payment-refactor
# ... работа над задачей ...
git checkout develop
git merge feature/payment-refactor
git branch -d feature/payment-refactor
```

| Термин | Определение |
|---|---|
| main/master | Стабильная production-ветка |
| develop | Интеграционная ветка для накопления готовых фич |
| feature/* | Короткоживущая ветка под конкретную задачу |

**Ответ для интервьюера:** main отражает стабильный production-код, develop — интеграционная ветка, куда стекаются завершённые фичи перед релизом, а feature-ветки — короткоживущие изолированные ветки под конкретную задачу, которые создаются от develop и удаляются после слияния.

### 11. Ветки stage и release
**Вопрос:** Для чего используются ветки stage и release?

**Ответ:** `release` — это ветка, создаваемая от develop, когда набор фич для следующего релиза заморожен: в ней допускаются только багфиксы, финальное тестирование, обновление номеров версий и документации, но не новая функциональность — после готовности она мержится и в main, и обратно в develop, а на main ставится тег версии. `stage` (staging) — это не столько Git-ветка в узком смысле, сколько окружение (и часто соответствующая ветка), максимально приближенное к production, куда деплоится release-ветка для финального ручного и автоматического тестирования перед реальным релизом, чтобы поймать проблемы до того, как код увидят пользователи.

```bash
git checkout develop
git switch -c release/2.4.0
# багфиксы, обновление версии, тестирование на stage-окружении
git checkout main
git merge release/2.4.0
git tag v2.4.0
git checkout develop
git merge release/2.4.0
```

| Термин | Определение |
|---|---|
| release-ветка | Ветка заморозки фич перед выпуском версии |
| stage (staging) | Предпродакшн-окружение для финального тестирования |
| Тег версии | Метка коммита, фиксирующая версию релиза |

**Ответ для интервьюера:** release-ветка создаётся от develop, когда фичи для версии заморожены, и в ней разрешены только багфиксы и подготовка к выпуску; после готовности она мержится в main с тегом версии и обратно в develop. stage — это предпродакшн-окружение (часто со своей веткой), куда деплоят release-ветку для финального тестирования перед реальным релизом на production.

## Командная разработка и процессы (Workflow)

### 12. Workflow работы над задачей
**Вопрос:** Опиши типичный Workflow работы над задачей: от создания ветки до слияния в основную ветку.

**Ответ:** Сначала обновляю основную/интеграционную ветку (`git pull`), затем создаю от неё новую feature-ветку с понятным именем, отражающим задачу (например, `feature/JIRA-123-add-auth`). Далее работаю итеративно: делаю небольшие логичные коммиты с понятными сообщениями, периодически подтягиваю изменения из основной ветки (rebase или merge), чтобы не накапливать большой дрейф. Когда задача готова, пушу ветку на удалённый репозиторий и открываю Pull Request, прохожу код-ревью и, если нужно, CI-проверки (тесты, линтеры), после чего ветка мержится в основную и удаляется.

```bash
git checkout develop && git pull
git switch -c feature/JIRA-123-add-auth
# ... работа, коммиты ...
git push -u origin feature/JIRA-123-add-auth
# открыть PR, пройти ревью и CI
# после аппрува — merge через PR, удалить ветку
```

| Термин | Определение |
|---|---|
| Feature-ветка | Ветка под конкретную задачу |
| Pull Request | Запрос на слияние ветки с код-ревью |
| CI-проверки | Автоматические тесты и линтеры перед мержем |

**Ответ для интервьюера:** Обновляю основную ветку, создаю от неё feature-ветку с именем задачи, делаю небольшие осмысленные коммиты, периодически синхронизируюсь с основной веткой, затем открываю Pull Request с прохождением код-ревью и CI. После аппрува ветка мержится в основную и удаляется, чтобы не копить неактуальные ветки.

### 13. Git Flow
**Вопрос:** Что такое Git Flow?

**Ответ:** Git Flow — это формализованная модель ветвления, предложенная Винсентом Дриссеном, с чётко определёнными типами веток и правилами их взаимодействия: долгоживущие `main` (production) и `develop` (интеграция), и короткоживущие `feature/*` (новая функциональность, от develop), `release/*` (подготовка релиза, от develop, мержится в main и develop) и `hotfix/*` (срочные фиксы production, от main, мержится в main и develop). Она хорошо подходит для проектов с чёткими версионными релизами и продолжительным циклом разработки, но считается избыточно сложной для проектов с непрерывной доставкой (CI/CD), где чаще используют более простые модели вроде GitHub Flow или Trunk-Based Development.

```bash
git flow init
git flow feature start login
git flow feature finish login
git flow release start 1.2.0
git flow release finish 1.2.0
git flow hotfix start critical-fix
```

| Термин | Определение |
|---|---|
| Git Flow | Модель ветвления с main, develop, feature, release, hotfix |
| hotfix-ветка | Срочный фикс production, создаётся от main |
| GitHub Flow / Trunk-Based | Более простые альтернативы Git Flow для CI/CD |

**Ответ для интервьюера:** Git Flow — модель ветвления с двумя долгоживущими ветками (main, develop) и тремя типами короткоживущих (feature, release, hotfix), каждая со строгими правилами, откуда создаётся и куда мержится. Хорошо подходит для версионных релизов, но избыточна для команд с непрерывным деплоем, где проще подходит GitHub Flow или Trunk-Based Development.

### 14. Pull Request / Merge Request
**Вопрос:** Что такое Pull Request (PR) или Merge Request (MR) и зачем они нужны в процессе разработки?

**Ответ:** Pull Request (в GitHub) или Merge Request (в GitLab) — это формальный запрос на слияние изменений из одной ветки в другую, создаваемый на уровне хостинга репозитория (не сама команда Git), который служит точкой синхронизации команды перед тем, как код попадёт в общую ветку. PR даёт площадку для код-ревью (комментарии к конкретным строкам, обсуждение подхода), автоматически запускает CI-проверки (тесты, линтеры, билд), показывает diff всех изменений и историю коммитов ветки, и требует аппрува от одного или нескольких ревьюеров прежде, чем merge станет возможен — это ключевой механизм контроля качества кода в команде.

| Термин | Определение |
|---|---|
| Pull/Merge Request | Запрос на слияние ветки с код-ревью на хостинге репозитория |
| Аппрув (approval) | Одобрение изменений ревьюером |
| CI checks | Автоматические проверки, запускаемые при открытии/обновлении PR |

**Ответ для интервьюера:** PR/MR — это запрос на слияние ветки, создаваемый на уровне хостинга (GitHub/GitLab), а не самим Git. Он служит точкой код-ревью, автоматических CI-проверок и обязательного аппрува перед мержем — основной механизм контроля качества и знаний о коде в команде.

### 15. Workflow в команде
**Вопрос:** Как организовать workflow работы с Git в команде?

**Ответ:** Выбор модели ветвления зависит от частоты релизов: для непрерывной доставки подходит Trunk-Based Development или GitHub Flow (короткоживущие ветки от main, частые небольшие PR), для версионных релизов — Git Flow. Общие практики независимо от модели: защита основной ветки (branch protection — запрет прямого push, обязательный PR и хотя бы один аппрув), обязательные CI-проверки перед мержем, единый стиль коммит-сообщений (например, Conventional Commits), маленькие и частые PR вместо огромных, и договорённость о стратегии мержа (merge commit, squash или rebase) для единообразной истории.

```bash
# Пример защиты ветки задаётся не Git-командами, а настройками хостинга,
# но локально команда придерживается конвенции коммитов:
git commit -m "feat(auth): добавить refresh token"
git commit -m "fix(payment): исправить округление суммы"
```

| Термин | Определение |
|---|---|
| Branch protection | Настройка хостинга, запрещающая прямой push в защищённую ветку |
| Trunk-Based Development | Модель с короткоживущими ветками и частой интеграцией в main |
| Conventional Commits | Соглашение о структуре сообщений коммитов |

**Ответ для интервьюера:** Выбираю модель ветвления под частоту релизов — Trunk-Based/GitHub Flow для непрерывной доставки, Git Flow для версионных релизов. Обязательно включаю защиту основной ветки, требую CI-проверки и аппрув перед мержем, придерживаюсь единого формата коммитов и стараюсь держать PR маленькими для быстрого ревью.

### 16. Git и Code Review
**Вопрос:** Какую роль играет Git в процессе Code Review?

**Ответ:** Git — техническая основа Code Review: изолированные ветки позволяют разработать изменения, не затрагивая основной код, а Pull/Merge Request на их основе даёт ревьюеру точный diff — что именно изменилось, строка за строкой, с возможностью оставлять комментарии к конкретным строкам. История коммитов внутри PR показывает логику работы автора, а требование зелёного CI (тесты, линтеры) перед ревью экономит время ревьюера, отсеивая очевидные проблемы автоматически. После правок по замечаниям автор коммитит исправления в ту же ветку, PR автоматически обновляется, и ревьюер видит именно новые изменения (incremental diff), а не всё заново.

| Термин | Определение |
|---|---|
| Diff | Построчное сравнение изменений между версиями файла |
| Incremental diff | Показ только новых изменений после правок по ревью |
| Inline comment | Комментарий ревьюера к конкретной строке кода |

**Ответ для интервьюера:** Git даёт техническую базу для ревью: изолированная ветка и PR с точным diff, комментариями к конкретным строкам и автоматическим CI перед ревью. После правок автор коммитит в ту же ветку, и ревьюер видит только новые изменения, а не весь код заново — это делает итеративное ревью быстрым.

## Git в инфраструктуре и CI/CD

### 17. Git и триггеры CI/CD
**Вопрос:** Как команды Git связаны с триггерами в CI/CD пайплайнах (например, при push или merge)?

**Ответ:** CI/CD-системы (GitHub Actions, GitLab CI, Jenkins) подписываются на Git-события через вебхуки хостинга репозитория: push в ветку, открытие или обновление Pull Request, merge в определённую ветку, создание тега — каждое из них может запускать свой пайплайн. Например, push в любую feature-ветку запускает линтеры и unit-тесты, открытие PR в main дополнительно гоняет интеграционные тесты и билд, а merge в main или создание тега `v*` триггерит деплой в production. Конфигурация описывается декларативно в самом репозитории (`.github/workflows/*.yml`, `.gitlab-ci.yml`), что делает пайплайн частью версионируемого кода.

```yaml
# .github/workflows/ci.yml
on:
  push:
    branches: [feature/**]
  pull_request:
    branches: [main]
  push:
    tags: ['v*']
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pytest
```

| Термин | Определение |
|---|---|
| Webhook | HTTP-уведомление о Git-событии, запускающее внешнюю систему |
| Триггер (on: push/pull_request) | Событие Git, запускающее CI/CD пайплайн |
| Тег (tag) | Неподвижная метка на коммите, часто обозначает версию релиза |

**Ответ для интервьюера:** CI/CD системы подписаны на Git-события через вебхуки: push, открытие/обновление PR, merge в конкретную ветку или создание тега — каждое запускает соответствующий пайплайн (линтеры/тесты на push, деплой на merge в main или тег версии). Конфигурация пайплайна хранится прямо в репозитории и версионируется вместе с кодом.

### 18. Процесс деплоя после main
**Вопрос:** Как организован процесс деплоя после того, как ветка попадает в main?

**Ответ:** После merge в main CI/CD-пайплайн автоматически запускается по триггеру push/merge: собирает приложение (билд, докер-образ), прогоняет полный набор тестов, при успехе публикует артефакт (образ в container registry) и разворачивает его на целевом окружении — напрямую или через стратегию постепенного выката (rolling update, blue-green, canary), чтобы минимизировать риск для пользователей при проблемах. Часто перед production-деплоем есть промежуточный шаг на staging-окружении для финальной проверки, а сам деплой в production может требовать ручного подтверждения (manual approval) как дополнительный gate. Если что-то пошло не так, используется автоматический или ручной rollback — откат к предыдущему успешно задеплоенному артефакту.

```yaml
# упрощённый пример job деплоя после merge в main
deploy:
  needs: test
  if: github.ref == 'refs/heads/main'
  runs-on: ubuntu-latest
  environment: production
  steps:
    - run: docker build -t app:${{ github.sha }} .
    - run: docker push registry/app:${{ github.sha }}
    - run: kubectl set image deployment/app app=registry/app:${{ github.sha }}
```

|Стратегия|Как работает|Риск для пользователей|
|---|---|---|
|**Rolling update**|Постепенная замена старых инстансов новыми|Средний — часть трафика какое-то время идёт на разные версии|
|**Blue-green**|Новая версия разворачивается параллельно, трафик переключается одним махом|Низкий — мгновенный откат переключением обратно|
|**Canary**|Новая версия сначала получает малую долю трафика, затем постепенно увеличивается|Минимальный — проблема затронет немногих пользователей|

| Термин | Определение |
|---|---|
| Rolling update / blue-green / canary | Стратегии постепенного безопасного выката новой версии |
| Manual approval | Ручное подтверждение перед деплоем в production как gate |
| Rollback | Откат к предыдущей рабочей версии при проблемах |

**Ответ для интервьюера:** После merge в main пайплайн автоматически собирает артефакт, прогоняет тесты и разворачивает его на окружении, обычно через безопасную стратегию выката (rolling update, blue-green или canary), иногда с ручным подтверждением перед production. При проблемах предусмотрен быстрый rollback к предыдущей рабочей версии.
