## Основы и проектирование БД

### 1. Реляционные vs нереляционные БД
**Вопрос:** Объясни разницу между реляционными и нереляционными базами данных.

**Ответ:** Реляционные БД (PostgreSQL, MySQL) хранят данные в таблицах со строгой заранее заданной схемой, связи между сущностями описываются через внешние ключи, а операции над данными гарантируют ACID. Это делает их предсказуемыми и надёжными для данных со сложными связями (финансы, учёт). Нереляционные БД (NoSQL) жертвуют жёсткой схемой и строгими связями ради гибкости, горизонтальной масштабируемости и высокой скорости записи/чтения, используя разные модели данных: документы (MongoDB), ключ-значение (Redis), колоночные (Cassandra, ClickHouse), графовые (Neo4j).

|Характеристика|Реляционные (SQL)|Нереляционные (NoSQL)|
|---|---|---|
|**Схема**|Строгая, заранее заданная|Гибкая или отсутствует|
|**Связи**|Через внешние ключи|Обычно денормализованы|
|**Гарантии**|ACID|Часто BASE / eventual consistency|
|**Масштабирование**|Преимущественно вертикальное|Горизонтальное «из коробки»|
|**Примеры**|PostgreSQL, MySQL|MongoDB, Redis, Cassandra, Neo4j|

| Термин | Определение |
|---|---|
| Схема (schema) | Жёстко заданная структура таблиц и типов данных |
| ACID | Гарантии надёжности транзакций в реляционных БД |
| Документная модель | Хранение данных в виде JSON/BSON-документов (MongoDB) |
| Горизонтальное масштабирование | Добавление серверов вместо увеличения мощности одного |

**Ответ для интервьюера:** Реляционные БД (PostgreSQL) хранят данные в таблицах со строгой схемой и гарантируют ACID-транзакции, что идеально для финансовых данных. NoSQL (MongoDB, Redis) жертвуют строгой схемой ради гибкости и горизонтального масштабирования, используя модели документов, ключ-значение или графов.

### 2. Первичный и внешний ключ
**Вопрос:** Что такое первичный ключ и внешний ключ?

**Ответ:** Первичный ключ (Primary Key, PK) — это поле или набор полей, однозначно идентифицирующих строку в таблице; он не может быть NULL и должен быть уникальным. Внешний ключ (Foreign Key, FK) — это поле, ссылающееся на первичный ключ другой (или той же) таблицы, оно связывает записи между таблицами и обеспечивает ссылочную целостность: нельзя вставить FK, которого нет в родительской таблице, и по умолчанию нельзя удалить родительскую запись, пока на неё есть ссылки.

```sql
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    customer_id INTEGER REFERENCES customers(id) ON DELETE CASCADE,
    total NUMERIC
);
```

| Термин | Определение |
|---|---|
| PRIMARY KEY | Уникальный идентификатор строки |
| FOREIGN KEY | Ссылка на PK другой таблицы |
| Ссылочная целостность | Гарантия, что FK всегда указывает на существующую запись |
| ON DELETE CASCADE | Автоматическое удаление зависимых строк при удалении родителя |

**Ответ для интервьюера:** Первичный ключ уникально идентифицирует строку в таблице и не может быть NULL. Внешний ключ ссылается на первичный ключ другой таблицы и обеспечивает ссылочную целостность — СУБД не даст создать «осиротевшую» запись.

### 3. Проблемы изменяемых PK
**Вопрос:** К каким проблемам может привести использование изменяемых строк в качестве первичных ключей?

**Ответ:** Если PK — это изменяемая строка (например, email или username), то при её изменении нужно каскадно обновлять все внешние ключи во всех связанных таблицах, что дорого и рискованно с точки зрения консистентности при высокой нагрузке. Кроме того, строковые ключи занимают больше места и работают медленнее в индексах, чем целочисленные или UUID, из-за более дорогого сравнения и большего размера B-Tree индекса. Стандартная практика — использовать суррогатный неизменяемый ключ (SERIAL/BIGSERIAL/UUID), а «естественный» изменяемый атрибут выносить в отдельное уникальное (UNIQUE), но не первичное поле.

| Термин | Определение |
|---|---|
| Естественный ключ | PK на основе бизнес-данных (email, ИНН) |
| Суррогатный ключ | Искусственный неизменяемый идентификатор (id, UUID) |
| Каскадное обновление | Обновление FK во всех связанных таблицах при изменении PK |
| UNIQUE constraint | Ограничение уникальности без роли первичного ключа |

**Ответ для интервьюера:** Изменяемый первичный ключ требует каскадного обновления всех внешних ключей в связанных таблицах при каждом изменении, что дорого и рискует консистентностью. Поэтому используют суррогатный неизменяемый ключ (id/UUID), а изменяемые атрибуты помечают как UNIQUE, а не PRIMARY KEY.

### 4. Нормализация и нормальные формы (1NF–3NF)
**Вопрос:** Что такое нормализация данных и зачем она нужна? Опиши первые три нормальные формы (1NF, 2NF, 3NF).

**Ответ:** Нормализация — это процесс структурирования таблиц так, чтобы устранить избыточность данных и аномалии при вставке, обновлении и удалении. 1NF требует атомарности значений (в ячейке — одно значение, не список) и уникальности строк. 2NF требует выполнения 1NF плюс отсутствия частичной зависимости — каждый неключевой атрибут должен зависеть от всего составного первичного ключа, а не от его части. 3NF требует выполнения 2NF плюс отсутствия транзитивной зависимости — неключевые атрибуты не должны зависеть друг от друга, только от PK напрямую.

| Термин | Определение |
|---|---|
| 1NF | Атомарность значений, отсутствие повторяющихся групп |
| 2NF | Отсутствие частичной зависимости от составного ключа |
| 3NF | Отсутствие транзитивной зависимости неключевых атрибутов |
| Аномалия обновления | Несогласованность данных из-за дублирования |

**Ответ для интервьюера:** Нормализация убирает дублирование данных и аномалии при изменениях, разбивая данные на связанные таблицы. 1NF — атомарные значения, 2NF — нет зависимости атрибута от части составного ключа, 3NF — нет зависимости одного неключевого атрибута от другого.

### 5. Денормализация
**Вопрос:** Когда и зачем используется денормализация?

**Ответ:** Денормализация — это намеренное добавление избыточности (дублирование данных или предвычисленных агрегатов) для ускорения чтения за счёт усложнения записи и риска рассинхронизации. Её применяют, когда чтение доминирует над записью и JOIN между множеством нормализованных таблиц становится узким местом производительности — например, в аналитике, дашбордах, кэшах витрин данных. Обратная сторона — нужно поддерживать целостность вручную (через триггеры, батч-обновления или eventual consistency).

| Термин | Определение |
|---|---|
| Денормализация | Намеренное дублирование данных ради скорости чтения |
| Read-heavy нагрузка | Сценарий, где чтений значительно больше, чем записей |
| Витрина данных (data mart) | Денормализованное хранилище для аналитики |
| Eventual consistency | Согласованность данных с задержкой |

**Ответ для интервьюера:** Денормализацию используют, когда read-нагрузка значительно превышает write и JOIN-ы становятся узким местом — данные дублируют или предвычисляют агрегаты, ускоряя чтение ценой сложности поддержания консистентности при записи.

### 6. Критерии выбора БД / Polyglot Persistence
**Вопрос:** Объясни критерии выбора базы данных для проекта (PostgreSQL, Redis, MongoDB). Приведи пример гибридной архитектуры (Polyglot Persistence).

**Ответ:** Выбор БД зависит от структуры данных, требований к консистентности, паттерна доступа и нагрузки. PostgreSQL выбирают для структурированных данных со связями и требованием ACID (заказы, платежи, пользователи). Redis — для кэша, сессий, очередей, счётчиков и любых сценариев, где важна скорость доступа к данным в памяти и допустима более простая модель. MongoDB — для слабоструктурированных или часто меняющихся по форме данных (логи, каталоги товаров с разными атрибутами). Polyglot Persistence — это архитектура, где для разных подсистем одного приложения используются разные БД одновременно: например, PostgreSQL как основной источник истины, Redis как кэш горячих данных и сессий, а Elasticsearch/MongoDB — для полнотекстового поиска или логов.

|БД|Модель данных|Сильная сторона|Типовое применение|
|---|---|---|---|
|**PostgreSQL**|Реляционная, ACID|Строгая консистентность, связи|Заказы, платежи, пользователи|
|**Redis**|Ключ-значение, in-memory|Скорость доступа|Кэш, сессии, очереди, счётчики|
|**MongoDB**|Документная, гибкая схема|Гибкость структуры|Каталоги, логи, произвольные профили|

| Термин | Определение |
|---|---|
| Polyglot Persistence | Использование нескольких типов БД в одном проекте под разные задачи |
| Source of truth | Основное хранилище, которому доверяют при конфликте данных |
| In-memory store | Хранилище данных в оперативной памяти (Redis) |
| Паттерн доступа | Характер типичных запросов к данным (частота, тип, объём) |

**Ответ для интервьюера:** PostgreSQL для структурированных данных со строгой консистентностью, Redis — для кэша и быстрого доступа в памяти, MongoDB — для гибкой схемы. В реальных проектах их часто комбинируют (Polyglot Persistence): PostgreSQL как источник истины, Redis как кэш, Elasticsearch/MongoDB — для поиска и логов.

## Язык SQL и запросы

### 7. SELECT, INSERT, UPDATE, DELETE
**Вопрос:** Объясни разницу между SELECT, INSERT, UPDATE, DELETE.

**Ответ:** Это базовые команды DML (Data Manipulation Language). SELECT читает данные из таблиц без изменения их состояния. INSERT добавляет новые строки. UPDATE изменяет значения существующих строк по условию. DELETE удаляет строки по условию. Все, кроме SELECT, изменяют состояние БД и обычно выполняются внутри транзакции.

```sql
SELECT id, name FROM customers WHERE id = 1;
INSERT INTO customers (name) VALUES ('Иван');
UPDATE customers SET name = 'Пётр' WHERE id = 1;
DELETE FROM customers WHERE id = 1;
```

| Термин | Определение |
|---|---|
| DML | Data Manipulation Language — язык манипуляции данными |
| WHERE | Условие, ограничивающее набор затрагиваемых строк |
| Транзакция | Атомарный блок из одной или нескольких DML-операций |

**Ответ для интервьюера:** SELECT читает данные, INSERT добавляет новые строки, UPDATE изменяет существующие по условию, DELETE удаляет строки по условию — все четыре относятся к DML и обычно оборачиваются в транзакцию для атомарности.

### 8. DELETE vs TRUNCATE
**Вопрос:** В чем разница между командами DELETE и TRUNCATE?

**Ответ:** DELETE удаляет строки построчно, может использовать WHERE, пишет каждую удалённую строку в журнал транзакций (WAL) и может быть отменён (ROLLBACK), но из-за построчного логирования работает медленно на больших таблицах. TRUNCATE мгновенно очищает всю таблицу целиком, минуя построчное логирование (использует деаллокацию страниц), не поддерживает WHERE, сбрасывает счётчики автоинкремента и в большинстве СУБД тоже транзакционен, но требует более строгой блокировки таблицы.

|Характеристика|DELETE|TRUNCATE|
|---|---|---|
|**WHERE**|✅ Поддерживает|❌ Только вся таблица|
|**Скорость на больших таблицах**|Медленнее (построчный лог)|Быстрее (деаллокация страниц)|
|**Автоинкремент**|Не сбрасывает|Сбрасывает|
|**ROLLBACK**|✅|✅ (в большинстве СУБД)|

```sql
DELETE FROM logs WHERE created_at < NOW() - INTERVAL '30 days';
TRUNCATE TABLE logs;
```

| Термин | Определение |
|---|---|
| DELETE | Построчное удаление с журналированием, поддерживает WHERE |
| TRUNCATE | Быстрая полная очистка таблицы без построчного лога |
| WAL | Write-Ahead Log — журнал для восстановления и отката |
| Автоинкремент | Счётчик значений PK (сбрасывается TRUNCATE) |

**Ответ для интервьюера:** DELETE удаляет строки построчно с возможностью WHERE и журналированием, поэтому медленнее и обратим. TRUNCATE мгновенно очищает всю таблицу, минуя построчный лог, но не позволяет фильтровать строки и сбрасывает автоинкремент.

### 9. INNER JOIN vs LEFT JOIN
**Вопрос:** Объясни разницу между INNER JOIN и LEFT JOIN (визуализация через круги Эйлера).

**Ответ:** INNER JOIN возвращает только те строки, для которых есть совпадение в обеих таблицах — на диаграмме Эйлера это пересечение двух кругов. LEFT JOIN возвращает все строки из левой (первой) таблицы, а из правой — данные, если совпадение найдено, иначе NULL — это весь левый круг плюс пересечение. LEFT JOIN полезен, когда нужно сохранить все записи «главной» сущности, даже если у неё нет связанных данных (например, все клиенты, включая тех, у кого пока нет заказов).

|Характеристика|INNER JOIN|LEFT JOIN|
|---|---|---|
|**Возвращает**|Только совпадающие строки в обеих таблицах|Все строки левой таблицы + совпадения справа|
|**Строки без совпадения справа**|Исключаются|Остаются, поля справа — NULL|
|**На диаграмме Эйлера**|Пересечение кругов|Весь левый круг + пересечение|

```sql
-- Только клиенты, у которых есть заказы
SELECT c.name, o.id FROM customers c
INNER JOIN orders o ON o.customer_id = c.id;

-- Все клиенты, включая тех, у кого заказов нет (o.id будет NULL)
SELECT c.name, o.id FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id;
```

| Термин | Определение |
|---|---|
| INNER JOIN | Пересечение — только совпадающие строки |
| LEFT JOIN | Все строки левой таблицы + совпадения справа (или NULL) |
| ON | Условие соединения таблиц |

**Ответ для интервьюера:** INNER JOIN возвращает только совпадающие строки в обеих таблицах — пересечение множеств. LEFT JOIN возвращает все строки левой таблицы, подставляя NULL там, где совпадений справа нет — это удобно, чтобы не терять записи без связанных данных.

### 10. SQL-запрос: JOIN клиентов и заказов
**Вопрос:** Напиши SQL-запрос для соединения таблиц клиентов и заказов по ID покупателя.

**Ответ:** Задача решается через JOIN по внешнему ключу customer_id, ссылающемуся на customers.id. Выбор INNER или LEFT JOIN зависит от того, нужны ли клиенты без заказов в результате.

```sql
SELECT
    c.id AS customer_id,
    c.name,
    o.id AS order_id,
    o.total
FROM customers c
JOIN orders o ON o.customer_id = c.id
ORDER BY c.id;
```

| Термин | Определение |
|---|---|
| Алиас (c, o) | Короткое имя таблицы для читаемости запроса |
| customer_id | Внешний ключ в orders, ссылающийся на customers.id |
| ORDER BY | Сортировка результата |

**Ответ для интервьюера:** Соединяю таблицы через JOIN по условию `orders.customer_id = customers.id`, используя алиасы для краткости; если нужны и клиенты без заказов — заменю JOIN на LEFT JOIN.

### 11. Агрегатные функции
**Вопрос:** Что такое агрегатные функции (COUNT, SUM, AVG) и когда их использовать?

**Ответ:** Агрегатные функции вычисляют одно итоговое значение по набору строк: COUNT считает количество строк, SUM суммирует числовое поле, AVG вычисляет среднее, MIN/MAX — минимум/максимум. Их используют для аналитики и отчётности — например, посчитать выручку, число заказов на клиента или средний чек. Без GROUP BY агрегат считается по всей выборке; с GROUP BY — отдельно по каждой группе.

```sql
SELECT customer_id, COUNT(*) AS orders_count, SUM(total) AS revenue
FROM orders
GROUP BY customer_id;
```

| Термин | Определение |
|---|---|
| COUNT | Количество строк |
| SUM / AVG | Сумма / среднее значение числового поля |
| GROUP BY | Группировка строк для поколоночного агрегирования |

**Ответ для интервьюера:** Агрегатные функции сворачивают набор строк в одно число — COUNT считает строки, SUM суммирует, AVG усредняет. Обычно применяются вместе с GROUP BY для получения метрик по группам, например выручки по каждому клиенту.

---

### 12. GROUP BY и ORDER BY
**Вопрос:** Что такое GROUP BY и ORDER BY в SQL?

**Ответ:** GROUP BY группирует строки с одинаковыми значениями указанных колонок в одну группу, что позволяет применять агрегатные функции отдельно к каждой группе; все колонки в SELECT, не входящие в агрегат, должны быть перечислены в GROUP BY. ORDER BY сортирует итоговый результат по одной или нескольким колонкам (ASC по умолчанию, либо DESC) и выполняется после группировки и фильтрации.

```sql
SELECT customer_id, SUM(total) AS revenue
FROM orders
GROUP BY customer_id
ORDER BY revenue DESC;
```

| Термин | Определение |
|---|---|
| GROUP BY | Группировка строк по значению колонок |
| ORDER BY | Сортировка результата запроса |
| HAVING | Фильтрация групп после агрегации (в отличие от WHERE) |

**Ответ для интервьюера:** GROUP BY объединяет строки с одинаковым значением колонки в группы для агрегации, а ORDER BY сортирует финальный результат. Порядок выполнения: сначала WHERE и GROUP BY, затем HAVING, в конце ORDER BY.

### 13. SQL injection
**Вопрос:** Что такое SQL injection и как защититься от него?

**Ответ:** SQL injection — это атака, при которой злоумышленник вставляет вредоносный SQL-код через непроверенный пользовательский ввод, если запрос собирается конкатенацией строк, что позволяет обойти аутентификацию, читать чужие данные или удалять таблицы. Защита — никогда не строить запрос конкатенацией с пользовательским вводом, а использовать параметризованные запросы или ORM, которые экранируют значения на уровне драйвера, плюс валидация входных данных и минимальные права у пользователя БД.

```python
# Уязвимо
cursor.execute(f"SELECT * FROM users WHERE name = '{username}'")

# Безопасно — параметризованный запрос
cursor.execute("SELECT * FROM users WHERE name = %s", (username,))
```

| Термин | Определение |
|---|---|
| SQL injection | Внедрение вредоносного SQL через пользовательский ввод |
| Параметризованный запрос | Запрос с плейсхолдерами, значения передаются отдельно |
| Экранирование | Преобразование спецсимволов, чтобы они не интерпретировались как код |

**Ответ для интервьюера:** SQL injection — это внедрение вредоносного SQL через конкатенацию пользовательского ввода в запрос. Защита — всегда использовать параметризованные запросы или ORM вместо конкатенации строк, плюс валидация ввода и минимальные права БД-пользователя.

---

### 14. Параметризованные запросы
**Вопрос:** Как работает параметризованный запрос и почему он безопаснее?

**Ответ:** В параметризованном запросе SQL-текст с плейсхолдерами (%s, ?, $1) отправляется в БД отдельно от значений параметров. Драйвер СУБД компилирует структуру запроса один раз, а значения передаются как данные, а не как часть SQL-синтаксиса, поэтому даже если строка содержит кавычки или SQL-ключевые слова, она не может изменить структуру запроса. Это устраняет SQL injection «по построению» и дополнительно ускоряет повторные запросы за счёт кэширования плана выполнения.

```python
# SQLAlchemy Core
result = connection.execute(
    text("SELECT * FROM orders WHERE customer_id = :cid"),
    {"cid": customer_id}
)
```

| Термин | Определение |
|---|---|
| Плейсхолдер | Символ-заполнитель для значения в запросе (%s, $1, :name) |
| План выполнения | Скомпилированная стратегия выполнения запроса |
| Bind-параметр | Значение, привязываемое к плейсхолдеру при выполнении |

**Ответ для интервьюера:** Параметризованный запрос разделяет SQL-структуру и данные — плейсхолдеры компилируются как часть запроса, а значения передаются отдельно и никогда не интерпретируются как SQL-код, что исключает SQL injection и попутно ускоряет повторные вызовы за счёт переиспользования плана выполнения.

## Транзакции и ACID

### 15. Транзакции и ACID
**Вопрос:** Объясни, что такое транзакция и какие принципы ACID она должна соблюдать.

**Ответ:** Транзакция — это последовательность операций с БД, выполняющаяся как единое неделимое целое: либо применяются все изменения, либо ни одно. ACID — набор гарантий: Atomicity (атомарность — всё или ничего), Consistency (согласованность — БД переходит из одного валидного состояния в другое, не нарушая ограничения), Isolation (изоляция — параллельные транзакции не мешают друг другу, как будто выполняются последовательно), Durability (устойчивость — после COMMIT данные сохраняются даже при сбое).

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

| Термин      | Определение                                    |
| ----------- | ---------------------------------------------- |
| Atomicity   | Всё или ничего                                 |
| Consistency | Переход между валидными состояниями данных     |
| Isolation   | Изоляция параллельных транзакций друг от друга |
| Durability  | Сохранность данных после успешного commit      |

**Ответ для интервьюера:** Транзакция — атомарный блок операций, который либо выполняется целиком, либо откатывается. ACID гарантирует: атомарность (всё или ничего), согласованность (данные остаются валидными), изоляцию (параллельные транзакции не влияют друг на друга) и устойчивость (изменения не теряются после commit).

---

### 16. Уровни изоляции транзакций
**Вопрос:** Объясни, что такое уровни изоляции транзакций.

**Ответ:** Уровень изоляции определяет, насколько сильно параллельно выполняющиеся транзакции «видят» незафиксированные изменения друг друга. Чем выше уровень изоляции, тем меньше аномалий чтения возможно, но тем больше блокировок и ниже пропускная способность — это классический компромисс между консистентностью и производительностью. СУБД реализуют уровни изоляции через блокировки (locking) или многоверсионность (MVCC — Multi-Version Concurrency Control, как в PostgreSQL).

| Термин            | Определение                                                     |
| ----------------- | --------------------------------------------------------------- |
| Изоляция          | Степень видимости изменений других незафиксированных транзакций |
| MVCC              | Многоверсионное управление конкурентным доступом                |
| Блокировка (lock) | Механизм запрета конкурентного доступа к данным                 |

**Ответ для интервьюера:** Уровень изоляции определяет, какие аномалии параллельного доступа допустимы: чем строже уровень, тем меньше аномалий, но выше стоимость в виде блокировок и потерь пропускной способности. PostgreSQL реализует это преимущественно через MVCC, а не через тотальные блокировки.

---

### 17. Четыре уровня изоляции и аномалии чтения
**Вопрос:** Опиши четыре уровня изоляции (READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE) и проблемы, которые они решают (грязное чтение, фантомное чтение и др.).

**Ответ:** READ UNCOMMITTED допускает грязное чтение (dirty read) — видны незафиксированные изменения других транзакций; в PostgreSQL фактически ведёт себя как READ COMMITTED. READ COMMITTED (уровень по умолчанию в PostgreSQL) устраняет грязное чтение, но допускает неповторяющееся чтение (non-repeatable read) — повторный SELECT в той же транзакции может вернуть другие данные, если их успели изменить и закоммитить. REPEATABLE READ фиксирует снимок данных на начало транзакции, устраняя неповторяющееся чтение, но теоретически допускает фантомное чтение (phantom read) — появление новых строк, подходящих под условие WHERE (в PostgreSQL фантомы тоже устранены за счёт MVCC). SERIALIZABLE — самый строгий уровень, гарантирует, что результат параллельных транзакций эквивалентен их последовательному выполнению, устраняя все аномалии, но ценой возможных откатов по конфликту сериализации.

```sql
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- операции
COMMIT;
```

| Уровень изоляции | Dirty Read | Non-repeatable Read | Phantom Read                | Особенность в PostgreSQL                                                                                             |
| ---------------- | ---------- | ------------------- | --------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| READ UNCOMMITTED | Возможен   | Возможен            | Возможен                    | Отдельно не реализован — фактически работает как READ COMMITTED                                                      |
| READ COMMITTED   | Невозможен | Возможен            | Возможен                    | Уровень по умолчанию; каждый оператор видит свежий снимок на момент своего старта                                    |
| REPEATABLE READ  | Невозможен | Невозможен          | Возможен (по стандарту SQL) | Фантомы тоже устранены — снимок MVCC фиксируется на весь старт транзакции                                            |
| SERIALIZABLE     | Невозможен | Невозможен          | Невозможен                  | Реализован через SSI (Serializable Snapshot Isolation); при конфликте транзакция откатывается с ошибкой сериализации |
|                  |            |                     |                             |                                                                                                                      |

**Ответ для интервьюера:** READ UNCOMMITTED допускает грязное чтение, READ COMMITTED устраняет его, но допускает неповторяющееся чтение, REPEATABLE READ фиксирует снимок на начало транзакции, устраняя это, а SERIALIZABLE даёт полную изоляцию ценой возможных откатов при конфликтах. PostgreSQL по умолчанию использует READ COMMITTED и благодаря MVCC устраняет даже фантомные чтения уже на REPEATABLE READ, хотя стандарт SQL формально допускает их на этом уровне.

## Оптимизация и производительность

### 18. Действия при медленном запросе
**Вопрос:** Твои действия, если в приложении обнаружен медленный запрос.

**Ответ:** Сначала воспроизвожу запрос и снимаю план выполнения через EXPLAIN ANALYZE, чтобы увидеть, где реально уходит время (Seq Scan вместо Index Scan, дорогой Sort, Nested Loop на больших наборах). Дальше проверяю наличие и селективность индексов на колонках в WHERE/JOIN/ORDER BY, смотрю на актуальность статистики таблицы (ANALYZE), ищу N+1 проблему на уровне ORM, оцениваю, не стоит ли переписать запрос (например, заменить подзапрос на JOIN) или денормализовать/закэшировать данные. После изменения повторно снимаю план и сравниваю метрики (время выполнения, число прочитанных страниц).

```sql
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 42;
```

| Термин | Определение |
|---|---|
| Seq Scan | Последовательное сканирование всей таблицы |
| Index Scan | Чтение через индекс, обычно быстрее для селективных условий |
| Селективность | Доля строк, удовлетворяющих условию |
| ANALYZE | Обновление статистики таблицы для планировщика |

**Ответ для интервьюера:** Сначала снимаю EXPLAIN ANALYZE, чтобы понять, где реально теряется время — отсутствие индекса, N+1 от ORM, дорогая сортировка. Затем целенаправленно исправляю: добавляю индекс, переписываю запрос или кэширую результат, и заново замеряю план для проверки эффекта.

### 19. Индексы и производительность
**Вопрос:** Что такое индексы в базах данных и как они улучшают производительность?

**Ответ:** Индекс — это дополнительная структура данных (чаще всего B-Tree), которая хранит отсортированные значения колонки вместе со ссылками на строки таблицы, позволяя находить нужные строки за O(log n) вместо полного сканирования таблицы O(n). Индексы ускоряют SELECT с условиями WHERE, JOIN и ORDER BY по индексированным колонкам, но замедляют INSERT/UPDATE/DELETE, так как индекс нужно поддерживать в актуальном состоянии, и занимают дополнительное место на диске.

```sql
CREATE INDEX idx_orders_customer_id ON orders(customer_id);
```

| Термин         | Определение                                             |
| -------------- | ------------------------------------------------------- |
| B-Tree         | Сбалансированное дерево — стандартная структура индекса |
| Index Scan     | Поиск строк через индекс                                |
| Write overhead | Дополнительная стоимость записи на поддержание индекса  |

**Ответ для интервьюера:** Индекс — это структура (обычно B-Tree), ускоряющая поиск строк с O(n) до O(log n) за счёт хранения отсортированных значений колонки со ссылками на строки. Плата за это — замедление записи и дополнительное место на диске, поэтому индексы добавляют осознанно под реальные паттерны запросов.

---

### 20. Виды индексов, их применение и сложность
**Вопрос:** Какие бывают виды индексов, где они используются и какая у них сложность?

**Ответ:** Помимо B-Tree по умолчанию, PostgreSQL поддерживает несколько типов индексов, оптимизированных под разные операторы и структуры данных. 
**B-Tree** — сбалансированное дерево, поддерживает = , `<`, `>`, `BETWEEN`, `LIKE 'prefix%'` и сортировку; поиск, вставка и удаление — O(log n). Это универсальный индекс по умолчанию для чисел, дат, строк, диапазонов и ORDER BY. 
**Hash** — хеш-таблица, поддерживает только оператор равенства =; поиск в среднем O(1), но не умеет диапазоны и сортировку, поэтому почти всегда проигрывает B-Tree по универсальности и используется редко. 
**GiST** (Generalized Search Tree) — обобщённое дерево для сложных типов данных (геометрия, диапазоны `range`, полнотекстовый поиск), поддерживает операторы пересечения/включения (`@>`, `<@`, `&&`); поиск в среднем O(log n), но в отличие от B-Tree не гарантирует строгую сбалансированность для всех типов данных и в худшем случае может деградировать. 
**GIN** (Generalized Inverted Index) — обратный индекс: хранит для каждого элемента (слова, ключа JSON, элемента массива) список строк, где он встречается; поддерживает `@>`, `@@`, `?` для массивов, JSONB и полнотекстового поиска — поиск по ключу O(log n) плюс O(k) на слияние списков строк (posting lists), но обновления заметно дороже, чем у B-Tree, так как одна строка может затрагивать много ключей. 
**BRIN** (Block Range Index) — блочный индекс: хранит min/max не по строкам, а по диапазонам физических страниц таблицы; занимает на порядки меньше места, чем B-Tree, а обход — O(число диапазонов блоков), то есть линейно от размера таблицы, но с очень маленькой константой. Работает только когда данные физически коррелируют с порядком колонки (например, `created_at` в таблице логов, куда пишут по возрастанию времени) — иначе индекс перестаёт быть избирательным и становится бесполезен. Отдельно от типа структуры есть модификаторы, применимые к любому из них: составной индекс (несколько колонок, важен порядок — правило левого префикса), частичный индекс (`WHERE`, покрывает только подмножество строк) и покрывающий индекс (`INCLUDE`, добавляет колонки для Index Only Scan без чтения самой таблицы).

```sql
-- B-Tree (по умолчанию): равенство, диапазоны, сортировка
CREATE INDEX idx_users_created ON users(created_at);

-- Hash: только точное равенство
CREATE INDEX idx_users_email_hash ON users USING HASH (email);

-- GiST: диапазоны, геоданные
CREATE INDEX idx_bookings_period ON bookings USING GIST (period);

-- GIN: JSONB, массивы, полнотекстовый поиск
CREATE INDEX idx_products_tags ON products USING GIN (tags);

-- BRIN: огромные таблицы с естественной сортировкой (логи)
CREATE INDEX idx_logs_time ON logs USING BRIN (event_time);
```

| Тип    | Сложность поиска                            | Операторы                        | Когда использовать                                                    |
| ------ | ------------------------------------------- | -------------------------------- | --------------------------------------------------------------------- |
| B-Tree | O(log n)                                    | =, <, >, BETWEEN, LIKE 'prefix%' | По умолчанию: равенство, диапазоны, сортировка                        |
| Hash   | O(1) в среднем                              | =                                | Только точное равенство (используется редко)                          |
| GiST   | O(log n) в среднем                          | @>, <@, &&, ~                    | Геоданные, диапазоны (range), полнотекст                              |
| GIN    | O(log n) + O(k) слияние                     | @>, @@, ?                        | Массивы, JSONB, полнотекстовый поиск                                  |
| BRIN   | O(число блоков), линейно от размера таблицы | =, <, >                          | Огромные таблицы с физически отсортированными данными (логи, метрики) |

**Ответ для интервьюера:** Кроме B-Tree (O(log n), универсальный индекс по умолчанию для равенства, диапазонов и сортировки), есть Hash (O(1), только равенство, редко нужен), GiST (O(log n) в среднем, геоданные и диапазоны), GIN (обратный индекс для массивов/JSONB/полнотекста, O(log n) плюс слияние списков строк, но дороже на запись) и BRIN (индекс по диапазонам блоков, почти бесплатный по месту, линейная сложность от числа блоков, но требует физической отсортированности данных, как в логах). Выбор типа зависит от оператора в запросе и структуры данных, а не только от объёма таблицы.

### 21. Когда индекс вреден
**Вопрос:** В каких случаях создание индекса может быть излишним или вредным?

**Ответ:** Индекс вреден, когда он не используется планировщиком (низкая селективность колонки, например булево поле или поле с малым числом уникальных значений), когда таблица маленькая (последовательное сканирование дешевле поиска по индексу), или когда таблица испытывает интенсивную нагрузку на запись — каждый лишний индекс замедляет INSERT/UPDATE/DELETE и увеличивает размер БД и время бэкапа/восстановления. Избыточные и дублирующиеся индексы также усложняют планировщику выбор оптимального плана и тратят память на кэширование.

| Термин | Определение |
|---|---|
| Низкая селективность | Малое число уникальных значений в колонке |
| Write-heavy таблица | Таблица с частыми вставками/обновлениями |
| Дублирующийся индекс | Индекс, покрываемый другим более широким индексом |

**Ответ для интервьюера:** Индекс вреден на колонках с низкой селективностью, на маленьких таблицах и на таблицах с интенсивной записью, где каждый индекс дополнительно замедляет INSERT/UPDATE/DELETE. Перед добавлением индекса всегда проверяю реальный паттерн запросов через EXPLAIN, а не добавляю «про запас».

### 22. Как выбрать колонки для составного индекса
**Вопрос:** Как определить, по каким колонкам нужно построить составной индекс?

**Ответ:** Составной индекс строят на основе реальных запросов из production (через pg_stat_statements или лог медленных запросов), выбирая колонки, которые чаще всего вместе встречаются в WHERE/JOIN/ORDER BY. Порядок колонок критичен: сначала идут колонки с равенством (=) и более высокой селективностью, затем — колонки диапазонных условий (>, <, BETWEEN) или ORDER BY, потому что B-Tree индекс эффективен как префикс — индекс (a, b) ускорит запросы по a и по (a, b), но не по одному b.

```sql
-- Запрос: WHERE customer_id = ? AND status = 'paid' ORDER BY created_at
CREATE INDEX idx_orders_customer_status_created
ON orders(customer_id, status, created_at);
```

| Термин | Определение |
|---|---|
| Составной индекс | Индекс по нескольким колонкам сразу |
| Правило левого префикса | Индекс (a,b,c) полезен для запросов по a, (a,b) и (a,b,c) |
| pg_stat_statements | Расширение PostgreSQL для анализа реальных запросов |

**Ответ для интервьюера:** Колонки выбираю по реальным запросам (через pg_stat_statements), ставя вперёд колонки равенства с высокой селективностью, а диапазонные условия и ORDER BY — в конец, учитывая правило левого префикса B-Tree индекса.

### 23. EXPLAIN
**Вопрос:** Объясни, что такое EXPLAIN и как он используется для оптимизации запросов.

**Ответ:** EXPLAIN показывает план выполнения запроса — какие методы доступа к данным (Seq Scan, Index Scan, Bitmap Scan), способы соединения (Nested Loop, Hash Join, Merge Join) и порядок операций выбрал планировщик, а также оценочную (estimated) стоимость и число строк, без фактического выполнения запроса. Используется для быстрой проверки плана перед запуском тяжёлого запроса и для сравнения альтернативных формулировок запроса или индексов.

```sql
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;
```

| Термин | Определение |
|---|---|
| План выполнения | Последовательность операций для выполнения запроса |
| Nested Loop / Hash Join | Алгоритмы соединения таблиц |
| Estimated cost | Оценочная стоимость операции планировщиком |

**Ответ для интервьюера:** EXPLAIN показывает план выполнения запроса — какой метод доступа и соединения выберет планировщик и его оценочную стоимость, без реального выполнения. Использую его, чтобы заранее проверить план на тяжёлых запросах или сравнить эффект от добавления индекса.

### 24. EXPLAIN ANALYZE — интерпретация
**Вопрос:** Как интерпретировать результаты EXPLAIN ANALYZE?

**Ответ:** В отличие от EXPLAIN, EXPLAIN ANALYZE реально выполняет запрос и добавляет фактические (actual) время и число строк рядом с оценочными (estimated), что позволяет увидеть расхождения планировщика с реальностью — например, если estimated rows сильно отличается от actual rows, статистика таблицы устарела и нужен ANALYZE. Смотрю на самые «дорогие» узлы плана (обычно листья дерева с наибольшим actual time), ищу Seq Scan на больших таблицах, дорогие сортировки (Sort с большим объёмом) и Nested Loop с большим числом итераций — это типичные кандидаты на добавление индекса или переписывание запроса.

```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE customer_id = 42;
```

| Термин         | Определение                                        |
| -------------- | -------------------------------------------------- |
| actual time    | Реальное время выполнения узла плана               |
| estimated rows | Оценка планировщика по числу строк                 |
| BUFFERS        | Показывает число прочитанных страниц из кэша/диска |

**Ответ для интервьюера:** В EXPLAIN ANALYZE смотрю на actual time каждой операции — ищу самые медленные. Seq Scan на больших таблицах — сигнал к созданию индекса. Сравниваю estimated rows с actual rows: если сильно отличаются, нужна ANALYZE для обновления статистики. Также смотрю на Buffers: много read означает чтение с диска, а не из кэша.

### 25. Bulk Insert vs построчная вставка
**Вопрос:** Как реализовать быструю загрузку большого объема данных (например, через Bulk Insert) и чем это отличается от построчной вставки?

**Ответ:** Построчная вставка (один INSERT на строку в цикле) медленна, потому что каждая операция — это отдельный round-trip к БД, отдельная транзакция (если не обёрнута явно) и отдельное обновление индексов. Bulk Insert объединяет много строк в один запрос или использует специальный протокол массовой загрузки (COPY в PostgreSQL), резко сокращая число round-trip'ов и накладных расходов на транзакции, что ускоряет загрузку на порядки на больших объёмах.

```python
# Медленно: N round-trip'ов
for row in rows:
    cursor.execute("INSERT INTO orders (customer_id, total) VALUES (%s, %s)", row)

# Быстро: один запрос
cursor.executemany("INSERT INTO orders (customer_id, total) VALUES (%s, %s)", rows)

# Ещё быстрее в PostgreSQL: COPY
cursor.copy_expert("COPY orders (customer_id, total) FROM STDIN WITH CSV", file)
```

| Термин | Определение |
|---|---|
| Round-trip | Один цикл запрос-ответ между приложением и БД |
| Bulk Insert | Вставка множества строк одним запросом/операцией |
| COPY | Специальный протокол PostgreSQL для массовой загрузки данных |

**Ответ для интервьюера:** Построчный INSERT в цикле создаёт отдельный round-trip и транзакционные накладные расходы на каждую строку. Bulk Insert (executemany, multi-row INSERT или COPY в PostgreSQL) объединяет вставку в один или несколько крупных запросов, ускоряя загрузку на порядки.

### 26. Обработка данных чанками
**Вопрос:** Как реализовать обработку данных «чанками» (порциями) при работе с большими файлами и БД?

**Ответ:** Обработка чанками — это чтение и обработка данных небольшими порциями (например, по 1000–10000 строк) вместо загрузки всего набора в память сразу, что предотвращает переполнение памяти и позволяет делать промежуточные commit'ы, снижая размер транзакции и время удержания блокировок. Для файлов используют потоковое чтение (генераторы, итераторы), для БД — курсоры (server-side cursor) или пагинацию по ключу (keyset pagination) вместо OFFSET на больших объёмах.

```python
def process_in_chunks(query, chunk_size=1000):
    offset = 0
    while True:
        rows = fetch(query, limit=chunk_size, offset=offset)
        if not rows:
            break
        process(rows)
        offset += chunk_size
```

| Термин | Определение |
|---|---|
| Chunking | Обработка данных порциями фиксированного размера |
| Server-side cursor | Курсор БД, не загружающий весь результат в память клиента |
| Keyset pagination | Пагинация по значению ключа вместо OFFSET |

**Ответ для интервьюера:** Данные читаю и обрабатываю порциями фиксированного размера, используя генераторы или серверные курсоры, чтобы не держать весь набор в памяти и не открывать одну гигантскую транзакцию. Для больших таблиц предпочитаю keyset pagination вместо OFFSET, так как OFFSET деградирует по производительности на больших смещениях.

## ORM (SQLAlchemy, Django ORM) и миграции

### 27. ORM: плюсы и минусы против Raw SQL
**Вопрос:** Объясни, что такое ORM и каковы плюсы и минусы его использования по сравнению с Raw SQL.

**Ответ:** ORM (Object-Relational Mapping) связывает таблицы БД с классами Python, а строки — с объектами, позволяя работать с данными через код на Python вместо написания SQL вручную. Плюсы: меньше шаблонного кода, защита от SQL injection «из коробки», переносимость между разными СУБД, встроенные миграции и валидация на уровне моделей. Минусы: дополнительный слой абстракции может генерировать неэффективный SQL (классическая проблема N+1), сложные аналитические запросы удобнее и быстрее писать на чистом SQL, и есть кривая обучения самого ORM поверх знания SQL.

|Характеристика|ORM|Raw SQL|
|---|---|---|
|**Скорость разработки**|Выше, меньше шаблонного кода|Ниже, всё вручную|
|**Защита от SQL injection**|✅ Из коробки|Нужно делать вручную (параметризация)|
|**Переносимость между СУБД**|✅|❌ SQL может быть специфичен для СУБД|
|**Эффективность генерируемого SQL**|Может быть неоптимальной (N+1)|Полный контроль|
|**Сложная аналитика**|Неудобно|✅ Естественнее|

```python
# ORM
customer = session.query(Customer).filter_by(id=1).first()

# Raw SQL
customer = connection.execute(text("SELECT * FROM customers WHERE id = 1")).first()
```

| Термин | Определение |
|---|---|
| ORM | Object-Relational Mapping — связка классов Python и таблиц БД |
| Raw SQL | Прямое написание SQL-запросов без ORM |
| N+1 | Типичная проблема неэффективных запросов через ORM |

**Ответ для интервьюера:** ORM мапит таблицы на классы Python, ускоряя разработку и защищая от SQL injection по умолчанию, но добавляет слой абстракции, который может сгенерировать неоптимальный SQL (N+1). Для простого CRUD использую ORM, для сложной аналитики — Raw SQL или гибрид (SQLAlchemy Core).

### 28. Связь БД и классов Python (модели)
**Вопрос:** Как связать работу с БД и классы Python (на примере моделей)?

**Ответ:** Связь реализуется через декларативные модели ORM: класс Python наследуется от базового класса ORM (Base в SQLAlchemy, models.Model в Django), а его атрибуты описывают колонки таблицы через специальные поля с типами. ORM сама генерирует SQL DDL для создания таблицы и транслирует операции с объектами (создание, изменение, удаление) в соответствующие SQL-запросы через Session (SQLAlchemy) или Manager (Django).

```python
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

class Base(DeclarativeBase):
    pass

class Customer(Base):
    __tablename__ = "customers"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str]
```

| Термин | Определение |
|---|---|
| Декларативная модель | Класс Python, описывающий структуру таблицы |
| Mapped_column | Способ описания колонки в SQLAlchemy 2.0 |
| Session | Объект для управления транзакциями и запросами ORM |

**Ответ для интервьюера:** Каждый класс-модель наследуется от базового класса ORM, а его атрибуты описывают колонки таблицы с типами и ограничениями. ORM транслирует операции с объектами этого класса в SQL через Session, включая генерацию DDL для создания самой таблицы.

### 29. SQLAlchemy 1.x vs 2.0
**Вопрос:** В чем разница между SQLAlchemy 1.x и 2.0 (использование column vs mapped_column)?

**Ответ:** В SQLAlchemy 1.x колонки объявлялись через `Column(Integer, primary_key=True)` без строгой типизации на уровне Python — тип определялся только через SQLAlchemy-тип. В SQLAlchemy 2.0 введён `mapped_column()` в связке с аннотацией типа `Mapped[int]`, что даёт полноценную поддержку статической типизации (mypy/pyright может проверить типы атрибутов модели) и автоматически выводит nullable из Optional. 2.0 также унифицировал синтаксис запросов вокруг `select()` вместо устаревшего `Query API`, сделав Core и ORM более согласованными, и добавил нативную поддержку async/await через `AsyncSession`.

|Характеристика|SQLAlchemy 1.x|SQLAlchemy 2.0|
|---|---|---|
|**Объявление колонки**|`Column(Integer, ...)`|`mapped_column()` + `Mapped[T]`|
|**Статическая типизация**|❌|✅ mypy/pyright проверяют модели|
|**API запросов**|Query API|Унифицирован вокруг `select()`|
|**Async поддержка**|Ограниченная|✅ Нативная через AsyncSession|

```python
# SQLAlchemy 1.x
class Customer(Base):
    id = Column(Integer, primary_key=True)
    name = Column(String, nullable=False)

# SQLAlchemy 2.0
class Customer(Base):
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str]
```

| Термин | Определение |
|---|---|
| Column | Старый способ объявления колонки (1.x) |
| mapped_column + Mapped[] | Новый типобезопасный способ (2.0) |
| AsyncSession | Асинхронная сессия SQLAlchemy 2.0 |

**Ответ для интервьюера:** SQLAlchemy 2.0 заменил `Column` на `mapped_column()` в паре с аннотацией `Mapped[T]`, что даёт статическую типизацию моделей и автоматический вывод nullable. Также 2.0 унифицировал API запросов вокруг `select()` и добавил нативную поддержку async через AsyncSession.

### 30. N+1 проблема
**Вопрос:** Что такое N+1 проблема и как её решить?

**Ответ:** N+1 — это ситуация, когда ORM выполняет один запрос для получения N родительских объектов, а затем ещё по одному дополнительному запросу для получения связанных данных каждого объекта (итого N+1 запрос вместо одного-двух), что резко деградирует производительность при увеличении N. Возникает при обращении к связанному (relationship) атрибуту в цикле без явной предзагрузки. Решается через eager loading — явную предзагрузку связанных данных одним дополнительным запросом (selectinload) или через JOIN (joinedload) вместо ленивой подгрузки по одному объекту за раз.

```python
# N+1: для каждого customer отдельный запрос на orders
customers = session.query(Customer).all()
for c in customers:
    print(c.orders)  # отдельный SELECT на каждой итерации

# Решение: одна предзагрузка
from sqlalchemy.orm import selectinload
customers = session.query(Customer).options(selectinload(Customer.orders)).all()
```

| Термин | Определение |
|---|---|
| N+1 | 1 запрос на родителей + N запросов на их связи |
| selectinload | Предзагрузка через отдельный IN-запрос |
| joinedload | Предзагрузка через LEFT JOIN в одном запросе |

**Ответ для интервьюера:** N+1 — когда обращение к связанному атрибуту в цикле генерирует по отдельному запросу на каждый объект вместо одной пакетной загрузки. Решаю через eager loading — `selectinload` для one-to-many или `joinedload` для one-to-one/many-to-one, чтобы получить всё за один-два запроса.

### 31. Lazy vs Eager loading
**Вопрос:** Объясни разницу между Lazy и Eager loading в SQLAlchemy (joinedload, selectinload).

**Ответ:** Lazy loading (стратегия по умолчанию) откладывает загрузку связанных данных до первого обращения к атрибуту — удобно, если связанные данные нужны не всегда, но именно это порождает N+1 при обращении в цикле. Eager loading загружает связанные данные заранее, вместе с основным запросом или сразу после него: `joinedload` делает это через LEFT JOIN в одном SQL-запросе (хорош для to-one связей, но дублирует данные родителя при to-many), а `selectinload` делает отдельный второй запрос вида `WHERE parent_id IN (...)` (эффективнее для to-many связей, так как не дублирует строки).

|Стратегия|Когда грузит данные|Число запросов|Лучше для|
|---|---|---|---|
|**Lazy loading**|При первом обращении к атрибуту|1 + N (риск N+1)|Когда связь нужна не всегда|
|**joinedload**|Сразу, через LEFT JOIN|1|To-one связей|
|**selectinload**|Сразу, отдельным IN-запросом|2|To-many связей (без дублирования строк)|

```python
from sqlalchemy.orm import joinedload, selectinload

# to-one связь — один JOIN
session.query(Order).options(joinedload(Order.customer)).all()

# to-many связь — отдельный IN-запрос
session.query(Customer).options(selectinload(Customer.orders)).all()
```

| Термин | Определение |
|---|---|
| Lazy loading | Загрузка связи по требованию, при первом обращении |
| joinedload | Eager loading через LEFT JOIN в одном запросе |
| selectinload | Eager loading через второй запрос с IN |

**Ответ для интервьюера:** Lazy loading грузит связь только при обращении к атрибуту, что удобно, но провоцирует N+1 в циклах. Eager loading грузит связь заранее: joinedload — одним JOIN-запросом (хорош для to-one), selectinload — отдельным IN-запросом (эффективнее для to-many, не дублирует строки).

### 32. Закрытие соединений с БД / контекстные менеджеры
**Вопрос:** Как правильно закрывать соединения с БД в Python и зачем нужны контекстные менеджеры?

**Ответ:** Соединения и сессии с БД — ограниченный ресурс (пул соединений), поэтому их обязательно нужно закрывать после использования, иначе пул исчерпается и приложение начнёт зависать в ожидании свободного соединения. Контекстный менеджер (`with`) гарантирует закрытие ресурса даже при исключении внутри блока, потому что вызывает `__exit__` в любом случае — в отличие от ручного `open()/close()`, где исключение до `close()` оставит соединение открытым.

```python
# Плохо: при исключении соединение не закроется
conn = engine.connect()
result = conn.execute(query)
conn.close()

# Хорошо: соединение закроется даже при ошибке
with engine.connect() as conn:
    result = conn.execute(query)
```

| Термин | Определение |
|---|---|
| Пул соединений | Ограниченный набор переиспользуемых соединений с БД |
| Контекстный менеджер | Конструкция `with`, гарантирующая освобождение ресурса |
| __exit__ | Метод, вызываемый при выходе из блока `with`, даже при исключении |

**Ответ для интервьюера:** Соединения беру из пула через `with`, чтобы гарантировать освобождение ресурса даже при исключении внутри блока — `__exit__` вызовется в любом случае, в отличие от ручного `close()`, который легко пропустить при ошибке. Без этого пул соединений быстро исчерпывается под нагрузкой.

### 33. Миграции БД и Alembic
**Вопрос:** Что такое миграции БД и зачем нужен инструмент Alembic?

**Ответ:** Миграция — это версионированный скрипт, описывающий изменение схемы БД (создание таблицы, добавление колонки, индекса), который можно применить (`upgrade`) или откатить (`downgrade`), обеспечивая воспроизводимость структуры БД между окружениями (dev, staging, prod) и историю изменений схемы наравне с историей кода в Git. Alembic — стандартный инструмент миграций для SQLAlchemy: он умеет автогенерировать миграцию, сравнивая текущие модели с фактической схемой БД, хранит применённые версии в служебной таблице и применяет недостающие миграции по порядку.

```bash
alembic revision --autogenerate -m "add status to orders"
alembic upgrade head
alembic downgrade -1
```

| Термин              | Определение                                |
| ------------------- | ------------------------------------------ |
| Миграция            | Версионированный скрипт изменения схемы БД |
| upgrade / downgrade | Применение / откат миграции                |
| Alembic             | Инструмент миграций для SQLAlchemy         |


**Ответ для интервьюера:** Миграция — это версионированное изменение схемы БД с возможностью применить или откатить, что синхронизирует структуру БД между окружениями и хранит историю изменений рядом с кодом. Alembic автогенерирует такие миграции на основе разницы между моделями SQLAlchemy и текущей схемой и последовательно применяет их через служебную таблицу версий.

### 34. Engine в SQLAlchemy и асинхронные драйверы
**Вопрос:** Зачем нужен движок (Engine) в SQLAlchemy и какие асинхронные драйверы (например, asyncpg) вы знаете?

**Ответ:** Engine — это точка входа SQLAlchemy для взаимодействия с БД: он хранит строку подключения, управляет пулом соединений (connection pool) и диалектом конкретной СУБД, но сам по себе не открывает соединение, пока оно реально не понадобится (ленивая инициализация). Для асинхронной работы используется `create_async_engine` вместе с асинхронным драйвером конкретной СУБД: для PostgreSQL — `asyncpg` (самый быстрый) или `psycopg` (3-я версия, тоже поддерживает async), для MySQL — `aiomysql` или `asyncmy`, для SQLite — `aiosqlite`.

```python
from sqlalchemy.ext.asyncio import create_async_engine

engine = create_async_engine(
    "postgresql+asyncpg://user:pass@localhost/db",
    pool_size=10,
)
```

| Термин | Определение |
|---|---|
| Engine | Точка входа SQLAlchemy, управляющая пулом соединений |
| Connection pool | Набор переиспользуемых соединений с БД |
| asyncpg | Быстрый асинхронный драйвер PostgreSQL для Python |

**Ответ для интервьюера:** Engine управляет пулом соединений и диалектом СУБД, инициализируя реальное соединение лениво, при первом обращении. Для асинхронной работы использую `create_async_engine` с драйвером `asyncpg` для PostgreSQL — он самый быстрый среди доступных async-драйверов.

### 35. Пагинация (limit/offset)
**Вопрос:** Как реализовать пагинацию на бэкенде (параметры limit и offset/skip)?

**Ответ:** Классическая offset-пагинация задаёт `LIMIT` (сколько строк вернуть) и `OFFSET` (сколько строк пропустить), что просто в реализации, но деградирует по производительности на больших смещениях, так как СУБД всё равно проходит и отбрасывает все пропущенные строки. Альтернатива для больших наборов данных — keyset (cursor-based) пагинация: вместо OFFSET передаётся значение ключа последней увиденной записи, и следующая страница выбирается условием `WHERE id > last_id ORDER BY id LIMIT N`, что работает за постоянное время независимо от глубины страницы.

|Характеристика|Offset-пагинация|Keyset-пагинация|
|---|---|---|
|**Параметры**|LIMIT + OFFSET|LIMIT + значение ключа|
|**Производительность на глубоких страницах**|❌ Деградирует|✅ Постоянное время|
|**Переход на произвольную страницу**|✅ Просто|❌ Только последовательно|
|**Простота реализации**|✅ Проще|Сложнее|

```sql
-- Offset-пагинация
SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET 40;

-- Keyset-пагинация (быстрее на больших смещениях)
SELECT * FROM orders WHERE id > 1040 ORDER BY id LIMIT 20;
```

| Термин | Определение |
|---|---|
| LIMIT / OFFSET | Ограничение и сдвиг выборки |
| Keyset pagination | Пагинация по значению ключа вместо смещения |
| Cursor-based pagination | Синоним keyset-пагинации, курсор = значение ключа |

**Ответ для интервьюера:** Для небольших наборов данных использую простую `LIMIT/OFFSET` пагинацию, а для больших — keyset-пагинацию по значению ключа (`WHERE id > last_id`), потому что OFFSET на больших страницах вынуждает БД читать и отбрасывать все предыдущие строки, теряя производительность.

## NoSQL и масштабирование

### 36. Redis и кэширование
**Вопрос:** Объясни, что такое Redis и как он используется для кэширования.

**Ответ:** Redis — это in-memory хранилище данных типа ключ-значение, поддерживающее не только строки, но и структуры вроде списков, множеств, хэшей и sorted set, с опциональной персистентностью на диск (RDB-снапшоты, AOF-журнал). Для кэширования типичный паттерн — cache-aside: приложение сначала проверяет Redis по ключу, при промахе (cache miss) читает из основной БД и кладёт результат в Redis с TTL (временем жизни), чтобы данные автоматически устаревали и не расходились с источником истины надолго.

```python
value = redis_client.get(f"user:{user_id}")
if value is None:
    value = db.get_user(user_id)
    redis_client.set(f"user:{user_id}", value, ex=300)  # TTL 5 минут
```

| Термин | Определение |
|---|---|
| Cache-aside | Паттерн: приложение само проверяет кэш, затем БД |
| TTL | Time To Live — время жизни ключа в кэше |
| Cache miss / hit | Промах / попадание при поиске в кэше |

**Ответ для интервьюера:** Redis — in-memory key-value хранилище с поддержкой сложных структур данных и опциональной персистентностью на диск. Для кэша чаще всего используют паттерн cache-aside: читаем из Redis, при промахе идём в БД и кладём результат обратно с TTL, чтобы данные не устаревали бесконтрольно.

### 37. MongoDB
**Вопрос:** Объясни, что такое MongoDB и в каких случаях её использование оправдано.

**Ответ:** MongoDB — документоориентированная NoSQL БД, хранящая данные в виде BSON-документов (бинарный JSON) внутри коллекций без жёсткой единой схемы — разные документы одной коллекции могут иметь разный набор полей. Она оправдана, когда структура данных часто меняется или изначально неоднородна (каталоги товаров с разными атрибутами, логи событий, профили пользователей с произвольными полями), а также когда нужна высокая скорость записи и горизонтальное шардирование «из коробки». Менее подходит для данных с большим числом связей и требованием строгих ACID-транзакций между разными сущностями, где реляционная модель естественнее.

```javascript
db.products.insertOne({
    name: "Ноутбук",
    specs: { cpu: "i7", ram: 16 }  // произвольная вложенная структура
});
```

| Термин | Определение |
|---|---|
| Документ (BSON) | Единица хранения — бинарный JSON |
| Коллекция | Аналог таблицы, набор документов без жёсткой схемы |
| Schemaless | Отсутствие обязательной единой схемы для всех документов |

**Ответ для интервьюера:** MongoDB хранит данные как BSON-документы в коллекциях без жёсткой единой схемы, что удобно для неоднородных или часто меняющихся данных — каталогов, логов, произвольных профилей. Менее подходит для данных с большим числом связей и требованием строгих ACID-транзакций между разными сущностями, где реляционная модель естественнее.

### 38. ClickHouse и колоночные БД
**Вопрос:** Что такое ClickHouse, в чем особенность колоночных баз данных?

**Ответ:** ClickHouse — колоночная (column-oriented) аналитическая СУБД, оптимизированная под OLAP-нагрузку: быстрые агрегаты по огромным объёмам данных (SUM, AVG, COUNT по миллиардам строк). В отличие от строчного хранения (row-oriented, как в PostgreSQL для OLTP), где данные одной строки лежат рядом на диске, колоночное хранение кладёт рядом все значения одной колонки — это позволяет читать с диска только нужные колонки (а не всю строку) и эффективно сжимать однотипные данные, что критично для аналитических запросов, но неэффективно для точечных вставок/обновлений отдельных строк.

|Характеристика|Строчное хранение (OLTP, PostgreSQL)|Колоночное хранение (OLAP, ClickHouse)|
|---|---|---|
|**Данные на диске**|Строка целиком лежит рядом|Значения одной колонки лежат рядом|
|**Сильная сторона**|Точечные вставки/обновления строк|Агрегаты по огромным объёмам данных|
|**Сжатие**|Хуже (разнородные данные строки)|Лучше (однотипные данные колонки)|
|**Типовая нагрузка**|Транзакции (OLTP)|Аналитика (OLAP)|

| Термин | Определение |
|---|---|
| OLAP | Online Analytical Processing — аналитическая нагрузка |
| OLTP | Online Transaction Processing — транзакционная нагрузка |
| Колоночное хранение | Данные одной колонки хранятся смежно на диске |

**Ответ для интервьюера:** ClickHouse — колоночная СУБД для OLAP: данные одной колонки лежат рядом на диске, что ускоряет агрегатные запросы по огромным объёмам и хорошо сжимается, но плохо подходит для точечных транзакционных обновлений — там нужна строчная OLTP-СУБД вроде PostgreSQL.

### 39. Репликация vs шардирование
**Вопрос:** В чем разница между репликацией и шардированием? Когда использовать каждую технологию?

**Ответ:** Репликация — это копирование одних и тех же данных на несколько серверов (master-replica), что повышает отказоустойчивость (при падении мастера можно переключиться на реплику) и позволяет распределить нагрузку на чтение между репликами, но не увеличивает объём данных, которые можно хранить, и не ускоряет запись, так как она всё ещё идёт через один мастер. Шардирование — это разделение данных на непересекающиеся части (шарды) по ключу шардирования, которые хранятся на разных серверах, что позволяет масштабировать и объём хранения, и запись, распределяя нагрузку горизонтально, но усложняет запросы, требующие данных из нескольких шардов (JOIN, агрегаты между шардами).

|Характеристика|Репликация|Шардирование|
|---|---|---|
|**Данные на серверах**|Одинаковые копии|Непересекающиеся части|
|**Увеличивает объём хранения**|❌|✅|
|**Ускоряет запись**|❌ (идёт через мастер)|✅ Распределяется по шардам|
|**Ускоряет чтение**|✅ (между репликами)|✅ (параллельно по шардам)|
|**Отказоустойчивость**|✅ Переключение на реплику|Нужна отдельная репликация каждого шарда|
|**Сложность кросс-запросов**|Не усложняет|Усложняет (JOIN между шардами)|

| Термин            | Определение                                                       |
| ----------------- | ----------------------------------------------------------------- |
| Репликация        | Копирование одних и тех же данных на несколько серверов           |
| Шардирование      | Разделение данных на части по ключу, хранимые отдельно            |
| Master-replica    | Схема с одним основным сервером для записи и репликами для чтения |
| Ключ шардирования | Поле, по которому данные распределяются между шардами             |


**Ответ для интервьюера:** Репликация копирует одни и те же данные на несколько серверов ради отказоустойчивости и масштабирования чтения, но не увеличивает объём хранения и не ускоряет запись. Шардирование разбивает данные на части по ключу и хранит их на разных серверах, масштабируя и хранение, и запись, но усложняет кросс-шардовые запросы — часто их комбинируют вместе.
