#flashcards/СУБД
что такое Транзакции ?  :: это логическая единица работы с базой данных, которая должна быть выполнена **целиком и независимо** от других операций. Гарантирует целостность данных даже при сбоях.<br><br>**Ключевые свойства (ACID):**<br>1. **Aтомарность (Atomicity)** — "всё или ничего": либо выполняются **все** изменения транзакции, либо **ни одного** (откат).<br>2. **Согласованность (Consistency)** — транзакция переводит БД из одного **валидного** состояния в другое (соблюдаются все ограничения: уникальность, внешние ключи).<br>3. **Изолированность (Isolation)** — параллельные транзакции не мешают друг другу (уровни изоляции: Read Committed, Repeatable Read и др.).<br>4. **Долговечность (Durability)** — после фиксации (`COMMIT`) изменения сохраняются **навсегда**, даже при сбое оборудования.<br><br>**На практике:** Транзакция начинается с `BEGIN`, завершается `COMMIT` (при успехе) или `ROLLBACK` (при ошибке/отмене).<br><br>**Важно**<br>- ✅ Знать что такое ACID - Атомарность, Согласованность, Изолированность, Долговечность

**Подтермины**
- "Атомарность" - все операции выполнятся или ни одна операция не выполнится
- "Согласованность" - транзакция переводит БД из одного **валидного** состояния в другое (соблюдаются все ограничения: уникальность, внешние ключи)
- "Изолированность" - параллельные транзакции не мешают друг другу (уровни изоляции: Read Committed, Repeatable Read и др.).
- "Долговечность" - после фиксации (`COMMIT`) изменения сохраняются **навсегда**, даже при сбое оборудования.

```python
from sqlalchemy.orm import Session
from fastapi import Depends

# В endpoint FastAPI
@app.post("/transfer")
def transfer_money(
    from_id: int,
    to_id: int,
    amount: float,
    db: Session = Depends(get_db)  # Сессия БД
):
    try:
        # 1. Начало транзакции (неявно, при первом запросе)
        from_acc = db.query(Account).filter(Account.id == from_id).one()
        to_acc = db.query(Account).filter(Account.id == to_id).one()
        
        # 2. Бизнес-логика
        if from_acc.balance < amount:
            raise HTTPException(status_code=400, detail="Недостаточно средств")
        
        from_acc.balance -= amount
        to_acc.balance += amount
        
        # 3. КОММИТ (при успехе)
        db.commit()  # Фиксируем ВСЕ изменения выше
        return {"message": "Перевод успешен"}
    
    except Exception as e:
        # 4. РОЛЛБЭК (при ЛЮБОЙ ошибке)
        db.rollback()  # Откатываем ВСЕ изменения
        raise HTTPException(status_code=500, detail="Ошибка перевода")
    
    finally:
        db.close()
```

#flashcards/СУБД
что такое ACID ?  :: это набор гарантий, который позволяет разработчику полагаться на то, что транзакция либо полностью завершится, оставив данные в целостном и сохранённом состоянии, либо не оставит никакого следа. Расшифровка следующая: <br>**Aтомарность** - неделимая единица работы. При ошибке в любом её месте БД откатывает **все** изменения, сделанные этой транзакцией, как будто её не было.<br>**Согласованность** - Каждая транзакция переводит БД из одного состояния, удовлетворяющего **всем бизнес-правилам** (ограничениям целостности: PRIMARY KEY, FOREIGN KEY, UNIQUE, CHECK), в другое такое же состояние.<br>**Изолированность** -Параллельные транзакции не должны влиять на результаты друг друга. Уровни изоляции (Read Committed, Repeatable Read, Serializable) определяют степень этой защиты.<br>**Долговечность** - После успешного `COMMIT` изменения записываются в постоянное хранилище (жёсткий диск, журнал транзакций WAL) и не могут быть потеряны даже при сбое питания.

### Уровни изоляции и аномалии

**ПОЛНЫЙ РАЗВЕРНУТЫЙ ОТВЕТ на вопрос: «Какие бывают уровни изоляции транзакций?»** 🚀

---

### **1 READ UNCOMMITTED (Чтение незафиксированных данных)**
- **Допускаемые аномалии:** Грязное чтение, Неповторяемое чтение, Фантомное чтение
- **Описание:** Транзакция видит незакоммиченные изменения других транзакций
- **Пример использования:** Редко используется, только для аналитических отчётов, где точность не критична
- **Синтаксис (PostgreSQL):**
  ```sql
  BEGIN TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
  ```

### **2 READ COMMITTED (Чтение зафиксированных данных) ⭐**
- **Допускаемые аномалии:** Неповторяемое чтение, Фантомное чтение
- **Запрещает:** Грязное чтение
- **По умолчанию в:** PostgreSQL, Oracle, SQL Server
- **Как работает:** Каждый `SELECT` видит только закоммиченные данные на момент начала выполнения этого конкретного запроса
- **Пример проблемы (неповторяемое чтение):**
  ```sql
  -- Транзакция 1
  BEGIN;
  SELECT balance FROM accounts WHERE id = 1; -- Вернуло 1000
  
  -- Транзакция 2 (параллельно)
  BEGIN;
  UPDATE accounts SET balance = 900 WHERE id = 1;
  COMMIT;
  
  SELECT balance FROM accounts WHERE id = 1; -- Вернуло 900 (значение изменилось!)
  COMMIT;
  ```

### **3 REPEATABLE READ (Повторяемое чтение)**
- **Допускаемые аномалии:** Фантомное чтение
- **Запрещает:** Грязное чтение, Неповторяемое чтение
- **По умолчанию в:** MySQL InnoDB
- **Как работает:** Транзакция видит снапшот данных на момент своего начала
- **Пример проблемы (фантомное чтение):**
  ```sql
  -- Транзакция 1
  BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
  SELECT COUNT(*) FROM users WHERE age > 30; -- Вернуло 10
  
  -- Транзакция 2 (параллельно)
  INSERT INTO users (name, age) VALUES ('John', 35);
  COMMIT;
  
  SELECT COUNT(*) FROM users WHERE age > 30; -- Всё равно вернёт 10 (снапшот)
  -- Но если сделать SELECT ... FOR UPDATE, может увидеть новые строки
  ```

### **4 SERIALIZABLE (Сериализуемый)**
- **Допускаемые аномалии:** Нет
- **Запрещает:** Все аномалии (грязное, неповторяемое, фантомное чтение)
- **Как работает:** Полная изоляция, как если бы транзакции выполнялись последовательно
- **Цена:** Высокие накладные расходы, возможные дедлоки
- **Пример (PostgreSQL):**
  ```sql
  BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
  -- Если две параллельные транзакции конфликтуют,
  -- одна получит ошибку: "could not serialize access due to concurrent update"
  ```

## **Аномалии, которые предотвращают уровни изоляции**

### **1 Грязное чтение (Dirty Read)**
- **Описание:** Чтение незакоммиченных данных другой транзакции
- **Пример:** Транзакция A видит изменения транзакции B, которая потом откатилась
- **Защита:** READ COMMITTED и выше

### **2 Неповторяемое чтение (Non-repeatable Read)**
- **Описание:** Одна транзакция дважды читает одни данные, но получает разные значения (потому что другая транзакция изменила и закоммитила эти данные между чтениями)
- **Защита:** REPEATABLE READ и выше

### **3 Фантомное чтение (Phantom Read)**
- **Описание:** Появление новых строк между повторными чтениями
- **Пример:** `SELECT ... WHERE age > 30` возвращает разное количество строк
- **Защита:** SERIALIZABLE

## **3. Реализации в конкретных СУБД**

### **3.1 PostgreSQL**
- READ UNCOMMITTED → фактически работает как READ COMMITTED
- READ COMMITTED → по умолчанию
- REPEATABLE READ → использует снапшоты (MVCC)
- SERIALIZABLE → строгая сериализация

### **3.2 MySQL (InnoDB)**
- READ UNCOMMITTED → реальная реализация
- READ COMMITTED → доступен (не по умолчанию)
- REPEATABLE READ → по умолчанию
- SERIALIZABLE → блокировочная реализация

### **3.3 Установка уровня в коде (Python + SQLAlchemy)**
```python
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

engine = create_engine('postgresql://user:pass@localhost/db')
# Установка уровня изоляции для всей сессии
engine.execution_options(isolation_level='REPEATABLE_READ')

# Или для конкретной транзакции
with engine.begin() as conn:
    conn.execute("SET TRANSACTION ISOLATION LEVEL SERIALIZABLE")
    # ... операции
```

## **4. Выбор уровня изоляции на практике**

| Уровень | Производительность | Консистентность | Использование |
|---------|-------------------|-----------------|---------------|
| READ UNCOMMITTED | 🟢 Высокая | 🔴 Низкая | Аналитика, отчёты |
| READ COMMITTED | 🟡 Средняя | 🟡 Средняя | Веб-приложения (по умолчанию) |
| REPEATABLE READ | 🟡 Средняя | 🟢 Высокая | Финансовые операции |
| SERIALIZABLE | 🔴 Низкая | 🟢 Максимальная | Банковские переводы |
