#flashcards/СУБД 
что такое EXPLAIN?  :: это **инструкция SQL**, которая возвращает **план выполнения** запроса, составленный оптимизатором СУБД. Ключевая цель — **профилирование и оптимизация** запросов. `EXPLAIN` не выполняет запрос, а лишь анализирует его и моделирует план. Для реального выполнения с замерами времени используется `EXPLAIN ANALYZE`.<br>
<!--SR:!2026-04-19,12,190-->

**Зачем нужно Middle/Senior разработчику:**
- ✅ Определить **«бутылочные горлышки»** (bottlenecks) в запросах.
- ✅ Проверить, **используются ли индексы**, и какие именно.
- ✅ Оценить **стоимость (cost)** операций и выбрать оптимальный способ денормализации или переписывания запроса.
- ✅ Обосновать необходимость **добавления/изменения индексов** или изменения структуры данных.

**Точно НЕ**
- ❌ не выполняет запрос, а лишь анализирует его и моделирует план


```python
# Рассмотрим на реальном примере с таблицей `users` (1 млн записей):

EXPLAIN (ANALYZE, BUFFERS) 
SELECT * FROM users 
WHERE email = 'user@example.com' AND status = 'active';
```

  Index Cond (email = 'user@example.com'text)
  Filter (status = 'active'text)
  Buffers: shared hit=4
Planning Time: 0.150 ms
Execution Time: 0.045 ms

**Разбор ключевых элементов:**

|Элемент|Что означает|Практический смысл|
|---|---|---|
|**Index Scan**|Тип доступа: сканирование по индексу|Хорошо! Используется индекс, а не полная таблица|
|**using idx_users_email**|Имя используемого индекса|Показывает, какой именно индекс выбран оптимизатором|
|**cost=0.42..8.44**|**Стоимость выполнения** (условные единицы). Первое число — стоимость получения первой строки, второе — всех строк|Важен **относительный** рост. 0.42→8.44 = ~20x. Если бы было 0.42→1000 — повод задуматься|
|**rows=1**|Ожидаемое количество строк после фильтрации|Сравнивается с `actual rows`. Если сильно расходится — у оптимизатора неактуальная статистика|
|**width=64**|Средний размер строки в байтах|Помогает оценить объём данных, которые нужно передать|
|**actual time=0.025..0.026**|**Фактическое время** в мс (только в ANALYZE)|Реальные метрики производительности|
|**Index Cond**|Условие, применённое **на этапе индексного сканирования**|Самый эффективный фильтр — обработан вместе с индексом|
|**Filter**|Условие, применённое **после извлечения строк**|Менее эффективно. Если фильтруется много строк — возможно, нужен составной индекс|
|**Buffers: shared hit=4**|Количество обращений к кэшу (hit) и диску (read)|`shared hit` — чтение из кэша (быстро), `shared read` — чтение с диска (медленно)|
|**Planning Time**|Время, затраченное на **построение плана**|Сложные запросы могут долго планироваться|
|**Execution Time**|Время **непосредственного выполнения**|Целевой метрика для оптимизации|

**3. Основные типы операций (узлов плана), которые нужно знать**

| Тип операции         | Сложность    | Когда встречается                                                                   |
| -------------------- | ------------ | ----------------------------------------------------------------------------------- |
| **Seq Scan**         | O(N)         | Полное сканирование таблицы. На больших таблицах — красный флаг                     |
| **Index Scan**       | O(log N)     | Сканирование по индексу с последующим обращением к таблице за другими полами        |
| **Index Only Scan**  | O(log N)     | Идеальный вариант. Все нужные поля есть в индексе, обращение к таблице не требуется |
| **Bitmap Heap Scan** | O(log N + K) | Компромисс между Seq Scan и Index Scan. Используется при множественных условиях OR  |
| **Nested Loop**      | O(N*M)       | Соединение вложенными циклами. Для маленьких таблиц или при наличии индексов        |
| **Hash Join**        | O(N+M)       | Соединение через хэш-таблицу. Часто эффективнее для больших таблиц без индексов     |
| **Merge Join**       | O(N+M)       | Соединение слиянием. Требует предварительной сортировки данных                      |
| **Sort**             | O(N log N)   | Сортировка. Дорогая операция, часто требует много памяти                            |
**4. Практические шаги по оптимизации на основе EXPLAIN (Middle+ уровень)**

**Шаг 1: Ищем «красные флаги»**

- **Seq Scan на большой таблице** → нужен индекс.
    
- **Высокий cost у Filter** (условие не в Index Cond) → возможно, нужен составной индекс.
    
- **Сильное расхождение rows и actual rows** → `ANALYZE table_name` для обновления статистики.
    
- **Ненужные Sort операции** → можно ли добавить индекс с правильным порядком колонок?
    

**Шаг 2: Анализируем эффективность индексов**

```sql
-- Пример неоптимального индекса
CREATE INDEX idx_users_status ON users(status); -- status имеет низкую селективность
-- EXPLAIN покажет: Index Scan, но rows=500000 (половина таблицы) — индекс практически бесполезен

-- Оптимальный составной индекс
CREATE INDEX idx_users_email_status ON users(email, status);
-- Теперь оба условия (email и status) попадут в Index Cond

**Шаг 3: Работа с соединениями (JOIN)**
```
