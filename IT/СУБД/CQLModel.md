1. **КРАТКИЙ ОТВЕТ (для интервьюера):**
   - «**SQLModel** — это библиотека от автора FastAPI (Sebastián Ramírez), объединяющая **SQLAlchemy** (ORM) и **Pydantic** (валидация).
   - Позволяет использовать один класс и для таблицы БД, и для схемы API, уменьшая дублирование кода (boilerplate).
   - Под капотом это всё та же SQLAlchemy, поэтому доступны все её возможности 🚀».

3. **ПОЛНЫЙ ОТВЕТ (для глубокого понимания):**

   **👵 Как для бабушки:**
   - **Обычный подход (SQLAlchemy + Pydantic):** Тебе нужно два документа. Один — паспорт (для базы данных), второй — пропуск (для сайта). Данные в них одинаковые, но оформлять приходится дважды 📄.
   - **SQLModel:** Это универсальная карта. Один документ работает и как паспорт, и как пропуск. Ты заполняешь его один раз, и он везде подходит.
   - **Зачем:** Меньше бумажной волокиты, меньше ошибок при переписывании данных с одной бумажки на другую 🗺️.

   **👨‍💻 Для глубокого понимания (Technical Deep Dive):**
   Для Middle разработчика важно понимать, когда SQLModel упрощает жизнь, а когда ограничивает.

   **1. Архитектура (Under the Hood):**
   - SQLModel наследуется от `pydantic.BaseModel` и использует `sqlalchemy` внутри.
   - `table=True` указывает, что класс — это таблица БД. Без этого флага — это просто схема данных (Pydantic).
   ```python
   from sqlmodel import SQLModel, Field
   
   class Hero(SQLModel, table=True):
       id: int | None = Field(default=None, primary_key=True)
       name: str
       secret_name: str
       age: int | None = None
   ```

   **2. Преимущества (Pros):**
   - **DRY (Don't Repeat Yourself):** Не нужно создавать отдельно `UserModel` (DB) и `UserSchema` (API).
   - **Type Hints:** Полная поддержка типов Python (автодополнение в IDE).
   - **FastAPI Integration:** Идеально работает с `response_model` и валидацией запросов.
   - **Async Support:** Полная поддержка асинхронности (`AsyncSession`).

   **3. Недостатки и Нюансы (Cons & Pitfalls):**
   - **Coupling (Связанность):** Смешивает логику БД и API. Иногда нужно скрыть поле `password` в API, но оставить в БД.
     - **Решение:** Использовать наследование для схем (`class HeroCreate(Hero): ...`).
   - **Complex Queries:** Для очень сложных запросов синтаксис SQLAlchemy может быть удобнее.
   - **Legacy:** В старых проектах только SQLAlchemy.

   **4. Пример использования в FastAPI:**
   ```python
   @app.post("/heroes/", response_model=Hero)
   def create_hero(hero: Hero):  # Один класс для входа и выхода
       session.add(hero)
       session.commit()
       session.refresh(hero)
       return hero
   ```

   **🎯 Совет для Middle:**
   На собеседовании скажи: «Я использую SQLModel для новых FastAPI проектов, чтобы сократить бойлерплейт. Но для сложных схем ответа я всё равно создаю отдельные классы наследованием, чтобы не светить лишние поля БД в API».
