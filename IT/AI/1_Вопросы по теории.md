## Основы и архитектура агентов

### 1. Что такое AI-агент и как он отличается от чатбота
**Вопрос:** Объясни, что такое AI-агент и чем он отличается от обычного чатбота?

**Ответ:** AI-агент — это автономная система, которая воспринимает окружение, принимает решения на основе собственной логики и целей, выполняет действия (включая использование внешних инструментов), а затем обновляет своё представление о мире на основе результатов. Агент имеет собственную цель, которую активно преследует, используя доступные инструменты и информацию. Чатбот — это реактивная система, которая просто обрабатывает входные сообщения от пользователя и выдаёт ответ без самостоятельных целей, планов или использования инструментов. Агент может работать самостоятельно (автономно), в то время как чатбот зависит от пользовательского ввода. Агент может привлекать информацию из БД, вызывать API, использовать калькулятор или поисковую систему для решения задачи, чатбот просто генерирует текст на основе обученного на примерах шаблона.

```python
# Чатбот — просто отвечает на вопрос
chatbot_response = llm("Какая столица Франции?")

# Агент — имеет цель и использует инструменты
agent = Agent(
    goal="Найти лучший ресторан в Париже",
    tools=[search_restaurants, get_reviews, book_reservation]
)
agent.execute()  # автономно выполняет поиск и бронирование
```

| Термин | Определение |
|---|---|
| AI-агент | Автономная система с целью, восприятием, рассуждением и действием |
| Чатбот | Реактивная система, отвечающая на пользовательский ввод |
| Autonomy | Способность агента работать независимо без постоянного указания пользователя |
| Tool use | Использование агентом внешних инструментов для решения задач |

**Ответ для интервьюера:** AI-агент имеет собственную цель и автономно работает, используя инструменты и информацию для её достижения. Чатбот — это просто реактивная система, которая отвечает на пользовательские вопросы. Агент может инициировать действия, планировать и привлекать внешние ресурсы, чатбот — только реагирует на ввод.

### 2. Жизненный цикл агента (perception, reasoning, action)
**Вопрос:** Объясни жизненный цикл AI-агента: восприятие, рассуждение, действие.

**Ответ:** Жизненный цикл (loop) агента состоит из трёх основных фаз, повторяющихся циклически:
1. **Perception (Восприятие)** — агент собирает информацию из окружения: текст пользователя, результаты предыдущих действий, состояние инструментов, БД и т.д. Эта информация формирует состояние агента.
2. **Reasoning (Рассуждение)** — на основе воспринятой информации и текущего контекста, агент использует LLM для определения следующего действия: какой инструмент вызвать, какой запрос сделать, или выдать ли финальный ответ пользователю.
3. **Action (Действие)** — агент выполняет решение из фазы рассуждения: вызывает функцию, делает запрос к API, читает из памяти, пишет результаты. Результаты действия становятся новой информацией для восприятия в следующей итерации.

Цикл повторяется до тех пор, пока агент не решит, что задача выполнена, или до достижения лимита итераций.

```python
class Agent:
    def run(self):
        while not self.goal_reached and self.iteration < self.max_iterations:
            # Perception: собираем текущее состояние
            state = self.get_current_state()
            
            # Reasoning: LLM решает, что делать
            action = self.llm.decide_next_action(state)
            
            # Action: выполняем решение
            result = self.execute_action(action)
            
            # Обновляем память
            self.memory.add(f"Action: {action}, Result: {result}")
            
            self.iteration += 1
        
        return self.get_final_answer()
```

| Термин | Определение |
|---|---|
| Perception | Сбор информации из окружения и текущего состояния |
| Reasoning | Принятие решения о следующем действии с помощью LLM |
| Action | Выполнение решения (вызов инструментов, API, обновление состояния) |
| Agent loop | Циклический повтор perception-reasoning-action |

**Ответ для интервьюера:** Агент работает в цикле: на этапе восприятия собирает информацию из окружения, на этапе рассуждения LLM решает, какое действие предпринять, на этапе действия выполняет это решение (вызывает инструменты). Цикл повторяется до выполнения цели или достижения лимита итераций.

### 3. Архитектуры агентов (ReAct, Plan-and-Execute, Tool-Use)
**Вопрос:** Какие основные архитектуры агентов существуют? Расскажи про ReAct, Plan-and-Execute и Tool-Use.

**Ответ:** Существует несколько основных архитектур, определяющих, как агент рассуждает и действует:
- **ReAct (Reasoning + Acting)** — агент чередует фазы рассуждения (размышления вслух, промежуточные выводы) и действия (вызова инструментов), создавая цепь мыслей с интерпретацией результатов. Пример: "Мне нужно найти информацию о погоде. [Вызвал weather_api] Сейчас идёт дождь. Значит, нужно рекомендовать зонт." Это самая распространённая архитектура, простая и интерпретируемая.
- **Plan-and-Execute** — агент сначала составляет полный план решения задачи на основе LLM, а затем последовательно его выполняет, обновляя план при необходимости. Вначале думает "Чтобы ответить на этот вопрос, мне нужно: 1) найти документ X, 2) извлечь информацию Y, 3) обобщить результат", потом выполняет каждый шаг. Лучше для сложных многошаговых задач, даёт больше контроля.
- **Tool-Use** — специализированная архитектура, где LLM явно вызывает инструменты через структурированный формат (функции/API), и агент проверяет результат перед продолжением. Часто используется с function calling в OpenAI API или tool_choice в других моделях.

| Архитектура | Как работает | Когда использовать |
|---|---|---|
| **ReAct** | Чередует рассуждение и действие, создаёт цепь мыслей | Большинство задач, хорошо интерпретируемо |
| **Plan-and-Execute** | Сначала план, потом выполнение, адаптация | Сложные многошаговые задачи, критична последовательность |
| **Tool-Use** | LLM явно вызывает функции, структурированный формат | Когда нужна надёжная интеграция с инструментами |


**Примеры использования в Claude Code:**

Claude Code реализует все три архитектуры в зависимости от типа задачи:

**1. ReAct (Reasoning + Acting) — основной паттерн:**
```
Пользователь: "Добавь вопрос про LangChain в файл и обнови нумерацию"

Агент (Claude):
💭 Thinking: Мне нужно:
   1. Прочитать файл
   2. Найти место для вставки
   3. Вставить новый вопрос
   4. Обновить нумерацию

⚙️ Action 1: Read file (Bash grep)
📊 Result: Вижу структуру, вопросы #29-#36

💭 Thinking: Нужно вставить между LangChain и LlamaIndex

⚙️ Action 2: Insert (Bash sed)
📊 Result: Вопрос вставлен

💭 Thinking: Обновить номера всех последующих

⚙️ Action 3: Renumber (Bash sed multiple)
📊 Result: Нумерация обновлена

✅ Final: Задача выполнена
```

**2. Plan-and-Execute — для сложных многошаговых задач:**
```
Пользователь: "Рефакторь проект: переструктурируй файлы, 
обнови импорты, добавь типизацию"

📋 PLANNING PHASE:
1. Анализ: Read все файлы, понять текущую структуру
2. План действий:
   - Создать новые директории (src/core/, src/utils/)
   - Переместить файлы (Git mv)
   - Обновить импорты (Bash sed)
   - Добавить type hints (Edit файлы)
   - Запустить тесты (Bash pytest)
3. Последовательность: что критично, что можно параллелить
4. Риск-анализ: точки отката

🔨 EXECUTION PHASE:
Step 1: Создать структуру → Bash mkdir
Step 2: Переместить файлы → Git mv
Step 3: Обновить импорты → Bash sed + Edit
Step 4: Добавить типы → Edit каждый файл
Step 5: Запустить тесты → Bash pytest
Step 6: Commit → Git commit

🔄 ADAPTATION:
Если Step 5 фейлится:
- Понять что сломалось
- Модифицировать план (может нужны дополнительные импорты)
- Продолжить с Step 4
```

**3. Tool-Use (Structured Tool Calling):**
```
Пользователь: "Найди все использования функции foo() и замени на bar()"

Структурированный вызов инструментов:

🔧 Tool 1: Grep
- Pattern: foo\(\)
- Files: **/*.py
- Result: Найдено 47 использований в 12 файлах

🔧 Tool 2: Edit (для каждого файла)
- old_string: foo()
- new_string: bar()
- File: src/module.py
- Result: Заменено 5 использований

🔧 Tool 3: Bash (тесты)
- Command: pytest
- Result: ✓ All tests pass

🔧 Tool 4: Git (commit)
- Message: "Refactor: rename foo() to bar()"
- Result: Committed
```

**Сравнение архитектур в контексте Claude Code:**

| Архитектура | Claude Code пример | Преимущества | Ограничения |
|---|---|---|---|
| **ReAct** | Читаю файл → вижу проблему → действую → объясняю | Интерпретируемо, гибко | Может быть медленнее |
| **Plan-and-Execute** | Анализирую весь проект → составляю план → выполняю пошагово | Контролируемо, предсказуемо | Требует больше времени на планирование |
| **Tool-Use** | Явно вызываю Bash, Edit, Grep с параметрами | Надёжно, структурировано | Нужна точная спецификация инструментов |

**Ответ для интервьюера:** ReAct чередует рассуждение и действие, создавая цепь мыслей — простая и интерпретируемая архитектура. Plan-and-Execute сначала создаёт план, потом его выполняет — лучше для сложных задач. Tool-Use использует структурированный вызов функций LLM — надёжнее для интеграции с инструментами.

### 4. Состояние и память в агентах
**Вопрос:** Как управлять состоянием и памятью в AI-агенте?

**Ответ:** Состояние агента включает текущую цель, историю действий, доступные инструменты, и контекст из памяти. Память в агентах обычно разделяется на:
- **Working Memory (рабочая память)** — текущий контекст, последние действия, переменные, используемые прямо сейчас. Это то, что передаётся в LLM в каждом вызове. Размер ограничен контекстным окном.
- **Episodic Memory (эпизодическая память)** — история конкретных событий, действий и результатов в текущей сессии. "Я вызвал API и получил результат X", "Я попробовал инструмент Y и он не сработал".
- **Semantic Memory (семантическая память)** — факты, правила, общие знания о задаче, загруженные из БД или документов. "Капитал Франции — Париж", "Этот API требует ключ авторизации".

Управление памятью критично, так как LLM имеет ограниченный контекст: нужно отбирать релевантную информацию (через поиск, summarization, importance scoring) и отбрасывать неважное, чтобы оставить место для текущей задачи.

| Тип памяти | Что хранит | Когда используется |
|---|---|---|
| **Working** | Текущий контекст, последние действия | В каждом вызове LLM |
| **Episodic** | История событий сессии | Для обучения и адаптации |
| **Semantic** | Факты и знания | При поиске релевантной информации |

**Ответ для интервьюера:** Память агента разделяется на рабочую (текущий контекст), эпизодическую (история действий) и семантическую (знания). Управление памятью критично из-за ограничения контекстного окна — нужно отбирать релевантную информацию и отбрасывать неважное.

## Языковые модели и трансформеры

### 5. Как работают трансформеры и внимание (Attention)
**Вопрос:** Объясни, как работают трансформеры и механизм внимания (Attention).

**Ответ:** Трансформер — архитектура на основе механизма внимания (Attention), который позволяет модели связать любое слово с любым другим словом в последовательности за один проход, в отличие от RNN, которые обрабатывают слова последовательно. Механизм внимания вычисляет взвешенное резюме всего входа для каждой позиции: для каждого слова (query) модель вычисляет «важность» (вес внимания) каждого другого слова (key) и суммирует их значения (values) пропорционально этим весам. Это позволяет модели одновременно обрабатывать весь текст и находить дальние зависимости между словами.

Трансформер состоит из:
1. **Embedding слоя** — преобразует текст в векторы (embeddings)
2. **Encoder** (в BERT, T5) — обрабатывает весь текст сразу, понимая контекст каждого слова
3. **Decoder** (в GPT, T5, LLaMA) — генерирует ответ слово за словом, используя внимание к входу и к уже сгенерированным словам
4. **Head (голова классификации/генерации)** — преобразует выход в предсказание следующего слова или класса

Параллелизм трансформеров позволил масштабировать обучение на больших текстовых корпусах, создав современные LLM (GPT, LLaMA, Claude).

| Термин            | Определение                                                                 |
| ----------------- | --------------------------------------------------------------------------- |
| Attention         | Механизм взвешенного суммирования значений, взвешенных по релевантности     |
| Query, Key, Value | Матрицы для вычисления внимания: query ищет, key матчит, value возвращается |
| Self-Attention    | Внимание к своей же последовательности (в отличие от cross-attention)       |
| Transformer       | Архитектура на основе внимания без рекуррентности                           |

**Ответ для интервьюера:** Трансформер использует механизм внимания, позволяющий модели одновременно обрабатывать весь текст и связывать любые два слова напрямую. Query ищет, Key матчит, Value возвращается — для каждого слова модель вычисляет, какие другие слова важны для понимания его контекста, и суммирует их значения. Это позволило создать современные LLM, способные обрабатывать большие объёмы текста параллельно.

### 6. Токенизация и embedding
**Вопрос:** Что такое токенизация и embedding? Как они используются в LLM?

**Ответ:** **Токенизация** — процесс разбиения текста на маленькие единицы (токены), которые LLM может обрабатывать. Токеном может быть слово, подслово (subword, как в BPE — Byte Pair Encoding), символ или специальный маркер. Например, "Hello, world!" может быть токенизирован как ["Hello", ",", " world", "!"] или на более мелкие единицы. Разные модели используют разные токенизаторы. OpenAI GPT использует BPE, где частые последовательности байтов объединяются в новые токены. Размер vocabularity и способ токенизации влияют на эффективность модели — если токены редкие, модель будет тратить больше контекста.

**Embedding** — преобразование токена в плотный вектор (обычно размерностью 768-4096), где семантически похожие слова лежат близко друг к другу в пространстве. Embeddings обучаются вместе с моделью: похожие токены получают похожие вектора через процесс обучения. Embeddings — это "понимание" моделью смысла слова.

```python
# Токенизация
text = "Hello, world!"
tokens = tokenizer.encode(text)  # [9906, 11, 1917, 0]

# Embedding: токены преобразуются в векторы
embeddings = model.embed(tokens)  # shape: [4, 768]
```

| Термин     | Определение                                          |
| ---------- | ---------------------------------------------------- |
| Token      | Единица текста (слово, подслово или символ)          |
| Tokenizer  | Алгоритм разбиения текста на токены (BPE, WordPiece) |
| Embedding  | Плотный вектор, представляющий смысл токена          |
| Vocabulary | Набор всех возможных токенов модели                  |


**Ответ для интервьюера:** Токенизация разбивает текст на маленькие единицы (токены) — слова, подслова или символы. Embedding преобразует токен в плотный вектор, где семантически похожие слова лежат близко. Размер и способ токенизации влияют на эффективность: более мелкие токены требуют больше контекста, более крупные — менее гибкие.

### 7. Температура (temperature) и top-p sampling
**Вопрос:** Что такое температура и top-p sampling в LLM? Как они влияют на поведение модели?

**Ответ:** LLM предсказывает следующее слово, выдавая вероятность для каждого токена из vocabulary. Вместо всегда выбирать наиболее вероятный токен, можно выбирать случайно из распределения. **Температура** — параметр, контролирующий "уверенность" модели:
- **Температура = 0** (или argmax) — всегда выбирается наиболее вероятный токен (детерминированно, скучно)
- **Температура = 1.0** — используется исходное распределение вероятностей модели (разумный баланс)
- **Температура > 1.0** — распределение становится более "плоским", модель выбирает менее вероятные токены чаще (творчески, но может быть бредовой)

**Top-p (nucleus) sampling** — вместо выбора из всех токенов, выбирают из наименьшего набора токенов, суммарная вероятность которых >= p. Например, при p=0.9 может выбираться из топ-5 токенов, если их суммарная вероятность уже 90%. Это избегает выбора совсем неправдоподобных токенов, но остаёт место для разнообразия.

| Параметр | Значение | Эффект |
|---|---|---|
| **Temperature** | 0 | Детерминированно (скучно) |
| **Temperature** | 0.5-0.7 | Сбалансированно (для точных ответов) |
| **Temperature** | 1.0-1.5 | Творчески (для генерации) |
| **Top-p** | 0.9 | Разумное разнообразие без бреда |
| **Top-p** | 0.5 | Более консервативно |

**Ответ для интервьюера:** Температура контролирует "уверенность" модели — 0 = всегда наиболее вероятный токен (детерминированно), выше = более случайно и творчески. Top-p sampling выбирает из наименьшего набора токенов, чья суммарная вероятность >= p, избегая совсем неправдоподобных выборов. Для точных ответов используют низкую температуру, для творчества — высокую.

### 8. Context window и token limits
**Вопрос:** Что такое context window и token limits? Как они ограничивают возможности агента?

**Ответ:** **Context window** — максимальное количество токенов, которые LLM может обработать в одном вызове. Это включает как входные токены (prompt), так и выходные (completion). Разные модели имеют разные лимиты:
- GPT-3.5: 4K токенов
- GPT-4: 8K или 128K токенов
- Claude: 100K или 200K токенов
- LLaMA 2: 4K токенов

**Ограничения context window** критичны для агентов:
1. **Ограничение истории** — если история длиннее, чем контекст, нужно обрезать или суммировать старые действия
2. **Ограничение размера документов** — если документ больше контекста, нужно разбивать его на части
3. **Ограничение памяти** — долгосрочная память должна быть компактной
4. **Задержка и стоимость** — использование большего контекста медленнее и дороже

```python
# Проверка размера контекста
tokens_used = len(encode(prompt)) + len(encode(completion))
if tokens_used > model_context_window:
    # Нужно сжать prompt или chunkit документы
    prompt = summarize(prompt)
```

| Параметр | Значение | Примечание |
|---|---|---|
| **GPT-3.5** | 4K токенов | Старая модель, дешевле |
| **GPT-4** | 8K / 128K | 128K позволяет много контекста |
| **Claude** | 100K / 200K | Одна из самых больших |
| **LLaMA 2** | 4K / 4K | Локальная модель |

**Ответ для интервьюера:** Context window — максимум токенов в одном вызове (входящих + исходящих). Разные модели имеют разные лимиты (GPT-4 до 128K, Claude до 200K). Для агентов это критично — нужно управлять историей, документами и памятью, чтобы не превышать лимит, иначе информация обрезается или модель отказывает.

### 9. Prompt engineering: техники и best practices
**Вопрос:** Какие техники prompt engineering используются для улучшения качества ответов LLM?

**Ответ:** **Prompt engineering** — искусство формулирования входных текстов (промптов) таким образом, чтобы LLM выдавала лучшие ответы. Основные техники:

1. **Специфичность и ясность** — вместо "Напиши текст" лучше "Напиши маркетинговое письмо для SaaS продукта, целевая аудитория — CTO". Более точный промпт даёт более точный ответ.

2. **System prompt (системный промпт)** — специальное указание, описывающее роль и поведение модели. Например: "Ты опытный разработчик Python с 10 годами опыта. Отвечай техническими терминами, но понятно."

3. **Few-shot examples** — дать несколько примеров желаемого формата ввода-вывода, чтобы модель поняла паттерн.

4. **Chain-of-Thought (CoT)** — попросить модель объяснить свои рассуждения пошагово ("Давай решим это пошагово"). Часто даёт более правильные результаты.

5. **Role-playing** — дать модели роль ("Представь, что ты юрист") для получения специализированных ответов.

6. **Constraints и guardrails** — явно указать ограничения ("Ответ должен быть < 100 слов, на русском языке").

```python
# Системный промпт + пользовательский + структура
system_prompt = """Ты помощник для кодирования. 
Объясняй код ясно и кратко.
Если код некорректен, укажи ошибку и дай исправленную версию."""

user_prompt = """Объясни что этот код делает:
```python
def fib(n):
    return n if n <= 1 else fib(n-1) + fib(n-2)
```"""

response = llm.chat([
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_prompt}
])
```

| Техника | Использование | Результат |
|---|---|---|
| **Специфичность** | Конкретно описать задачу | Более точный ответ |
| **System prompt** | Определить роль модели | Консистентный стиль |
| **Few-shot** | Показать примеры | Лучше поймёт паттерн |
| **Chain-of-Thought** | Попросить объяснение | Лучшие рассуждения |

**Ответ для интервьюера:** Хорошие промпты должны быть специфичными, содержать систему указания (role), примеры если нужно, и просить объяснение рассуждений (CoT). Более точный промпт даёт более точный ответ — это считается больше искусством, чем наукой, но есть закономерности.

### 10. Few-shot learning vs zero-shot
**Вопрос:** Что такое few-shot learning и zero-shot? В чём разница и когда что использовать?

**Ответ:** **Zero-shot** — модель решает задачу без каких-либо примеров, опираясь только на описание задачи и свои обученные знания. Например: "Классифицируй этот текст как положительный или отрицательный: 'Отличный фильм!'" — модель сразу даёт ответ. Это работает благодаря большому объёму данных, на которых модель была обучена.

**Few-shot** — модель получает несколько примеров (обычно 2-5) решения задачи, а потом применяет эту логику к новой задаче. Например: "Вот примеры классификации: 'Отлично!' → Положительный, 'Ужасно!' → Отрицательный. Теперь классифицируй: 'Хороший продукт'". Это часто улучшает качество для специализированных или нетривиальных задач.

**One-shot** (частный случай few-shot) — только один пример.

| Подход | Примеры | Когда использовать | Результат |
|---|---|---|---|
| **Zero-shot** | Нет | Простые общие задачи | Быстро, может быть неточно |
| **Few-shot** | 2-5 | Специфичные задачи, коррекция формата | Точнее, но требует примеров |
| **Fine-tuning** | Тысячи | Полная переподготовка на своих данных | Самое точное, но дорого |

**Ответ для интервьюера:** Zero-shot — модель решает задачу без примеров, опираясь на знания. Few-shot — модель получает несколько примеров и потом применяет логику. Few-shot часто улучшает результат для специфичных задач, особенно если формат ответа нестандартный.

---

### 11. Chain-of-Thought (CoT) промпты
**Вопрос:** Что такое Chain-of-Thought и почему это улучшает результаты?

**Ответ:** **Chain-of-Thought (CoT)** — техника, при которой просят LLM объяснить свои рассуждения пошагово, вместо того чтобы просто выдать финальный ответ. Исследования показали, что это значительно улучшает качество ответов, особенно на сложных логических задачах, математике и рассуждениях.

Почему это работает:
1. **Промежуточные шаги** — модель может исправлять себя на ходу, если сразу заметит ошибку
2. **Разбиение сложности** — разбивает сложную задачу на более простые подшаги
3. **Прозрачность** — можно видеть, где ошибка, если ответ неправильный

```python
# Без CoT
prompt_direct = "2 человека работали 3 часа. 3 человека работали 5 часов. Сколько всего часов?"
# Ответ может быть неправильным: просто сложит 3 + 5 = 8

# С CoT
prompt_cot = """Давай решим эту задачу пошагово:
2 человека работали 3 часа. Сколько всего часов работы? 2 * 3 = 6 часов.
3 человека работали 5 часов. Сколько всего часов работы? 3 * 5 = 15 часов.
Сколько всего? 6 + 15 = 21 час."""
# Модель с большей вероятностью даст правильный ответ
```

**Self-Consistency** — расширение CoT, где вызывают модель несколько раз с разными цепочками рассуждений и выбирают самый частый ответ. Это ещё больше улучшает качество.

| Подход | Сложность | Качество | Использование |
|---|---|---|---|
| **Прямой ответ** | Простой | Ниже | Простые задачи |
| **Chain-of-Thought** | Средний | Выше | Логика, математика |
| **Self-Consistency** | Дорогой | Самое высокое | Критичные задачи |

**Ответ для интервьюера:** Chain-of-Thought просит модель объяснить свои рассуждения пошагово, а не выдать сразу ответ. Это значительно улучшает качество на сложных логических задачах, потому что модель может исправлять себя. Self-Consistency ещё больше улучшает результат, вызывая модель несколько раз и выбирая наиболее часто встречающийся ответ.

---

### 12. Fine-tuning vs in-context learning
**Вопрос:** В чём разница между fine-tuning и in-context learning? Когда что использовать?

**Ответ:** **Fine-tuning** — переподготовка модели на новых данных (обычно тысячи примеров), обновляя её веса. Это требует вычислительных ресурсов, времени и денег, но модель становится "знающей" о специфичной задаче. После fine-tuning модель может решать задачу лучше, даже без явных примеров в промпте.

**In-context learning** — предоставление примеров прямо в промпте (few-shot), без переподготовки модели. Это быстро, дешево и гибко, но зависит от качества примеров и контекстного окна.

| Аспект | Fine-tuning | In-context learning |
|---|---|---|
| **Стоимость** | Высокая (GPU, время) | Низкая (API вызов) |
| **Время** | Часы/дни | Секунды |
| **Гибкость** | Низкая (переподготовка нужна для новой задачи) | Высокая (просто измени промпт) |
| **Качество** | Часто лучше | Достаточно для большинства |
| **Требует данных** | Тысячи примеров | 2-5 примеров |
| **Когда использовать** | Очень специфичная задача, большой объём | Большинство случаев |

**Ответ для интервьюера:** Fine-tuning переподготавливает модель на новых данных, что требует ресурсов, но даёт лучший результат для очень специфичных задач. In-context learning (few-shot) даёт примеры в промпте, это быстро и дешево. Для большинства задач достаточно in-context learning; fine-tuning нужен, если специфичная задача и много данных.

### 13. Function calling / Tool use в LLM
**Вопрос:** Что такое function calling и как его использовать в агентах?

**Ответ:** **Function calling** (tool use) — способность LLM вызывать внешние функции/инструменты, описанные в JSON-схеме. Вместо просто генерирования текста, модель может "выбрать" функцию, которая нужна, и предоставить параметры для её вызова. Это ключевой компонент agent loop.

Поток:
1. Определить доступные функции в JSON-схеме (название, описание, параметры)
2. Отправить user prompt и схемы функций в LLM
3. LLM возвращает выбранную функцию и параметры (как JSON)
4. Агент вызывает функцию в коде (не сама LLM)
5. Результат передаётся обратно в LLM для следующего шага

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Получить погоду в городе",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "Название города"},
                    "units": {"type": "string", "enum": ["celsius", "fahrenheit"]}
                }
            }
        }
    }
]

# LLM возвращает:
# {"type": "function_call", "function": "get_weather", "arguments": {"city": "Paris", "units": "celsius"}}

# Агент вызывает функцию:
result = get_weather(city="Paris", units="celsius")
```

| Компонент | Описание |
|---|---|
| Tool definition | JSON-схема функции (название, описание, параметры) |
| Tool call | Выбор функции и параметры (возвращает LLM) |
| Tool execution | Вызов функции в коде (выполняет агент) |
| Tool result | Результат, передаваемый обратно в LLM |

**Ответ для интервьюера:** Function calling позволяет LLM "выбирать" инструменты из доступного набора (определённого в JSON-схеме) и предоставлять параметры. Агент вызывает функцию в коде, а результат передаётся обратно в LLM для следующего шага. Это надёжнее, чем парсить естественный язык для вызова функций.

### 14. RAG (Retrieval-Augmented Generation)
**Вопрос:** Что такое RAG и как его использовать в агентах?

**Ответ:** **RAG (Retrieval-Augmented Generation)** — техника, при которой перед генерацией ответа, система сначала извлекает (retrieves) релевантные документы из базы знаний, а потом передаёт их в LLM вместе с вопросом. Это позволяет модели давать ответы на основе актуальной информации, не переподготавливаясь.

Поток RAG:
1. **Retrieval (Индексация + Поиск)** — преобразовать вопрос в embedding и найти похожие документы в vector DB
2. **Augmentation (Контекст)** — добавить найденные документы в контекст промпта
3. **Generation (Генерация)** — LLM генерирует ответ, опираясь на полученные документы

Примеры: поиск в документации компании, ответы на вопросы о своих данных, QA на основе корпуса документов.

#### Подвопрос 1: RAG это что? Его нужно импортировать или это просто механизм?

**Ответ:** **RAG не является библиотекой или модулем**, которые нужно импортировать. Это **архитектурный паттерн/механизм** — способ организации работы с LLM. RAG реализуется через комбинацию различных компонентов:

- **Vector DB** — хранилище embeddings (pgvector, Pinecone, Weaviate, Milvus)
- **Embedding модель** — преобразует текст в вектор (sentence-transformers, OpenAI embeddings)
- **LLM** — генерирует ответ (Claude, GPT, LLaMA)
- **Фреймворки для оркестрации** — LangChain, LlamaIndex помогают собрать RAG pipeline

RAG можно реализовать несколькими способами:
1. **С LangChain** — готовые компоненты, быстро поднять MVP
2. **С LlamaIndex** — специализирован для работы с документами
3. **С нуля** — вручную писать retrieval, augmentation, generation

```python
# Способ 1: С LangChain (используется готовая библиотека)
from langchain.vectorstores import PgVector
from langchain.embeddings import OpenAIEmbeddings
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA

# RAG механизм собирается из компонентов
embeddings = OpenAIEmbeddings()  # импортируется
vector_db = PgVector(connection_string=..., embedding_function=embeddings)
llm = ChatOpenAI()  # импортируется

# Это уже готовый RAG
rag_chain = RetrievalQA.from_chain_type(llm=llm, retriever=vector_db.as_retriever())

# Способ 2: Вручную (никакой специальной библиотеки для RAG)
import psycopg2
from sentence_transformers import SentenceTransformer
from anthropic import Anthropic

# Ты сам реализуешь каждый шаг
embeddings_model = SentenceTransformer('model')
db_conn = psycopg2.connect(...)
llm_client = Anthropic()

# Вручную вызываешь каждый шаг (retrieval → augmentation → generation)
```

**Вывод:** RAG — это паттерн, не библиотека. Ты импортируешь компоненты (vector DB, embedding модель, LLM), которые выполняют RAG, или используешь фреймворки вроде LangChain, которые уже собрали RAG для тебя.

#### Пример: RAG-ассистент для селлеров в проекте "Мир хобби"

Реализован RAG-ассистент для селлеров на базе **LangChain + FastAPI**:

```python
# tech stack: LangChain, FastAPI, PostgreSQL (pgvector), SSE streaming

from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from langchain.vectorstores.pgvector import PgVector
from langchain.embeddings import OpenAIEmbeddings
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA
from langchain.callbacks.streaming_aiter import AsyncIteratorCallbackHandler
import asyncio

app = FastAPI()

# Инициализация компонентов
embeddings = OpenAIEmbeddings()
vector_store = PgVector(
    connection_string="postgresql://user:pass@localhost/hobbies_db",
    embedding_function=embeddings,
    collection_name="marketplace_knowledge"  # база знаний: правила маркетплейсов, API, гайды
)
retriever = vector_store.as_retriever(search_kwargs={"k": 5})
llm = ChatOpenAI(model="gpt-3.5-turbo", temperature=0.7)

# RAG chain с streaming callbacks
rag_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",
    retriever=retriever,
    return_source_documents=True
)

async def stream_response_generator(query: str):
    """Стриминг ответов через SSE"""
    handler = AsyncIteratorCallbackHandler()
    
    try:
        # Retrieval: поиск релевантных документов
        # (база знаний: правила маркетплейсов, API-документация, гайды по ценообразованию)
        relevant_docs = retriever.get_relevant_documents(query)
        
        # Generation с streaming
        task = asyncio.create_task(
            rag_chain.acall(
                {"query": query},
                callbacks=[handler]
            )
        )
        
        async for token in handler.aiter():
            yield f"data: {token}\n\n"
        
        result = await task
        yield f"data: [DONE]\n\n"
        
    except Exception as e:
        yield f"data: Error: {str(e)}\n\n"

@app.post("/ask")
async def ask_assistant(question: str):
    """Endpoint для RAG запроса с SSE стримингом"""
    return StreamingResponse(
        stream_response_generator(question),
        media_type="text/event-stream"
    )

# Метрики в продакшене:
# - Обрабатывает ~1500 запросов в сутки
# - p95 latency < 2 сек (в основном благодаря pgvector индексам и кэшированию embeddings)
# - Точность поиска (recall@5): 92%
# - Пользовательская оценка ответов: 4.2/5.0
```

**Архитектура:**
```
Вопрос селлера
    ↓
[Embedding] → преобразовать вопрос в вектор
    ↓
[pgvector Search] → поиск в базе знаний (PostgreSQL + pgvector)
    ↓
[Retrieval] → получить топ-5 релевантных документов
    ↓
[Augmentation] → вставить документы в контекст
    ↓
[LLM (GPT-3.5)] → генерировать ответ
    ↓
[SSE Streaming] → отправить ответ клиенту потоком (не ждём полного ответа)
```

**Ключевые оптимизации для продакшена:**
1. **pgvector индексы (HNSW)** — быстрый поиск на больших объёмах (1500 req/день)
2. **Кэширование embeddings** — не пересчитываем одинаковые вопросы
3. **SSE streaming** — не ждём полного ответа LLM, отправляем токены по мере генерации
4. **Retriever с k=5** — баланс между качеством (больше контекста) и скоростью (не слишком много)
5. **Async/await** — параллельная обработка множества запросов

| Метрика | Значение |
|---|---|
| **Requests per day** | ~1500 |
| **p95 Latency** | < 2 сек |
| **Recall@5** | 92% |
| **User rating** | 4.2/5.0 |
| **Uptime** | 99.8% |

| Компонент    | Описание                                                       |
| ------------ | -------------------------------------------------------------- |
| Retrieval    | Поиск релевантных документов в vector DB (pgvector)            |
| Ranking      | Сортировка документов по релевантности (косинусное расстояние) |
| Augmentation | Добавление документов в промпт LLM                             |
| Generation   | LLM генерирует ответ, опираясь на контекст                     |
| Streaming    | SSE для потокового отправления ответа клиенту                  |

**Ответ для интервьюера:** RAG — архитектурный паттерн, не библиотека. В проекте "Мир хобби" реализован RAG на LangChain + FastAPI с pgvector для поиска по базе знаний селлеров. Обрабатывает 1500 запросов в сутки с p95 latency < 2 сек благодаря оптимизациям (индексирование, кэширование, streaming). Основные компоненты: retrieval (pgvector поиск) → augmentation (добавление контекста) → generation (LLM ответ) → streaming (SSE).

### 15. Vector databases и embeddings
**Вопрос:** Как работают vector databases и embeddings для поиска подобных документов?

**Ответ:** **Embeddings** — плотные векторы (обычно 384-4096 размерности), которые представляют смысл текста. Похожие по смыслу тексты имеют похожие embeddings (близко расположены в векторном пространстве). Embeddings создаются специальными моделями (например, `text-embedding-3-small` от OpenAI, `all-MiniLM-L6-v2` от Hugging Face).

**Vector database** (например, Pinecone, Weaviate, Milvus, ChromaDB) — хранилище, оптимизированное для поиска подобных векторов. Вместо полнотекстового поиска (где ищут по ключевым словам), они ищут по семантическому смыслу.

```python
# Создать embeddings
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('all-MiniLM-L6-v2')
documents = [
    "Как установить Python?",
    "Установка PHP на Linux",
    "Погода сегодня в Москве"
]
embeddings = model.encode(documents)  # shape: [3, 384]

# Поиск похожего документа
query = "Как скачать Python?"
query_embedding = model.encode([query])[0]

# Найти ближайшие документы
similarities = cosine_similarity([query_embedding], embeddings)[0]
```

| Компонент | Описание |
|---|---|
| Embedding model | Модель, преобразующая текст в вектор |
| Vector DB | Хранилище для быстрого поиска похожих векторов |
| Similarity search | Поиск ближайших векторов (cosine, euclidean) |
| Index | Структура (HNSW, IVF) для быстрого поиска |

**Ответ для интервьюера:** Embeddings представляют смысл текста как плотные векторы, похожие тексты имеют похожие embeddings. Vector DB хранит эти векторы и позволяет быстро найти похожие документы через поиск ближайших соседей, вместо полнотекстового поиска по ключевым словам.

### 16. Semantic search vs keyword search
**Вопрос:** В чём разница между semantic search и keyword search?

**Ответ:** **Keyword search** — традиционный поиск, где система ищет точное совпадение ключевых слов в документах (как Google до neural networks). Быстро, точно для точных совпадений, но не понимает смысл. Например, поиск по "как установить" не найдёт "инструкция по установке", даже если смысл одинаковый.

**Semantic search** — поиск, основанный на смысле, а не на ключевых словах. Система преобразует запрос и документы в embeddings и ищет ближайшие по смыслу документы. Находит семантически похожие документы, даже если нет точного совпадения слов. Например, "как установить" найдёт "инструкция по установке", "процесс инсталляции" и т.д.

| Аспект | Keyword | Semantic |
|---|---|---|
| **Как работает** | Совпадение слов | Embeddings + similarity |
| **Скорость** | Очень быстро | Медленнее (нужны embeddings) |
| **Понимание смысла** | Нет | Да |
| **Синонимы** | Не находит | Находит |
| **Точные совпадения** | Отличный результат | Хороший результат |

```python
# Keyword search
results = db.query("SELECT * FROM docs WHERE text LIKE '%install%'")

# Semantic search
query_emb = embed_model.encode("How to set up?")
results = vector_db.search(query_emb, top_k=5)
```

**Ответ для интервьюера:** Keyword search ищет точные совпадения слов (быстро, точно для точных совпадений). Semantic search ищет по смыслу через embeddings (находит синонимы и перефразировки). Для агентов обычно используют semantic search в RAG, так как нужно понимать смысл вопроса.

## Память и контекст

### 17. Типы памяти в агентах (working, episodic, semantic)
**Вопрос:** Какие типы памяти используются в AI-агентах?

**Ответ:** Агенты используют несколько типов памяти, заимствованные из когнитивной психологии:

1. **Working Memory (рабочая память)** — текущий контекст, который находится "в фокусе" агента. Это информация, переданная в LLM в текущем вызове. Размер ограничен контекстным окном модели. Включает последние действия, текущую задачу, релевантную историю.

2. **Episodic Memory (эпизодическая память)** — история конкретных событий и действий в текущей сессии. "Я попробовал API и получил ошибку 404", "Я спросил у пользователя и он сказал...". Используется для обучения и адаптации агента в текущей сессии. Может быть сохранена для последующего анализа.

3. **Semantic Memory (семантическая память)** — факты, правила, знания, не связанные с конкретным эпизодом. "Столица Франции — Париж", "Этот API требует ключ авторизации", "Пользователь предпочитает короткие ответы". Может быть загружена из БД, документов или изучена из истории.

| Память       | Содержимое                           | Хранение                  | Использование                     |
| ------------ | ------------------------------------ | ------------------------- | --------------------------------- |
| **Working**  | Текущий контекст, последние действия | LLM context               | В каждом вызове LLM               |
| **Episodic** | История событий сессии               | In-memory, DB             | Для обучения и адаптации          |
| **Semantic** | Факты и знания                       | Vector DB, Knowledge Base | Для поиска релевантной информации |

**Ответ для интервьюера:** Рабочая память — текущий контекст (ограничен контекстным окном). Эпизодическая — история действий в сессии (для адаптации). Семантическая — факты и знания (из БД или документов). Агент должен управлять всеми тремя, решая, какую информацию хранить в контексте.

### 18. Управление контекстом и окно памяти
**Вопрос:** Как управлять контекстом в агенте, если история становится больше context window?

**Ответ:** Когда история действий растёт, а context window ограничен, нужно выбирать, какую информацию сохранять в текущем контексте. Основные стратегии:

1. **Sliding window** — хранить только последние N сообщений/действий. Простая, но может потерять важную информацию из начала.

2. **Summarization** — периодически суммировать старую историю в краткое резюме. Например: "История до этого момента: пользователь спросил про X, я сделал Y, результат был Z". Новые действия добавляются целиком.

3. **Importance scoring** — оценить каждый элемент истории по важности (вероятно ли он понадобится в будущем?) и хранить только высокоимпортантные.

4. **Hierarchical memory** — разные слои памяти для разных временных горизонтов. Недавние события в working memory, все события в episodic memory, обобщение в semantic memory.

5. **Compression** — использовать LLM для сжатия истории: "Переформулируй эту историю в одном абзаце, сохраняя ключевые факты".

```python
class Agent:
    def add_to_memory(self, action):
        self.memory.append(action)
        
        # Проверить размер контекста
        tokens = count_tokens(self.memory)
        if tokens > self.context_limit * 0.8:  # 80% лимита
            # Суммировать старую историю
            old_history = self.memory[:-5]  # всё кроме последних 5
            summary = self.llm.summarize(old_history)
            self.memory = [summary] + self.memory[-5:]
```

| Стратегия | Простота | Качество | Использование |
|---|---|---|---|
| **Sliding window** | Высокая | Низкое | Когда История не критична |
| **Summarization** | Средняя | Высокое | Для длинных сессий |
| **Importance scoring** | Низкая | Очень высокое | Критичные задачи |
| **Hierarchical** | Низкая | Очень высокое | Сложные системы |

**Ответ для интервьюера:** Когда история растёт, используют sliding window (только последние события), summarization (суммировать старую историю), или importance scoring (хранить только важные события). Summarization обычно лучше всего — сохраняет смысл, экономит токены.

### 19. Summarization для долгосрочной памяти
**Вопрос:** Как использовать summarization для управления долгосрочной памятью агента?

**Ответ:** Summarization — техника, при которой LLM сжимает длинную историю в краткое резюме, сохраняя ключевую информацию. Это позволяет агенту "помнить" важные факты без хранения полной истории.

Стратегии:
1. **Incremental summarization** — каждые N действий суммировать историю. Новое резюме = старое резюме + действия с момента последнего резюме.
2. **Hierarchical summarization** — несколько уровней: детальная история последних действий, месячное резюме, годовое резюме.
3. **Selective summarization** — суммировать только менее важные части, оставляя недавние действия в полном виде.

```python
# Incremental summarization
history = [action1, action2, ..., action100]
if len(history) > 50:  # Каждые 50 действий
    old_history = history[:len(history)-10]
    summary = llm.summarize(old_history, max_tokens=100)
    # Новая история: [резюме, последние 10 действий]
    history = [{"type": "summary", "content": summary}] + history[-10:]
```

| Аспект | Без summarization | С summarization |
|---|---|---|
| **Размер памяти** | Растёт линейно | Растёт логарифмически |
| **Качество** | Лучше (всё помнит) | Хорошо (суть помнит) |
| **Скорость** | Медленнее (больше токенов) | Быстрее (меньше токенов) |
| **Информация потери** | Нет | Минимальная |

**Ответ для интервьюера:** Summarization сжимает долгую историю в резюме, экономя токены и контекст. Использу incremental (каждые N действий) или hierarchical (разные уровни детализации) summarization. Позволяет агентам работать в длинных сессиях.

### 20. Importance scoring для памяти
**Вопрос:** Как оценивать важность информации в памяти агента?

**Ответ:** Importance scoring — техника, при которой оценивают каждый элемент памяти по вероятности его использования в будущем, а потом сохраняют только высокооценённые элементы. Разные стратегии:

1. **LLM-based scoring** — попросить LLM оценить важность каждого события для текущей задачи. "На шкале 1-10, насколько важно для решения задачи, что...?"

2. **Heuristic scoring** — простые правила. Недавние события важнее старых, события с ошибками важнее успешных, события, упомянутые пользователем, важнее автоматических.

3. **Learned scoring** — обучить модель оценивать важность на основе истории какие события потом действительно пригодились.

4. **Keyword-based** — события, содержащие ключевые слова задачи, важнее.

```python
def score_importance(event, task):
    """Оценить важность события для задачи (1-10)"""
    score = 0
    
    # Recency: недавние события важнее
    time_delta = now() - event.timestamp
    score += 5 - min(time_delta / 1_hour, 5)  # Макс 5 баллов за новизну
    
    # Relevance: упоминается ли в задаче?
    if any(keyword in event.text for keyword in task.keywords):
        score += 3
    
    # Errors: ошибки часто важны
    if event.error:
        score += 2
    
    return score

# Сохранить только события с score > 3
important_events = [e for e in memory if score_importance(e, task) > 3]
```

**Ответ для интервьюера:** Importance scoring оценивает каждый элемент по вероятности использования в будущем. Можно использовать LLM, эвристики или обученные модели. Позволяет сохранять в памяти только действительно нужную информацию, экономя токены.

## Мультиагентные системы

### 21. Мультиагентные системы и координация
**Вопрос:** Как работают мультиагентные системы и как они координируются?

**Ответ:** **Мультиагентная система** — несколько независимых агентов, работающих вместе для достижения общей или индивидуальных целей. Каждый агент может иметь свои роли, знания, инструменты и цели. Координация критична для эффективности.

Способы координации:
1. **Hierarchical** — один координирующий агент назначает задачи другим (manager-worker).
2. **Peer-to-peer** — агенты общаются напрямую и договариваются о действиях.
3. **Blackboard** — общая доска (например, в памяти или БД), где все агенты пишут и читают информацию.
4. **Auction** — агенты "торгуются" за задачи, выбирая те, в которых они сильнее всего.

Примеры:
- **Planning agent** составляет план
- **Execution agents** выполняют части плана параллельно
- **Monitoring agent** следит за прогрессом и адаптирует план

| Способ | Сложность | Эффективность | Адаптивность |
|---|---|---|---|
| **Hierarchical** | Низкая | Средняя | Низкая (менеджер бутылочное горлышко) |
| **Peer-to-peer** | Высокая | Высокая | Высокая |
| **Blackboard** | Средняя | Высокая | Средняя |
| **Auction** | Средняя | Очень высокая | Высокая |

**Ответ для интервьюера:** Мультиагентные системы координируют несколько агентов для общей цели. Способы: hierarchical (менеджер назначает задачи), peer-to-peer (агенты общаются), blackboard (общая доска), auction (торговля задачами). Выбор зависит от сложности задачи и требований к адаптивности.

### 22. Коммуникация между агентами
**Вопрос:** Как агенты обмениваются информацией в мультиагентной системе?

**Ответ:** Коммуникация между агентами может быть:

1. **Synchronous (синхронная)** — агент A отправляет сообщение агенту B и ждёт ответа. Простая, но медленная (блокирует A).

2. **Asynchronous (асинхронная)** — агент A отправляет сообщение и продолжает работать. Быстрее, но нужно обрабатывать ответ позже. Обычно используется очередь сообщений (message queue).

3. **Broadcast** — один агент отправляет сообщение всем остальным. Используется для оповещений.

4. **Protocol-based** — агенты следуют определённому протоколу общения (например, Request-Reply, Publish-Subscribe).

```python
# Асинхронная коммуникация с очередью
from asyncio import Queue

message_queue = Queue()

async def agent_a():
    await message_queue.put({"from": "A", "task": "find_data"})
    # продолжает работу, не ждя ответа
    
async def agent_b():
    msg = await message_queue.get()  # получает задачу
    result = process(msg["task"])
    await message_queue.put({"from": "B", "result": result})
```

| Тип | Скорость | Сложность | Использование |
|---|---|---|---|
| **Synchronous** | Медленнее | Проще | Простые системы |
| **Asynchronous** | Быстрее | Сложнее | Масштабируемые системы |
| **Broadcast** | Быстро | Средне | Оповещения |
| **Protocol-based** | Среднее | Средне | Структурированная кооперация |

**Ответ для интервьюера:** Синхронная коммуникация проста, но блокирует агента. Асинхронная быстрее (через очередь сообщений), но сложнее. Broadcast отправляет всем. Protocol-based общение следует определённым правилам. Выбор зависит от требований к масштабируемости.

### 23. Специализация и разделение ответственности
**Вопрос:** Как разделить ответственность между агентами в мультиагентной системе?

**Ответ:** Хорошая специализация повышает эффективность: вместо одного "всемогущего" агента, несколько специализированных агентов лучше справляются с разными задачами.

Примеры специализации:
1. **По функции** — один агент отвечает за поиск, другой за анализ, третий за презентацию результатов.
2. **По домену** — HR-агент отвечает за кадровые вопросы, IT-агент за технические.
3. **По инструментам** — один агент работает с БД, другой с API, третий с файлами.
4. **По сложности** — simple-solver для простых задач, complex-solver для сложных.

Преимущества:
- Каждый агент может быть оптимизирован для своей задачи
- Легче отлаживать и тестировать
- Можно использовать специализированные LLM или инструменты
- Ошибка в одном не срывает всю систему

```python
class SearchAgent:
    tools = [google_search, internal_db_search]
    
class AnalysisAgent:
    tools = [summarizer, key_points_extractor, sentiment_analyzer]
    
class PresenterAgent:
    tools = [format_pdf, create_slides, send_email]

# Мультиагентный workflow
search_result = SearchAgent.search(query)
analysis = AnalysisAgent.analyze(search_result)
presentation = PresenterAgent.present(analysis)
```

**Ответ для интервьюера:** Специализация агентов повышает эффективность. Можно разделить по функциям (поиск, анализ, презентация), доменам (HR, IT) или инструментам. Каждый агент лучше справляется с узкой задачей, чем один агент со всем.

### 24. Emergent behavior в мультиагентных системах
**Вопрос:** Что такое emergent behavior и как оно возникает в мультиагентных системах?

**Ответ:** **Emergent behavior** — свойства и поведение, которые возникают из взаимодействия простых компонентов и не могут быть предсказаны из анализа отдельных компонентов. Например, "интеллект толпы" — когда несколько менее умных агентов, взаимодействуя, демонстрируют коллективный интеллект.

Примеры в мультиагентных системах:
1. **Синхронизация** — агенты синхронизируют свои действия без централизованного приказа (как птицы в стае).
2. **Самоорганизация** — система организует себя без внешнего управления (как муравьи находят оптимальный путь).
3. **Конфликт и консенсус** — агенты с разными целями находят компромисс.

Это может быть как полезным (быстрое решение сложных задач), так и вредным (нежелательное поведение, недостижимые цели).

| Явление | Причина | Результат |
|---|---|---|
| **Синхронизация** | Локальные правила взаимодействия | Коллективное движение |
| **Самоорганизация** | Распределённая информация | Оптимальные решения |
| **Флокирование** | Стремление к консенсусу | Иногда полезное, иногда вредное |

**Ответ для интервьюера:** Emergent behavior — свойства, которые возникают из взаимодействия агентов и не предсказуемы из анализа отдельных агентов. Примеры: синхронизация, самоорганизация, консенсус. Может быть полезным (быстрое решение) или вредным (нежелательное поведение). Важно мониторить в сложных системах.

### 25. Hallucinations в LLM и способы их снижения
**Вопрос:** Что такое галлюцинации и как их предотвращать?

**Ответ:** **Hallucinations (галлюцинации)** — когда LLM генерирует информацию, которая звучит правдоподобно, но на самом деле неправда, неточна или вообще выдумана. Это происходит, потому что модель предсказывает статистически вероятные токены, а не проверяет факты. Примеры: выдуманные цитаты, несуществующие названия, неправильные даты.

Способы снижения:
1. **RAG (Retrieval-Augmented Generation)** — предоставить LLM релевантные документы, чтобы она ответила на основе фактов, а не выдумки.
2. **Few-shot examples** — показать правильные примеры ответов.
3. **Prompt engineering** — явно просить ответить на основе предоставленной информации, не добавлять.
4. **Chain-of-Thought** — просить объяснить рассуждения, это помогает выявить ошибки.
5. **Fact-checking** — добавить фильтр, который проверяет факты в ответе.
6. **Temperature reduction** — снизить температуру, чтобы модель была консервативнее.
7. **Explicit instruction** — "Если ты не знаешь, скажи 'я не знаю'".

```python
prompt = """На основе этого текста ответь на вопрос. 
Если информация не в тексте, ответь "я не знаю".

Текст: {context}

Вопрос: {question}"""
```

| Способ | Эффективность | Сложность |
|---|---|---|
| **RAG** | Очень высокая | Средняя |
| **Few-shot** | Средняя | Низкая |
| **Prompt engineering** | Средняя | Низкая |
| **Chain-of-Thought** | Средняя | Низкая |
| **Fact-checking** | Высокая | Высокая |

**Ответ для интервьюера:** галлюцинации — когда LLM выдумывает факты. Способы борьбы: RAG (ответ на основе документов), CoT (объяснить рассуждения), низкая температура, явное указание "скажи 'я не знаю'". RAG самый эффективный, так как привязывает ответ к фактам.

### 26. Guardrails и constraints в агентах
**Вопрос:** Как устанавливать guardrails для предотвращения нежелательного поведения агента?

**Ответ:** **Guardrails** — ограничения и правила, которые предотвращают агент от нежелательного поведения (например, раскрытие конфиденциальной информации, оскорбления, выполнение опасных действий).

Типы guardrails:
1. **Input validation** — проверить входные данные перед обработкой (фильтр спама, проверка прав).
2. **Output filtering** — фильтровать ответы перед отправкой пользователю (удалять конфиденциальное, проверять тон).
3. **Action constraints** — ограничить действия, которые агент может выполнять (только чтение, не запись).
4. **Prompt-based guardrails** — добавить инструкции в систему промпт. "Никогда не раскрывай пароли", "Будь вежливым".
5. **Model-based detection** — использовать отдельную модель для обнаружения нежелательных ответов.

```python
class Agent:
    def guardrails(self, action):
        # Проверить, не опасное ли действие
        if action.type == "delete_database":
            raise ValueError("Agent не может удалять БД")
        
        # Проверить, не раскрывается ли конфидециальное
        if any(secret in action.params for secret in self.secrets):
            raise ValueError("Конфиденциальная информация")
        
        return True

def filter_output(response):
    """Удалить конфиденциальное из ответа"""
    for secret in secrets_list:
        response = response.replace(secret, "[REDACTED]")
    return response
```

| Guardrail                 | Эффективность | Использование         |
| ------------------------- | ------------- | --------------------- |
| **Input validation**      | Высокая       | Всегда                |
| **Output filtering**      | Средняя       | Для публичных ответов |
| **Action constraints**    | Очень высокая | Для опасных операций  |
| **Prompt-based**          | Средняя       | Базовое поведение     |
| **Model-based detection** | Высокая       | Сложное поведение     |

**Ответ для интервьюера:** Ограничения предотвращают нежелательное поведение. Типы: input validation (проверка входа), output filtering (фильтр выхода), action constraints (ограничить действия), prompt-based (инструкции), model-based detection (отдельная модель). Комбинация нескольких guardrails — лучше всего.

### 27. Оценка качества ответов (BLEU, ROUGE, пользовательская метрика)
**Вопрос:** Как оценивать качество ответов AI-агента?

**Ответ:** Качество ответов можно оценивать несколькими способами:

1. **BLEU (Bilingual Evaluation Understudy)** — метрика для сравнения сгенерированного текста с эталонным. Считает совпадения n-грамм. Хороша для машинного перевода, но может быть недостаточна для разговора.

2. **ROUGE (Recall-Oriented Understudy for Gisting Evaluation)** — похожа на BLEU, но ориентирована на recall. Часто используется для оценки суммаризации.

3. **Пользовательская оценка (Human Evaluation)** — люди оценивают качество. Дорого, но самое надёжное. Обычно оценивают точность, полнотность, тон, структуру.

4. **Метрики, специфичные для задачи**:
   - Для QA: точность (правильно/неправильно), частичная правильность
   - Для рекомендаций: relevance, diversity, serendipity
   - Для кода: syntactic correctness, runtime correctness

5. **LLM-based evaluation** — используют другую LLM для оценки качества. Быстро, дешево, хорошо коррелирует с человеческой оценкой.

```python
from rouge_score import rouge_scorer

# Оценка ROUGE
scorer = rouge_scorer.RougeScorer(['rouge1', 'rougeL'], use_stemmer=True)
scores = scorer.score(reference, generated)

# Пользовательская оценка
evaluation = {
    "correctness": 8,  # 1-10
    "completeness": 7,
    "tone": 9,
    "structure": 8
}
overall_score = sum(evaluation.values()) / len(evaluation)
```

| Метрика | Автоматическая | Полнота | Использование |
|---|---|---|---|
| **BLEU** | ✅ | Средняя | Машинный перевод |
| **ROUGE** | ✅ | Средняя | Суммаризация |
| **Human** | ❌ | Очень высокая | Критичные системы |
| **LLM-based** | ✅ | Высокая | Быстрая оценка |
| **Task-specific** | ✅ | Очень высокая | Конкретная задача |

**Ответ для интервьюера:** BLEU и ROUGE — автоматические метрики для сравнения с эталоном. Пользовательская оценка (люди) — самая надёжная, но дорогая. LLM-based evaluation — хороший компромисс. Лучше всего использовать комбинацию метрик и регулярно собирать feedback от пользователей.

### 28. Обработка ошибок и fallbacks
**Вопрос:** Как обрабатывать ошибки в AI-агентах?

**Ответ:** Обработка ошибок критична для надёжности агента. Стратегии:

1. **Retry logic** — повторить запрос, если он не удался. Часто API временно недоступна.

2. **Fallback strategy** — если первый способ не сработал, попробовать второй. Например, если поиск в DB не вернул результатов, попробовать поиск в интернете.

3. **Graceful degradation** — если что-то не сработало, выдать менее точный, но работающий ответ. Например, вернуть кэшированный результат вместо свежих данных.

4. **Error recovery** — попытаться восстановиться от ошибки. Например, если API требует повторную авторизацию, повторить логин.

5. **User notification** — сообщить пользователю об ошибке честно. "Я не смог найти свежую информацию, использую кэшированные данные от вчера".

```python
class Agent:
    async def call_api(self, url, retries=3):
        for attempt in range(retries):
            try:
                return await api.get(url)
            except TimeoutError:
                if attempt < retries - 1:
                    await asyncio.sleep(2 ** attempt)  # exponential backoff
                else:
                    # Fallback: вернуть кэшированный результат
                    return self.cache.get(url)
            except AuthError:
                # Error recovery: повторить логин
                await self.re_authenticate()
                continue
```

| Стратегия | Надёжность | Сложность |
|---|---|---|
| **Retry** | Низкая | Низкая |
| **Fallback** | Средняя | Средняя |
| **Graceful degradation** | Высокая | Средняя |
| **Error recovery** | Высокая | Высокая |

**Ответ для интервьюера:** Обработка ошибок включает retry logic (повторить), fallback strategy (альтернативный способ), graceful degradation (менее точный, но рабочий результат), error recovery (восстановиться от ошибки). Важно честно сообщать пользователю об ошибках.

### 30. LangChain vs LangGraph: выбор фреймворка
**Вопрос:** В чём разница между LangChain и LangGraph? Когда использовать каждый?

**Ответ:** **LangChain** и **LangGraph** — оба фреймворки от LangChain Inc. для работы с LLM, но они решают разные задачи.

**LangChain** — общий фреймворк для построения приложений с LLM. Предоставляет:
- LLM обёртки (unified interface для разных моделей)
- Prompt templates
- Chains (последовательные операции)
- Agents с инструментами
- Memory management
- Retrievers для RAG

LangChain хорош для простых и средних приложений, линейных workflows.

**LangGraph** — специализированный фреймворк для построения сложных agent-based систем. Фокусируется на:
- Graph-based workflows (узлы и грани вместо линейных chains)
- Сложная логика принятия решений
- Multi-agent coordination
- Циклические процессы (loops)
- State management между узлами
- Визуализация workflows

LangGraph лучше для сложных систем с ветвлениями, циклами, мультиагентной координацией.

**Сравнение:**

| Критерий                                     | LangChain              | LangGraph                 |
| -------------------------------------------- | ---------------------- | ------------------------- |
| **Сложность**                                | Простая-средняя        | Сложная                   |
| **Структура**                                | Линейные chains        | Graph с узлами/гранями    |
| **Loops (циклы)**                            | Ограниченная поддержка | Полная поддержка          |
| **State management (управление состоянием)** | Простой                | Продвинутый               |
| **Multi-agent**                              | Базовая поддержка      | Встроенная поддержка      |
| **Лучше всего для**                          | RAG, simple chatbots   | Complex agents, workflows |

**Кодовый пример:**

```python
# LangChain — линейный workflow
from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate

chain = LLMChain(llm=llm, prompt=PromptTemplate(...))
result = chain.run(input="question")

# LangGraph — graph-based workflow
from langgraph.graph import StateGraph, START, END

graph = StateGraph(State)
graph.add_node("agent", agent_node)
graph.add_node("tools", tool_node)
graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", should_continue)
graph.add_edge("tools", "agent")
graph.add_edge("agent", END)

workflow = graph.compile()
result = workflow.invoke({"messages": [...]})
```

**Когда что выбирать:**

**Используй LangChain если:**
- Простой чатбот или QA система
- RAG приложение со стандартным workflow
- Проект не требует сложной логики
- Нужна быстрая разработка

**Используй LangGraph если:**
- Многоагентная система
- Нужны циклы и ветвления
- Сложная логика принятия решений
- Нужна полная визуализация workflow
- Требуется сложное управление состоянием

**Ответ для интервьюера:** LangChain — фреймворк для построения простых и средних приложений с LLM (chains, agents, RAG). LangGraph — специализированный фреймворк для сложных агентов с graph-based workflows, поддержкой циклов и мультиагентной координации. LangChain лучше для стандартных приложений, LangGraph — для сложных систем.

### 32. AgentLoop и orchestration
**Вопрос:** Как организовать agent loop и оркестрировать несколько агентов?

**Ответ:** **Agent loop** — основной цикл, где агент воспринимает, рассуждает, действует. **Orchestration** — координация нескольких агентов.

Простой agent loop:

```python
class Agent:
    def run(self, task):
        self.memory.add(task)
        
        while not self.done:
            # Perception
            state = self.get_state()
            
            # Reasoning
            action = self.llm.decide_action(state)
            
            # Action
            result = self.execute(action)
            self.memory.add(result)
            
            self.iteration += 1
            if self.iteration >= self.max_iterations:
                break
        
        return self.get_result()
```

Orchestration нескольких агентов:

```python
class Orchestrator:
    def run(self, task):
        # Разложить задачу
        subtasks = self.planner.plan(task)
        
        # Назначить агентам
        results = {}
        for subtask in subtasks:
            agent = self.select_agent(subtask)
            results[subtask] = await agent.run(subtask)  # параллельно
        
        # Собрать результаты
        final = self.aggregator.aggregate(results)
        return final
```

| Компонент | Назначение |
|---|---|
| **Perception** | Сбор информации о состоянии |
| **Reasoning** | Принятие решения о действии |
| **Action** | Выполнение действия |
| **Orchestrator** | Координация нескольких агентов |

**Ответ для интервьюера:** Agent loop — это когда агент постоянно крутится в цикле: сначала посмотрит что происходит, потом подумает что делать, потом это сделает. И так по кругу. Оркестрация — это управление несколькими агентами сразу. Суть проста: берёшь большую задачу, режешь её на части, даёшь каждому агенту свой кусок (они работают параллельно, не ждут друг друга), а потом собираешь все результаты вместе. Типа как команда программистов — каждый кодит свой модуль, но в итоге всё должно скомпилироваться.

### 33. Асинхронное выполнение в агентах
**Вопрос:** Как использовать асинхронное выполнение в AI-агентах для скорости?

**Ответ:** Асинхронное выполнение позволяет нескольким операциям выполняться параллельно, не блокируя друг друга. Особенно полезно при работе с внешними API.

```python
import asyncio

class AsyncAgent:
    async def call_multiple_tools(self, tools):
        """Вызвать несколько инструментов параллельно"""
        tasks = [asyncio.create_task(tool()) for tool in tools]
        results = await asyncio.gather(*tasks)
        return results
    
    async def run_with_timeout(self, task, timeout=5):
        """Выполнить с таймаутом"""
        try:
            result = await asyncio.wait_for(task, timeout=timeout)
            return result
        except asyncio.TimeoutError:
            return self.fallback_result()

# Использование
agent = AsyncAgent()
results = await agent.call_multiple_tools([
    agent.search(),
    agent.fetch_db(),
    agent.call_api()
])
```

| Подход | Скорость | Сложность |
|---|---|---|
| **Синхронный** | Медленнее (N инструментов = N времени) | Проще |
| **Асинхронный** | Быстрее (N инструментов ≈ время самого медленного) | Сложнее |

**Ответ для интервьюера:** Асинхронное выполнение позволяет вызывать несколько инструментов параллельно. Используем `asyncio.gather()` для параллельных задач. Особенно полезно при API вызовах, где ждать приходится долго. Скорость N инструментов ≈ время самого медленного, вместо суммы всех.

### 34. Кэширование запросов к LLM

**Вопрос:** Как оптимизировать скорость и стоимость через кэширование запросов к LLM?

**Ответ:** Кэширование — сохранение результатов часто повторяемых запросов, чтобы повторно не вызывать LLM.

**Типы кэширования:**

1. **Exact caching** — кэшировать только точные совпадения. Быстро, но менее гибко. Идеален для повторяющихся вопросов.

2. **Semantic caching** — кэшировать на основе смысла запроса, а не точного совпадения. Если новый запрос семантически похож на кэшированный (similarity > 0.9), вернуть кэшированный результат.

3. **Prompt-level caching** — некоторые LLM (например, Claude, GPT-4) поддерживают caching на уровне промпта. Сохраняются промежуточные представления, экономит токены на повторные запросы.

**Базы данных для кэширования:**

| БД                        | Использование            | Скорость                  | Персистентность                   | Когда использовать                                 |
| ------------------------- | ------------------------ | ------------------------- | --------------------------------- | -------------------------------------------------- |
| **Redis**                 | In-memory cache          | Очень быстро (~1ms)       | ❌ Нет (если не настроить RDB/AOF) | Горячее кэширование, сессии                        |
| **Memcached**             | In-memory cache          | Очень быстро              | ❌ Нет                             | Распределённое кэширование                         |
| **PostgreSQL + pgvector** | Vector similarity search | Быстро с индексами (HNSW) | ✅ Да                              | RAG, semantic search, долгосрочное кэширование     |
| **SQLite**                | Local persistent cache   | Быстро                    | ✅ Да                              | Development, single-machine, локальное кэширование |
| **MongoDB**               | Document cache + TTL     | Средне                    | ✅ Да                              | Гибкая схема, сложные объекты                      |
| **DuckDB**                | In-memory analytics      | Очень быстро              | ✅ Опционально                     | Кэш с аналитикой, OLAP запросы                     |

```python
# Exact caching с Redis
from functools import lru_cache
import redis
import hashlib
import json

class RedisCachedLLM:
    def __init__(self):
        self.redis = redis.Redis(host='localhost', port=6379)
        self.ttl = 3600 * 24  # 24 часа
    
    async def call(self, prompt):
        # Compute hash of prompt
        key = hashlib.sha256(prompt.encode()).hexdigest()
        
        # Проверить Redis
        cached = self.redis.get(f"llm:{key}")
        if cached:
            return json.loads(cached)
        
        # Call LLM
        result = await llm.generate(prompt)
        
        # Сохранить в Redis
        self.redis.setex(f"llm:{key}", self.ttl, json.dumps(result))
        return result

# Semantic caching с PostgreSQL + pgvector
import psycopg2
from pgvector.psycopg2 import register_vector

class SemanticCachedLLM:
    def __init__(self, db_connection):
        self.db = db_connection
        register_vector(self.db)
    
    async def call(self, query):
        # 1. Embed запрос
        query_embedding = embed_model.encode(query)
        
        # 2. Поиск похожего кэша в PostgreSQL (pgvector)
        cursor = self.db.cursor()
        cursor.execute("""
            SELECT response, similarity FROM llm_cache
            WHERE embedding <-> %s < 0.1
            ORDER BY embedding <-> %s
            LIMIT 1
        """, (query_embedding, query_embedding))
        
        result = cursor.fetchone()
        if result:
            response, similarity = result
            if similarity > 0.9:  # Достаточно похожа
                return response
        
        # 3. Иначе вызвать LLM
        response = await llm.call(query)
        
        # 4. Сохранить в БД
        cursor.execute("""
            INSERT INTO llm_cache (query, embedding, response)
            VALUES (%s, %s, %s)
        """, (query, query_embedding, response))
        self.db.commit()
        
        return response
```

**Кэширование в RAG-агентах:**

RAG состоит из 3 этапов, каждый можно оптимизировать через кэширование:

```
Query (вопрос селлера)
  ↓
[1. Retrieval] → Поиск в БД знаний (кэшируется по запросу)
  ↓
[2. Augmentation] → Сборка промпта с документами (кэшируется полностью)
  ↓
[3. Generation] → LLM генерирует ответ (кэшируется)
  ↓
Response
```

**Пример: RAG кэширование в "Мир хобби"**

```python
class RAGWithCaching:
    def __init__(self):
        # Кэш для retrieval результатов
        self.redis = redis.Redis()
        # Кэш для LLM вызовов
        self.llm_cache = SemanticCachedLLM(db_connection)
    
    async def ask(self, question: str):
        # Этап 1: Retrieval (кэшируется по хешу вопроса)
        retrieval_key = hashlib.sha256(question.encode()).hexdigest()
        cached_docs = self.redis.get(f"retrieval:{retrieval_key}")
        
        if cached_docs:
            documents = json.loads(cached_docs)
        else:
            # Поиск в pgvector базе знаний
            documents = vector_db.search(question, top_k=5)
            self.redis.setex(f"retrieval:{retrieval_key}", 3600, 
                           json.dumps(documents))
        
        # Этап 2: Augmentation (собрать промпт)
        context = "\n".join([doc.text for doc in documents])
        augmented_prompt = f"""На основе этих документов ответь:
Документы:
{context}

Вопрос: {question}"""
        
        # Этап 3: Generation (кэшируется семантически)
        # Если похожий вопрос уже был, вернуть кэшированный ответ
        response = await self.llm_cache.call(augmented_prompt)
        
        return response

# Результат: "Мир хобби" экономит:
# - Retrieval: не пересчитывает embeddings для одинаковых вопросов
# - LLM calls: не вызывает LLM для похожих вопросов
# - Пропускная способность: ~1500 req/day вместо 3000+ без кэша
```

**Экономия через кэширование:**

```
Без кэширования:
  1500 запросов/день × $0.002 за вызов LLM = $3/день
  
С exact caching (50% попадания):
  1500 × 0.5 = 750 реальных вызовов = $1.50/день ✓ (Сэкономили $1.50)
  
С semantic + exact (70% попадания):
  1500 × 0.3 = 450 реальных вызовов = $0.90/день ✓ (Сэкономили $2.10)
  
С prompt-level caching (экономия токенов):
  Система с длинным system prompt (2000 токенов):
  Кэширование сэкономит на всех повторениях = -70% стоимости
```

| Тип | Скорость | Экономия | Гибкость | Сложность |
|---|---|---|---|---|
| **Exact** | Очень быстро | Средняя (50%) | Низкая | Низкая |
| **Semantic** | Быстро | Средняя (30-50%) | Высокая | Средняя |
| **Prompt-level** | Быстро | Очень высокая (70%) | Высокая | Средняя |
| **Redis** | ~1ms | Для горячих данных | - | Низкая |
| **PostgreSQL** | ~50ms | Для точного поиска | - | Средняя |

**Ответ для интервьюера:** Кэширование экономит токены и деньги. Есть несколько типов: exact (точное совпадение, простой), semantic (похожие запросы, гибче), prompt-level (экономит токены системного промпта). Для хранения используем Redis (горячее кэширование, быстро), PostgreSQL с pgvector (персистентное, для RAG), SQLite (локальное). В RAG агентах кэшируются все три этапа: retrieval (поиск документов), augmentation (сборка промпта), generation (ответ LLM). В "Мир хобби" благодаря кэшированию получается 70% экономия и такая же скорость, потому что похожие вопросы селлеров часто повторяются.

### 35. Cost optimization при работе с LLM API
**Вопрос:** Как оптимизировать стоимость при работе с LLM API?

**Ответ:** LLM API дорогие, стоимость зависит от числа токенов. Стратегии оптимизации:

1. **Выбор модели** — более дешёвые модели для простых задач (GPT-3.5 вместо GPT-4), локальные модели (LLaMA) если возможно.

2. **Prompt optimization** — писать лаконичнее, убирать лишнее. Короче промпт = меньше токенов.

3. **Batch processing** — обрабатывать несколько запросов одним вызовом, если возможно.

4. **Caching** — как обсуждали выше.

5. **Token limits** — ограничить max_tokens в промпте, чтобы не платить за слишком длинные ответы.

6. **Rate limiting** — не перегружать API частыми вызовами.

```python
class CostOptimizedAgent:
    def __init__(self):
        self.model = "gpt-3.5-turbo"  # Дешевле, чем GPT-4
        self.cache = {}
    
    async def call(self, prompt, max_tokens=100):
        # Оптимизировать промпт
        optimized_prompt = self.optimize_prompt(prompt)
        
        # Проверить кэш
        if optimized_prompt in self.cache:
            return self.cache[optimized_prompt]
        
        # Вызвать LLM с ограничением токенов
        result = await llm.create(
            model=self.model,
            messages=[{"role": "user", "content": optimized_prompt}],
            max_tokens=max_tokens  # Ограничить расходы
        )
        
        self.cache[optimized_prompt] = result
        return result
```

| Стратегия | Экономия | Эффект |
|---|---|---|
| **Дешёвая модель** | 10-90% | Медленнее, менее точно |
| **Prompt optimization** | 20-50% | Требует времени |
| **Caching** | 50-90% | Требует расходов памяти |
| **Token limits** | 20-40% | Может обрезать ответы |

**Ответ для интервьюера:** Оптимизировать стоимость можно через: выбор более дешёвой модели, сокращение промпта, кэширование, ограничение токенов, batch processing (несколько запросов). Комбинация нескольких методов даёт наибольшую экономию.

### 36. Monitoring, logging и отладка агентов
**Вопрос:** Как мониторить и отлаживать AI-агенты в продакшене?

**Ответ:** Мониторинг критичен для надёжности и оптимизации агентов.

Что мониторить:
1. **Performance** — время выполнения, количество итераций, использованные токены.
2. **Quality** — точность ответов, успешность задач, hallucination rate.
3. **Errors** — failed actions, timeouts, API errors.
4. **Cost** — количество потраченных токенов, стоимость API вызовов.

```python
import logging
from datetime import datetime

class MonitoredAgent:
    def __init__(self):
        self.logger = logging.getLogger(__name__)
        self.metrics = {
            "total_tasks": 0,
            "successful_tasks": 0,
            "total_tokens": 0,
            "errors": []
        }
    
    async def run(self, task):
        start_time = datetime.now()
        
        try:
            result = await self._execute(task)
            
            # Логирование успеха
            self.logger.info(f"Task completed: {task}, time: {datetime.now() - start_time}")
            self.metrics["successful_tasks"] += 1
            
            return result
        
        except Exception as e:
            # Логирование ошибки
            self.logger.error(f"Task failed: {task}, error: {str(e)}")
            self.metrics["errors"].append((task, str(e)))
            raise
        
        finally:
            self.metrics["total_tasks"] += 1
    
    def get_metrics(self):
        return {
            "success_rate": self.metrics["successful_tasks"] / self.metrics["total_tasks"],
            "avg_cost": self.metrics["total_tokens"] / self.metrics["total_tasks"],
            **self.metrics
        }
```

| Метрика | Цель | Действие |
|---|---|---|
| **Success rate** | > 95% | Alert if < 95% |
| **Avg response time** | < 5s | Optimize if > 5s |
| **Error rate** | < 1% | Investigate if > 1% |
| **Cost per task** | Минимизировать | Optimize prompt/model |

**Ответ для интервьюера:** Мониторить performance (время, токены), quality (точность), errors, cost. Используем logging для всех операций, собираем метрики, ставим alerts на аномалии. Регулярно анализируем логи, чтобы выявить проблемы и оптимизировать.

### 37. Latency optimization и streaming responses
**Вопрос:** Как оптимизировать latency и использовать streaming для улучшения пользовательского опыта?

**Ответ:** **Latency** — задержка от запроса до ответа. В LLM это может быть заметно (несколько секунд). **Streaming** — отправлять ответ частями по мере генерации, вместо ожидания полного ответа.

Способы снижения latency:
1. **Streaming responses** — LLM генерирует токены, мы сразу отправляем их клиенту (вместо ожидания полного ответа).
2. **Parallel processing** — запустить несколько операций параллельно.
3. **Caching** — вернуть кэшированный результат вместо нового запроса.
4. **Model optimization** — использовать более быструю модель.

```python
async def stream_response(prompt):
    """Стримить ответ по мере генерации"""
    async for chunk in llm.stream(prompt):
        # Отправить пользователю прямо сейчас
        yield chunk

# Использование в FastAPI
@app.get("/stream")
async def stream_endpoint(query: str):
    return StreamingResponse(
        stream_response(query),
        media_type="text/event-stream"
    )
```

Преимущества streaming:
- Пользователь видит начало ответа сразу, не ждёт конца
- Воспринимается быстрее (даже если реальное время одинаково)
- Лучше для mobile (может начать показывать ответ до полной загрузки)

| Подход | Latency | UX |
|---|---|---|
| **Без streaming** | Ждёт полный ответ | Кажется медленно |
| **Streaming** | Ждёт первый токен | Кажется быстро |

**Ответ для интервьюера:** Streaming отправляет ответ частями по мере генерации вместо ожидания полного ответа. Первый токен приходит быстрее (меньше latency), пользователь сразу видит начало ответа, что кажется быстрее. Особенно полезно для длинных ответов и мобильной версии.

### 38. SSE (Server-Sent Events) для streaming в AI-агентах

**Вопрос:** Что такое SSE и как его использовать для streaming ответов от AI-агентов?

**Ответ:** **SSE (Server-Sent Events)** — протокол для отправки данных от сервера клиенту в режиме реального времени (streaming) через HTTP подключение. Server отправляет данные в формате `data: ...\n\n`, а Browser слушает и обновляет интерфейс по мере получения данных.

**Как работает SSE:**
```
Browser → (HTTP GET /stream) → Server
         ← (data: token1\n\n) ← (продолжает слушать)
         ← (data: token2\n\n)
         ← (data: token3\n\n)
         ← ([DONE]\n\n) и закрывает соединение
```

**Основные характеристики:**
- Односторонняя коммуникация (только Server → Client)
- Автоматическое переподключение при разрыве соединения
- Простой протокол (HTTP, не нужно WebSocket)
- Идеален для streaming LLM ответов
- Встроенная поддержка в браузерах (EventSource API)

**Примеры использования SSE:**
1. **Streaming LLM ответов** — отправлять токены по мере генерации
2. **Real-time notifications** — живые уведомления от сервера
3. **Data feeds** — котировки, погода, биржевые данные
4. **Progress tracking** — отслеживание прогресса long-running операций

**Code example: FastAPI + SSE для RAG-ассистента**

```python
# Backend: FastAPI

### 39. Метрики производительности AI-агентов: как считать и мониторить

**Вопрос:** Как считать и мониторить метрики производительности AI-агентов (RPS, latency, p95, p99)?

**Ответ:** **Метрики производительности** — количественные показатели, характеризующие работу системы. Для AI-агентов критичны: пропускная способность (RPS), задержка (latency), надёжность (uptime), качество (accuracy).

**Основные метрики:**

1. **RPS (Requests Per Second)** или **RPD (Requests Per Day)**
   - Сколько запросов система обрабатывает в единицу времени
   - Пример: "~1500 запросов в сутки" = 1500 / (24*3600) ≈ 0.017 RPS

2. **Latency** — время от запроса до ответа
   - **Min latency** — самый быстрый ответ
   - **Max latency** — самый медленный ответ
   - **Median (p50)** — 50% запросов быстрее, 50% медленнее
   - **p95** — 95% запросов обработаны быстрее, чем это значение
   - **p99** — 99% запросов быстрее (самые медленные случаи)
   - Пример: "p95 latency < 2 сек" = 95% запросов получают ответ быстрее 2 секунд

3. **Throughput** — объём обработанных данных в единицу времени (MB/s, tokens/s)

4. **Error rate** — процент ошибочных запросов

5. **Uptime** — процент времени, когда система доступна

**Как считать метрики на примере RAG-ассистента "Мир хобби":**

```python
import time
import statistics
from datetime import datetime, timedelta
from collections import defaultdict

class MetricsCollector:
    """Сборщик метрик для AI-ассистента"""

    def __init__(self):
        self.request_times = []
        self.requests_per_day = defaultdict(int)
        self.error_count = 0
        self.total_requests = 0
        self.start_time = datetime.now()

    def log_request(self, request_id: str, query: str, response_time: float,
                   success: bool, tokens_used: int = 0):
        """Логировать запрос с метриками"""
        self.total_requests += 1
        self.request_times.append(response_time)
        
        today = datetime.now().date()
        self.requests_per_day[today] += 1
        
        if not success:
            self.error_count += 1

    def get_latency_percentiles(self):
        """Получить процентили latency"""
        if not self.request_times:
            return {}
        
        sorted_times = sorted(self.request_times)
        n = len(sorted_times)
        
        return {
            'min': min(sorted_times),
            'max': max(sorted_times),
            'mean': statistics.mean(sorted_times),
            'median': statistics.median(sorted_times),
            'p95': sorted_times[int(n * 0.95)],
            'p99': sorted_times[int(n * 0.99)],
        }

    def get_rps(self):
        """Получить RPS (запросов в секунду)"""
        elapsed = (datetime.now() - self.start_time).total_seconds()
        return self.total_requests / elapsed if elapsed > 0 else 0

    def get_error_rate(self):
        """Получить процент ошибок"""
        if self.total_requests == 0:
            return 0
        return (self.error_count / self.total_requests) * 100

# Пример вывода:
# Всего запросов: 1500
# RPS (текущий): 0.017 req/s
# Latency p95: 1950.8ms (95% запросов < 2 сек)
# Latency p99: 4120.2ms (99% запросов < 4.1 сек)
# Ошибки: 15/1500 (1.00%)
```

**Интерпретация метрик "1500 запросов, p95 < 2 сек":**

```
За день обработано 1500 запросов
    ↓
1500 запросов / 86400 сек ≈ 0.017 RPS

Распределение latency:
└─ 1425 запросов (95%) обработаны < 2 сек ✓
└─ 75 запросов (5%) обработаны дольше (медленные хвосты)
```

### 40. Нагрузочное тестирование AI-систем: как определить максимум запросов?

**Вопрос:** Как провести нагрузочное тестирование AI-агента? Как определить, сколько максимум запросов в день может обработать система (на примере "Мир хобби")?

**Ответ:** **Нагрузочное тестирование (Load Testing)** — проверка производительности системы при постепенно возрастающей нагрузке, чтобы понять её пределы и найти узкие места.

**Цели нагрузочного тестирования:**
1. Найти максимум RPS (requests per second)
2. Измерить latency при разных нагрузках
3. Найти точку отказа системы
4. Определить, где деградирует производительность
5. Планировать capacity (сколько серверов нужно)

**Как провести тест с Locust (Python):**

```python
from locust import HttpUser, task, between
import time
import random

class RAGUser(HttpUser):
    wait_time = between(1, 5)

    @task
    def ask_rag(self):
        """Имитация пользователя, задающего вопрос"""
        questions = [
            "Как установить цену?",
            "Какие документы нужны?",
            "Как оптимизировать доставку?",
            "Какая комиссия?",
        ]

        question = random.choice(questions)
        start = time.time()

        response = self.client.post(
            "/ask",
            json={"question": question},
            timeout=10
        )

        latency = time.time() - start
        if response.status_code != 200:
            print(f"✗ Error {response.status_code}")

# Запуск:
# locust -f load_test.py --host=http://localhost:8000 -u 100 -r 10 -t 10m
# -u 100: 100 пользователей
# -r 10: добавлять 10 пользователей/сек
# -t 10m: тестировать 10 минут
```

**Инструменты для нагрузочного тестирования:**

| Инструмент | Язык | Простота | Когда использовать |
|---|---|---|---|
| **Locust** | Python | Низкая | Быстрые тесты, LLM API |
| **k6** | JavaScript | Низкая | Cloud тестирование |
| **vegeta** | Go | Очень низкая | CLI, быстро |

**Самый простой тест (vegeta):**

```bash
# Генерировать 100 запросов в секунду 10 минут
vegeta attack -rate=100 -duration=10m -body='{"question":"test"}' \
  http://localhost:8000/ask | vegeta report

# Результаты:
# Requests: 600000 (100 RPS)
# Latency p95: 480ms
# Status 200: 599500, Errors: 500
```

**Как определить максимум запросов в день:**

**Этап 1: Провести тест с растущей нагрузкой**

```
Phase 1: 10 RPS × 1 min
  → p95 latency: 1.2s ✓ Норма

Phase 2: 50 RPS × 1 min
  → p95 latency: 1.8s ✓ Норма

Phase 3: 100 RPS × 1 min
  → p95 latency: 2.5s ⚠ Начинает замедляться

Phase 4: 150 RPS × 1 min
  → p95 latency: 3.1s, Errors: 0.1% ⚠

Phase 5: 200 RPS × 1 min
  → p95 latency: 4.2s, Errors: 1.5% ✗ Критично

🛑 Breaking point: 200 RPS (система начинает отказывать)
```

**Этап 2: Найти Safe RPS**

- Breaking point: 200 RPS (когда error rate > 1% или p95 > 5s)
- Safe RPS: 100 RPS (где всё ещё норма, p95 < 2.5s, errors < 0.1%)

**Этап 3: Применить коэффициент безопасности (70% от safe)**

```
Safe RPS: 100
Коэффициент: 70%
Используемый RPS: 100 * 0.7 = 70 RPS
```

**Этап 4: Расчёт максимума в день**

```
Формула:
Requests per day = RPS * Seconds per day
                 = 70 * (24 * 3600)
                 = 70 * 86400
                 = 6,048,000 запросов/день

На один сервер максимум: 6M запросов/день
```

**Пример для "Мир хобби":**

```
Реальный трафик:
- Пиковые часы (9-18): 200 селлеров × 0.5 вопр/час = 100 вопр/час
- Ночные часы (18-9): 50 вопросов/час
- За день: 100*9 + 50*15 = 900 + 750 = 1650 ≈ 1500 (с кэшем)

Пиковый RPS:
- 100 вопросов/час = 100/3600 = 0.028 RPS

Тест показал:
- Safe RPS: 100
- Требуется: 0.028 RPS
- Запас: 100 / 0.028 = 3571x раз

Вывод: 
- Один микро-сервер справляется с большим запасом
- На 10x рост трафика: всё равно хватит одного сервера
- На 100x рост трафика: нужно 2 сервера
```

**Метрики для анализа:**

| Метрика | ✓ Хорошо | ⚠ Требует оптимизации | ✗ Критично |
|---|---|---|---|
| **p95 Latency** | < 2s | 2-5s | > 5s |
| **p99 Latency** | < 5s | 5-10s | > 10s |
| **Error Rate** | < 0.1% | 0.1-1% | > 1% |
| **CPU Usage** | < 70% | 70-85% | > 85% |
| **Memory** | Стабильна | Медленный рост | Утечка |

**Шаги для расчёта максимума:**

1. **Провести нагрузочный тест** (staging сервер)
   ```bash
   locust -f test.py --host=http://staging -u 200 -r 20 -t 30m
   ```

2. **Найти breaking point** — где система начинает отказывать

3. **Найти safe RPS** — максимум при нормальных метриках (p95 < 2.5s)

4. **Применить коэффициент 70%** — используемый RPS = safe * 0.7

5. **Рассчитать максимум:**
   ```
   Max requests/day = RPS * 86400
   ```

6. **Добавить серверы, если нужно:**
   ```
   Servers needed = (Required traffic) / (Max per server)
   ```

**Конкретный расчёт для "Мир хобби":**

```
Нагрузочный тест показал:
- Safe RPS: 100 (p95: 2.5s, errors: 0.1%)
- Максимум на день: 100 * 86400 = 8,640,000

Реальный трафик:
- Требуется: 1500 запросов/день = 0.017 RPS
- Это 0.017 / 100 = 0.017% от capacity

Вывод:
- На одном сервере можно обслужить 5760x больше трафика
- Цена: $10/мес за микро-сервер
- Может обслужить 500+ копий "Мир хобби"

Рекомендация по масштабированию:
- До 10K запросов/день: 1 микро-сервер ($10/мес)
- До 100K запросов/день: 1 small сервер ($20/мес)
- До 1M запросов/день: 2-3 medium сервера ($50-100/мес)
- До 10M запросов/день: load balancer + 5-10 серверов
```

**Ответ для интервьюера:** Нагрузочное тестирование — это постепенное увеличение нагрузки до отказа. Стартуем с 10 RPS, растём (50, 100, 200...) пока не упадёт latency > 5s или error rate > 1%. Находим safe RPS (где всё нормально), применяем коэффициент 70% и считаем: макс запросов = RPS × 86400. Для "Мир хобби" тест показал 100 safe RPS = 8.6М запросов/день, а нужно 1500 = 0.017 RPS. Это в 5760 раз ниже лимита! Огромный запас, значит один микро-сервер легко справляется и можно додавать фич без деградации. На 100x рост трафика понадобится 2 сервера.
