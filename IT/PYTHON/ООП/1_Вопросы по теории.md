## Основы ООП и его принципы

### 1. Четыре принципа ООП
**Вопрос:** Объясни четыре принципа ООП: инкапсуляция, наследование, полиморфизм, абстракция.

**Ответ:** Инкапсуляция — объединение данных и методов работы с ними в одном классе и ограничение прямого доступа к внутреннему состоянию (в Python — по соглашению через `_`/`__` и `@property`). Наследование — создание нового класса на основе существующего с переиспользованием и расширением его поведения. Полиморфизм — возможность единообразно работать с объектами разных классов через общий интерфейс, когда конкретное поведение определяется реальным типом объекта во время выполнения. Абстракция — выделение существенных характеристик объекта и сокрытие деталей реализации за понятным интерфейсом (в Python — через `abc.ABC` и абстрактные методы).

```python
from abc import ABC, abstractmethod

class Shape(ABC):              # абстракция
    @abstractmethod
    def area(self): ...

class Circle(Shape):            # наследование
    def __init__(self, r):
        self._r = r              # инкапсуляция
    def area(self):
        return 3.14 * self._r ** 2

for shape in [Circle(2), Circle(3)]:
    print(shape.area())          # полиморфизм — единый интерфейс area()
```

| Термин | Определение |
|---|---|
| Инкапсуляция | Сокрытие внутреннего состояния объекта |
| Наследование | Переиспользование поведения родительского класса |
| Полиморфизм | Единый интерфейс для объектов разных типов |
| Абстракция | Сокрытие деталей реализации за понятным интерфейсом |

**Ответ для интервьюера:** Инкапсуляция скрывает состояние объекта за методами, наследование переиспользует поведение родителя, полиморфизм позволяет работать с разными типами через общий интерфейс, а абстракция выделяет существенное и прячет детали реализации. В Python эти принципы реализуются менее строго, чем в языках со строгой типизацией, полагаясь во многом на соглашения и duck typing.

### 2. Класс vs объект
**Вопрос:** В чем разница между классом и объектом (экземпляром)?

**Ответ:** Класс — это шаблон (blueprint), описывающий структуру данных (атрибуты) и поведение (методы) для целой категории сущностей; объект (экземпляр) — конкретная реализация этого шаблона в памяти, со своими собственными значениями атрибутов. Один класс может породить сколько угодно объектов, каждый из которых независим по состоянию, но разделяет одну и ту же логику методов, определённую в классе. В Python сам класс тоже является объектом (экземпляром метакласса, обычно `type`), что подчёркивает единообразие модели данных языка.

```python
class Dog:
    def __init__(self, name):
        self.name = name
    def bark(self):
        return f"{self.name} гавкает"

rex = Dog("Рекс")      # объект — конкретный экземпляр
bobik = Dog("Бобик")   # другой независимый объект того же класса
```

| Термин | Определение |
|---|---|
| Класс | Шаблон, описывающий структуру и поведение |
| Объект (экземпляр) | Конкретная реализация класса в памяти |
| type | Метакласс, экземпляром которого является обычный класс |

**Ответ для интервьюера:** Класс — шаблон с описанием атрибутов и методов, объект — конкретный экземпляр этого шаблона со своим состоянием в памяти. Из одного класса можно создать произвольное число независимых по данным, но одинаковых по логике объектов.

### 3. Когда использовать наследование
**Вопрос:** Когда целесообразно использовать наследование?

**Ответ:** Наследование оправдано, когда между классами существует настоящее отношение "является" (is-a) — например, `Dog` является `Animal` — и подкласс действительно расширяет или уточняет поведение родителя, не нарушая его контракта (принцип подстановки Барбары Лисков). Если же связь скорее "имеет" (has-a) — например, `Car` имеет `Engine` — правильнее использовать композицию: включить объект как атрибут, а не наследоваться от него. Злоупотребление наследованием (особенно глубокими иерархиями) увеличивает связность кода и усложняет изменения, поэтому в спорных случаях предпочитают композицию ("favor composition over inheritance").

|Характеристика|Наследование (is-a)|Композиция (has-a)|
|---|---|---|
|**Отношение**|"Является" (Dog is Animal)|"Имеет" (Car has Engine)|
|**Связность**|Выше|Ниже|
|**Гибкость изменений**|Ниже (жёсткая иерархия)|Выше (можно подменить объект)|
|**Когда предпочесть**|Подкласс реально расширяет контракт родителя|В спорных случаях — "favor composition over inheritance"|

```python
# is-a — наследование уместно
class Animal:
    def speak(self): ...
class Dog(Animal):
    def speak(self):
        return "Гав"

# has-a — предпочтительнее композиция
class Engine:
    def start(self): ...
class Car:
    def __init__(self):
        self.engine = Engine()   # Car "имеет" Engine, а не "является" им
```

| Термин | Определение |
|---|---|
| is-a | Отношение "является" — кандидат на наследование |
| has-a | Отношение "имеет" — кандидат на композицию |
| Принцип подстановки Лисков | Подкласс не должен нарушать контракт родителя |

**Ответ для интервьюера:** Наследование уместно при отношении is-a, когда подкласс действительно расширяет поведение родителя, не нарушая его контракт. При отношении has-a или в спорных случаях предпочитаю композицию — она даёт меньшую связность и более гибкую архитектуру, чем глубокие иерархии наследования.

### 4. Инкапсуляция в Python
**Вопрос:** Почему в Python нет строгой инкапсуляции и как это компенсируется?

**Ответ:** В Python нет модификаторов доступа уровня языка (`private`/`protected`, как в Java или C++) — любой атрибут технически доступен извне класса, потому что философия языка ("we are all consenting adults here") полагается на соглашения, а не на принудительные запреты. Компенсируется это соглашением об именовании: одиночное подчёркивание `_attr` сигнализирует "внутреннее использование, не трогай снаружи" (но ничего не блокирует), двойное `__attr` включает name mangling, физически усложняя случайный доступ извне класса. Дополнительно используют `@property` для контролируемого доступа к атрибуту через геттеры/сеттеры с валидацией, что даёт инкапсуляцию поведения при сохранении простого синтаксиса.

```python
class Account:
    def __init__(self, balance):
        self._balance = balance   # соглашение: "внутреннее"

    @property
    def balance(self):
        return self._balance

    @balance.setter
    def balance(self, value):
        if value < 0:
            raise ValueError("Баланс не может быть отрицательным")
        self._balance = value
```

| Термин | Определение |
|---|---|
| _attr | Соглашение "внутренний атрибут", не принудительное |
| __attr | Name mangling — усложняет, но не блокирует доступ |
| @property | Контролируемый доступ к атрибуту через методы |

**Ответ для интервьюера:** Python не запрещает доступ к «приватным» атрибутам на уровне языка — вместо этого используются соглашения: `_attr` сигнализирует внутреннее использование, `__attr` включает name mangling. Реальную инкапсуляцию поведения (валидацию, вычисляемые значения) обеспечивает `@property`, которая выглядит как обычный атрибут, но выполняет код при доступе.

### 5. Duck typing
**Вопрос:** Что такое «утиная типизация» (Duck typing)?

**Ответ:** Duck typing — подход, при котором тип объекта не важен явно (через `isinstance`/наследование), важно лишь то, поддерживает ли объект нужное поведение — "если что-то крякает как утка, значит это утка". В Python это проявляется повсеместно: любой объект с методом `__len__` работает с `len()`, любой объект с `__iter__` работает в цикле `for`, независимо от того, к какой иерархии классов он принадлежит. Это даёт гибкость и меньшую связность кода по сравнению со строгой номинативной типизацией, но требует более тщательного тестирования, так как ошибки несовместимости типов обнаруживаются только в рантайме.

```python
class Duck:
    def quack(self): return "Кря!"

class Person:
    def quack(self): return "Я притворяюсь уткой"

def make_it_quack(thing):
    print(thing.quack())   # неважно, какой это класс — важен только метод

make_it_quack(Duck())
make_it_quack(Person())
```

| Термин | Определение |
|---|---|
| Duck typing | Значение имеет только поведение объекта, а не его тип |
| Номинативная типизация | Совместимость по явной иерархии типов (противоположность) |
| Protocol (typing) | Способ статически описать duck typing через структурную типизацию |

**Ответ для интервьюера:** Duck typing — принцип "если объект ведёт себя как нужно, его конкретный класс неважен": Python проверяет наличие метода/поведения, а не принадлежность к иерархии классов. Это даёт гибкость кода, но переносит проверку совместимости типов в рантайм, поэтому её иногда формализуют статически через `typing.Protocol`.

### 6. Виды полиморфизма
**Вопрос:** Какие виды полиморфизма существуют в Python?

**Ответ:** Полиморфизм подтипов (subtype polymorphism) — переопределение метода в подклассах, когда вызов одного и того же метода на объектах разных классов даёт разное поведение (`shape.area()` для `Circle` и `Square`). Duck typing / полиморфизм на основе интерфейса — объекты разных, никак не связанных наследованием классов работают одинаково, если поддерживают нужный протокол (`__len__`, `__iter__`). Параметрический полиморфизм (generic) — код, работающий одинаково для любого типа данных без знания о нём заранее (обобщённые контейнеры, `typing.Generic`, дженерик-функции). Оператор- / ad-hoc полиморфизм — перегрузка операторов через магические методы (`__add__` ведёт себя по-разному для чисел и строк).

```python
class Circle:
    def area(self): return 3.14 * 2 ** 2
class Square:
    def area(self): return 4 ** 2

for shape in [Circle(), Square()]:
    print(shape.area())   # subtype polymorphism
```

| Термин | Определение |
|---|---|
| Subtype polymorphism | Переопределение метода в подклассах |
| Duck typing polymorphism | Совместимость по поведению, а не иерархии |
| Параметрический полиморфизм | Обобщённый код, независимый от конкретного типа |
| Ad-hoc полиморфизм | Перегрузка операторов через магические методы |

**Ответ для интервьюера:** В Python встречаются полиморфизм подтипов (переопределение методов в наследниках), duck typing (совместимость по поведению без общей иерархии), параметрический полиморфизм (обобщённый код через generics) и ad-hoc полиморфизм через перегрузку операторов магическими методами вроде `__add__`.

### 7. Миксины (Mixins)
**Вопрос:** Что такое миксины (Mixins) и для каких задач они используются?

**Ответ:** Миксин — небольшой класс, добавляющий конкретную переиспользуемую функциональность (например, сериализацию, логирование, сравнение) через множественное наследование, но не предназначенный для самостоятельного создания экземпляров — он не описывает "что это за сущность", а лишь "что она умеет". Миксины позволяют комбинировать поведение из нескольких независимых источников без глубокой иерархии наследования, следуя принципу композиции поведения. Порядок наследования важен — миксины обычно ставят перед основным базовым классом, чтобы их методы корректно встраивались в MRO.

```python
class JSONSerializableMixin:
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class LoggingMixin:
    def log(self, msg):
        print(f"[{self.__class__.__name__}] {msg}")

class User(JSONSerializableMixin, LoggingMixin):
    def __init__(self, name):
        self.name = name

u = User("Иван")
u.log("создан")
print(u.to_json())
```

| Термин                     | Определение                                                      |
| -------------------------- | ---------------------------------------------------------------- |
| Миксин                     | Класс с переиспользуемым поведением, не самостоятельная сущность |
| Множественное наследование | Механизм, через который подключаются миксины                     |
| MRO                        | Порядок разрешения методов, важный при нескольких миксинах       |

**Ответ для интервьюера:** Миксин — небольшой класс с конкретной переиспользуемой функциональностью, который подключают через множественное наследование, не создавая его экземпляры напрямую. Это способ компоновать поведение из независимых источников (сериализация, логирование) без построения глубокой и жёсткой иерархии наследования.

## Классы, атрибуты и self

### 8. self в Python
**Вопрос:** Что такое self в Python и зачем он нужен?

**Ответ:** `self` — первый параметр методов экземпляра, через который метод получает ссылку на конкретный объект, на котором он вызван, и может читать/изменять его атрибуты. Это не зарезервированное слово, а лишь соглашение об имени — можно назвать параметр иначе, но `self` — общепринятая конвенция, которую ожидают все разработчики Python. Когда метод вызывается через точку (`obj.method()`), Python автоматически подставляет `obj` как первый аргумент — это работа дескриптора функций (bound method), а не магия синтаксиса `.`.

```python
class Counter:
    def __init__(self):
        self.count = 0
    def increment(self):
        self.count += 1   # self ссылается на конкретный экземпляр

c1, c2 = Counter(), Counter()
c1.increment()
print(c1.count, c2.count)   # 1 0 — независимые состояния
```

| Термин | Определение |
|---|---|
| self | Ссылка на текущий экземпляр внутри метода |
| Bound method | Метод, автоматически привязанный к конкретному объекту |
| Соглашение об имени | self — не зарезервированное слово, а конвенция |

**Ответ для интервьюера:** `self` — первый параметр метода экземпляра, через который метод обращается к состоянию конкретного объекта, на котором он вызван. Python автоматически подставляет объект как `self` при вызове через точку благодаря механизму bound method — дескриптору функций внутри класса.

### 9. Публичные, защищенные и приватные атрибуты
**Вопрос:** Объясни разницу между публичными, защищенными и приватными атрибутами в Python.

**Ответ:** Публичный атрибут (`name`) не имеет префикса и доступен без ограничений откуда угодно — часть открытого API класса. "Защищённый" атрибут (`_name`) с одним подчёркиванием — соглашение "для внутреннего использования модулем/классом и наследниками, не гарантирую стабильность", но интерпретатор ничего не блокирует. "Приватный" атрибут (`__name`) с двумя подчёркиваниями запускает name mangling — Python физически переименовывает его в `_ClassName__name`, что затрудняет случайный доступ и коллизии имён при наследовании, но не делает его по-настоящему недоступным.

|Атрибут|Синтаксис|Реальное ограничение доступа|Смысл|
|---|---|---|---|
|**Публичный**|`name`|Нет|Часть открытого API|
|**"Защищённый"**|`_name`|Нет, только соглашение|"Не трогай снаружи"|
|**"Приватный"**|`__name`|Name mangling (усложняет, не блокирует)|Защита от коллизий имён при наследовании|

```python
class Base:
    def __init__(self):
        self.public = 1
        self._protected = 2
        self.__private = 3

b = Base()
print(b.public, b._protected)     # доступны без проблем
print(b._Base__private)            # доступен только через mangled-имя
```

| Термин | Определение |
|---|---|
| Публичный (name) | Открытый атрибут без ограничений |
| Защищённый (_name) | Соглашение "для внутреннего использования" |
| Приватный (__name) | Name mangling усложняет, но не блокирует доступ |

**Ответ для интервьюера:** В Python нет настоящей приватности — деление на публичные, защищённые (`_name`) и приватные (`__name`) атрибуты основано на соглашениях. `_name` — сигнал "не трогай снаружи", `__name` дополнительно физически переименовывается через name mangling, что защищает от случайных коллизий при наследовании, но не от намеренного доступа.

### 10. Name Mangling
**Вопрос:** Что такое механизм Name Mangling?

**Ответ:** Name mangling — механизм, при котором интерпретатор автоматически переименовывает атрибут с двумя ведущими подчёркиваниями (и не более чем одним завершающим) внутри тела класса из `__attr` в `_ClassName__attr`. Основная цель — не сокрытие данных, а предотвращение случайного конфликта имён атрибутов при множественном наследовании: если родительский и дочерний класс независимо используют `self.__value`, mangling гарантирует, что они физически обращаются к разным атрибутам (`_Parent__value` и `_Child__value`), не перезаписывая друг друга.

```python
class Parent:
    def __init__(self):
        self.__value = "parent"
    def show_parent(self):
        return self.__value   # реально self._Parent__value

class Child(Parent):
    def __init__(self):
        super().__init__()
        self.__value = "child"   # реально self._Child__value, не конфликтует с Parent

c = Child()
print(c.show_parent())   # "parent" — атрибуты не пересеклись
```

| Термин | Определение |
|---|---|
| Name mangling | Переименование __attr в _ClassName__attr |
| Конфликт имён | Проблема, которую решает mangling при наследовании |
| __attr | Синтаксис, запускающий mangling |

**Ответ для интервьюера:** Name mangling автоматически переименовывает `__attr` в `_ClassName__attr`, чтобы избежать конфликта одноимённых атрибутов у родителя и наследника при множественном наследовании. Это защита от случайных коллизий имён, а не механизм безопасности — атрибут по-прежнему доступен, если знать mangled-имя.

### 11. __slots__
**Вопрос:** Зачем нужен атрибут __slots__ и как он влияет на работу с памятью и наследование?

**Ответ:** По умолчанию у каждого экземпляра класса есть `__dict__` — словарь для хранения произвольных атрибутов, что гибко, но расходует существенно больше памяти на каждый объект. `__slots__` объявляет фиксированный набор допустимых атрибутов, из-за чего интерпретатор выделяет память под них компактным фиксированным слотом вместо словаря, экономя память (особенно заметно при создании миллионов экземпляров) и слегка ускоряя доступ к атрибутам. Ограничения: нельзя добавлять новые атрибуты вне объявленных в `__slots__` (если явно не добавить `__dict__` обратно), и при множественном наследовании от нескольких классов с непустыми `__slots__` возникает конфликт компоновки памяти.

```python
class PointDict:
    def __init__(self, x, y):
        self.x, self.y = x, y   # есть __dict__ — гибко, но затратно

class PointSlots:
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x, self.y = x, y   # нет __dict__ — компактно

p = PointSlots(1, 2)
p.z = 3   # AttributeError — z не объявлен в __slots__
```

| Термин | Определение |
|---|---|
| __dict__ | Словарь атрибутов экземпляра по умолчанию |
| __slots__ | Фиксированный набор атрибутов без __dict__ |
| Экономия памяти | Основная выгода __slots__ при массовом создании объектов |

**Ответ для интервьюера:** `__slots__` заменяет `__dict__` экземпляра на фиксированный набор слотов, экономя память и ускоряя доступ к атрибутам, но лишая возможности добавлять произвольные новые атрибуты. Использую это для классов, создаваемых в больших количествах (например, узлов структуры данных), помня, что множественное наследование от нескольких классов с непустыми `__slots__` может вызвать конфликт.

### 12. dir() и __dict__
**Вопрос:** Как работают функции dir() и атрибут __dict__?

**Ответ:** `dir(obj)` возвращает отсортированный список всех имён, доступных через точку у объекта — включая собственные атрибуты, унаследованные методы и атрибуты класса, собирая их из `__dict__` объекта, всех классов по MRO и, при наличии, из `__dir__`, если он переопределён. `__dict__` объекта — фактический словарь, где хранятся именно собственные атрибуты экземпляра (не унаследованные), а `ClassName.__dict__` — отдельный словарь атрибутов и методов самого класса. Разница: `dir()` даёт полную картину для интроспекции/автодополнения, `__dict__` — точное "сырое" состояние конкретного объекта или класса.

```python
class Dog:
    species = "Canis"
    def __init__(self, name):
        self.name = name

d = Dog("Рекс")
print(d.__dict__)        # {'name': 'Рекс'} — только атрибуты экземпляра
print(Dog.__dict__.keys())  # включает species, __init__ и др.
print("bark" in dir(d))    # проверка наличия имени через dir()
```

| Термин | Определение |
|---|---|
| dir() | Полный отсортированный список доступных имён объекта |
| __dict__ (экземпляра) | Словарь собственных атрибутов конкретного объекта |
| __dict__ (класса) | Словарь атрибутов и методов, определённых в классе |

**Ответ для интервьюера:** `dir()` возвращает полный список всех доступных объекту имён, включая унаследованные, собирая их из `__dict__` объекта и классов по MRO. `__dict__` самого объекта содержит только его собственные атрибуты экземпляра, а атрибуты и методы класса лежат отдельно в `ClassName.__dict__`.

### 13. type() vs isinstance()
**Вопрос:** В чем разница между проверкой через type() и isinstance()?

**Ответ:** `type(obj) == SomeClass` проверяет точное совпадение класса, игнорируя иерархию наследования — объект подкласса не пройдёт такую проверку для родительского класса. `isinstance(obj, SomeClass)` учитывает наследование (и `__instancecheck__`/ABC-регистрацию), возвращая `True` также для экземпляров подклассов, что соответствует принципу подстановки Лисков и духу полиморфизма в ООП. Поэтому `isinstance()` — предпочтительный способ проверки типа в идиоматичном Python-коде, а `type() ==` используют реже, только когда нужна именно точная проверка класса без учёта наследников.

```python
class Animal: pass
class Dog(Animal): pass

d = Dog()
print(type(d) == Animal)      # False — точное совпадение не выполняется
print(isinstance(d, Animal))   # True — Dog является Animal через наследование
```

| Термин | Определение |
|---|---|
| type() | Точный тип объекта без учёта иерархии |
| isinstance() | Проверка типа с учётом наследования |
| __instancecheck__ | Метод, позволяющий кастомизировать поведение isinstance |

**Ответ для интервьюера:** `type(obj) == X` проверяет точный класс без учёта наследования, а `isinstance(obj, X)` учитывает иерархию — вернёт True и для подклассов X. В идиоматичном Python почти всегда используют `isinstance()`, так как это соответствует принципу подстановки Лисков и корректно работает с полиморфизмом.

## Методы и свойства

### 14. Instance vs classmethod vs staticmethod
**Вопрос:** В чем разница между методом экземпляра, @classmethod и @staticmethod?

**Ответ:** Обычный метод экземпляра принимает `self` первым аргументом и работает с конкретным объектом, имея доступ к его состоянию. `@classmethod` принимает `cls` (сам класс) вместо `self` и обычно используется как альтернативный конструктор или для операций, затрагивающих класс в целом, а не конкретный экземпляр — при этом `cls` в наследнике будет указывать на реальный вызывающий подкласс, а не на класс, где метод определён. `@staticmethod` не принимает ни `self`, ни `cls` — это обычная функция, логически связанная с классом по смыслу (пространство имён), но не обращающаяся ни к состоянию объекта, ни к классу.

| Характеристика                 | Instance method              | @classmethod                | @staticmethod                             |
| ------------------------------ | ---------------------------- | --------------------------- | ----------------------------------------- |
| **Первый аргумент**            | `self`                       | `cls`                       | Нет                                       |
| **Доступ к состоянию объекта** | ✅                            | ❌                           | ❌                                         |
| **Доступ к классу**            | Через `self.__class__`       | ✅ Напрямую                  | ❌                                         |
| **Типичное применение**        | Работа с конкретным объектом | Альтернативные конструкторы | Утилитарная функция в пространстве класса |

```python
class Pizza:
    def __init__(self, toppings):
        self.toppings = toppings

    @classmethod
    def margherita(cls):             # альтернативный конструктор
        return cls(["сыр", "томаты"])

    @staticmethod
    def validate_topping(name):      # не зависит от self/cls
        return name in ("сыр", "томаты", "оливки")

p = Pizza.margherita()
print(Pizza.validate_topping("сыр"))
```

| Термин | Определение |
|---|---|
| Instance method | Метод с self, работает с конкретным объектом |
| @classmethod | Метод с cls, работает с классом (альтернативные конструкторы) |
| @staticmethod | Функция внутри класса без доступа к self/cls |

**Ответ для интервьюера:** Метод экземпляра получает `self` и работает с конкретным объектом, `@classmethod` получает `cls` и обычно используется для альтернативных конструкторов, корректно работая и с наследниками, а `@staticmethod` — обычная функция в пространстве имён класса без доступа к self или cls.

### 15. Аргументы разных типов методов
**Вопрос:** Какие аргументы принимают разные типы методов?

**Ответ:** Метод экземпляра автоматически получает `self` — ссылку на объект, на котором вызван, плюс любые дополнительные аргументы, переданные явно при вызове. `@classmethod` автоматически получает `cls` — ссылку на класс (а не экземпляр), через который вызван метод, что важно при наследовании: `cls` будет указывать на реальный подкласс, вызвавший метод. `@staticmethod` не получает ничего автоматически — все аргументы, включая необходимость доступа к классу или экземпляру (если он нужен), должны передаваться явно.

```python
class Base:
    @classmethod
    def who(cls):
        return cls.__name__

class Child(Base):
    pass

print(Child.who())   # "Child" — cls указывает на реальный вызывающий класс
```

| Термин | Определение |
|---|---|
| self | Автоматически передаваемый экземпляр |
| cls | Автоматически передаваемый класс (реальный, с учётом наследования) |
| Явные аргументы | Всё, что staticmethod должен получать без автоподстановки |

**Ответ для интервьюера:** Метод экземпляра автоматически получает `self`, classmethod — `cls` (с учётом реального вызывающего подкласса при наследовании), а staticmethod не получает ничего автоматически — это просто функция в пространстве имён класса.

### 16. Декоратор @property
**Вопрос:** Как работает декоратор @property и для чего используются геттеры, сеттеры и делетеры?

**Ответ:** `@property` превращает метод в дескриптор, доступ к которому выглядит как к обычному атрибуту (без скобок вызова), но фактически выполняет код метода — это позволяет добавить вычисляемые значения, валидацию или ленивые вычисления, не меняя внешний интерфейс класса. Геттер (`@property`) определяет чтение атрибута, сеттер (`@attr.setter`) — присваивание значения (обычно с валидацией), делетер (`@attr.deleter`) — поведение при `del obj.attr`. Использование `@property` вместо публичного атрибута с самого начала — хорошая практика, позволяющая позже добавить логику без изменения кода, который уже использует класс.

```python
class Temperature:
    def __init__(self, celsius):
        self._celsius = celsius

    @property
    def celsius(self):
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("Ниже абсолютного нуля")
        self._celsius = value

    @celsius.deleter
    def celsius(self):
        del self._celsius

t = Temperature(25)
t.celsius = 30   # вызывает сеттер с валидацией
```

| Термин        | Определение                                  |
| ------------- | -------------------------------------------- |
| @property     | Геттер — доступ на чтение как к атрибуту     |
| @attr.setter  | Присваивание значения с возможной валидацией |
| @attr.deleter | Поведение при удалении атрибута через del    |

**Ответ для интервьюера:** `@property` позволяет обращаться к методу как к обычному атрибуту, добавляя вычисление или валидацию прозрачно для внешнего кода. Геттер отвечает за чтение, сеттер — за запись с проверками, делетер — за поведение при `del`, что даёт полный контроль над жизненным циклом атрибута без изменения публичного API класса.

### 17. Валидация через свойства
**Вопрос:** Как реализовать валидацию данных с помощью свойств?

**Ответ:** Валидацию помещают в сеттер свойства (`@attr.setter`): при каждом присваивании значения выполняется проверка, и если она не проходит, выбрасывается исключение (обычно `ValueError`), не давая объекту оказаться в невалидном состоянии. Это гарантирует инвариант класса в любой момент времени, независимо от того, как и откуда присваивается значение — при инициализации в `__init__` (если он тоже идёт через `self.attr = value`) или позже в коде.

```python
class Person:
    def __init__(self, age):
        self.age = age   # проходит через сеттер уже при инициализации

    @property
    def age(self):
        return self._age

    @age.setter
    def age(self, value):
        if not isinstance(value, int) or value < 0:
            raise ValueError("Возраст должен быть неотрицательным целым числом")
        self._age = value

p = Person(25)
p.age = -5   # ValueError
```

| Термин | Определение |
|---|---|
| Инвариант класса | Условие, которое должно быть истинным для объекта всегда |
| Сеттер (setter) | Место для проверки данных перед сохранением |
| ValueError | Типичное исключение при невалидных данных |

**Ответ для интервьюера:** Валидацию помещаю в сеттер свойства — при каждом присваивании значение проверяется, и при нарушении условия выбрасывается исключение, не позволяя объекту перейти в невалидное состояние. Если в `__init__` присваивание идёт через `self.attr = value`, валидация автоматически срабатывает и при создании объекта.

### 18. Дескрипторы (descriptors)
**Вопрос:** Что такое дескрипторы и как они работают?

**Ответ:** Дескриптор — объект класса, который определяет поведение доступа (чтение, запись, удаление) к атрибутам другого класса через магические методы `__get__`, `__set__` и `__delete__`. При обращении к атрибуту `obj.attr`, если атрибут класса `obj.__class__.attr` является дескриптором, Python вместо прямого возврата значения вызывает `attr.__get__(obj, type(obj))`, передавая ему экземпляр и его класс — это позволяет перехватить логику доступа к атрибуту. `@property` и `@classmethod` — частные случаи дескрипторов, реализованные встроенно; пользовательские дескрипторы используют для сложной валидации, ленивой загрузки, отслеживания доступа или создания вычисляемых атрибутов, зависящих от состояния других объектов.

```python
class ValidatedInt:
    def __init__(self, name, min_val=None, max_val=None):
        self.name = name
        self.min_val = min_val
        self.max_val = max_val
    
    def __get__(self, obj, type=None):
        if obj is None:
            return self
        return obj.__dict__.get(self.name)
    
    def __set__(self, obj, value):
        if not isinstance(value, int):
            raise TypeError(f"{self.name} должен быть целым числом")
        if self.min_val is not None and value < self.min_val:
            raise ValueError(f"{self.name} не может быть меньше {self.min_val}")
        if self.max_val is not None and value > self.max_val:
            raise ValueError(f"{self.name} не может быть больше {self.max_val}")
        obj.__dict__[self.name] = value

class Person:
    age = ValidatedInt("age", min_val=0, max_val=150)
    def __init__(self, age):
        self.age = age

p = Person(25)
print(p.age)   # 25
p.age = 200    # ValueError
```

| Термин | Определение |
|---|---|
| Дескриптор | Объект с __get__, __set__, __delete__ для контроля доступа к атрибуту |
| __get__ | Вызывается при чтении атрибута дескриптора |
| __set__ | Вызывается при записи значения в атрибут дескриптора |
| __delete__ | Вызывается при удалении атрибута через del |
| Data descriptor | Дескриптор с __set__ или __delete__ (приоритет выше, чем __dict__) |
| Non-data descriptor | Дескриптор только с __get__ (приоритет ниже __dict__) |

**Ответ для интервьюера:** Дескриптор — объект с методами `__get__`, `__set__`, `__delete__`, который перехватывает доступ к атрибутам класса, позволяя реализовать сложную валидацию, ленивую загрузку или вычисляемые атрибуты. `@property` и `@classmethod` — встроенные дескрипторы; пользовательские дескрипторы нужны для более продвинутого контроля над атрибутами, чем может обеспечить простое свойство.

## Магические методы и жизненный цикл объекта

### 19. Магические (дандер) методы
**Вопрос:** Что такое магические (дандер) методы? Приведи примеры и объясни их назначение.

**Ответ:** Магические (dunder, double underscore) методы — методы с именами вида `__name__`, которые Python вызывает неявно в ответ на встроенные операции языка, а не через прямой вызов пользователем: `__init__` — при создании объекта, `__str__`/`__repr__` — при преобразовании в строку и отображении в отладчике, `__len__` — при вызове `len(obj)`, `__add__` — при использовании оператора `+`, `__eq__` — при сравнении `==`. Они позволяют пользовательским классам интегрироваться со стандартным синтаксисом языка и встроенными функциями, ведя себя как "родные" типы Python.

```python
class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)
    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

print(Vector(1, 2) + Vector(3, 4))   # Vector(4, 6)
```

| Термин | Определение |
|---|---|
| Dunder-метод | Метод вида __name__, вызываемый неявно интерпретатором |
| __repr__ / __str__ | Строковое представление объекта |
| Операторная перегрузка | Поведение операторов через магические методы |

**Ответ для интервьюера:** Магические методы — методы вида `__name__`, которые интерпретатор вызывает неявно за встроенным синтаксисом: `__init__` при создании, `__add__` при `+`, `__len__` при `len()`, `__eq__` при `==`. Они позволяют пользовательским классам вести себя как встроенные типы, интегрируясь со стандартными операторами и функциями языка.

### 19. __new__ vs __init__
**Вопрос:** В чем разница между методами __new__ и __init__?

**Ответ:** `__new__` — статический по своей природе метод, отвечающий за создание самого объекта в памяти (аллокацию) и возвращающий новый экземпляр; он вызывается первым и получает класс (`cls`) как первый аргумент. `__init__` — отвечает за инициализацию уже созданного объекта (заполнение атрибутов начальными значениями) и ничего не возвращает (точнее, обязан вернуть `None`); вызывается автоматически сразу после `__new__`, но только если `__new__` вернул экземпляр именно того класса (или его подкласса). `__new__` переопределяют редко — в основном для неизменяемых типов (наследники `tuple`, `str`) или для реализации паттерна Singleton, где нужно контролировать сам факт создания объекта, а не только его состояние.

|Характеристика|__new__|__init__|
|---|---|---|
|**Задача**|Создание (аллокация) объекта|Инициализация уже созданного объекта|
|**Первый аргумент**|`cls`|`self`|
|**Возвращает**|Новый экземпляр|`None`|
|**Вызывается**|Первым|Сразу после `__new__`|
|**Переопределяют**|Редко (Singleton, неизменяемые типы)|Часто|

```python
class Singleton:
    _instance = None
    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

a = Singleton()
b = Singleton()
print(a is b)   # True — один и тот же объект
```

| Термин | Определение |
|---|---|
| __new__ | Создание (аллокация) объекта, возвращает экземпляр |
| __init__ | Инициализация уже созданного объекта, возвращает None |
| Singleton | Паттерн, где __new__ контролирует единственность экземпляра |

**Ответ для интервьюера:** `__new__` создаёт сам объект и возвращает его, вызывается первым и получает класс как аргумент. `__init__` инициализирует уже созданный объект и ничего не возвращает. `__new__` переопределяют редко — например, для реализации Singleton или наследования от неизменяемых типов.

### 20. Класс без __init__
**Вопрос:** Что произойдет, если не определить __init__ в классе?

**Ответ:** Если класс не определяет `__init__` явно, Python использует унаследованный `__init__` от родительского класса, а если класс не наследуется явно ни от чего — от `object`, чей `__init__` не делает ничего (не принимает дополнительных аргументов, кроме `self`). Объект в этом случае можно создать без аргументов, но он не будет иметь никаких заранее заданных атрибутов экземпляра — только атрибуты класса и унаследованные методы будут доступны через `obj.attr`, пока их явно не установят позже.

```python
class Empty:
    pass

e = Empty()          # используется object.__init__ — работает без ошибок
e.value = 42          # атрибут можно добавить динамически после создания
print(e.value)
```

| Термин | Определение |
|---|---|
| object.__init__ | Базовая реализация __init__ по умолчанию, ничего не делает |
| Динамический атрибут | Атрибут, добавленный объекту уже после создания |
| Наследование __init__ | Если не переопределён — берётся у родителя |

**Ответ для интервьюера:** Если `__init__` не определён, используется унаследованный `__init__`, а в конце цепочки — `object.__init__`, который ничего не делает. Объект можно создать без аргументов, но никаких предустановленных атрибутов экземпляра у него не будет, пока их не добавят явно.

### 21. __hash__ и __eq__
**Вопрос:** Зачем нужны методы __hash__ и __eq__ при работе с коллекциями?

**Ответ:** `__eq__` определяет логику сравнения объектов на равенство (что значит "два объекта одинаковы" для конкретного класса — по умолчанию `object.__eq__` сравнивает по идентичности `id()`). `__hash__` определяет, как объект хэшируется для использования в `set` и как ключ `dict` — по умолчанию хэш тоже основан на `id()`. Если переопределить `__eq__`, не переопределив `__hash__`, объект автоматически становится нехэшируемым (Python устанавливает `__hash__ = None`), потому что нарушается критичный инвариант: равные по `__eq__` объекты обязаны иметь одинаковый хэш, иначе хэш-таблица не сможет корректно находить "равные" ключи.

```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __eq__(self, other):
        return (self.x, self.y) == (other.x, other.y)
    def __hash__(self):
        return hash((self.x, self.y))   # обязателен вместе с __eq__

points = {Point(1, 2), Point(1, 2)}   # без __hash__ будет TypeError
print(len(points))   # 1 — считаются одинаковыми
```

| Термин | Определение |
|---|---|
| __eq__ | Определяет логическое равенство объектов |
| __hash__ | Определяет хэш объекта для set/dict |
| Инвариант hash/eq | Равные объекты обязаны иметь равный хэш |

**Ответ для интервьюера:** `__eq__` определяет логику сравнения объектов, `__hash__` — их хэш для использования в set и как ключ словаря. Если переопределить `__eq__` без `__hash__`, Python сделает объект нехэшируемым, потому что иначе нарушится инвариант «равные объекты обязаны иметь равный хэш» и хэш-таблица начнёт работать некорректно.

### 22. Метод __call__
**Вопрос:** За что отвечает магический метод __call__?

**Ответ:** `__call__` делает экземпляр класса вызываемым как функцию — синтаксис `obj(args)` транслируется в `obj.__call__(args)`. Это применяют для объектов, которые хранят состояние между вызовами (например, счётчики, кэширующие декораторы, параметризованные фабрики) или когда объект логически представляет "операцию", но с настраиваемым при создании поведением, в отличие от обычной функции без состояния.

```python
class Multiplier:
    def __init__(self, factor):
        self.factor = factor
    def __call__(self, x):
        return x * self.factor

double = Multiplier(2)
print(double(5))   # 10 — double(5) эквивалентно double.__call__(5)
```

| Термин | Определение |
|---|---|
| __call__ | Делает экземпляр вызываемым как функцию |
| Callable | Любой объект, поддерживающий вызов через () |
| Объект с состоянием | Callable-объект, хранящий данные между вызовами |

**Ответ для интервьюера:** `__call__` позволяет вызывать экземпляр класса как функцию через `obj(args)`, что транслируется в `obj.__call__(args)`. Использую это для callable-объектов с сохраняемым между вызовами состоянием — например, параметризованных фабрик или классов-декораторов.

## Наследование и MRO

### 23. Множественное наследование
**Вопрос:** Как работает множественное наследование в Python?

**Ответ:** Python позволяет классу наследоваться сразу от нескольких родительских классов, перечисленных через запятую в определении класса — `class Child(ParentA, ParentB)`. При обращении к методу или атрибуту интерпретатор ищет его по определённому порядку разрешения методов (MRO, Method Resolution Order), построенному алгоритмом C3-линеаризации, который гарантирует консистентный и предсказуемый порядок обхода даже в сложных иерархиях с общими предками. Множественное наследование мощно, но легко ведёт к запутанным иерархиям — на практике его чаще применяют ограниченно, через миксины с чёткой единственной ответственностью каждого.

```python
class A:
    def greet(self): return "A"
class B:
    def greet(self): return "B"
class C(A, B):
    pass

print(C().greet())        # "A" — A стоит первым в списке родителей
print(C.__mro__)           # порядок поиска методов
```

| Термин | Определение |
|---|---|
| Множественное наследование | Наследование от нескольких классов одновременно |
| MRO | Порядок разрешения методов |
| C3-линеаризация | Алгоритм построения MRO |

**Ответ для интервьюера:** Класс может наследоваться сразу от нескольких родителей, а порядок поиска атрибутов и методов определяется MRO, построенным алгоритмом C3-линеаризации — он гарантирует консистентный порядок обхода даже при сложных пересекающихся иерархиях. На практике множественное наследование чаще используют ограниченно, через простые миксины.

---

### 24. MRO и C3-линеаризация
**Вопрос:** Что такое MRO и алгоритм C3-линеаризации?

**Ответ:** MRO (Method Resolution Order) — линейный порядок, в котором Python просматривает классы иерархии при поиске метода или атрибута. Он строится алгоритмом C3-линеаризации, который сохраняет два ключевых свойства: локальный порядок предшествования (родитель всегда стоит после ребёнка в списке) и монотонность (порядок, заданный в объявлении класса, не может быть нарушен результатом линеаризации) — это устраняет неоднозначности, возможные при наивном обходе в глубину (как было в старом Python 2 classic classes). Список MRO можно посмотреть через `ClassName.__mro__` или `ClassName.mro()`.

```python
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print(D.__mro__)
# (D, B, C, A, object) — C3-линеаризация
```

| Термин | Определение |
|---|---|
| MRO | Линейный порядок обхода классов при поиске атрибута |
| C3-линеаризация | Алгоритм построения MRO с гарантией монотонности |
| __mro__ | Атрибут класса со списком порядка разрешения |

**Ответ для интервьюера:** MRO — линейный порядок обхода классов при поиске метода, а C3-линеаризация — алгоритм, который его строит, гарантируя, что родитель всегда стоит после потомка и порядок, заданный в объявлении класса, не нарушается. Посмотреть итоговый порядок можно через `ClassName.__mro__`.

### 25. Проблема ромба
**Вопрос:** Как решается «проблема ромба» при наследовании?

**Ответ:** "Проблема ромба" возникает, когда класс `D` наследуется от `B` и `C`, а оба они, в свою очередь, наследуются от общего класса `A` — без чёткого правила неясно, какую версию метода `A`, переопределённую в `B` и `C`, должен получить `D`. C3-линеаризация решает это, гарантируя, что `A` встретится в MRO только один раз, после обоих своих потомков (`B` и `C`), а порядок между `B` и `C` определяется порядком их перечисления при объявлении `D(B, C)`. Благодаря этому `super()` в методах `B` и `C` корректно передаёт вызов дальше по цепочке MRO, а не порождает дублирование вызовов `A`.

```python
class A:
    def method(self): print("A")
class B(A):
    def method(self):
        print("B")
        super().method()
class C(A):
    def method(self):
        print("C")
        super().method()
class D(B, C):
    def method(self):
        print("D")
        super().method()

D().method()
# D -> B -> C -> A, каждый вызывается ровно один раз
```

| Термин | Определение |
|---|---|
| Проблема ромба | Неоднозначность метода при общем предке двух родителей |
| Единственный вызов A | C3-линеаризация гарантирует один проход по общему предку |
| super() в MRO | Передаёт вызов следующему классу по линеаризованному порядку |

**Ответ для интервьюера:** Проблема ромба — неоднозначность, какую реализацию общего предка использовать, когда два родителя класса сами наследуются от одного класса. C3-линеаризация решает это, помещая общего предка в MRO единственный раз, после обоих потомков, благодаря чему цепочка вызовов `super()` проходит каждый класс ровно один раз без дублирования.

### 26. Функция super()
**Вопрос:** Как работает функция super()?

**Ответ:** `super()` возвращает прокси-объект, который перенаправляет вызовы методов к следующему классу в MRO текущего объекта, начиная с позиции после текущего класса, а не обязательно к "прямому родителю" в классическом ООП-смысле — что критично при множественном наследовании, где следующий класс в MRO зависит от полной иерархии. Без аргументов внутри метода `super()` автоматически определяет текущий класс и `self` через специальный механизм (`__class__` замыкание), что и есть форма без аргументов, доступная с Python 3; явная форма `super(CurrentClass, self)` работает так же, но подходит и для Python 2.

```python
class Base:
    def __init__(self):
        print("Base init")

class Child(Base):
    def __init__(self):
        super().__init__()   # эквивалент super(Child, self).__init__()
        print("Child init")

Child()
```

| Термин | Определение |
|---|---|
| super() | Прокси к следующему классу по MRO текущего объекта |
| Явная форма | super(CurrentClass, self) — работает и в Python 2 |
| __class__ замыкание | Механизм, позволяющий super() без аргументов работать в Python 3 |

**Ответ для интервьюера:** `super()` возвращает прокси-объект, перенаправляющий вызов к следующему классу по MRO текущего экземпляра, а не обязательно к прямому родителю — это важно при множественном наследовании. Форма без аргументов в Python 3 работает благодаря неявному `__class__` замыканию внутри метода.

## Метаклассы

### 27. Что такое метаклассы
**Вопрос:** Что такое метаклассы и какую роль они играют в создании классов?

**Ответ:** Метакласс — это "класс класса": так же, как обычный класс определяет, как устроены и ведут себя его экземпляры, метакласс определяет, как устроены и ведут себя сами классы. Когда интерпретатор выполняет инструкцию `class`, он фактически вызывает метакласс как функцию с именем класса, кортежем базовых классов и словарём атрибутов — метакласс может модифицировать эти данные перед созданием класса (добавлять методы, проверять соглашения об именовании, регистрировать класс в реестре). Метаклассы используют для продвинутых сценариев: ORM (автогенерация полей из объявлений), валидация структуры класса на этапе определения, паттерн Singleton на уровне класса.

```python
class Meta(type):
    def __new__(mcs, name, bases, namespace):
        namespace["created_by"] = "Meta"
        return super().__new__(mcs, name, bases, namespace)

class MyClass(metaclass=Meta):
    pass

print(MyClass.created_by)   # "Meta"
```

| Термин    | Определение                                  |
| --------- | -------------------------------------------- |
| Метакласс | Класс, экземплярами которого являются классы |
| type      | Метакласс по умолчанию в Python              |
| namespace | Словарь атрибутов создаваемого класса        |

**Ответ для интервьюера:** Метакласс — «класс класса»: он контролирует процесс создания самих классов, так же как класс контролирует создание своих экземпляров. Используется для продвинутых сценариев вроде автогенерации полей в ORM, валидации структуры класса при определении или реализации Singleton на уровне класса.

---

### 28. Метакласс по умолчанию (type)
**Вопрос:** Какой метакласс используется в Python по умолчанию?

**Ответ:** По умолчанию метаклассом любого класса в Python 3 является `type` — сам `type` является одновременно и функцией (при вызове с одним аргументом возвращает тип объекта), и метаклассом (при вызове с тремя аргументами — именем, базами и namespace — создаёт новый класс динамически). Это подчёркивает единообразие модели данных Python: классы — тоже объекты, экземпляры `type`, а `type` — экземпляр самого себя, замыкая цепочку.

```python
class Foo:
    pass

print(type(Foo))    # <class 'type'>
Bar = type("Bar", (), {"x": 1})   # динамическое создание класса через type
print(Bar().x)        # 1
```

| Термин | Определение |
|---|---|
| type | Метакласс по умолчанию для всех классов |
| Динамическое создание класса | type(name, bases, dict) как альтернатива class |
| type — экземпляр себя | type является собственным метаклассом |

**Ответ для интервьюера:** Метакласс по умолчанию — `type`, который одновременно возвращает тип объекта при вызове с одним аргументом и динамически создаёт новый класс при вызове с тремя (имя, базы, namespace). Это отражает единообразие модели Python: классы сами являются объектами — экземплярами `type`.

---

### 29. Собственный метакласс
**Вопрос:** Как задать собственный метакласс для класса?

**Ответ:** Собственный метакласс задают наследованием от `type` и переопределением `__new__` (для модификации создаваемого класса до его финального создания) и/или `__init__` (для действий после создания класса). Класс, который должен использовать этот метакласс, указывает его через параметр `metaclass=` в определении класса. Всё наследники такого класса автоматически тоже будут использовать этот метакласс, если явно не переопределят его.

```python
class ValidatingMeta(type):
    def __new__(mcs, name, bases, namespace):
        if "required_method" not in namespace:
            raise TypeError(f"{name} должен определить required_method")
        return super().__new__(mcs, name, bases, namespace)

class Valid(metaclass=ValidatingMeta):
    def required_method(self): pass

class Invalid(metaclass=ValidatingMeta):   # TypeError при определении класса
    pass
```

| Термин | Определение |
|---|---|
| metaclass= | Параметр указания метакласса при определении класса |
| __new__ метакласса | Модификация класса до его финального создания |
| Наследование метакласса | Подклассы наследуют метакласс родителя |

**Ответ для интервьюера:** Создаю класс, наследующий `type`, переопределяю в нём `__new__` или `__init__` для нужной логики, а затем указываю его через `metaclass=` при определении целевого класса. Все наследники этого класса будут автоматически использовать тот же метакласс, если явно не переопределят его.

### 30. Конфликт метаклассов
**Вопрос:** Как разрешить конфликт метаклассов при наследовании?

**Ответ:** Конфликт метаклассов возникает, когда класс наследуется от двух родителей с разными, несовместимыми метаклассами (ни один из которых не является подклассом другого) — Python не может однозначно выбрать, чей метакласс использовать, и бросает `TypeError: metaclass conflict`. Решение — создать новый метакласс, который наследуется от обоих конфликтующих метаклассов, объединяя их поведение, и явно указать его как `metaclass=` для итогового класса; в большинстве случаев так удаётся построить непротиворечивую иерархию метаклассов, зеркалирующую иерархию обычных классов.

```python
class MetaA(type): pass
class MetaB(type): pass

class A(metaclass=MetaA): pass
class B(metaclass=MetaB): pass

# class C(A, B): pass   # TypeError: metaclass conflict

class MetaC(MetaA, MetaB):   # объединяющий метакласс
    pass

class C(A, B, metaclass=MetaC):   # теперь работает
    pass
```

| Термин | Определение |
|---|---|
| Metaclass conflict | Ошибка при несовместимых метаклассах родителей |
| Объединяющий метакласс | Новый метакласс, наследующий оба конфликтующих |
| TypeError | Исключение, которое Python бросает при конфликте |

**Ответ для интервьюера:** Конфликт возникает, когда родительские классы используют разные несовместимые метаклассы — Python бросает `TypeError`. Решаю созданием нового метакласса, наследующего оба конфликтующих, и явно указываю его через `metaclass=` для итогового класса, зеркалируя иерархию обычных классов на уровне метаклассов.

### 31. Метаклассы и производительность
**Вопрос:** Влияют ли метаклассы на производительность приложения?

**Ответ:** Метаклассы влияют на производительность только в момент создания класса (импорт модуля), а не на работу уже созданных экземпляров — логика метакласса выполняется один раз при определении класса, а не при каждом создании объекта или вызове метода. Поэтому даже сложная логика в метаклассе (валидация, автогенерация методов) практически не сказывается на рантайм-производительности приложения, если только сам метакласс не выполняет что-то дорогое при каждом импорте модуля (что при большом числе классов может немного замедлить старт приложения).

| Термин | Определение |
|---|---|
| Время создания класса | Момент, когда выполняется логика метакласса |
| Время выполнения (рантайм) | Работа уже созданных объектов, метакласс не участвует |
| Время старта приложения | Может слегка увеличиться при дорогих метаклассах и множестве классов |

**Ответ для интервьюера:** Метакласс выполняется один раз при создании класса, во время импорта модуля, а не при каждом обращении к экземплярам или вызове методов — поэтому на рантайм-производительность работающего приложения он практически не влияет. Единственный заметный эффект — возможное небольшое замедление старта приложения, если метакласс выполняет что-то дорогое при определении множества классов.

## SOLID и проектирование

### 32. Принципы SOLID
**Вопрос:** Подробно объясни каждый из пяти принципов SOLID.

**Ответ:** S (Single Responsibility) — у класса должна быть только одна причина для изменения, то есть одна чётко очерченная ответственность. O (Open/Closed) — код должен быть открыт для расширения, но закрыт для изменения: новое поведение добавляется через новые классы/наследование, а не правкой существующего проверенного кода. L (Liskov Substitution) — объект подкласса должен быть взаимозаменяем с объектом базового класса без нарушения корректности программы, то есть подкласс не должен ослаблять предусловия или усиливать постусловия родителя. I (Interface Segregation) — лучше несколько узких специализированных интерфейсов, чем один большой "интерфейс на все случаи", чтобы классы не зависели от методов, которые они не используют. D (Dependency Inversion) — модули верхнего уровня не должны зависеть от модулей нижнего уровня напрямую, оба должны зависеть от абстракций (интерфейсов), что снижает связность и упрощает замену реализации.

```python
from abc import ABC, abstractmethod

class NotificationSender(ABC):     # D — абстракция
    @abstractmethod
    def send(self, message: str): ...

class EmailSender(NotificationSender):   # O — расширение без изменения базового кода
    def send(self, message: str):
        print(f"Email: {message}")

class OrderService:                  # D — зависит от абстракции, не от конкретной реализации
    def __init__(self, sender: NotificationSender):
        self.sender = sender
```

|Буква|Принцип|Суть|
|---|---|---|
|**S**|Single Responsibility|У класса одна причина для изменения|
|**O**|Open/Closed|Открыт для расширения, закрыт для изменения|
|**L**|Liskov Substitution|Подкласс взаимозаменяем с базовым классом|
|**I**|Interface Segregation|Узкие интерфейсы вместо одного большого|
|**D**|Dependency Inversion|Зависимость от абстракций, а не от реализаций|

| Термин | Определение |
|---|---|
| SRP | Одна ответственность на класс |
| OCP | Расширение без изменения существующего кода |
| LSP | Подкласс взаимозаменяем с базовым классом |
| DIP | Зависимость от абстракций, а не от конкретных реализаций |

**Ответ для интервьюера:** SOLID — пять принципов: единственная ответственность класса (SRP), открытость для расширения при закрытости для изменения (OCP), взаимозаменяемость подклассов с базовым классом (LSP), узкие специализированные интерфейсы вместо одного большого (ISP), и зависимость от абстракций, а не от конкретных реализаций (DIP). Вместе они снижают связность кода и упрощают его изменение и тестирование.

### 33. Организация кода через классы и модули
**Вопрос:** Как организовать код с использованием классов и модулей?

**Ответ:** Классы группируют связанные данные и поведение в единую сущность с чёткой ответственностью (следуя SRP), а модули (`.py`-файлы) и пакеты (директории с `__init__.py`) группируют логически связанные классы и функции на уровне файловой структуры проекта, разделяя код по слоям (например, `models/`, `services/`, `repositories/`, `api/`) или по доменным областям (`users/`, `orders/`). Хорошая организация минимизирует связность между модулями (низкий coupling) и максимизирует связность внутри модуля (высокий cohesion), а также избегает циклических импортов между модулями.

```python
# project/
#   models/order.py       — класс Order (данные)
#   services/order_service.py  — бизнес-логика над Order
#   repositories/order_repo.py — доступ к БД для Order
#   api/orders.py           — HTTP-эндпоинты

from models.order import Order
from repositories.order_repo import OrderRepository
```

| Термин | Определение |
|---|---|
| Coupling (связность между модулями) | Степень зависимости модулей друг от друга |
| Cohesion (связность внутри модуля) | Насколько логично сгруппирован код одного модуля |
| Слоистая архитектура | Разделение кода на models/services/repositories/api |

**Ответ для интервьюера:** Классы инкапсулируют данные и поведение с единственной ответственностью, а модули и пакеты группируют логически связанные классы по слоям или доменным областям. Стремлюсь к низкой связности между модулями и высокой — внутри модуля, избегая циклических импортов и смешения ответственности разных слоёв.

### 34. Инверсия зависимостей
**Вопрос:** Что такое инверсия зависимостей и как она реализуется через абстрактные классы?

**Ответ:** Инверсия зависимостей (Dependency Inversion, последний принцип SOLID) требует, чтобы модули верхнего уровня (бизнес-логика) зависели от абстракций, а не от конкретных реализаций модулей нижнего уровня (детали вроде конкретной БД или внешнего API) — при этом сами абстракции не должны зависеть от деталей, а детали должны зависеть от абстракций. В Python это реализуют через `abc.ABC` и абстрактные методы: сервис принимает в конструктор объект абстрактного типа (интерфейса), а конкретную реализацию передают снаружи (dependency injection), что позволяет подменять реализацию (например, на mock в тестах) без изменения кода сервиса.

```python
from abc import ABC, abstractmethod

class PaymentGateway(ABC):
    @abstractmethod
    def charge(self, amount: float) -> bool: ...

class StripeGateway(PaymentGateway):
    def charge(self, amount: float) -> bool:
        print(f"Списано {amount} через Stripe")
        return True

class OrderService:
    def __init__(self, gateway: PaymentGateway):   # зависимость от абстракции
        self.gateway = gateway

service = OrderService(StripeGateway())   # конкретная реализация передана снаружи
```

| Термин | Определение |
|---|---|
| Dependency Inversion | Зависимость от абстракций, а не от конкретных реализаций |
| abc.ABC | Базовый класс для создания абстрактных классов в Python |
| Dependency Injection | Передача зависимости снаружи, а не создание внутри класса |

**Ответ для интервьюера:** Инверсия зависимостей означает, что бизнес-логика зависит от абстракции (интерфейса), а не от конкретной реализации, которую передают снаружи через конструктор — dependency injection. В Python абстракцию задают через `abc.ABC` с абстрактными методами, что позволяет легко подменять реализацию, например, на mock-объект в тестах.

## Исключения и контекстные менеджеры

### 35. try/except vs собственное исключение
**Вопрос:** Объясни, когда использовать try/except, а когда стоит создать собственное исключение.

**Ответ:** `try/except` со встроенными исключениями (`ValueError`, `KeyError`, `FileNotFoundError`) достаточно, когда природа ошибки стандартна и потребителю кода понятен смысл встроенного типа исключения. Собственное исключение (наследник `Exception` или более специфичного встроенного класса) стоит создавать, когда ошибка специфична для домена приложения и нужно, чтобы вызывающий код мог отличить её от других ошибок и обработать целенаправленно (`except InsufficientFundsError`), а также когда полезно приложить дополнительный контекст (атрибуты) к самому исключению, а не только текстовое сообщение.

|Характеристика|Встроенное исключение|Собственное исключение|
|---|---|---|
|**Когда использовать**|Стандартная, типовая ошибка|Ошибка специфична для домена приложения|
|**Понятность вызывающему коду**|Сразу понятна по имени типа|Нужен собственный класс с говорящим именем|
|**Дополнительный контекст**|Только текст сообщения|Можно приложить свои атрибуты|
|**Точечная обработка**|Ограничена стандартными типами|`except InsufficientFundsError` и т.п.|

```python
class InsufficientFundsError(Exception):
    def __init__(self, balance, amount):
        self.balance, self.amount = balance, amount
        super().__init__(f"Недостаточно средств: баланс {balance}, нужно {amount}")

def withdraw(balance, amount):
    if amount > balance:
        raise InsufficientFundsError(balance, amount)
    return balance - amount
```

| Термин | Определение |
|---|---|
| Встроенное исключение | Стандартный тип ошибки Python (ValueError и др.) |
| Пользовательское исключение | Класс-наследник Exception под конкретный домен |
| Контекст исключения | Дополнительные атрибуты, приложенные к ошибке |

**Ответ для интервьюера:** Встроенных исключений достаточно для типовых ошибок с понятной семантикой. Собственное исключение создаю, когда ошибка специфична для бизнес-логики приложения и вызывающему коду важно отличить именно её от прочих ошибок — плюс это позволяет приложить структурированный контекст к исключению, а не просто текст сообщения.

### 36. Исключения при работе с файлами
**Вопрос:** Как использовать исключения при работе с файлами?

**Ответ:** Работу с файлами оборачивают в `try/except`, перехватывая конкретные ожидаемые исключения — `FileNotFoundError` (файл не существует), `PermissionError` (нет прав доступа), `IsADirectoryError`, `UnicodeDecodeError` (проблема кодировки) — вместо широкого `except Exception`, чтобы не скрыть неожиданные баги. Сам файл всегда открывают через `with open(...)`, чтобы файл гарантированно закрылся даже при исключении внутри блока, независимо от того, где именно оно возникло.

```python
try:
    with open("data.txt", encoding="utf-8") as f:
        content = f.read()
except FileNotFoundError:
    print("Файл не найден")
except PermissionError:
    print("Нет прав на чтение файла")
except UnicodeDecodeError:
    print("Ошибка кодировки файла")
```

| Термин | Определение |
|---|---|
| FileNotFoundError | Исключение при отсутствии файла |
| PermissionError | Исключение при недостатке прав доступа |
| with open() | Гарантированное закрытие файла даже при исключении |

**Ответ для интервьюера:** Файл всегда открываю через `with open()`, чтобы гарантировать закрытие даже при ошибке, а исключения перехватываю точечно — `FileNotFoundError`, `PermissionError`, `UnicodeDecodeError` — вместо широкого `except Exception`, чтобы не маскировать неожиданные баги.

### 37. Контекстные менеджеры (повтор)
**Вопрос:** Как работают контекстные менеджеры?

**Ответ:** Контекстный менеджер — объект с методами `__enter__` и `__exit__`, используемый через `with` для надёжного управления ресурсом: `__enter__` выполняется при входе в блок (например, открывает соединение), `__exit__` — при выходе, гарантированно, даже если внутри блока произошло исключение. В контексте работы с исключениями и файлами это ключевой инструмент: `with` избавляет от необходимости вручную писать `try/finally` для закрытия ресурса, инкапсулируя эту логику внутри самого объекта-менеджера.

```python
import sqlite3

with sqlite3.connect("app.db") as conn:
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM users")
# соединение закрыто автоматически, даже если execute бросит исключение
```

| Термин | Определение |
|---|---|
| __enter__ | Вызывается при входе в блок with |
| __exit__ | Вызывается при выходе, гарантированно, даже при исключении |
| with | Синтаксис, инкапсулирующий try/finally для ресурса |

**Ответ для интервьюера:** Контекстный менеджер через `__enter__`/`__exit__` гарантирует освобождение ресурса при выходе из блока `with`, даже если внутри произошло исключение — это избавляет от ручного написания `try/finally` для каждого места использования ресурса, инкапсулируя логику очистки внутри самого объекта.

---

### 38. Подавление исключения в контекстном менеджере
**Вопрос:** Как обработать или подавить исключение внутри контекстного менеджера?

**Ответ:** Внутри `__exit__` можно проверить тип исключения через переданный `exc_type` и вернуть `True`, чтобы подавить его (не пробрасывать дальше), либо обработать и вернуть `False`/`None`, чтобы оно продолжило распространяться. Для типового случая подавления заранее известных исключений Python предоставляет готовый `contextlib.suppress`, который делает это в одну строку без написания собственного класса.

```python
from contextlib import suppress

with suppress(FileNotFoundError):
    import os
    os.remove("temp.txt")   # если файла нет — исключение тихо подавляется
print("Код продолжает выполняться в любом случае")
```

| Термин | Определение |
|---|---|
| exc_type в __exit__ | Тип переданного исключения для проверки |
| return True | Подавляет исключение внутри __exit__ |
| contextlib.suppress | Готовый менеджер для подавления заданных исключений |

**Ответ для интервьюера:** Подавляю исключение, возвращая `True` из `__exit__`, предварительно проверив его тип через `exc_type`, чтобы не подавить случайно что-то непредвиденное. Для типовых случаев использую готовый `contextlib.suppress(ExceptionType)` вместо написания собственного класса менеджера.

### 39. else и finally в обработке ошибок
**Вопрос:** Для чего нужны блоки else и finally в конструкции обработки ошибок?

**Ответ:** `else` в конструкции `try/except` выполняется только тогда, когда `try` не выбросил исключений, что позволяет чётко отделить "код, который может упасть" от "кода, который выполняется только при успехе", не рискуя случайно перехватить в `except` ошибку из уже успешно завершившейся операции. `finally` гарантированно выполняется всегда — при успехе, при перехваченном исключении и даже при непойманном исключении (перед его дальнейшим распространением) или при `return`/`break`/`continue` внутри блока — и обычно используется для финальной очистки состояния, логирования или освобождения ресурсов, не зависящих от результата.

```python
def process(value):
    try:
        result = 100 / value
    except ZeroDivisionError:
        print("Деление на ноль")
        return None
    else:
        print("Вычисление прошло успешно")
        return result
    finally:
        print("Обработка завершена")   # выполнится в любом случае
```

| Термин | Определение |
|---|---|
| else (try) | Выполняется только при отсутствии исключений в try |
| finally | Выполняется всегда, независимо от результата |
| Разделение успеха и ошибки | Основная цель наличия отдельного else |

**Ответ для интервьюера:** `else` выполняется только при успешном завершении `try`, отделяя основную логику от кода обработки ошибок и защищая от случайного перехвата исключений из уже успешного пути. `finally` выполняется всегда — успех, обработанное или необработанное исключение — и используется для гарантированной очистки, не зависящей от исхода.
