## Синхронность против Асинхронности

### 1. Синхронный vs асинхронный код
**Вопрос:** Объясни разницу между синхронным и асинхронным кодом.

**Ответ:** Синхронный код выполняется строго последовательно — каждая операция блокирует выполнение потока до своего завершения, прежде чем начнётся следующая. Асинхронный код позволяет "поставить операцию на паузу" (`await`) в точке ожидания результата (обычно I/O), вернуть управление циклу событий, который в это время может выполнять другие корутины, и вернуться к продолжению именно этой функции, когда результат будет готов — всё это в рамках одного потока, без реального параллелизма ОС. Выигрыш асинхронности проявляется именно на I/O-bound нагрузке с большим числом одновременных ожиданий (сетевые запросы, БД), а не на вычислительных задачах.

```python
import time, asyncio

def sync_task():
    time.sleep(1)
    return "done"

async def async_task():
    await asyncio.sleep(1)
    return "done"

# 3 синхронных вызова: суммарно ~3с
# 3 асинхронных через gather: суммарно ~1с
```

| Термин | Определение |
|---|---|
| Синхронный код | Последовательное блокирующее выполнение |
| Асинхронный код | Кооперативное переключение на паузах ожидания |
| await | Точка, где корутина отдаёт управление циклу событий |

**Ответ для интервьюера:** Синхронный код блокирует поток на каждой операции до её завершения. Асинхронный код в точке `await` отдаёт управление циклу событий, позволяя выполнять другие задачи во время ожидания I/O, и возвращается к продолжению по готовности результата — всё в одном потоке, без реального параллелизма ОС, поэтому выигрыш заметен именно на I/O-bound нагрузке.

### 2. Когда использовать sync/async
**Вопрос:** Когда стоит использовать каждый из этих подходов и в чем заключается выигрыш в производительности.

**Ответ:** Синхронный подход достаточен для простых скриптов, CLI-утилит и случаев с малым числом последовательных операций, где сложность асинхронного кода не окупается. Асинхронный подход выгоден при большом числе одновременных I/O-bound операций — веб-сервер, обрабатывающий тысячи соединений, множественные HTTP-запросы к внешним API, работа с БД — потому что во время ожидания ответа от сети или диска процессор простаивает, а асинхронность позволяет использовать это время для обработки других запросов в том же потоке, вместо создания дорогого отдельного потока/процесса на каждое ожидание. Выигрыш измеряется не в ускорении одной операции, а в пропускной способности (throughput) — числе одновременно обслуживаемых операций при том же объёме ресурсов.

| Термин | Определение |
|---|---|
| Throughput | Число операций, обслуживаемых за единицу времени |
| I/O-bound | Задача, где время уходит на ожидание внешних ресурсов |
| Конкурентность | Одновременное продвижение множества задач в одном потоке |

**Ответ для интервьюера:** Синхронный подход хорош для простых последовательных сценариев, асинхронный — когда нужно обслуживать множество одновременных I/O-bound операций, не тратя ресурсы на отдельный поток под каждое ожидание. Выигрыш измеряется в пропускной способности — числе одновременно обрабатываемых задач, а не в скорости одной конкретной операции.

### 3. I/O-bound vs CPU-bound
**Вопрос:** Что такое I/O-bound и CPU-bound задачи и как они влияют на выбор способа обработки данных.

**Ответ:** I/O-bound задача — та, где основное время уходит на ожидание внешнего ресурса (диск, сеть, БД), а не на вычисления самого процессора; для таких задач эффективны asyncio или threading, потому что во время ожидания можно выполнять другую работу. CPU-bound задача — та, где узкое место именно вычисления процессора (обработка изображений, численные расчёты, парсинг больших объёмов данных); для неё ни asyncio, ни threading не дают реального ускорения из-за GIL — нужен multiprocessing, распределяющий вычисления по нескольким процессам, каждый со своим интерпретатором и GIL.

|Характеристика|I/O-bound|CPU-bound|
|---|---|---|
|**Узкое место**|Ожидание внешнего ресурса (сеть, диск, БД)|Вычисления процессора|
|**Что помогает**|asyncio, threading|multiprocessing|
|**Ограничение GIL**|Не мешает (GIL освобождается на время ожидания)|Мешает — нужен отдельный процесс|
|**Пример**|HTTP-запросы, чтение файлов|Обработка изображений, численные расчёты|

```python
# I/O-bound — эффективен async/threading
async def fetch_url(session, url):
    async with session.get(url) as resp:
        return await resp.text()

# CPU-bound — эффективен multiprocessing
def compute_heavy(n):
    return sum(i * i for i in range(n))
```

| Термин | Определение |
|---|---|
| I/O-bound | Время уходит на ожидание внешних ресурсов |
| CPU-bound | Время уходит на вычисления процессора |
| GIL | Ограничение, из-за которого threading не ускоряет CPU-bound |

**Ответ для интервьюера:** I/O-bound задачи упираются в ожидание внешних ресурсов и хорошо решаются asyncio или threading. CPU-bound задачи упираются в вычисления процессора, и для них из-за GIL ни asyncio, ни threading не дают параллелизма — нужен multiprocessing с несколькими независимыми процессами.

### 4. Блокирующий вызов в асинхронном коде
**Вопрос:** Что произойдет, если вызвать блокирующую (синхронную) задачу или функцию (например, time.sleep()) внутри асинхронного кода.

**Ответ:** Блокирующий вызов (`time.sleep()`, синхронный `requests.get()`, тяжёлые вычисления) внутри корутины выполняется как обычный синхронный код — он не отдаёт управление циклу событий, потому что не является `await`-точкой, и полностью блокирует единственный поток, в котором крутится Event Loop. В результате все остальные корутины, ожидающие своей очереди в этом же цикле событий, замирают на всё время блокировки, даже если формально у них уже готовы результаты — это одна из самых частых и опасных ошибок в асинхронном коде. Решение — заменить блокирующий вызов на асинхронный аналог (`asyncio.sleep()`, `aiohttp` вместо `requests`) или вынести блокирующий код в отдельный поток/процесс через `run_in_executor`/`asyncio.to_thread`.

```python
import asyncio, time

async def bad():
    time.sleep(2)   # блокирует ВЕСЬ event loop на 2 секунды

async def good():
    await asyncio.sleep(2)   # не блокирует, другие корутины продолжают работать
```

| Термин | Определение |
|---|---|
| Блокирующий вызов | Операция, не отдающая управление циклу событий |
| Event Loop starvation | «Голодание» других задач из-за блокировки цикла |
| asyncio.to_thread | Способ вынести блокирующий код в отдельный поток |

**Ответ для интервьюера:** Блокирующий вызов внутри корутины не отдаёт управление циклу событий и замораживает весь Event Loop на время своего выполнения, останавливая абсолютно все остальные корутины в этом потоке. Решение — использовать асинхронный аналог вызова или выносить блокирующий код в отдельный поток через `asyncio.to_thread`/`run_in_executor`.

## Синтаксис и внутренняя работа asyncio

### 5. async и await
**Вопрос:** Объясни, что такое async и await в Python и как они работают вместе.

**Ответ:** `async def` объявляет корутинную функцию — при вызове она не выполняется сразу, а возвращает объект-корутину, требующий последующего запуска через `await`, `asyncio.run()` или обёртку в `Task`. `await` приостанавливает выполнение текущей корутины в этой точке до завершения ожидаемого объекта (другой корутины, `Task`, `Future`), одновременно отдавая управление циклу событий, чтобы он мог выполнить другую готовую к работе задачу — и это именно то, что делает выполнение кооперативным, а не вытесняющим (preemptive), как в потоках ОС.

```python
import asyncio

async def fetch_data():
    print("Начало запроса")
    await asyncio.sleep(1)   # точка приостановки, управление уходит в event loop
    print("Данные получены")
    return 42

asyncio.run(fetch_data())
```

| Термин | Определение |
|---|---|
| async def | Объявление корутинной функции |
| await | Точка приостановки с передачей управления Event Loop |
| Кооперативная многозадачность | Задачи сами уступают управление, в отличие от вытесняющей |

**Ответ для интервьюера:** `async def` создаёт корутинную функцию, вызов которой возвращает объект-корутину, требующий явного запуска. `await` приостанавливает текущую корутину до готовности ожидаемого результата, отдавая управление Event Loop — это основа кооперативной многозадачности, где задачи сами решают, в какой момент уступить процессор.

### 6. Event Loop
**Вопрос:** Что такое Event Loop (цикл событий) и какова его роль в управлении асинхронными задачами.

**Ответ:** Event Loop — центральный диспетчер asyncio, работающий в одном потоке: он хранит очередь готовых к выполнению задач и колбэков, по кругу выбирает следующую готовую задачу, выполняет её код до ближайшей точки `await` (или до завершения), затем переключается на следующую готовую задачу, а для задач, ожидающих I/O, использует системные механизмы уведомления (epoll/kqueue/IOCP через модуль `selectors`), чтобы не тратить процессор на активное ожидание (busy-waiting). Именно Event Loop создаёт иллюзию параллелизма внутри одного потока, эффективно переиспользуя процессорное время во время простоя I/O-операций.

```python
import asyncio

loop = asyncio.new_event_loop()
# в современном коде обычно используется asyncio.run(), который создаёт
# и корректно закрывает event loop сам
asyncio.run(main())
```

| Термин | Определение |
|---|---|
| Event Loop | Диспетчер, управляющий выполнением корутин в одном потоке |
| epoll/kqueue/IOCP | Системные механизмы уведомления о готовности I/O |
| Busy-waiting | Активное ожидание, которого Event Loop избегает |

**Ответ для интервьюера:** Event Loop — центральный диспетчер asyncio в одном потоке: он выполняет готовые задачи до точки `await`, переключается на следующую готовую, а для I/O-ожидания опирается на системные механизмы вроде epoll, чтобы не тратить процессор впустую. Именно он создаёт иллюзию параллелизма внутри единственного потока.

---

### 7. Переключение задач в Event Loop
**Вопрос:** Как именно Event Loop переключается между задачами в моменты ожидания.

**Ответ:** Когда корутина доходит до `await` на операции, требующей ожидания (сетевой сокет, таймер, файл), она регистрирует свой интерес в готовности этого события у Event Loop и возвращает управление ему (yield control), не блокируя поток. Event Loop добавляет соответствующий файловый дескриптор/таймер в список отслеживаемых через системный вызов селектора (`select`/`epoll`/`kqueue`) и переходит к выполнению следующей задачи из очереди готовых. Когда операция завершается (данные пришли, таймер истёк), селектор сообщает об этом Event Loop, тот помечает соответствующую корутину как готовую и в следующей итерации своего цикла возобновляет её выполнение именно с того места, где был `await`.

| Термин | Определение |
|---|---|
| Yield control | Возврат управления Event Loop в точке await |
| Селектор (selector) | Системный механизм отслеживания готовности файловых дескрипторов |
| Очередь готовых задач | Список корутин, которые Event Loop выполнит следующими |

**Ответ для интервьюера:** В точке `await` корутина регистрирует ожидаемое событие у Event Loop и возвращает ему управление, не блокируя поток; Event Loop переходит к следующей готовой задаче, а само ожидание отслеживается системным селектором (epoll/kqueue). Когда событие наступает, селектор уведомляет Event Loop, и тот возобновляет корутину точно с места остановки на следующей итерации цикла.

### 8. Coroutine vs Task vs Future
**Вопрос:** В чем разница между корутиной (coroutine), задачей (task) и объектом Future.

**Ответ:** Корутина (`coroutine`) — результат вызова `async def` функции, "рецепт" вычисления, который сам по себе ничего не выполняет, пока его явно не запустят через `await` или не обернут в `Task`. `Task` — обёртка над корутиной, которая планирует её выполнение в Event Loop немедленно (не дожидаясь `await`), позволяя нескольким корутинам выполняться конкурентно, и хранит состояние выполнения и результат. `Future` — более низкоуровневый примитив, представляющий результат, который станет доступен в будущем (обычно результат асинхронной операции более низкого уровня, например, callback-based API) — `Task` фактически является подклассом `Future`, добавляющим логику выполнения корутины поверх него.

|Характеристика|Coroutine|Task|Future|
|---|---|---|---|
|**Что это**|"Рецепт" вычисления|Обёртка над корутиной|Низкоуровневый результат "в будущем"|
|**Запускается сразу**|❌ Нет, пока не await/create_task|✅ Немедленно планируется в Event Loop|Зависит от источника|
|**Отношение**|Оборачивается в Task|Подкласс Future|Базовый класс для Task|

```python
import asyncio

async def compute():
    await asyncio.sleep(1)
    return 42

async def main():
    coro = compute()               # просто объект-корутина, ещё не запущен
    task = asyncio.create_task(compute())  # запланирован для выполнения СЕЙЧАС
    result = await task
    print(result)

asyncio.run(main())
```

| Термин | Определение |
|---|---|
| Coroutine | Результат вызова async def, "рецепт" без немедленного запуска |
| Task | Запланированная в Event Loop обёртка над корутиной |
| Future | Низкоуровневый объект-результат, доступный в будущем |

**Ответ для интервьюера:** Корутина — это ленивый «рецепт» вычисления, который не выполняется, пока его не запустят. `Task` планирует корутину на немедленное выполнение в Event Loop, позволяя нескольким корутинам работать конкурентно. `Future` — более низкоуровневый примитив «результата в будущем», а `Task` фактически является его подклассом, добавляющим логику выполнения корутины.

### 9. Обёртка корутины в Task
**Вопрос:** Для чего нужна обертка корутины в Task и какие состояния она может принимать.

**Ответ:** Простой `await coroutine` выполняет корутины строго последовательно — следующая начнётся только после завершения предыдущей. Обёртка в `Task` через `asyncio.create_task()` немедленно планирует корутину в Event Loop, позволяя запустить сразу несколько задач конкурентно, не дожидаясь их последовательного завершения — именно так достигается параллелизм ожидания I/O в asyncio. Task проходит через состояния: `Pending` (запланирована, но ещё не завершена), `Running` (выполняется в данный момент внутри цикла событий), `Done` (завершена — успешно, с исключением или отменой), `Cancelled` (отменена явным вызовом `cancel()`).

```python
import asyncio

async def worker(n):
    await asyncio.sleep(n)
    return n

async def main():
    task = asyncio.create_task(worker(2))
    print(task.done())      # False — Pending
    await task
    print(task.done())      # True — Done
    print(task.result())     # 2

asyncio.run(main())
```

| Термин | Определение |
|---|---|
| Pending | Задача запланирована, ещё не завершена |
| Done | Задача завершена (успешно/с ошибкой/отменена) |
| Cancelled | Задача явно отменена через cancel() |

**Ответ для интервьюера:** Обёртка в `Task` позволяет запустить корутину немедленно и конкурентно с другими, а не ждать её последовательного завершения через простой `await`. Task проходит состояния Pending, Running, Done и Cancelled, а его результат или исключение можно получить методами `result()`/`exception()` после завершения.

### 10. Асинхронные итераторы и контекстные менеджеры
**Вопрос:** Как реализованы асинхронные итераторы и контекстные менеджеры.

**Ответ:** Асинхронный итератор реализует протокол `__aiter__`/`__anext__`, где `__anext__` — корутина, которая может приостанавливаться на `await` (например, ожидая следующую порцию данных из сети) и бросает `StopAsyncIteration` в конце; перебирают такой объект через `async for`. Асинхронный контекстный менеджер реализует `__aenter__`/`__aexit__` — оба тоже корутины, что позволяет выполнять асинхронные операции при входе и выходе (например, асинхронное открытие/закрытие сетевого соединения); используется через `async with`. Как и в синхронном случае, для генераторной формы есть удобный декоратор `@contextlib.asynccontextmanager`.

```python
class AsyncCounter:
    def __init__(self, limit):
        self.i, self.limit = 0, limit
    def __aiter__(self):
        return self
    async def __anext__(self):
        if self.i >= self.limit:
            raise StopAsyncIteration
        await asyncio.sleep(0.1)
        self.i += 1
        return self.i

async def main():
    async for value in AsyncCounter(3):
        print(value)
```

| Термин | Определение |
|---|---|
| __aiter__ / __anext__ | Протокол асинхронного итератора |
| __aenter__ / __aexit__ | Протокол асинхронного контекстного менеджера |
| async for / async with | Синтаксис для работы с асинхронными протоколами |

**Ответ для интервьюера:** Асинхронный итератор реализует `__aiter__`/`__anext__`, где `__anext__` — корутина, способная приостанавливаться на `await`, и перебирается через `async for`. Асинхронный контекстный менеджер реализует `__aenter__`/`__aexit__` как корутины и используется через `async with`, что позволяет выполнять асинхронные операции при входе и выходе из блока, например открытие и закрытие сетевого соединения.

## Библиотека asyncio и ее методы

### 11. asyncio.run/gather/sleep
**Вопрос:** Объясни назначение и использование основных функций: asyncio.run(), asyncio.gather(), asyncio.sleep().

**Ответ:** `asyncio.run(coro)` — точка входа для запуска асинхронной программы из синхронного кода: создаёт новый Event Loop, выполняет переданную корутину до завершения и корректно закрывает цикл — используется один раз, обычно в `if __name__ == "__main__"`. `asyncio.gather(*coros)` запускает несколько корутин конкурентно и возвращает список их результатов в том же порядке, в котором были переданы аргументы, дожидаясь завершения всех. `asyncio.sleep(seconds)` — неблокирующий асинхронный аналог `time.sleep()`, приостанавливающий текущую корутину на заданное время, не блокируя Event Loop, часто используется как точка `await` в демонстрационных примерах или для намеренной задержки.

```python
import asyncio

async def fetch(n):
    await asyncio.sleep(n)
    return f"result-{n}"

async def main():
    results = await asyncio.gather(fetch(1), fetch(2), fetch(3))
    print(results)   # ['result-1', 'result-2', 'result-3'], суммарно ~3с, а не 6с

asyncio.run(main())
```

| Термин | Определение |
|---|---|
| asyncio.run | Точка входа, запускающая Event Loop для главной корутины |
| asyncio.gather | Конкурентный запуск нескольких корутин с ожиданием всех |
| asyncio.sleep | Неблокирующий асинхронный аналог time.sleep |

**Ответ для интервьюера:** `asyncio.run()` — единая точка входа, создающая и корректно закрывающая Event Loop для запуска главной корутины. `asyncio.gather()` запускает несколько корутин конкурентно и возвращает результаты в порядке аргументов. `asyncio.sleep()` — неблокирующая пауза, не останавливающая весь цикл событий, в отличие от `time.sleep()`.

### 12. gather() vs create_task()
**Вопрос:** В чем разница между asyncio.gather() и asyncio.create_task().

**Ответ:** `asyncio.create_task(coro)` немедленно планирует одну корутину как `Task` в Event Loop и сразу возвращает объект `Task`, который можно сохранить, передать дальше или дождаться позже — управление задачей полностью в руках вызывающего кода. `asyncio.gather(*coros)` — это удобная обёртка более высокого уровня для запуска сразу нескольких awaitable-объектов конкурентно и одновременного ожидания их всех, возвращая список результатов; внутри `gather` тоже оборачивает переданные корутины в задачи. `create_task` дают больший контроль (можно отменить конкретную задачу, проверить её состояние в произвольный момент), `gather` — удобнее для типового сценария "запустить N штук и дождаться всех".

|Характеристика|create_task()|gather()|
|---|---|---|
|**Число корутин**|Одна|Несколько сразу|
|**Возвращает**|Объект Task для гибкого контроля|Список результатов после ожидания всех|
|**Отмена конкретной задачи**|✅ Просто|Сложнее, нужно хранить ссылки на задачи|
|**Типовой сценарий**|Динамическое порождение задач по одной|Фиксированный набор независимых операций|

```python
import asyncio

async def main():
    # create_task: запуск + гибкий контроль над каждой задачей отдельно
    t1 = asyncio.create_task(worker(1))
    t2 = asyncio.create_task(worker(2))
    await t1
    await t2

    # gather: компактный запуск и ожидание сразу нескольких
    results = await asyncio.gather(worker(1), worker(2))
```

| Термин | Определение |
|---|---|
| create_task | Немедленное планирование одной корутины с контролем над ней |
| gather | Конкурентный запуск и ожидание нескольких awaitable сразу |
| Awaitable | Объект, который можно передать в await (корутина, Task, Future) |

**Ответ для интервьюера:** `create_task` планирует одну корутину и сразу возвращает управляемый объект Task, что даёт гибкий контроль — отмену, проверку состояния в произвольный момент. `gather` — обёртка более высокого уровня для запуска сразу нескольких awaitable конкурентно с ожиданием всех и возвратом списка результатов, удобная для типового сценария "запустить и дождаться всех".

### 13. Результаты и исключения в gather()
**Вопрос:** Как работает метод gather() с точки зрения возврата результатов и обработки исключений.

**Ответ:** По умолчанию `gather()` возвращает список результатов строго в том же порядке, в каком были переданы awaitable-объекты, независимо от фактического порядка их завершения. Если хотя бы одна из корутин бросает исключение, по умолчанию `gather()` немедленно пробрасывает это исключение наружу (после того как остальные ещё выполняющиеся задачи получат шанс завершиться в фоне, но их результаты будут потеряны для вызывающего кода). Чтобы получить результаты и исключения всех задач без немедленного прерывания, используют `gather(*coros, return_exceptions=True)` — тогда исключения возвращаются в списке результатов как обычные объекты вместо того, чтобы прерывать выполнение.

```python
import asyncio

async def may_fail(n):
    if n == 2:
        raise ValueError("Ошибка на 2")
    return n

async def main():
    results = await asyncio.gather(
        may_fail(1), may_fail(2), may_fail(3),
        return_exceptions=True
    )
    print(results)   # [1, ValueError('Ошибка на 2'), 3]

asyncio.run(main())
```

| Термин | Определение |
|---|---|
| Порядок результатов | Соответствует порядку переданных аргументов, не завершения |
| return_exceptions=True | Возвращает исключения как значения вместо прерывания |
| Проброс исключения | Поведение gather() по умолчанию при ошибке в одной из задач |

**Ответ для интервьюера:** `gather()` возвращает результаты в порядке переданных аргументов, а не порядке завершения. По умолчанию исключение любой задачи немедленно пробрасывается наружу; чтобы получить результаты и ошибки всех задач без прерывания, передаю `return_exceptions=True` — тогда исключения приходят как обычные элементы результирующего списка.

### 14. Отмена асинхронной задачи
**Вопрос:** Как можно отменить выполнение запущенной асинхронной задачи.

**Ответ:** Задачу отменяют вызовом `task.cancel()`, который асинхронно бросает исключение `asyncio.CancelledError` внутри корутины в ближайшей точке `await` — задача не останавливается мгновенно, а получает шанс перехватить `CancelledError` для очистки ресурсов (обычно в `finally`), после чего исключение обязано быть проброшено дальше, иначе отмена не завершится корректно. После отмены `task.cancelled()` возвращает `True`, а попытка получить `task.result()` бросит `CancelledError`.

```python
import asyncio

async def worker():
    try:
        await asyncio.sleep(10)
    except asyncio.CancelledError:
        print("Задача отменена, освобождаю ресурсы")
        raise   # важно пробросить дальше

async def main():
    task = asyncio.create_task(worker())
    await asyncio.sleep(1)
    task.cancel()
    await asyncio.gather(task, return_exceptions=True)

asyncio.run(main())
```

| Термин | Определение |
|---|---|
| task.cancel() | Инициирует отмену задачи |
| CancelledError | Исключение, бросаемое в корутину при отмене |
| Точка отмены | Ближайший await, где CancelledError фактически возникает |

**Ответ для интервьюера:** Отмена вызывается через `task.cancel()`, что бросает `CancelledError` внутрь корутины в ближайшей точке `await`, давая ей шанс освободить ресурсы в `finally` перед тем, как исключение обязано быть проброшено дальше. Это кооперативная отмена — задача не прерывается мгновенно посреди синхронного кода между точками `await`.

### 15. Когда что использовать (run/gather/sleep)
**Вопрос:** Приведи примеры задач, для которых лучше всего подходит использование этих функций.

**Ответ:** `asyncio.run()` используют один раз как единственную точку входа приложения или скрипта — например, запуск CLI-утилиты, делающей несколько сетевых запросов. `asyncio.gather()` идеален, когда нужно выполнить заранее известный, фиксированный набор независимых операций конкурентно и дождаться всех — например, одновременно запросить данные из трёх разных API для одной страницы. `create_task()` больше подходит для динамического сценария, где задачи создаются по одной в процессе работы программы (например, обработчик, порождающий новую задачу на каждое входящее сообщение из очереди) и нужен независимый контроль над каждой.

```python
async def build_dashboard():
    weather, news, stocks = await asyncio.gather(
        fetch_weather(), fetch_news(), fetch_stocks()
    )
    return {"weather": weather, "news": news, "stocks": stocks}

asyncio.run(build_dashboard())
```

| Термин | Определение |
|---|---|
| asyncio.run | Единая точка входа приложения |
| Фиксированный набор задач | Типовой сценарий для gather() |
| Динамическое порождение задач | Типовой сценарий для create_task() |

**Ответ для интервьюера:** `asyncio.run()` — единственная точка входа программы. `gather()` хорош для фиксированного набора независимых операций, которые нужно выполнить конкурентно и дождаться всех разом — например, сбор данных для дашборда из нескольких API. `create_task()` больше подходит, когда задачи порождаются динамически по одной в процессе работы и каждой нужен независимый контроль.

## GIL (Global Interpreter Lock)

### 16. Что такое GIL
**Вопрос:** Что такое GIL и как он влияет на параллелизм в Python.

**Ответ:** GIL (Global Interpreter Lock) — глобальная блокировка в CPython, гарантирующая, что байткод Python в любой момент времени выполняет только один поток, даже если в процессе запущено несколько потоков ОС. Она существует потому, что управление памятью CPython (в первую очередь reference counting) не потокобезопасно само по себе, и GIL — самый простой способ защитить внутренние структуры интерпретатора от гонок данных без переработки всей модели памяти. Следствие: несколько потоков Python не дают реального параллелизма для CPU-bound вычислений на многоядерном процессоре — в любой момент реально считает только один поток, они лишь чередуются.

```python
import threading

def cpu_heavy():
    x = 0
    for _ in range(10_000_000):
        x += 1

threads = [threading.Thread(target=cpu_heavy) for _ in range(4)]
# на 4 потоках эта работа НЕ ускорится в ~4 раза из-за GIL
```

| Термин | Определение |
|---|---|
| GIL | Глобальная блокировка интерпретатора CPython |
| Reference counting | Основная причина существования GIL |
| CPU-bound параллелизм | Не достигается через потоки Python из-за GIL |

**Ответ для интервьюера:** GIL — глобальная блокировка в CPython, разрешающая исполнять байткод Python только одному потоку одновременно, введённая ради потокобезопасности reference counting без переработки всей модели управления памятью. Из-за неё многопоточность в Python не даёт реального параллелизма для CPU-bound задач на нескольких ядрах.

---

### 17. GIL и счётчик ссылок
**Вопрос:** Как GIL защищает счетчик ссылок в памяти.

**Ответ:** Счётчик ссылок каждого объекта инкрементируется и декрементируется при каждом присваивании, передаче в функцию, удалении ссылки — эта операция не атомарна на уровне процессора без дополнительной синхронизации, а значит без защиты два потока могли бы одновременно изменить счётчик, потеряв одно из изменений (классическая гонка данных, race condition) — что привело бы либо к преждевременному освобождению объекта, на который ещё есть ссылки (use-after-free, критическая уязвимость), либо к утечке памяти. GIL гарантирует, что в любой момент байткод выполняет только один поток, поэтому операции инкремента/декремента счётчика ссылок фактически всегда сериализованы и не требуют отдельной блокировки на каждый объект (что было бы значительно дороже по производительности).

| Термин | Определение |
|---|---|
| Атомарность операции | Гарантия, что операция не будет прервана посреди выполнения |
| Race condition | Гонка данных при одновременном изменении общего состояния |
| Cериализация через GIL | Все байткод-операции выполняются строго по очереди |

**Ответ для интервьюера:** Инкремент/декремент счётчика ссылок объекта не атомарен на уровне процессора, поэтому без защиты два потока могли бы гонять его одновременно и потерять изменение, что привело бы к use-after-free или утечке памяти. GIL сериализует выполнение байткода одним потоком за раз, делая эти операции безопасными без необходимости ставить отдельную блокировку на каждый объект.

---

### 18. Почему threading не даёт параллелизма для CPU-bound
**Вопрос:** Почему Threading (многопоточность) не дает настоящего параллелизма для CPU-bound задач.

**Ответ:** Даже на многоядерном процессоре GIL позволяет исполнять байткод Python только одному потоку в конкретный момент времени — остальные потоки Python ждут своей очереди, независимо от числа свободных ядер. Для CPU-bound задачи (все время уходит на вычисления, а не на ожидание) это означает, что потоки не ускоряют вычисление, а лишь чередуются, конкурируя за единственный "билет" на исполнение — суммарное время выполнения нескольких CPU-bound задач в потоках может быть даже больше, чем при последовательном выполнении, из-за накладных расходов на переключение контекста и передачу GIL между потоками.

```python
import time, threading

def cpu_task(n):
    x = 0
    for _ in range(n):
        x += 1

start = time.perf_counter()
cpu_task(50_000_000)   # однопоточно
print("Один поток:", time.perf_counter() - start)

start = time.perf_counter()
t1 = threading.Thread(target=cpu_task, args=(25_000_000,))
t2 = threading.Thread(target=cpu_task, args=(25_000_000,))
t1.start(); t2.start(); t1.join(); t2.join()
print("Два потока:", time.perf_counter() - start)  # не быстрее, часто медленнее
```

| Термин | Определение |
|---|---|
| Переключение контекста | Накладные расходы на смену выполняемого потока |
| Передача GIL | Механизм поочерёдной выдачи блокировки потокам |
| Чередование (interleaving) | Потоки выполняются по очереди, а не одновременно |

**Ответ для интервьюера:** GIL разрешает выполнять байткод Python только одному потоку одновременно вне зависимости от числа ядер, поэтому потоки для CPU-bound задачи не ускоряют вычисления, а лишь чередуются. Из-за накладных расходов на переключение контекста и передачу GIL суммарное время может даже увеличиться по сравнению с последовательным выполнением.

### 19. Когда поток отпускает GIL
**Вопрос:** В какой момент поток отпускает GIL при выполнении I/O-операций.

**Ответ:** Перед выполнением блокирующей системной операции ввода-вывода (чтение файла, сетевой запрос, `time.sleep()`) реализация функции в CPython (обычно на уровне C-расширения) явно освобождает GIL перед вызовом системного вызова и захватывает его обратно после получения результата — поскольку во время ожидания ответа от ОС интерпретатору всё равно нечего исполнять в этом потоке, и GIL можно безопасно отдать другому потоку Python для полезной работы. Для чисто вычислительного кода на чистом Python GIL также периодически освобождается интерпретатором через фиксированный интервал (настраиваемый `sys.setswitchinterval()`, по умолчанию 5 мс) для обеспечения справедливого переключения между потоками, даже если явного I/O нет.

```python
import sys
print(sys.getswitchinterval())   # интервал принудительного переключения GIL, сек
```

| Термин | Определение |
|---|---|
| Освобождение GIL перед I/O | Явный отказ от блокировки на время системного вызова |
| sys.setswitchinterval | Интервал принудительного переключения GIL между потоками |
| Захват GIL обратно | Поток снова берёт блокировку после получения результата I/O |

**Ответ для интервьюера:** Перед блокирующим системным вызовом (сеть, файл, sleep) поток явно освобождает GIL, так как во время ожидания ответа ОС интерпретатору всё равно нечего делать, и захватывает блокировку обратно после получения результата. Для чисто вычислительного кода GIL также периодически освобождается по таймеру (`sys.setswitchinterval`, по умолчанию 5 мс) ради справедливого переключения между потоками.

### 20. Как обойти ограничения GIL
**Вопрос:** Как можно обойти ограничения GIL для выполнения тяжелых вычислений.

**Ответ:** Основной способ — `multiprocessing`: каждый процесс получает собственный интерпретатор Python со своим GIL, поэтому процессы реально выполняются параллельно на разных ядрах, ценой более дорогого создания процесса и необходимости сериализации (pickle) данных при обмене между ними. Альтернативы: вынести тяжёлые вычисления в C-расширение или библиотеку вроде NumPy, которые внутри освобождают GIL на время выполнения нативного кода (векторные операции NumPy выполняются вне GIL); использовать альтернативные реализации без GIL (эксперимент free-threaded CPython, доступный начиная с Python 3.13 как опциональная сборка); либо распределить вычисления на внешние воркеры (Celery, отдельные сервисы) вместо потоков внутри одного процесса.

```python
from multiprocessing import Pool

def cpu_heavy(n):
    return sum(i * i for i in range(n))

if __name__ == "__main__":
    with Pool(processes=4) as pool:
        results = pool.map(cpu_heavy, [10_000_000] * 4)  # реальный параллелизм
```

| Термин | Определение |
|---|---|
| multiprocessing | Параллелизм через отдельные процессы, каждый со своим GIL |
| NumPy без GIL | Векторные вычисления, освобождающие GIL в нативном коде |
| Free-threaded CPython | Экспериментальная сборка Python без GIL (3.13+) |

**Ответ для интервьюера:** Главный способ — `multiprocessing`, где каждый процесс получает собственный GIL и реально исполняется параллельно на разных ядрах, ценой накладных расходов на создание процессов и сериализацию данных. Также помогают библиотеки вроде NumPy, которые освобождают GIL на время выполнения нативного кода, и появившаяся в Python 3.13 экспериментальная free-threaded сборка без GIL.

## Threading (Многопоточность)

### 21. Что такое threading
**Вопрос:** Объясни, что такое threading и в каких ситуациях его целесообразно использовать.

**Ответ:** Модуль `threading` предоставляет потоки ОС внутри одного процесса Python — лёгкие по сравнению с процессами (общая память, быстрое создание), но ограниченные GIL для CPU-bound кода. Целесообразно использовать threading для I/O-bound задач в приложениях, где по каким-то причинам не подходит asyncio (например, работа со старыми синхронными библиотеками, не имеющими async-версии) — во время ожидания I/O GIL освобождается, и другие потоки могут продолжать работу, что даёт реальный выигрыш в параллельном ожидании нескольких I/O-операций, даже без истинного параллелизма вычислений.

```python
import threading, requests

def download(url):
    resp = requests.get(url)
    print(url, resp.status_code)

urls = ["https://example.com"] * 5
threads = [threading.Thread(target=download, args=(u,)) for u in urls]
for t in threads: t.start()
for t in threads: t.join()
```

| Термин | Определение |
|---|---|
| threading | Модуль для работы с потоками ОС внутри процесса |
| Общая память потоков | Потоки одного процесса делят адресное пространство |
| I/O-bound с синхронными библиотеками | Типичный сценарий применения threading вместо asyncio |

**Ответ для интервьюера:** threading даёт потоки ОС внутри одного процесса с общей памятью — лёгкие, но ограниченные GIL для вычислений. Целесообразен для I/O-bound задач, особенно когда используются синхронные библиотеки без async-аналога — во время ожидания I/O GIL освобождается, и потоки реально работают параллельно над ожиданием, даже не имея параллелизма вычислений.

---

### 22. Как потоки делят ресурсы процесса
**Вопрос:** Как потоки распределяют ресурсы процесса (ядра процессора и память).

**Ответ:** Все потоки одного процесса разделяют общее адресное пространство памяти — глобальные переменные, объекты в куче доступны любому потоку напрямую, без копирования, что делает обмен данными между потоками дешёвым, но требует ручной синхронизации (Lock и др.) для защиты общих изменяемых данных от гонок. По ядрам процессора: ОС может физически распределять потоки одного процесса по разным ядрам, но в CPython GIL всё равно разрешает исполнять байткод Python только одному потоку одновременно — поэтому реальной параллельной загрузки нескольких ядер вычислениями Python-кода не происходит, в отличие от потоков C-расширений, которые могут работать вне GIL.

| Термин | Определение |
|---|---|
| Общее адресное пространство | Потоки одного процесса делят память напрямую |
| Синхронизация | Необходимость защищать общие данные при работе из нескольких потоков |
| Планировщик ОС | Распределяет потоки по ядрам, но GIL ограничивает Python-байткод |

**Ответ для интервьюера:** Потоки одного процесса делят общее адресное пространство памяти, что делает обмен данными дешёвым, но требует явной синхронизации через Lock для защиты общих данных. ОС может раскидывать потоки по разным ядрам, но GIL всё равно разрешает исполнять Python-байткод только одному потоку за раз, так что реальной многоядерной загрузки чистым Python-кодом не получается.

### 23. Пример threading для I/O
**Вопрос:** Приведи пример использования модуля threading для выполнения параллельных I/O-операций.

**Ответ:** Типичный сценарий — параллельная загрузка нескольких файлов или URL, где каждый поток независимо ждёт сетевой ответ, и пока один поток ждёт (GIL освобождён), другой поток может выполнять свою часть работы. Итоговое время близко к времени самого долгого отдельного запроса, а не к сумме времён всех запросов, как было бы при последовательном выполнении.

```python
import threading
import requests
import time

urls = ["https://httpbin.org/delay/1"] * 5

def fetch(url, results, index):
    resp = requests.get(url)
    results[index] = resp.status_code

results = [None] * len(urls)
threads = [threading.Thread(target=fetch, args=(u, results, i)) for i, u in enumerate(urls)]

start = time.perf_counter()
for t in threads: t.start()
for t in threads: t.join()
print(f"Заняло {time.perf_counter() - start:.2f}с")  # ~1с, а не ~5с
```

| Термин | Определение |
|---|---|
| Thread.start() | Запуск потока на выполнение |
| Thread.join() | Ожидание завершения потока |
| Параллельное ожидание | Несколько I/O-запросов ждут результата одновременно |

**Ответ для интервьюера:** Для параллельной загрузки нескольких URL создаю по потоку на запрос, запускаю их через `start()` и дожидаюсь через `join()` — поскольку GIL освобождается на время сетевого ожидания, все запросы фактически ждут одновременно, и итоговое время близко к самому долгому запросу, а не к их сумме.

### 24. ThreadPoolExecutor
**Вопрос:** Что такое ThreadPoolExecutor и в чем его преимущество перед ручным созданием потоков.

**Ответ:** `ThreadPoolExecutor` из модуля `concurrent.futures` — пул заранее созданных и переиспользуемых потоков фиксированного размера, которым можно отправлять задачи через `submit()`/`map()`, получая объект `Future` для отслеживания результата, вместо ручного создания, запуска и `join()` каждого `Thread` по отдельности. Преимущества: не нужно вручную управлять жизненным циклом потоков и следить за их количеством (что легко привело бы к перегрузке системы при создании тысяч потоков на тысячи задач), пул автоматически переиспользует потоки между задачами, а единый интерфейс `Future` унифицирован с `ProcessPoolExecutor` и с asyncio через `run_in_executor`.

```python
from concurrent.futures import ThreadPoolExecutor
import requests

urls = ["https://httpbin.org/delay/1"] * 5

with ThreadPoolExecutor(max_workers=5) as executor:
    futures = [executor.submit(requests.get, url) for url in urls]
    results = [f.result() for f in futures]
```

| Термин | Определение |
|---|---|
| ThreadPoolExecutor | Пул переиспользуемых потоков из concurrent.futures |
| submit() / map() | Способы отправки задач в пул |
| Future | Объект-обёртка над результатом асинхронно выполняемой задачи |

**Ответ для интервьюера:** ThreadPoolExecutor управляет пулом заранее созданных потоков фиксированного размера, которым отправляют задачи через `submit()`, получая `Future` для результата, вместо ручного создания и `join()` каждого потока. Это ограничивает число одновременных потоков, переиспользует их между задачами и даёт единый интерфейс с ProcessPoolExecutor и asyncio.

## Multiprocessing (Многопроцессность)

### 25. Что такое multiprocessing
**Вопрос:** Объясни, что такое multiprocessing и чем он принципиально отличается от многопоточности.

**Ответ:** Модуль `multiprocessing` создаёт полностью независимые процессы ОС, каждый со своим интерпретатором Python, своим GIL и своим изолированным адресным пространством памяти — в отличие от потоков, которые делят память и GIL одного процесса. Благодаря изоляции процессы реально выполняются параллельно на разных ядрах даже для CPU-bound кода, но за это платят более дорогим созданием (полное копирование состояния интерпретатора или fork), отсутствием прямого доступа к памяти друг друга (обмен данными требует сериализации через IPC — очереди, пайпы, общую память) и большим потреблением памяти на каждый процесс.

|Характеристика|Потоки (threading)|Процессы (multiprocessing)|
|---|---|---|
|**Память**|Общая (одно адресное пространство)|Изолированная у каждого процесса|
|**GIL**|Один общий на процесс|Свой у каждого процесса|
|**Параллелизм CPU-bound**|❌ Нет (ограничен GIL)|✅ Реальный, на разных ядрах|
|**Стоимость создания**|Дешевле|Дороже|
|**Обмен данными**|Напрямую через общую память|Через сериализацию (IPC): Queue, Pipe|

| Термин | Определение |
|---|---|
| Процесс | Независимая единица исполнения со своим адресным пространством |
| IPC | Inter-Process Communication — обмен данными между процессами |
| Изоляция памяти | Процессы не имеют прямого доступа к памяти друг друга |

**Ответ для интервьюера:** multiprocessing создаёт независимые процессы, каждый со своим интерпретатором, GIL и памятью, поэтому в отличие от потоков они реально работают параллельно на разных ядрах даже для CPU-bound кода. Плата за это — более дорогое создание процессов и необходимость сериализовывать данные для обмена между ними через IPC, так как памяти они не делят.

---

### 26. Процессы, память и ядра
**Вопрос:** Как процессы работают с оперативной памятью и ядрами процессора.

**Ответ:** Каждый процесс получает собственное изолированное виртуальное адресное пространство памяти, недоступное напрямую другим процессам — операционная система физически разделяет их через механизмы виртуальной памяти, что даёт безопасность (один процесс не может случайно испортить память другого), но требует явного механизма обмена данными (сериализация через pickle и передача через очередь, пайп или сегмент общей памяти). По ядрам: планировщик ОС может назначить каждый процесс на отдельное физическое ядро, и поскольку GIL действует только внутри одного интерпретатора/процесса, несколько процессов Python реально выполняют вычисления одновременно на разных ядрах.

| Термин | Определение |
|---|---|
| Виртуальное адресное пространство | Изолированная память каждого процесса |
| Сериализация (pickle) | Преобразование данных для передачи между процессами |
| Реальный параллелизм ядер | Достижим через процессы, но не через потоки Python |

**Ответ для интервьюера:** Каждый процесс получает изолированное виртуальное адресное пространство памяти, физически разделённое ОС, поэтому обмен данными между процессами требует сериализации и явных механизмов IPC. Планировщик ОС размещает процессы на разных ядрах, и, поскольку GIL действует только внутри одного процесса, несколько процессов Python реально считают параллельно на разных ядрах.

---

### 27. Обмен данными между процессами
**Вопрос:** Как организовать обмен данными между независимыми процессами (очереди, пайпы, общая память).

**Ответ:** `multiprocessing.Queue` — потокобезопасная и процессобезопасная очередь FIFO для передачи произвольных сериализуемых через pickle объектов между процессами, самый универсальный и часто используемый способ. `multiprocessing.Pipe` — двунаправленный канал связи между ровно двумя процессами, более лёгкий, чем очередь, но менее гибкий при большем числе участников. `multiprocessing.Value`/`Array` (и модуль `shared_memory`) дают доступ к области памяти, физически разделяемой процессами без сериализации при каждом обращении — быстрее для больших объёмов данных или частых обновлений, но требуют ручной синхронизации через `Lock`, так как несколько процессов пишут в одну и ту же память.

```python
from multiprocessing import Process, Queue

def worker(q):
    q.put("результат из процесса")

if __name__ == "__main__":
    q = Queue()
    p = Process(target=worker, args=(q,))
    p.start()
    print(q.get())
    p.join()
```

| Термин | Определение |
|---|---|
| Queue | Процессобезопасная очередь для обмена сериализуемыми объектами |
| Pipe | Двунаправленный канал между двумя процессами |
| Shared memory (Value/Array) | Физически общая память без сериализации при обращении |

**Ответ для интервьюера:** Для обмена данными между процессами использую `Queue` — универсальную процессобезопасную очередь с сериализацией через pickle, `Pipe` — для лёгкой связи между двумя процессами, или `shared_memory`/`Value`/`Array`, когда нужен быстрый доступ к общей памяти без сериализации на каждое обращение, но с ручной синхронизацией через Lock.

### 28. Пример multiprocessing
**Вопрос:** Приведи пример использования multiprocessing для ускорения параллельных вычислений.

**Ответ:** Для CPU-bound задачи (например, вычисление чего-то тяжёлого для набора входных значений) `multiprocessing.Pool` распределяет работу по нескольким процессам-воркерам, каждый из которых считает свою часть данных на отдельном ядре независимо от GIL, а результаты потом собираются обратно в основном процессе. На многоядерной машине это даёт близкое к линейному ускорение относительно числа задействованных ядер, в отличие от threading, который для такой задачи ускорения не даст вовсе.

```python
from multiprocessing import Pool
import time

def is_prime(n):
    if n < 2:
        return False
    return all(n % i != 0 for i in range(2, int(n ** 0.5) + 1))

if __name__ == "__main__":
    numbers = list(range(10_000_000, 10_010_000))

    start = time.perf_counter()
    with Pool(processes=4) as pool:
        results = pool.map(is_prime, numbers)   # реальный параллелизм на 4 ядрах
    print(f"Найдено простых: {sum(results)} за {time.perf_counter() - start:.2f}с")
```

| Термин | Определение |
|---|---|
| Pool | Пул процессов для распределения задач |
| pool.map() | Распределение элементов итерируемого объекта по воркерам |
| Линейное ускорение | Рост производительности пропорционально числу ядер |

**Ответ для интервьюера:** Использую `multiprocessing.Pool.map()`, чтобы распределить тяжёлые CPU-bound вычисления по нескольким процессам-воркерам, каждый из которых реально работает на своём ядре независимо от GIL. Это даёт близкое к линейному ускорение относительно числа ядер, чего threading для вычислительной нагрузки достичь не может.

## Проблемы параллелизма и примитивы синхронизации

### 29. Race Condition
**Вопрос:** Что такое Race Condition (состояние гонки) и почему оно возникает.

**Ответ:** Race condition возникает, когда несколько потоков (или процессов) одновременно читают и изменяют общие данные без синхронизации, и итоговый результат зависит от непредсказуемого порядка их фактического переключения — то есть от "кто кого обогнал". Классический пример — операция `counter += 1` кажется атомарной, но на самом деле состоит из трёх шагов (прочитать значение, увеличить, записать обратно), и переключение потока между любыми двумя из них может привести к потере одного из инкрементов, если два потока прочитали одно и то же старое значение до того, как кто-то из них успел записать новое.

```python
import threading

counter = 0

def increment():
    global counter
    for _ in range(100_000):
        counter += 1   # НЕ атомарно: read -> add -> write

threads = [threading.Thread(target=increment) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)   # обычно меньше 200000 из-за гонки
```

| Термин | Определение |
|---|---|
| Race condition | Результат зависит от непредсказуемого порядка переключения потоков |
| Неатомарная операция | Операция из нескольких шагов, между которыми возможно переключение |
| Общие данные | Изменяемое состояние, доступное нескольким потокам одновременно |

**Ответ для интервьюера:** Race condition возникает, когда несколько потоков одновременно читают и пишут общие данные без синхронизации, и результат зависит от случайного порядка их переключения. Даже с виду простая операция вроде `counter += 1` на самом деле состоит из нескольких шагов — чтения, изменения и записи — и переключение потока между ними может привести к потере части обновлений.

---

### 30. Deadlock
**Вопрос:** Что такое Deadlock (взаимная блокировка) и как её обнаружить.

**Ответ:** Deadlock — ситуация, когда два или более потока (процесса) навсегда блокируют друг друга, ожидая ресурсы, удерживаемые друг другом: поток A держит `Lock1` и ждёт `Lock2`, а поток B в этот момент держит `Lock2` и ждёт `Lock1` — оба замирают навсегда, потому что ни один не отпустит удерживаемый ресурс. Обнаружить на практике — по зависанию программы без ошибок и без потребления CPU (потоки просто ждут); для отладки используют профилировщики стека потоков (`faulthandler`, `py-spy`), которые показывают, на чём именно застряли потоки. Профилактика — всегда захватывать несколько блокировок в одном и том же глобальном порядке во всех местах кода, использовать таймауты в `acquire(timeout=...)` или заменять множественные блокировки на одну более крупнозернистую.

```python
import threading

lock1, lock2 = threading.Lock(), threading.Lock()

def task_a():
    with lock1:
        with lock2:   # если task_b захватит в обратном порядке — deadlock
            pass

def task_b():
    with lock2:
        with lock1:
            pass
```

| Термин | Определение |
|---|---|
| Deadlock | Взаимная блокировка потоков, ожидающих ресурсы друг друга |
| Порядок захвата блокировок | Ключевое правило профилактики deadlock |
| acquire(timeout=...) | Способ избежать вечного ожидания блокировки |

**Ответ для интервьюера:** Deadlock — когда два потока держат по одной блокировке и одновременно ждут блокировку, удерживаемую другим, замирая навсегда. Обнаруживается по зависанию программы без ошибок и без нагрузки на CPU; профилактика — всегда захватывать несколько блокировок в едином глобальном порядке или использовать таймауты `acquire(timeout=...)`.

### 31. Lock, RLock, Semaphore, Event
**Вопрос:** Объясни работу примитивов синхронизации: Lock, RLock, Semaphore, Event.

**Ответ:** `Lock` — простейшая взаимоисключающая блокировка (mutex): захвачена одним потоком — все остальные, пытающиеся захватить её же, блокируются до освобождения; повторный захват тем же потоком, который уже её держит, приведёт к deadlock самому с собой. `RLock` (reentrant lock) снимает это ограничение — один и тот же поток может захватывать её многократно (например, при рекурсивных вызовах метода с блокировкой), просто нужно освободить столько же раз, сколько захватить. `Semaphore` разрешает не одному, а сразу N потокам одновременно владеть ресурсом, что полезно для ограничения параллельного доступа к пулу из нескольких одинаковых ресурсов (например, максимум 5 одновременных соединений). `Event` — простой флаг-сигнал между потоками: одни потоки ждут (`wait()`), пока другой не установит его (`set()`), удобно для сценария "жди, пока не произойдёт событие".

|Примитив|Сколько владельцев|Повторный захват тем же потоком|Назначение|
|---|---|---|---|
|**Lock**|1|❌ Deadlock|Простой mutex|
|**RLock**|1|✅ Разрешён|Рекурсивные вызовы с блокировкой|
|**Semaphore**|N|—|Ограничение доступа к пулу ресурсов|
|**Event**|—|—|Флаг-сигнал между потоками|

```python
import threading

lock = threading.Lock()
rlock = threading.RLock()
sem = threading.Semaphore(3)   # не более 3 потоков одновременно
event = threading.Event()

def wait_for_signal():
    event.wait()      # блокируется, пока кто-то не вызовет event.set()
    print("Сигнал получен")
```

| Термин | Определение |
|---|---|
| Lock | Простой mutex, эксклюзивный доступ одному потоку |
| RLock | Реентерабельная блокировка, тот же поток может захватывать повторно |
| Semaphore | Ограничение числа одновременных владельцев ресурса |
| Event | Флаг-сигнал между потоками |

**Ответ для интервьюера:** Lock — простая взаимоисключающая блокировка на одного владельца, RLock допускает повторный захват тем же потоком без deadlock (полезно при рекурсии). Semaphore ограничивает число одновременных владельцев ресурса до N, а Event — простой флаг-сигнал, позволяющий одним потокам ждать, пока другой не подаст сигнал через `set()`.

### 32. Пример использования Lock
**Вопрос:** Приведи пример ситуации, где необходимо использовать Lock для защиты общих данных (например, словаря).

**Ответ:** Если несколько потоков одновременно читают и модифицируют общий словарь (например, счётчики посещений по ключу или общий кэш), операции вроде `d[key] = d.get(key, 0) + 1` не атомарны и подвержены race condition — `Lock` гарантирует, что в любой момент только один поток выполняет критическую секцию кода, работающую с общими данными, сериализуя доступ и устраняя гонку.

```python
import threading

counters = {}
lock = threading.Lock()

def increment(key):
    with lock:              # критическая секция — только один поток за раз
        counters[key] = counters.get(key, 0) + 1

threads = [threading.Thread(target=increment, args=("visits",)) for _ in range(1000)]
for t in threads: t.start()
for t in threads: t.join()
print(counters["visits"])   # ровно 1000 благодаря Lock
```

| Термин | Определение |
|---|---|
| Критическая секция | Участок кода, работающий с общими данными, требующий защиты |
| with lock | Идиоматичный способ гарантированного освобождения блокировки |
| Сериализация доступа | Только один поток одновременно работает с общими данными |

**Ответ для интервьюера:** Когда несколько потоков читают и изменяют общий словарь, операции вроде инкремента счётчика по ключу неатомарны и подвержены гонке — оборачиваю такой участок кода (критическую секцию) в `with lock`, чтобы гарантировать, что только один поток одновременно работает с общими данными, устраняя race condition.

### 33. run_in_executor / to_thread
**Вопрос:** Как запустить синхронный блокирующий код внутри асинхронного приложения, не блокируя Event Loop (run_in_executor, to_thread).

**Ответ:** `loop.run_in_executor(executor, func, *args)` отправляет блокирующую синхронную функцию на выполнение в отдельный поток (по умолчанию — во встроенный `ThreadPoolExecutor`) или процесс (если передать `ProcessPoolExecutor`), возвращая awaitable, который можно ждать через `await`, не блокируя при этом сам Event Loop — цикл событий продолжает обрабатывать другие корутины, пока блокирующий код выполняется в фоновом потоке. `asyncio.to_thread(func, *args)` (с Python 3.9) — упрощённая обёртка над тем же механизмом специально для потоков, более читаемый и короткий способ записи типового случая.

```python
import asyncio
import time

def blocking_io():
    time.sleep(2)   # синхронная блокирующая операция
    return "готово"

async def main():
    result = await asyncio.to_thread(blocking_io)   # не блокирует Event Loop
    print(result)

asyncio.run(main())
```

| Термин | Определение |
|---|---|
| run_in_executor | Выполнение блокирующего кода в отдельном потоке/процессе из asyncio |
| asyncio.to_thread | Упрощённая обёртка над run_in_executor для потоков |
| ThreadPoolExecutor по умолчанию | Executor, используемый run_in_executor без явного указания |

**Ответ для интервьюера:** Блокирующий синхронный код выношу в отдельный поток через `asyncio.to_thread()` (или более гибкий `loop.run_in_executor()`), что возвращает awaitable и не блокирует Event Loop — цикл событий продолжает обрабатывать другие корутины, пока блокирующая функция выполняется в фоне. Это стандартный способ интегрировать легаси-синхронный код в асинхронное приложение.

### 34. Критерии выбора async/threading/multiprocessing
**Вопрос:** Каковы критерии выбора между async/await, threading и multiprocessing для конкретного проекта.

**Ответ:** Главный критерий — природа нагрузки: для I/O-bound с большим числом одновременных ожиданий и контролем над всем стеком (написан с нуля, есть async-библиотеки) выбирают asyncio — минимальные накладные расходы, тысячи конкурентных задач в одном потоке. Для I/O-bound, но с обязательными синхронными/легаси библиотеками без async-аналога — threading, как более простая интеграция ценой больших накладных расходов на поток по сравнению с корутиной. Для CPU-bound вычислений — только multiprocessing, так как ни asyncio, ни threading не дают реального параллелизма вычислений из-за GIL. Дополнительно учитывают сложность отладки (multiprocessing сложнее из-за IPC и сериализации), объём переиспользуемых существующих библиотек и требуемую масштабируемость (тысячи соединений — только asyncio).

|Критерий|asyncio|threading|multiprocessing|
|---|---|---|---|
|**Тип нагрузки**|I/O-bound|I/O-bound (легаси-код)|CPU-bound|
|**Параллелизм вычислений**|❌|❌ (ограничен GIL)|✅|
|**Накладные расходы на единицу**|Минимальные (корутина)|Средние (поток)|Высокие (процесс)|
|**Масштаб конкурентности**|Тысячи задач|Сотни потоков|Число ядер CPU|
|**Совместимость с sync-библиотеками**|Нужны async-аналоги|✅ Работает как есть|✅ Работает как есть|

| Термин | Определение |
|---|---|
| Природа нагрузки | Главный критерий выбора: I/O-bound vs CPU-bound |
| Накладные расходы на единицу конкурентности | Корутина дешевле потока, поток дешевле процесса |
| Экосистема библиотек | Наличие async-версий нужных библиотек влияет на выбор |

**Ответ для интервьюера:** Выбор определяется природой нагрузки: I/O-bound с async-совместимыми библиотеками — asyncio, I/O-bound с легаси-синхронным кодом — threading, CPU-bound вычисления — только multiprocessing, так как GIL не даёт параллелизма вычислений ни asyncio, ни threading. Дополнительно учитываю накладные расходы на единицу конкурентности — корутина дешевле потока, поток дешевле процесса.

### 35. Примеры задач для каждого подхода
**Вопрос:** Приведи примеры реальных задач, где наиболее эффективен каждый из подходов.

**Ответ:** asyncio эффективен для веб-сервера с тысячами одновременных HTTP-соединений (FastAPI, aiohttp-сервер), веб-скрапера, параллельно опрашивающего сотни URL, или чат-бота с множеством одновременных WebSocket-соединений. threading подходит для GUI-приложения, где нужно не заморозить интерфейс при фоновой синхронной операции, или интеграции с синхронной legacy-библиотекой (некоторые драйверы БД, устаревшие SDK) в рамках уже написанного приложения. multiprocessing эффективен для обработки изображений/видео, численных расчётов (научные вычисления, ML-инференс на CPU), парсинга и трансформации больших датасетов — везде, где требуется реальная параллельная загрузка нескольких ядер вычислениями.

| Термин | Определение |
|---|---|
| Веб-сервер с массой соединений | Типовой сценарий для asyncio |
| GUI без заморозки интерфейса | Типовой сценарий для threading |
| Обработка изображений/видео | Типовой сценарий для multiprocessing |

**Ответ для интервьюера:** asyncio — для веб-серверов с тысячами одновременных соединений и массового опроса внешних API. threading — для GUI-приложений, где нельзя замораживать интерфейс, и интеграции с синхронными legacy-библиотеками. multiprocessing — для CPU-тяжёлой обработки изображений, численных расчётов и трансформации больших датасетов, где нужна реальная параллельная загрузка нескольких ядер.

### 36. aiohttp для асинхронных запросов
**Вопрос:** Как использовать библиотеку aiohttp для выполнения асинхронных HTTP-запросов.

**Ответ:** `aiohttp` — асинхронная HTTP-библиотека, где основная сущность — `ClientSession`, которую создают один раз (обычно через `async with`) и переиспользуют для множества запросов, так как она держит пул соединений (connection pooling) и переиспользует TCP-соединения между запросами к одному хосту (keep-alive), что заметно эффективнее создания нового соединения на каждый запрос. Сам запрос делают через `session.get()`/`session.post()` внутри `async with`, а тело ответа читают асинхронно через `await response.text()`/`.json()`.

```python
import aiohttp
import asyncio

async def fetch(session, url):
    async with session.get(url) as response:
        return await response.json()

async def main():
    async with aiohttp.ClientSession() as session:
        data = await fetch(session, "https://api.example.com/data")
        print(data)

asyncio.run(main())
```

| Термин | Определение |
|---|---|
| ClientSession | Переиспользуемая сессия с пулом соединений aiohttp |
| Connection pooling | Переиспользование TCP-соединений между запросами |
| async with response | Асинхронное чтение тела HTTP-ответа |

**Ответ для интервьюера:** В aiohttp создаю одну `ClientSession` через `async with` и переиспользую её для всех запросов, так как она держит пул TCP-соединений с keep-alive. Сами запросы делаю через `session.get()`/`post()` внутри `async with`, а тело ответа читаю асинхронно через `await response.json()` или `.text()`.

### 37. aiohttp/httpx vs requests
**Вопрос:** В чем преимущества aiohttp или httpx по сравнению со стандартной библиотекой requests.

**Ответ:** `requests` — синхронная библиотека: каждый запрос блокирует поток до получения ответа, поэтому N последовательных запросов займут сумму их времён. `aiohttp` и `httpx` (в асинхронном режиме) позволяют выполнять множество запросов конкурентно в одной корутине через `asyncio.gather`, дожидаясь самого долгого, а не суммы всех — критично при большом числе внешних вызовов (агрегация данных из нескольких API). `httpx` дополнительно интересен тем, что поддерживает и синхронный, и асинхронный API с почти идентичным интерфейсом (в отличие от aiohttp, который асинхронен изначально и целиком), что упрощает миграцию существующего синхронного кода на requests-подобный API в асинхронный.

|Характеристика|requests|aiohttp|httpx|
|---|---|---|---|
|**Модель**|Синхронная|Асинхронная|Синхронная и асинхронная|
|**N запросов, суммарное время**|Сумма всех|~Время самого долгого|~Время самого долгого (в async-режиме)|
|**API похож на requests**|—|Отличается|✅ Почти идентичен|

```python
import httpx, asyncio

async def main():
    async with httpx.AsyncClient() as client:
        responses = await asyncio.gather(
            client.get("https://api.example.com/a"),
            client.get("https://api.example.com/b"),
        )
```

| Термин | Определение |
|---|---|
| Синхронный запрос (requests) | Блокирует поток до получения ответа |
| Конкурентные запросы | Множество запросов выполняются одновременно в одной корутине |
| httpx.AsyncClient | Асинхронный клиент httpx с requests-подобным API |

**Ответ для интервьюера:** requests блокирует поток на каждом запросе, поэтому N запросов выполняются суммарно за N времён. aiohttp и httpx позволяют выполнять запросы конкурентно через `asyncio.gather`, занимая время самого долгого запроса, а не их суммы, что критично при множественных вызовах внешних API — httpx при этом даёт почти идентичный синхронный и асинхронный API.

### 38. Параллельные HTTP-запросы
**Вопрос:** Приведи пример выполнения нескольких параллельных HTTP-запросов.

**Ответ:** Несколько независимых HTTP-запросов выполняют конкурентно через `asyncio.gather`, передав список корутин с запросами внутри одной `ClientSession` — итоговое время близко к самому долгому запросу вместо суммы всех, а на практике ограничивать число одновременных запросов стоит через `asyncio.Semaphore`, чтобы не перегружать целевой сервер и не упереться в лимиты на число открытых соединений.

```python
import aiohttp
import asyncio

async def fetch(session, url, semaphore):
    async with semaphore:
        async with session.get(url) as response:
            return await response.json()

async def main():
    urls = [f"https://api.example.com/items/{i}" for i in range(20)]
    semaphore = asyncio.Semaphore(5)   # не более 5 одновременных запросов

    async with aiohttp.ClientSession() as session:
        results = await asyncio.gather(*[fetch(session, u, semaphore) for u in urls])
    return results

asyncio.run(main())
```

| Термин | Определение |
|---|---|
| asyncio.gather | Конкурентный запуск нескольких запросов сразу |
| asyncio.Semaphore | Ограничение числа одновременно выполняемых запросов |
| Единая ClientSession | Общий пул соединений для всех запросов |

**Ответ для интервьюера:** Собираю список корутин с запросами внутри одной `ClientSession` и запускаю их конкурентно через `asyncio.gather`, что даёт время выполнения около самого долгого запроса вместо суммы всех. Число одновременных запросов дополнительно ограничиваю через `asyncio.Semaphore`, чтобы не перегрузить целевой сервер или не упереться в лимит соединений.

### 39. Тестирование асинхронного кода
**Вопрос:** Как организовать тестирование асинхронного кода.

**Ответ:** Основной инструмент — плагин `pytest-asyncio`, который позволяет писать тестовые функции как `async def` с маркером `@pytest.mark.asyncio`, а сам плагин берёт на себя запуск Event Loop для теста. Для мокирования внешних асинхронных вызовов (HTTP-запросы, БД) используют `unittest.mock.AsyncMock` вместо обычного `Mock`, так как обычный мок не поддерживает `await`. Для проверки поведения при конкурентности (race condition, deadlock) полезно тестировать сценарии явно через `asyncio.gather` с несколькими одновременными вызовами, а не полагаться только на изолированные единичные тесты.

```python
import pytest
from unittest.mock import AsyncMock

@pytest.mark.asyncio
async def test_fetch_data():
    mock_session = AsyncMock()
    mock_session.get.return_value.__aenter__.return_value.json.return_value = {"ok": True}

    result = await fetch_data(mock_session, "https://api.example.com")
    assert result == {"ok": True}
```

| Термин | Определение |
|---|---|
| pytest-asyncio | Плагин для тестирования async-функций в pytest |
| AsyncMock | Мок, поддерживающий await, в отличие от обычного Mock |
| Тестирование конкурентности | Проверка поведения при одновременных вызовах через gather |

**Ответ для интервьюера:** Тестирую асинхронный код через `pytest-asyncio`, помечая тестовые функции `@pytest.mark.asyncio` — плагин сам запускает Event Loop для теста. Для мокирования асинхронных зависимостей использую `AsyncMock` вместо обычного `Mock`, так как только он поддерживает `await`, а поведение при конкурентности проверяю отдельными тестами с `asyncio.gather`.

## Разобрать по темам

### 40. gather vs последовательные await
**Вопрос:** asyncio.gather vs последовательные await — когда что использовать

**Ответ:** Последовательные `await` (`result1 = await task1(); result2 = await task2()`) выполняют корутины строго одну за другой — суммарное время равно сумме их времён, и это оправдано, только если вторая операция логически зависит от результата первой. `asyncio.gather` запускает независимые корутины конкурентно, и суммарное время близко к самому долгому из вызовов — используют, когда операции независимы друг от друга и порядок их выполнения не важен, что почти всегда даёт выигрыш в реальных сценариях с несколькими независимыми I/O-вызовами.

|Характеристика|Последовательные await|asyncio.gather|
|---|---|---|
|**Суммарное время**|Сумма времён операций|Время самой долгой операции|
|**Когда уместно**|Операции зависят друг от друга|Операции независимы|

```python
# Последовательно: суммарно ~2с (зависимость есть или её ошибочно не убрали)
user = await get_user(user_id)
orders = await get_orders(user_id)

# Конкурентно: суммарно ~1с (операции независимы)
user, orders = await asyncio.gather(get_user(user_id), get_orders(user_id))
```

| Термин | Определение |
|---|---|
| Последовательный await | Суммарное время равно сумме времён операций |
| asyncio.gather | Суммарное время равно времени самой долгой операции |
| Зависимость между операциями | Ключевой критерий выбора между подходами |

**Ответ для интервьюера:** Последовательные `await` использую только тогда, когда одна операция реально зависит от результата другой. Если операции независимы, всегда предпочитаю `asyncio.gather` — это сводит суммарное время к времени самого долгого вызова вместо их суммы, что почти всегда даёт заметный выигрыш при нескольких независимых I/O-операциях.

### 41. Забытый await
**Вопрос:** Забытый await

**Ответ:** Если вызвать корутинную функцию без `await` (`result = fetch_data()` вместо `result = await fetch_data()`), она не выполнится — вызов лишь создаёт объект-корутину, который так и останется незапущенным, а `result` будет содержать сам объект `coroutine`, а не ожидаемое значение. Python обычно предупреждает об этом через `RuntimeWarning: coroutine 'fetch_data' was never awaited` при сборке мусора, но код при этом не падает с ошибкой сразу в месте вызова, из-за чего такую ошибку легко не заметить, особенно если результат корутины не используется напрямую (например, "запустить и забыть" по ошибке вместо `create_task`).

```python
async def fetch_data():
    await asyncio.sleep(1)
    return 42

async def main():
    result = fetch_data()      # забыли await!
    print(result)                # <coroutine object fetch_data at 0x...>
    print(await result)          # так — правильно, 42
```

| Термин | Определение |
|---|---|
| Незапущенная корутина | Результат вызова async def без await или create_task |
| RuntimeWarning | Предупреждение о never awaited корутине |
| never awaited | Признак забытого await, обнаруживаемый при сборке мусора |

**Ответ для интервьюера:** Забытый `await` означает, что корутина не выполнилась — вызов лишь создал объект-корутину без запуска, и переменная содержит сам этот объект, а не результат. Python выдаст `RuntimeWarning: coroutine was never awaited` при сборке мусора, но ошибка не проявляется немедленно в месте вызова, поэтому такой баг легко пропустить без внимательного code review или линтера.

### 42. asyncio.Lock vs обычный лок
**Вопрос:** Разница asyncio.Lock vs обычный лок

**Ответ:** `threading.Lock` — блокирующий примитив уровня ОС: поток, пытающийся захватить занятую блокировку, физически приостанавливается (блокируется), что для asyncio-приложения означало бы заморозку всего Event Loop (все корутины в этом потоке встали бы). `asyncio.Lock` — кооперативный примитив: при попытке захватить занятую блокировку корутина не блокирует поток физически, а отдаёт управление Event Loop через `await`, позволяя другим корутинам работать, пока блокировка не освободится. Использовать `threading.Lock` внутри корутин нельзя (это заблокирует весь цикл событий), а `asyncio.Lock` бессмысленно использовать между настоящими потоками ОС — она не даёт межпоточной защиты, только между корутинами одного Event Loop.

|Характеристика|threading.Lock|asyncio.Lock|
|---|---|---|
|**При ожидании**|Физически блокирует поток ОС|Отдаёт управление Event Loop через await|
|**Использовать в корутинах**|❌ Заморозит весь Event Loop|✅|
|**Защита между потоками ОС**|✅|❌ Только между корутинами одного Event Loop|

```python
import asyncio

lock = asyncio.Lock()
shared_state = {}

async def update_state(key, value):
    async with lock:              # не блокирует поток, а await'ит освобождение
        shared_state[key] = value
        await asyncio.sleep(0.1)
```

| Термин | Определение |
|---|---|
| threading.Lock | Блокирует поток физически на уровне ОС |
| asyncio.Lock | Кооперативная блокировка, отдающая управление Event Loop |
| async with lock | Идиоматичный захват asyncio.Lock без блокировки потока |

**Ответ для интервьюера:** `threading.Lock` физически блокирует поток ОС при ожидании, что внутри корутины заморозило бы весь Event Loop. `asyncio.Lock` — кооперативный примитив: при ожидании корутина отдаёт управление циклу событий через `await` вместо блокировки потока, поэтому в асинхронном коде для защиты общих данных между корутинами нужно использовать именно `asyncio.Lock`, а не `threading.Lock`.
