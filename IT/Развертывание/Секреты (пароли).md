### 🔹 Где хранить секреты? (По возрастанию безопасности)

| Место                 | Пример                               | Когда использовать               |
| --------------------- | ------------------------------------ | -------------------------------- |
| ❌ **В коде**          | `password = "123"`                   | **Никогда** (утечка в git)       |
| ❌ **Локальный .env**  | `.env` без `.gitignore`              | Только локальная отладка         |
| ⚠️ **Docker Secrets** | `docker secret create`               | Swarm-кластеры, но сложно в K8s  |
| ✅ **CI/CD Variables** | GitHub Secrets, GitLab CI/CD         | **Стандарт для пайплайнов**      |
| 🔐 **Vault**          | HashiCorp Vault, AWS Secrets Manager | Enterprise, динамические секреты |

### 🔹 Как это работает в пайплайне?
```yaml
# GitHub Actions пример
jobs:
  deploy:
    steps:
      - name: Deploy
        env:
          DB_PASSWORD: ${{ secrets.DB_PASSWORD }}  # ✅ Подстановка из Settings
        run: python deploy.py
```
> **Важно:** В логах секрет маскируется (`***`), чтобы не утечь.

### 🔹 Best Practices
1.  **Принцип наименьших привилегий:** Доступ к редактированию — только лидам/админам.
2.  **Rotation:** Регулярная смена токенов/паролей.
3.  **Не логировать:** Проверять, что секреты не попадают в `print()` или логи приложения.
4.  **Разделение окружений:** Разные секреты для dev/stage/prod.

### 🔹 Для интервью
> «Секреты хранятся в **защищённом хранилище CI/CD** (GitHub Secrets, GitLab Variables). Они подставляются в рантайме как env-переменные, маскируются в логах и не попадают в репозиторий. Для продакшена с высокими требованиями используют HashiCorp Vault.»

4. **ОСТАТОК СИМВОЛОВ** (~300) 🚀🔐

Секреты хранят в Settings → Secrets/Variables, подставляют в job.
