## Основы Docker и контейнеризации

### 1. Что такое Docker и его преимущества
**Вопрос:** Что такое Docker и какие преимущества он дает при разработке и развертывании?

**Ответ:** Docker — это платформа для контейнеризации приложений, упаковывающая код вместе со всеми зависимостями, системными библиотеками и настройками окружения в единый переносимый образ. Контейнеры используют ядро хостовой ОС (в отличие от виртуальных машин с полноценным гипервизором), поэтому запускаются за секунды и потребляют значительно меньше ресурсов. Главные преимущества: воспроизводимость окружения («работает у меня» перестаёт быть проблемой, потому что dev, staging и prod используют один и тот же образ), изоляция зависимостей разных приложений друг от друга, быстрое горизонтальное масштабирование и упрощённый CI/CD за счёт единого артефакта развёртывания.

```bash
docker build -t myapp:1.0 .
docker run -d -p 8000:8000 myapp:1.0
```

| Термин | Определение |
|---|---|
| Контейнеризация | Упаковка приложения с зависимостями в изолированный переносимый юнит |
| Образ (image) | Неизменяемый шаблон для создания контейнеров |
| Воспроизводимость окружения | Идентичное поведение приложения на всех этапах развёртывания |

**Ответ для интервьюера:** Docker упаковывает приложение вместе со всеми зависимостями в переносимый образ, который работает одинаково на любой машине с Docker. Это решает проблему рассинхронизации окружений между разработкой и продакшеном, а также ускоряет деплой и масштабирование за счёт лёгкости контейнеров по сравнению с виртуальными машинами.

### 2. Docker-образ vs контейнер
**Вопрос:** В чем разница между Docker-образом и контейнером?

**Ответ:** Образ (image) — это неизменяемый шаблон только для чтения, состоящий из слоёв файловой системы, инструкций и метаданных, который описывает, что должно быть внутри и как приложение запускается. Контейнер — это работающий (или остановленный) экземпляр образа: при запуске Docker добавляет поверх слоёв образа тонкий изменяемый слой (writable layer), в который пишутся все изменения во время работы. Из одного образа можно запустить сколько угодно независимых контейнеров, точно так же как из одного класса можно создать множество объектов.

|Характеристика|Image|Container|
|---|---|---|
|**Изменяемость**|Неизменяемый (read-only)|Есть writable-слой поверх образа|
|**Состояние**|Статичный шаблон|Запущенный (или остановленный) экземпляр|
|**Количество из одного шаблона**|—|Множество независимых контейнеров|
|**Аналогия из ООП**|Класс|Объект (экземпляр)|

```bash
docker images            # список образов
docker ps -a              # список контейнеров (включая остановленные)
docker run --name c1 myapp:1.0
docker run --name c2 myapp:1.0   # второй независимый контейнер из того же образа
```

| Термин | Определение |
|---|---|
| Image | Неизменяемый шаблон файловой системы и метаданных |
| Container | Запущенный экземпляр образа с writable-слоем |
| Writable layer | Изменяемый верхний слой контейнера, уничтожается при удалении контейнера |

**Ответ для интервьюера:** Образ — это статичный неизменяемый шаблон, а контейнер — его запущенный экземпляр с добавленным поверх writable-слоем для изменений во время работы. Аналогия — класс и объект: из одного образа можно поднять множество независимых контейнеров.

### 3. Изоляция среды исполнения в Docker
**Вопрос:** Как Docker обеспечивает изоляцию среды исполнения?

**Ответ:** Docker использует механизмы ядра Linux — namespaces и cgroups. Namespaces изолируют видимость ресурсов: PID namespace скрывает процессы других контейнеров, network namespace даёт отдельный сетевой стек и IP, mount namespace — свою файловую систему, UTS namespace — своё имя хоста. Cgroups (control groups) ограничивают и учитывают потребление ресурсов — CPU, память, I/O, — не позволяя одному контейнеру исчерпать ресурсы хоста в ущерб остальным. В отличие от виртуальных машин, все контейнеры на одном хосте делят одно ядро ОС, что и делает их значительно легче.

```bash
docker run -d --memory=512m --cpus=1.5 myapp:1.0
```

| Термин | Определение |
|---|---|
| Namespace | Механизм ядра, изолирующий видимость ресурсов (PID, network, mount) |
| cgroups | Control groups — ограничение и учёт ресурсов CPU/памяти/I-O |
| Общее ядро ОС | Все контейнеры хоста используют одно ядро, в отличие от VM |

**Ответ для интервьюера:** Изоляция строится на namespaces ядра Linux, которые разделяют видимость процессов, сети и файловой системы между контейнерами, и на cgroups, которые ограничивают потребление CPU и памяти каждым контейнером. При этом все контейнеры делят одно ядро хоста, что делает их значительно легче полноценных виртуальных машин.

### 4. Docker vs venv
**Вопрос:** Чем Docker отличается от обычного виртуального окружения Python (venv)?

**Ответ:** venv изолирует только Python-зависимости и интерпретатор внутри уже существующей ОС — системные библиотеки, версия ОС, переменные окружения и прочие процессы остаются общими с хостом. Docker изолирует всё окружение целиком: свою файловую систему, сетевой стек, системные пакеты и даже, при необходимости, другую версию дистрибутива Linux, независимо от того, что установлено на хосте. Поэтому venv решает только проблему конфликта Python-пакетов между проектами, а Docker решает проблему воспроизводимости всего окружения приложения на любой машине.

|Характеристика|venv|Docker|
|---|---|---|
|**Что изолирует**|Только Python-пакеты и интерпретатор|Всё окружение: ФС, сеть, системные пакеты, ОС|
|**Системные библиотеки**|Общие с хостом|Изолированы|
|**Переносимость на другую машину**|Не гарантирована|Гарантирована (тот же образ)|
|**Решает проблему**|Конфликт версий Python-пакетов|Полную воспроизводимость окружения|

```bash
# venv — изолирует только пакеты Python
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Docker — изолирует всё окружение целиком
docker build -t myapp .
docker run myapp
```

| Термин | Определение |
|---|---|
| venv | Изолированное окружение Python-пакетов внутри существующей ОС |
| Изоляция уровня ОС | Docker изолирует файловую систему, сеть и системные библиотеки |
| Переносимость | Способность запускаться идентично на любом хосте с Docker |

**Ответ для интервьюера:** venv изолирует только зависимости и интерпретатор Python внутри той же ОС, тогда как Docker упаковывает всё окружение целиком — системные библиотеки, сеть, файловую систему — и гарантирует одинаковое поведение независимо от хостовой машины. venv решает конфликт версий пакетов, Docker решает полную воспроизводимость окружения.

### 5. Основные инструкции Dockerfile
**Вопрос:** Назовите основные инструкции Dockerfile и их назначение.

**Ответ:** `FROM` задаёт базовый образ, от которого строится новый. `WORKDIR` устанавливает рабочую директорию внутри образа для последующих команд. `COPY`/`ADD` копируют файлы с хоста в образ (ADD дополнительно умеет распаковывать архивы и скачивать по URL, поэтому COPY предпочтительнее как более явный и предсказуемый). `RUN` выполняет команду на этапе сборки образа и создаёт новый слой. `ENV` задаёт переменные окружения. `EXPOSE` документирует, какой порт слушает приложение (не публикует его сам по себе). `CMD` задаёт команду по умолчанию при запуске контейнера (может быть переопределена), а `ENTRYPOINT` задаёт неизменяемую точку входа, часто в связке с CMD как аргументами по умолчанию.

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
ENV PYTHONUNBUFFERED=1
EXPOSE 8000
CMD ["python", "app.py"]
```

| Термин | Определение |
|---|---|
| FROM | Базовый образ для сборки |
| RUN | Команда, выполняемая на этапе сборки образа |
| CMD / ENTRYPOINT | Команда запуска контейнера по умолчанию / неизменяемая точка входа |
| EXPOSE | Документирование порта приложения (без реальной публикации) |

**Ответ для интервьюера:** FROM задаёт базовый образ, WORKDIR — рабочую директорию, COPY переносит файлы с хоста, RUN выполняет команды на этапе сборки создавая новый слой, а CMD/ENTRYPOINT определяют, что выполнится при запуске контейнера. CMD легко переопределить снаружи, ENTRYPOINT — фиксированная точка входа, их часто комбинируют.

## Оптимизация Docker-образов

### 6. Порядок копирования файлов в Dockerfile
**Вопрос:** Зачем в Dockerfile сначала копируются файлы зависимостей (например, requirements.txt), а только потом код приложения?

**Ответ:** Docker кэширует каждый слой сборки и переиспользует кэш, пока содержимое соответствующей инструкции и предыдущих слоёв не изменилось. Код приложения меняется практически при каждом коммите, а зависимости — гораздо реже, поэтому если скопировать requirements.txt и установить зависимости отдельным, более ранним слоем, а сам код — отдельным поздним слоем, то при изменении только кода Docker переиспользует закэшированный слой с уже установленными зависимостями и не будет переустанавливать их заново. Это резко ускоряет повторные сборки образа в CI и локально.

```dockerfile
# Правильно: зависимости кэшируются отдельно от кода
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

# Неправильно: любое изменение кода инвалидирует кэш установки зависимостей
COPY . .
RUN pip install --no-cache-dir -r requirements.txt
```

| Термин | Определение |
|---|---|
| Layer caching | Переиспользование ранее собранных слоёв при неизменном содержимом |
| Инвалидация кэша | Сброс кэша слоя при изменении входных данных инструкции |
| Build context | Набор файлов, отправляемых демону Docker при сборке |

**Ответ для интервьюера:** Docker кэширует слои по порядку, и если изменился слой, все последующие пересобираются заново. Зависимости меняются реже кода, поэтому их устанавливают отдельным ранним слоем — тогда изменение кода не инвалидирует кэш установки пакетов, и пересборка образа проходит намного быстрее.

### 7. Кэширование слоев Docker
**Вопрос:** Как работает механизм кэширования слоев в Docker?

**Ответ:** Каждая инструкция Dockerfile создаёт отдельный слой файловой системы, и Docker вычисляет для него хэш на основе самой инструкции и содержимого затрагиваемых файлов (для COPY/ADD — хэш содержимого копируемых файлов). Если хэш совпадает с ранее собранным слоем и родительские слои тоже не изменились, Docker берёт слой из кэша вместо повторного выполнения инструкции. Как только один слой инвалидируется (например, изменился файл или сама команда), все последующие слои пересобираются заново, даже если их содержимое формально не изменилось бы — поэтому порядок инструкций в Dockerfile напрямую влияет на скорость сборки.

```bash
docker build --no-cache -t myapp .     # принудительная пересборка без кэша
docker build --progress=plain -t myapp .  # подробный вывод, видно cached/не cached слои
```

| Термин | Определение |
|---|---|
| Слой (layer) | Неизменяемый снимок файловой системы, созданный одной инструкцией |
| Хэш слоя | Контрольная сумма инструкции и её входных данных |
| Каскадная инвалидация | Сброс кэша всех последующих слоёв после первого изменённого |

**Ответ для интервьюера:** Каждая инструкция создаёт слой с хэшем от своего содержимого; если хэш совпадает с предыдущей сборкой, Docker переиспользует слой из кэша. Как только один слой меняется, все слои после него пересобираются заново, поэтому редко меняющиеся инструкции (установка зависимостей) ставят раньше часто меняющихся (копирование кода).

### 8. Alpine, Slim и полные базовые образы
**Вопрос:** В чем разница между базовыми образами Alpine, Slim и полными версиями?

**Ответ:** Полный образ (например, `python:3.12`) построен на полноценном дистрибутиве (Debian) со множеством предустановленных утилит и библиотек, весит сотни мегабайт, но максимально совместим — почти любой пакет соберётся без проблем. Slim-образ (`python:3.12-slim`) — тот же Debian, но с минимальным набором пакетов, без документации, компиляторов и лишних утилит, весит в разы меньше, но иногда требует доустановки build-зависимостей для сборки некоторых Python-пакетов с C-расширениями. Alpine-образ построен на Alpine Linux с musl libc вместо glibc и весит ещё меньше (единицы-десятки МБ), но musl несовместим с некоторыми скомпилированными wheel-пакетами (например, часть научных библиотек), из-за чего сборка иногда идёт дольше или требует дополнительных системных пакетов через apk.

|Образ|Размер|Библиотека C|Совместимость|
|---|---|---|---|
|**Full**|~1 ГБ|glibc|Максимальная|
|**Slim**|~150 МБ|glibc|Хорошая, иногда нужны build-зависимости|
|**Alpine**|~50 МБ|musl|Могут ломаться некоторые C-расширения/wheel|

```dockerfile
FROM python:3.12            # ~1 ГБ, максимальная совместимость
FROM python:3.12-slim       # ~150 МБ, компромисс
FROM python:3.12-alpine     # ~50 МБ, musl libc, возможны проблемы совместимости
```

| Термин | Определение |
|---|---|
| Debian-based (slim/full) | Образ на базе Debian с glibc |
| Alpine | Минималистичный дистрибутив с musl libc вместо glibc |
| musl libc | Альтернативная реализация стандартной библиотеки C, менее совместима с некоторыми wheel |

**Ответ для интервьюера:** Полный образ — максимальная совместимость ценой размера, slim — тот же Debian без лишних пакетов и значительно легче, alpine — самый маленький, но на musl libc, что иногда ломает сборку C-расширений Python. На практике чаще всего выбирают slim как разумный баланс размера и совместимости.

### 9. Multi-stage builds
**Вопрос:** Что такое Multi-stage builds и для чего они применяются?

**Ответ:** Multi-stage build — это Dockerfile с несколькими последовательными секциями `FROM`, где на промежуточных стадиях выполняется сборка (компиляция, установка build-зависимостей, сборка фронтенда), а в финальный образ через `COPY --from=<stage>` переносятся только нужные артефакты — без компиляторов, кэша пакетных менеджеров и промежуточных файлов. Это позволяет держать в финальном образе только рантайм-зависимости, резко уменьшая его размер и площадь атаки, при этом не жертвуя удобством использования тяжёлых инструментов сборки на промежуточных стадиях.

```dockerfile
# Стадия сборки
FROM python:3.12 AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# Финальная стадия — только рантайм
FROM python:3.12-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY . .
ENV PATH=/root/.local/bin:$PATH
CMD ["python", "app.py"]
```

| Термин | Определение |
|---|---|
| Stage | Отдельный этап сборки, начинающийся с FROM |
| COPY --from | Копирование артефактов из другой стадии в текущую |
| Площадь атаки | Совокупность лишнего ПО в образе, потенциально уязвимого |

**Ответ для интервьюера:** Multi-stage build разделяет Dockerfile на несколько стадий: на одной собираются зависимости или бинарники со всеми build-инструментами, а в финальную стадию через COPY --from переносятся только готовые артефакты. Итоговый образ не содержит компиляторов и промежуточного мусора, что уменьшает его размер и снижает площадь атаки.

### 10. Оптимизация размера Docker-образа
**Вопрос:** Как можно оптимизировать размер итогового Docker-образа?

**Ответ:** Основные приёмы: выбирать минимальный подходящий базовый образ (slim/alpine вместо full), использовать multi-stage builds, чтобы в финальный образ не попадали build-зависимости и компиляторы, объединять связанные RUN-команды в один слой с очисткой кэша пакетного менеджера в той же команде (иначе кэш попадёт в промежуточный слой и не уменьшит итоговый размер, даже если удалить его следующей инструкцией), исключать ненужные файлы через .dockerignore, и ставить зависимости с флагами `--no-cache-dir` (pip) или аналогичными для других менеджеров пакетов.

```dockerfile
# Объединение команд + очистка кэша в одном слое
RUN apt-get update && \
    apt-get install -y --no-install-recommends gcc && \
    pip install --no-cache-dir -r requirements.txt && \
    apt-get purge -y gcc && \
    apt-get autoremove -y && \
    rm -rf /var/lib/apt/lists/*
```

| Термин | Определение |
|---|---|
| --no-cache-dir | Флаг pip, запрещающий сохранение кэша скачанных пакетов |
| Объединение RUN | Слияние команд в один RUN, чтобы очистка кэша реально уменьшала слой |
| .dockerignore | Исключение ненужных файлов из контекста сборки |

**Ответ для интервьюера:** Уменьшаю размер через выбор минимального базового образа, multi-stage builds, чтобы build-инструменты не попадали в финальный слой, объединение RUN-команд с очисткой кэша пакетного менеджера в той же инструкции, и .dockerignore, чтобы лишние файлы вообще не попадали в контекст сборки.

### 11. Файл .dockerignore
**Вопрос:** Для чего нужен файл .dockerignore?

**Ответ:** .dockerignore задаёт список файлов и директорий, которые не отправляются в build context — набор данных, который клиент Docker передаёт демону при сборке образа. Это ускоряет сборку (меньше данных передаётся и меньше потенциальных инвалидаций кэша от случайных изменений в файлах вроде .git или локальных логов), уменьшает риск случайно закопировать в образ секреты (.env, ключи), кэш пакетных менеджеров или артефакты локальной разработки (venv, __pycache__, node_modules), и снижает итоговый размер образа, если такие файлы всё же попадают в COPY.

```
# .dockerignore
.git
.env
__pycache__/
*.pyc
.venv/
node_modules/
tests/
README.md
```

| Термин | Определение |
|---|---|
| Build context | Набор файлов, передаваемых демону Docker при docker build |
| .dockerignore | Файл с шаблонами исключаемых из контекста путей |
| Утечка секретов | Случайное попадание конфиденциальных файлов в образ |

**Ответ для интервьюера:** .dockerignore исключает файлы из build context — не даёт случайно скопировать в образ .env с секретами, .git, локальные venv или кэши, а заодно ускоряет саму сборку, потому что демону передаётся меньше данных и меньше шансов зря инвалидировать кэш слоёв.

## Docker Compose и оркестрация

### 12. docker-compose и основные команды
**Вопрос:** Что такое docker-compose и для чего он нужен? Назовите основные команды.

**Ответ:** Docker Compose — инструмент для описания и запуска многоконтейнерных приложений через единый декларативный YAML-файл, где для каждого сервиса задаются образ или Dockerfile, переменные окружения, порты, volumes и зависимости между сервисами. Он избавляет от необходимости вручную запускать `docker run` с длинным списком флагов для каждого контейнера и автоматически создаёт общую сеть между сервисами. Основные команды: `docker compose up` (поднять все сервисы, `-d` — в фоне), `docker compose down` (остановить и удалить контейнеры и сеть), `docker compose build` (пересобрать образы), `docker compose logs` (посмотреть логи), `docker compose ps` (статус сервисов).

```yaml
services:
  app:
    build: .
    ports:
      - "8000:8000"
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

| Термин | Определение |
|---|---|
| Сервис (service) | Описание одного контейнера в docker-compose.yml |
| depends_on | Указание порядка запуска сервисов |
| docker compose up / down | Поднять / остановить и удалить всё окружение |

**Ответ для интервьюера:** Docker Compose описывает многоконтейнерное приложение одним YAML-файлом — сервисы, сети, volumes, зависимости между ними — и поднимает всё одной командой `docker compose up` вместо серии ручных `docker run`. Основные команды — up, down, build, logs, ps.

### 13. Переменные окружения в Docker и Compose
**Вопрос:** Как в Docker и Docker Compose организована работа с переменными окружения?

**Ответ:** В Dockerfile переменные задаются инструкцией `ENV` и становятся частью образа (видны на этапе сборки и во всех контейнерах из него). При запуске контейнера значения можно переопределить или добавить через `docker run -e KEY=value` или `--env-file`. В Docker Compose переменные задаются в секции `environment` сервиса или через `env_file`, указывающий на файл (обычно `.env`), причём Compose также автоматически подхватывает файл `.env` из корня проекта для подстановки переменных прямо в сам docker-compose.yml (например, `${DB_PASSWORD}`). Секреты в переменные окружения напрямую в docker-compose.yml класть нежелательно — их держат в .env, исключённом из Git, или используют Docker secrets/внешние секрет-хранилища.

```yaml
services:
  app:
    build: .
    env_file:
      - .env
    environment:
      - LOG_LEVEL=info
      - DB_HOST=db
```

| Термин | Определение |
|---|---|
| ENV (Dockerfile) | Переменная, зафиксированная в образе на этапе сборки |
| env_file | Файл со списком переменных окружения для сервиса |
| Подстановка ${VAR} | Compose подставляет значения из .env в сам YAML-файл |

**Ответ для интервьюера:** В Dockerfile ENV фиксирует переменные в образе, при запуске их можно переопределить через -e или --env-file. В Compose переменные задаются в environment или через env_file, а Compose дополнительно сам подхватывает .env из корня проекта для подстановки значений в YAML. Секреты в .env держат вне репозитория, добавляя файл в .gitignore.

### 14. Связь контейнеров в одну сеть
**Вопрос:** Как связать несколько контейнеров (например, приложение и базу данных) в одну сеть?

**Ответ:** Docker Compose по умолчанию создаёт отдельную bridge-сеть для проекта и подключает к ней все описанные сервисы, после чего они могут обращаться друг к другу по имени сервиса как по DNS-имени (встроенный DNS Docker резолвит имя сервиса в его текущий внутренний IP) — не нужно ни знать IP-адрес контейнера, ни публиковать порт наружу для внутреннего взаимодействия. При работе с чистым Docker (без Compose) для этого нужно вручную создать пользовательскую сеть через `docker network create` и подключить к ней контейнеры флагом `--network`, поскольку контейнеры в сети `bridge` по умолчанию не резолвят друг друга по имени.

```bash
docker network create app-net
docker run -d --name db --network app-net postgres:16
docker run -d --name app --network app-net -p 8000:8000 myapp
```

```yaml
services:
  app:
    build: .
    environment:
      - DB_HOST=db   # имя сервиса = DNS-имя внутри сети Compose
  db:
    image: postgres:16
```

| Термин | Определение |
|---|---|
| Bridge-сеть | Виртуальная сеть, изолирующая и связывающая контейнеры |
| Встроенный DNS Docker | Резолвинг имени сервиса/контейнера в его внутренний IP |
| --network | Флаг подключения контейнера к пользовательской сети |

**Ответ для интервьюера:** Compose автоматически создаёт общую сеть для всех сервисов проекта и позволяет обращаться друг к другу по имени сервиса благодаря встроенному DNS Docker. В чистом Docker для этого нужно вручную создать пользовательскую сеть через docker network create и подключить к ней контейнеры флагом --network, так как контейнеры сети bridge по умолчанию не резолвятся по имени.

## CI/CD пайплайны

### 15. Артефакты и кэш в CI
**Вопрос:** Что такое артефакты и кэш в CI?

**Ответ:** Артефакт — это файл или набор файлов, полученный на одном этапе пайплайна (собранный дистрибутив, Docker-образ, отчёт о тестах, coverage-файл) и сохранённый для использования на последующих этапах или для скачивания после завершения пайплайна. Кэш — это переиспользуемые между запусками пайплайна данные, которые дорого пересоздавать заново каждый раз (установленные зависимости pip/npm, скачанные Docker-слои): кэш привязывается к ключу (обычно хэшу lock-файла) и восстанавливается, если ключ совпал с предыдущим запуском, что ускоряет пайплайн. Ключевое отличие — артефакт всегда актуален для конкретного запуска и часто является конечным результатом, а кэш — это оптимизация скорости, и его отсутствие не должно ломать пайплайн, а лишь замедлять его.

|Характеристика|Артефакт|Кэш|
|---|---|---|
|**Назначение**|Результат этапа для использования далее/скачивания|Ускорение повторных запусков|
|**Критичность**|Часто конечный результат пайплайна|Опциональная оптимизация|
|**Поведение при отсутствии**|Пайплайн не может продолжиться|Пайплайн просто работает медленнее|
|**Привязка**|К конкретному запуску пайплайна|К ключу (обычно хэш lock-файла)|

```yaml
# GitHub Actions
- uses: actions/cache@v4
  with:
    path: ~/.cache/pip
    key: pip-${{ hashFiles('requirements.txt') }}

- uses: actions/upload-artifact@v4
  with:
    name: coverage-report
    path: coverage.xml
```

| Термин | Определение |
|---|---|
| Артефакт | Результат этапа пайплайна, сохранённый для использования далее |
| Кэш | Переиспользуемые данные для ускорения повторных запусков |
| Cache key | Ключ, по которому кэш восстанавливается между запусками |

**Ответ для интервьюера:** Артефакт — это результат работы этапа пайплайна (билд, отчёт, образ), который передаётся дальше по пайплайну или сохраняется для скачивания. Кэш — это оптимизация скорости: переиспользуемые данные вроде установленных зависимостей, привязанные к ключу и не критичные для корректности пайплайна при промахе кэша.

### 16. Этапы пайплайна для Python-приложения в Docker
**Вопрос:** Какие этапы обычно входят в пайплайн для Python-приложения в Docker? Назовите минимум 4.

**Ответ:** Типичный пайплайн включает: (1) lint — статическая проверка кода линтерами (ruff, flake8) и форматтерами; (2) test — запуск юнит- и интеграционных тестов с подсчётом покрытия; (3) build — сборка Docker-образа приложения; (4) push — публикация образа в container registry с тегом (обычно по SHA коммита или версии); (5) deploy — выкладка нового образа на целевое окружение (staging/prod), часто с дополнительным шагом smoke-теста после деплоя. Этапы обычно выстраиваются последовательно с ранним падением (fail-fast) — если lint или тесты упали, сборка и деплой не запускаются, чтобы не тратить время и не выкатывать заведомо сломанный код.

```yaml
stages:
  - lint
  - test
  - build
  - push
  - deploy
```

| Термин | Определение |
|---|---|
| Fail-fast | Остановка пайплайна на первом упавшем этапе |
| Container registry | Хранилище Docker-образов (Docker Hub, GHCR, ECR) |
| Smoke test | Быстрая проверка после деплоя, что сервис в принципе жив |

**Ответ для интервьюера:** Минимальный набор — lint, test, build, push образа в registry и deploy на целевое окружение. Этапы идут последовательно по принципу fail-fast: если упали линтер или тесты, сборка образа и деплой не запускаются, чтобы не тратить ресурсы и не выкатить сломанный код.

### 17. Где хранится конфигурация пайплайна
**Вопрос:** Где хранится конфигурация пайплайна?

**Ответ:** Конфигурация CI/CD хранится как код (Pipeline as Code) прямо в репозитории проекта рядом с исходным кодом — например, `.github/workflows/*.yml` для GitHub Actions, `.gitlab-ci.yml` для GitLab CI, `Jenkinsfile` для Jenkins. Такой подход даёт версионирование конфигурации пайплайна вместе с кодом, code review изменений пайплайна через обычный pull request и возможность иметь разную конфигурацию в разных ветках. CI-система сама подхватывает файл конфигурации при событии-триггере (push, pull request) без ручной настройки через внешний UI для каждого изменения.

```yaml
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest
```

| Термин | Определение |
|---|---|
| Pipeline as Code | Хранение конфигурации CI/CD в виде файла в репозитории |
| .github/workflows | Директория конфигурации GitHub Actions |
| Jenkinsfile | Файл описания пайплайна для Jenkins |

**Ответ для интервьюера:** Конфигурация хранится как код прямо в репозитории — .github/workflows для GitHub Actions, .gitlab-ci.yml для GitLab CI, Jenkinsfile для Jenkins. Это даёт версионирование и code review изменений пайплайна наравне с обычным кодом.

### 18. Триггеры в CI/CD
**Вопрос:** Для чего нужны триггеры в CI/CD и какие условия их срабатывания бывают?

**Ответ:** Триггер определяет, какое событие запускает пайплайн, чтобы не гонять его вручную и не выполнять на каждый чих. Частые типы триггеров: push в определённую ветку или по маске (например, только в main или feature/*), открытие или обновление pull/merge request (обычно запускает lint+test, но не деплой), создание тега или релиза (часто триггерит деплой в prod), расписание по cron (ночные регрессионные прогоны), и ручной запуск (workflow_dispatch в GitHub Actions) для контролируемого деплоя по кнопке. Разные события обычно запускают разные подмножества этапов — деплой в prod почти никогда не триггерится автоматически на каждый push, только на тег/релиз или ручное подтверждение.

```yaml
on:
  push:
    branches: [main]
  pull_request:
  schedule:
    - cron: "0 3 * * *"
  workflow_dispatch:
```

| Термин | Определение |
|---|---|
| Триггер (trigger) | Событие, запускающее выполнение пайплайна |
| workflow_dispatch | Ручной запуск пайплайна по кнопке в GitHub Actions |
| cron-триггер | Запуск пайплайна по расписанию |

**Ответ для интервьюера:** Триггеры определяют, какое событие запускает пайплайн: push в ветку, открытие pull request, создание тега/релиза, cron-расписание или ручной запуск. Обычно разные события запускают разные наборы этапов — например, деплой в prod триггерится только тегом или ручным подтверждением, а не любым push.

### 19. Хранение секретов в пайплайне
**Вопрос:** Как и где безопасно хранить секреты (пароли, токены) в пайплайне?

**Ответ:** Секреты никогда не хранят в коде или в самом файле конфигурации пайплайна — их кладут в защищённое хранилище секретов CI-системы (GitHub Actions Secrets, GitLab CI/CD Variables с флагом masked/protected, HashiCorp Vault, AWS Secrets Manager) и обращаются к ним через переменные окружения, которые CI подставляет во время выполнения, не выводя значение в логи. Дополнительные меры: ограничение доступа секретов только на нужные ветки/окружения (protected branches/environments), ротация токенов, минимальные права у выдаваемых учётных данных и запрет на вывод секретов в лог командой echo или через set -x.

```yaml
# GitHub Actions
steps:
  - name: Deploy
    env:
      DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
    run: ./deploy.sh
```

| Термин | Определение |
|---|---|
| Secrets store | Защищённое хранилище чувствительных значений в CI |
| Masked variable | Переменная, значение которой скрывается в логах CI |
| Protected environment | Окружение с ограниченным доступом к его секретам |

**Ответ для интервьюера:** Секреты храню в защищённом хранилище CI-системы (GitHub Actions Secrets, GitLab CI Variables, Vault), а не в коде или конфиге пайплайна, и подставляю их как переменные окружения во время выполнения — CI-система автоматически маскирует их в логах. Дополнительно ограничиваю доступ к секретам через protected environments и минимальные права токенов.

### 20. Проверка покрытия тестами и линтеры
**Вопрос:** На каком этапе пайплайна целесообразно проводить проверку покрытия тестами и работу линтеров?

**Ответ:** Линтеры и статический анализ (ruff, flake8, mypy) целесообразно запускать первым этапом, сразу после checkout кода, до тестов и тем более до сборки образа — это самая быстрая проверка, и нет смысла тратить время на запуск тестов или сборку Docker-образа, если код не проходит даже базовые синтаксические и стилевые правила. Проверку покрытия тестами запускают вместе с самими тестами (test-этап), сразу после lint, до build — падение по недостаточному покрытию (например, coverage ниже порога в 80%) должно останавливать пайплайн так же, как и падение самих тестов, ещё до того, как собирается образ для деплоя.

```yaml
- name: Lint
  run: ruff check .
- name: Test with coverage
  run: pytest --cov=app --cov-fail-under=80
```

| Термин | Определение |
|---|---|
| Статический анализ | Проверка кода без его выполнения (линтеры, mypy) |
| --cov-fail-under | Порог покрытия, ниже которого сборка считается неуспешной |
| Fail-fast порядок | lint → test/coverage → build → deploy |

**Ответ для интервьюера:** Линтеры запускаю первым этапом сразу после checkout — это самая дешёвая проверка, нет смысла тратить время на тесты и сборку, если код не проходит базовый стиль. Покрытие тестами проверяю вместе с самими тестами, до сборки образа, с порогом (--cov-fail-under), ниже которого пайплайн падает так же, как при упавших тестах.

## Инфраструктура и процессы развертывания

### 21. Workflow от ветки в Git до деплоя
**Вопрос:** Опиши типичный Workflow работы над задачей: от создания ветки в Git до автоматического деплоя.

**Ответ:** Разработчик создаёт ветку от актуального main по конвенции именования (feature/..., fix/...), пишет код и коммитит, затем открывает pull request — это автоматически триггерит CI-пайплайн с lint и тестами, а также запрашивает code review у коллег. После одобрения review и зелёного статуса CI ветку сливают в main (обычно squash-merge), что триггерит следующий пайплайн: сборку Docker-образа, его публикацию в registry с тегом по коммиту, и деплой на staging — часто автоматический. Деплой в prod либо происходит автоматически после успешных проверок на staging, либо требует ручного подтверждения (approval gate) или создания git-тега/релиза, после чего CD-система выкатывает новую версию (rolling update, blue-green или canary) и по необходимости позволяет быстро откатиться на предыдущий образ.

```
feature/branch → commit → PR (CI: lint+test) → code review →
merge to main → CI (build+push image) → deploy to staging →
manual approval / tag → deploy to prod
```

| Термин | Определение |
|---|---|
| Pull request | Запрос на слияние ветки с ревью и автоматическими проверками |
| Approval gate | Ручное подтверждение перед выкладкой в prod |
| Rolling update | Постепенная замена старых экземпляров новыми без простоя |

**Ответ для интервьюера:** Ветка от main, коммиты, pull request с автоматическим CI (lint, тесты) и код-ревью, слияние в main триггерит сборку и публикацию образа и деплой на staging, а в prod выкатка идёт либо автоматически после проверок, либо через ручное подтверждение или тег релиза, с возможностью быстрого отката.

### 22. Роль балансировщика нагрузки (Nginx)
**Вопрос:** Какую роль в инфраструктуре играет балансировщик нагрузки (например, Nginx)?

**Ответ:** Балансировщик нагрузки принимает весь входящий трафик на единую точку входа и распределяет запросы между несколькими одинаковыми экземплярами (репликами) приложения по алгоритму (round-robin, least connections, по хэшу IP), что позволяет горизонтально масштабировать сервис и повышает отказоустойчивость — если один инстанс упал, балансировщик через health check перестаёт направлять на него трафик и продолжает работу с оставшимися. Nginx как балансировщик также часто совмещает эту роль с terminate TLS (HTTPS), отдачей статики, сжатием ответов и reverse proxy к бэкенду, скрывая внутреннюю топологию сервисов от внешнего клиента.

```nginx
upstream app_backend {
    server app1:8000;
    server app2:8000;
    server app3:8000;
}

server {
    listen 80;
    location / {
        proxy_pass http://app_backend;
    }
}
```

| Термин | Определение |
|---|---|
| Load balancing | Распределение запросов между несколькими экземплярами сервиса |
| Health check | Периодическая проверка живости бэкенд-инстанса |
| Reverse proxy | Прокси, принимающий запросы от клиента и перенаправляющий их бэкенду |

**Ответ для интервьюера:** Балансировщик распределяет входящий трафик между репликами приложения, обеспечивая горизонтальное масштабирование и отказоустойчивость — упавший инстанс исключается из ротации по health check. Nginx как reverse proxy дополнительно часто берёт на себя TLS-терминацию, отдачу статики и сжатие ответов.

### 23. API Gateway и взаимодействие с сервисами
**Вопрос:** Что такое API Gateway и как он взаимодействует с внутренними сервисами?

**Ответ:** API Gateway — это единая точка входа для внешних клиентов в микросервисную архитектуру: он принимает все запросы, а затем маршрутизирует каждый к нужному внутреннему сервису на основе пути, домена или заголовков, скрывая от клиента реальную топологию и количество сервисов за собой. Помимо маршрутизации Gateway обычно берёт на себя сквозные (cross-cutting) задачи, которые иначе пришлось бы дублировать в каждом сервисе: аутентификацию и авторизацию, rate limiting, логирование и трейсинг запросов, агрегацию ответов нескольких сервисов в один для клиента, и версионирование API. Внутренние сервисы при этом обычно недоступны напрямую снаружи и общаются либо напрямую по внутренней сети, либо через service mesh.

```yaml
# упрощённый пример маршрутизации API Gateway
routes:
  - path: /api/users/*
    service: user-service:8001
  - path: /api/orders/*
    service: order-service:8002
```

| Термин | Определение |
|---|---|
| API Gateway | Единая точка входа, маршрутизирующая запросы к микросервисам |
| Cross-cutting concerns | Сквозные задачи: аутентификация, rate limiting, логирование |
| Service mesh | Инфраструктурный слой для управления взаимодействием сервисов |

**Ответ для интервьюера:** API Gateway — единая точка входа для клиентов, которая маршрутизирует запросы к внутренним микросервисам и берёт на себя общие для всех задачи: аутентификацию, rate limiting, логирование, агрегацию ответов. Внутренние сервисы напрямую снаружи недоступны и скрыты за Gateway, что упрощает эволюцию архитектуры без изменений на стороне клиента.

### 24. Docker vs системы оркестрации (Kubernetes)
**Вопрос:** В чем разница между деплоем в Docker и использованием систем оркестрации, таких как Kubernetes?

**Ответ:** Голый Docker (или Docker Compose) хорошо подходит для запуска приложения на одной машине — он не умеет сам распределять контейнеры между несколькими серверами, автоматически перезапускать упавшие контейнеры на другом хосте, балансировать нагрузку между узлами кластера или делать zero-downtime rolling update без ручного вмешательства. Kubernetes — это система оркестрации контейнеров поверх кластера серверов: она автоматически распределяет поды (группы контейнеров) по узлам, следит за их состоянием и перезапускает упавшие (self-healing), масштабирует количество реплик по нагрузке (autoscaling), делает rolling update и откат версий декларативно, и предоставляет встроенные service discovery и балансировку между подами. Плата за эти возможности — существенно более высокая сложность настройки и эксплуатации по сравнению с обычным Docker.

|Характеристика|Docker / Docker Compose|Kubernetes|
|---|---|---|
|**Масштаб**|Одна машина|Кластер серверов|
|**Self-healing**|❌ Ручной перезапуск|✅ Автоматический|
|**Autoscaling**|❌|✅ По нагрузке|
|**Rolling update / rollback**|Вручную|✅ Декларативно|
|**Service discovery**|Через Compose-сеть (в рамках хоста)|✅ Встроенный, в масштабе кластера|
|**Сложность эксплуатации**|Низкая|Высокая|

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:1.0
          ports:
            - containerPort: 8000
```

| Термин | Определение |
|---|---|
| Оркестрация | Автоматическое управление жизненным циклом контейнеров в кластере |
| Self-healing | Автоматический перезапуск упавших контейнеров/подов |
| Pod | Минимальная единица развёртывания в Kubernetes (один или несколько контейнеров) |

**Ответ для интервьюера:** Docker и Compose хороши для запуска на одной машине, но не умеют сами распределять нагрузку между серверами, перезапускать упавшие контейнеры на другом узле и делать rolling update без ручных действий. Kubernetes добавляет оркестрацию поверх кластера: self-healing, autoscaling, декларативные обновления и service discovery, ценой значительно более высокой сложности эксплуатации.

### 25. Фиксация версий зависимостей
**Вопрос:** Зачем нужно фиксировать версии зависимостей (например, через pip freeze или poetry.lock) при сборке проекта?

**Ответ:** Без фиксации версий каждая установка зависимостей может подтянуть новые (мажорные или минорные) версии пакетов, что ломает воспроизводимость сборки: код может работать на машине разработчика и падать в CI или на проде из-за незаметного изменения поведения транзитивной зависимости. Lock-файл (poetry.lock, Pipfile.lock, requirements.txt с точными версиями через `pip freeze`) фиксирует не только прямые, но и все транзитивные зависимости с точными версиями и хэшами, гарантируя, что на любой машине и в любой момент времени будет установлен побитово тот же набор пакетов. Это особенно критично в Docker-образах, чтобы пересборка образа через полгода не привела к неожиданно другому поведению приложения из-за случайно обновившейся зависимости.

```bash
# requirements.txt с фиксированными версиями
pip freeze > requirements.txt

# Poetry — воспроизводимая установка строго по lock-файлу
poetry lock
poetry install --no-root
```

| Термин | Определение |
|---|---|
| Lock-файл | Файл с точными версиями всех прямых и транзитивных зависимостей |
| Транзитивная зависимость | Зависимость зависимости, не указанная явно проектом |
| Воспроизводимая сборка | Идентичный набор пакетов при установке в любой момент времени |

**Ответ для интервьюера:** Без фиксации версий сборка не воспроизводима — новая мажорная версия транзитивной зависимости может тихо сломать поведение приложения в CI или на проде. Lock-файл (poetry.lock, requirements.txt с точными версиями) фиксирует весь граф зависимостей с точными версиями и хэшами, гарантируя одинаковый набор пакетов при любой пересборке образа.
