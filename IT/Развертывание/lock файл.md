### 🔹 Зачем нужен lock-файл?
**Основная задача:** Гарантировать **идентичность окружения** на всех стадиях (dev, test, prod).

| Проблема без lock-файла | Решение с lock-файлом |
|------------------------|----------------------|
| `requirements.txt` с версиями `>=1.0` | `poetry.lock` с точными версиями `1.2.3` |
| Сегодня установилось `1.2`, завтра `1.3` (сломало прод) | Всегда устанавливается `1.2.3` |
| Разные версии транзитивных зависимостей | Фиксируется **всё дерево зависимостей** |

### 🔹 Что внутри?
```toml
# poetry.lock
[[package]]
name = "requests"
version = "2.28.1"  # ✅ Точная версия
description = "..."
files = [...]       # ✅ Хэши для проверки целостности
```

### 🔹 Best Practices
1.  **Commit to VCS:** `poetry.lock` и `requirements.txt` (сгенерированный) должны быть в git.
2.  **Не редактировать вручную:** Обновлять через `poetry update` или `pip-compile`.
3.  **CI/CD:** Использовать `poetry install --no-root` или `pip install -r requirements.txt` для гарантии воспроизводимости.

### 🔹 Сравнение инструментов
| Инструмент | Файл | Особенности |
|------------|------|-------------|
| **pip** | `requirements.txt` | Нужно генерировать через `pip-tools` для.lock-эффекта |
| **Poetry** | `poetry.lock` | Автоматически, фиксирует всё дерево |
| **Pipenv** | `Pipfile.lock` | Аналогично Poetry, JSON формат |

> **Для интервью:** «Lock-файл фиксирует точные версии всех пакетов (включая транзитивные), чтобы сборка была детерминированной. Без него возможна ситуация "у меня работает, а на проде нет" из-за автообновления зависимостей.»
