**Q2: Почему Kafka для потока событий, а не RabbitMQ?**

Скелет ответа:

- Kafka: replayability (перемотка потока) — ключевое требование
- При изменении формул аналитики — перечитываем события с нужного offset
- RabbitMQ: сообщения удаляются после consume, нет rewind
- 500K событий/ч — Kafka лучше для throughput
- Trade-off: Kafka сложнее в эксплуатации (ZooKeeper/KRaft, партиции)

---

**Q3: Как устроены приоритетные очереди в Celery?**

Скелет ответа:

- Celery поддерживает multiple queues с разным priority
- Критичные задачи (уведомления об остатках) → high-priority queue
- Обычные задачи (отчёты) → default queue
- Воркеры обрабатывают high-priority первыми
- Конфиг: `task_routes`, `task_queues`, `worker_prefetch_multiplier`

**Q7: Как работает RabbitMQ с DLQ?**

Скелет ответа:

- persistent: сообщения сохраняются на диск, не теряются при рестарте
- ack: подтверждение после обработки (не auto-ack)
- DLQ (Dead Letter Queue): при N retry — сообщение уходит в DLQ
- Сценарий: пиковая нагрузка → воркер не успевает → в DLQ
- Потом: ручной или автоматический replay из DLQ
- Результат: 0 потерь данных при пиковых нагрузках

### Продукт для потребителя

**SaaS-дашборд «Маяк»** — веб-приложение для селлеров, где они видят:

- Аналитику продаж: выручка, маржа, динамика по SKU и категориям
- Мониторинг конкурентов: цены, наличие, рекламные кампании
- Рекомендации по ценообразованию: «подними цену на 8% — спрос не упадёт»

Пользователь подключает свой магазин WB/Ozon через API-ключи → платформа собирает данные → дашборд показывает инсайты → рекомендации помогают повысить маржу.

### Внутренние сервисы

1. **API Gateway (FastAPI)** — точка входа для фронтенда и внешних интеграций. Принимает запросы от дашборда, маршрутизирует к backend-сервисам, агрегирует ответы.
    
2. **Data Collector** — сервис сбора данных с API WB/Ozon. Периодически опрашивает маркетплейсы, забирает данные о продажах, ценах конкурентов, остатках. Выгружает в Kafka.
    
3. **Analytics Engine** — ядро аналитики: агрегирует сырые данные, считает метрики (выручка, маржа, оборачиваемость), строит отчёты. Работает с большими таблицами, использует CTE и оконные функции.
    
4. **Recommendation Service (FastAPI)** — сервис рекомендаций по ценам. Анализирует историю продаж, цены конкурентов, сезонность. Выдаёт рекомендованную цену для максимизации маржи.
    
5. **Event Pipeline (Kafka)** — поток событий: сбор данных → обработка → уведомление → дашборд. Обеспечивает replay для пересчёта аналитики при изменении формул.
    
6. **Task Queue (Celery + RabbitMQ)** — фоновые задачи: генерация отчётов, пересчёт метрик, уведомления. Приоритетные очереди: критичные задачи (остатки) обрабатываются первыми.
    
7. **Telegram Bot (Aiogram)** — мгновенные уведомления селлерам: остатки < 10 шт, конкурент снизил цену на 15%, бюджет израсходован на 90%.

**Клиент/домен:** Издательство настольных игр «Мир хобби» — розничные магазины + онлайн-продажи + маркетплейсы.

**Бизнес-контекст:** Компания развивает онлайн-продажи через собственный интернет-магазин и интеграции с маркетплейсами (WB, Ozon). Внутренняя платформа объединяет каталог товаров, складской учёт, агрегацию данных. Каталог тормозит при 30+ фильтрах, данные о остатках рассинхронизированы между сайтом, 1С, складами и маркетплейсами. Баги в продакшене — 5 в неделю, регресс-тестирование занимает 4 часа вручную.

### Продукт для потребителя

**Интернет-магазин настольных игр** — каталог с 30+ фильтрами (жанр, цена, кол-во игроков, время партии, возраст, издатель, наличие). Пользователь выбирает игруку, оформляет заказ, получает доставку. Также товары продаются на WB/Ozon — и остатки должны быть синхронизированы между всеми каналами, чтобы не было «товар закончился, а на сайте есть».

### Внутренние сервисы

1. **Каталог товаров (FastAPI)** — REST API каталога с поддержкой 30+ фильтров. Обслуживает запросы от фронтенда интернет-магазина. Каждый запрос — фильтрация, сортировка, пагинация. Напрямую работает с PostgreSQL.
    
2. **Поисковая система (Elasticsearch)** — полнотекстовый поиск по каталогу, автодополнение, фасетный поиск. Данные из PostgreSQL индексируются в Elasticsearch асинхронно через RabbitMQ.
    
3. **Агрегатор данных маркетплейсов** — сервис, который каждые 30 минут сверяет остатки одного товара на сайте, в 1С, на складах WB и Ozon. Выявляет расхождения, формирует задачи на синхронизацию.
    
4. **ETL Pipeline** — пайплайн сбора данных с API маркетплейсов (WB, Ozon). Забирает данные о продажах, остатках, ценах. Обрабатывает до 1 млн записей в день.
    
5. **Message Bus (RabbitMQ)** — асинхронное взаимодействие между микросервисами: изменение остатка на сайте → обновление на маркетплейсах, новый заказ → резерв на складе. Гарантированная доставка через persistent + ack + DLQ.
    
6. **Кэш-слой (Redis)** — кэширование результатов популярных запросов к каталогу, сессии пользователей, блокировки для предотвращения race conditions.

### Архитектура и масштаб

Микросервисы на Python (FastAPI) → PostgreSQL (основные данные) + MongoDB (неструктурированные данные) + Elasticsearch (поиск) → RabbitMQ (асинхронное взаимодействие) + Redis (кэш) → Jenkins CI/CD. 1 млн записей/день через ETL, 15мин → 5-10сек задержка поиска.

### Мой вклад (5 достижений):

- **API каталога 30+ фильтров** — FastAPI, select_related/prefetch_related + составные индексы, 1,5с → 400мс, -35% БД
- **PostgreSQL → Elasticsearch** — асинхронная индексация через RabbitMQ, 15мин → 5-10сек

**Стэк:** Python, FastAPI, PostgreSQL, MongoDB, SQLAlchemy, Alembic, Pytest, unittest, Elasticsearch, RabbitMQ, Jenkins, Redis

##### Реализовал сбор данных с API WB/Ozon, генерацию отчётов, пересчёт метрик в фоновые задачи Celery + RabbitMQ. Внедрил приоритетные очереди: критичные задачи (уведомления об остатках) обрабатываются в первую очередь. Результат: время ответа API сократилось с 2,5 с до 400 мс (p95), пиковая нагрузка обрабатывается без деградации.

> **Celery:** Это распределённая очередь задач на Python. Архитектура: брокер (RabbitMQ) хранит задачи, отдельные worker-процессы забирают и выполняют их, результат пишется в бэкенд (Redis/DB). Это позволяет вынести тяжёлые синхронные операции (парсинг, генерация отчётов) из event loop FastAPI, освобождая его для клиентских запросов.
> 
> **Замер p95:** В FastAPI я использовал кастомный middleware, который замеряет `request_start` и `response_end`. Метрики экспортировались в Prometheus через `prometheus-fastapi-instrumentator`. На графиках в Grafana смотрел `http_request_duration_seconds{quantile="0.95"}`. Именно так мы зафиксировали снижение с 2.5с до 400мс.
> 
   **Пиковая нагрузка:** Это не частота обновления данных, а резкий рост RPS (например, утро понедельника, запуск рекламной кампании, массовая выгрузка отчётов). Раньше при пиках RabbitMQ забивался, воркеры не успевали, начинались таймауты и HTTP 503.
> 
   **Деградация и фикс:** Проблема была в том, что статусы задач и метрики писались в БД отдельными синхронными запросами, создавая lock contention и фрагментацию индексов. Я перевёл запись на Celery с использованием `priority queues` и пакетной обработки через транзакции/Redis Pipeline. Да, каждая операция стала на 10-20мс дольше из-за атомарности, но система перестала падать под нагрузкой, а API стабилизировался на p95 < 400мс.


	**1. Async vs Celery:** `async def` работает в event loop (кооперативная многозадачность в одном потоке), идеален для IO-bound (сеть, БД). Celery — это отдельные OS-процессы (`prefork`/`gevent`), выполняющие задачи параллельно на разных ядрах. Подходит для CPU-bound (тяжёлые расчёты, парсинг) и долгих IO-операций, не блокируя основной поток FastAPI.
	
	**2. Забитая очередь:** Запускаем отдельные инстансы воркеров под разные очереди: `celery -A proj worker -Q notifications -c 10` и `celery -A proj worker -Q reports -c 2`. Для `notifications` ставим `prefetch_count=1`, чтобы воркер не захватывал 100К задач в RAM. Либо используем `x-max-priority` в RabbitMQ.
	
	**3. PromQL p95:** `histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[15m])) by (le))`. Функция `rate` считает скорость роста бакетов, `histogram_quantile` математически вычисляет 95-й перцентиль за 15 минут.
	
	**4. Транзакции:** Снижают throughput из-за overhead на блокировки, MVCC-снапшоты и `fsync` WAL. Но повышают стабильность, исключая гонки данных, частичные записи и каскадные ошибки при падении соединений.

##### Оптимизировал аналитические запросы для дашбордов: заменил N+1 запросы SQLAlchemy на CTE и оконные функции в raw SQL, добавил составные индексы по (seller_id, date, category). Результат: время загрузки отчёта «Продажи по SKU» сократилось с 6,5 с до 800 мс, нагрузка на БД в пик упала на 40%.

> «**N+1 в SQLAlchemy** возникает при ленивой загрузке (`lazy="select"`): ORM делает 1 запрос на получение сущностей и N отдельных запросов на связанные объекты. Раньше я решал это через `joinedload` или `selectinload`, но для аналитических дашбордов это всё равно создавало сложные `JOIN`-цепочки и временные таблицы.
> 
> **Почему перешёл на CTE + оконные функции + raw SQL:** Для отчётов «Продажи по SKU» нужно было считать ранги, скользящие средние и доли рынка в одном проходе. Оконные функции (`SUM() OVER(PARTITION BY ...)`) делают это без `GROUP BY` и без выноса агрегаций в память приложения. CTE (`WITH ... AS`) позволил структурировать запрос и дать оптимизатору PostgreSQL материализовать промежуточные выборки.
> 
> **Составные индексы:** Создал индекс `(seller_id, date, category)`. Порядок критичен: сначала фильтрация по = (`seller_id`), затем диапазон (`date`), затем точное совпадение (`category`). Это работает благодаря B-Tree Left-Prefix Rule.
> 
> **Замеры:** Латентность API фиксировал middleware + Prometheus-экспортер (`p95 < 800ms`). Нагрузку на БД контролировал через `pg_stat_statements` (топ тяжёлых запросов), `pg_stat_activity` (блокировки/коннекты) и метрики `postgres_exporter` в Grafana. Пиковую нагрузку валидировал нагрузочными тестами в K6/Яндекс.Танк.»

**1. N+1 vs Оконные функции vs CTE:**

```sql
-- ❌ N+1 (SQLAlchemy lazy load)
SELECT * FROM sellers; 
-- Для каждого seller_id: SELECT * FROM orders WHERE seller_id = ?;

-- ✅ Оконная функция (решает задачу аналитики, а не N+1)
SELECT 
    sku, 
    sale_date, 
    revenue,
    SUM(revenue) OVER(PARTITION BY seller_id ORDER BY sale_date) AS running_total,
    RANK() OVER(PARTITION BY category ORDER BY revenue DESC) AS rank_in_category
FROM sales;

```

CTE (`WITH monthly_stats AS (...)`) позволяет разбить монолитный запрос на логические блоки. В PG12+ CTE по умолчанию инлайнится, но `MATERIALIZED` форсирует кеширование промежуточного результата.

**2. Составные индексы (Left-Prefix Rule):** Индекс `(A, B, C)` эффективен для:

- `WHERE A = ?`
- `WHERE A = ? AND B > ?`
- `WHERE A = ? AND B = ? AND C = ?` **Не эффективен** для: `WHERE B = ?` или `WHERE C = ?` или `WHERE B = ? AND C = ?`. Это связано с тем, как B-Tree строит отсортированное дерево: сначала сортировка по A, внутри A — по B, внутри B — по C.

**3. Мониторинг и валидация оптимизации:**

- **EXPLAIN (ANALYZE, BUFFERS):** Показывает реальный план выполнения, число просканированных строк (`rows removed by filter`), использование кэша (`Shared Hit`).
- **pg_stat_statements:** Расширение PG, логирующее статистику по каждому уникальному запросу (calls, total_time, rows, shared_blks_hit).
- **Prometheus:** `pg_up`, `pg_stat_activity_count`, `pg_stat_database_tup_fetched`, `postgres_locks`.

	«**joinedload vs selectinload:** `joinedload` добавляет `LEFT OUTER JOIN` к основному запросу. Это один SQL-запрос, что хорошо для связей 1:1 или 1:N с малым числом записей. Минус: риск "взрыва строк" (Cartesian Explosion) и дублирования данных в памяти. `selectinload` делает два запроса: первый получает основные сущности, второй — связанные через `WHERE fk_id IN (...)`. Это предотвращает дублирование, эффективнее при больших 1:N связях и потребляет меньше RAM.
	
	**Индекс (status, created_at):** Запрос `WHERE created_at > '2025-01-01'` **не использует** этот индекс эффективно из-за правила левого префикса (Left-Prefix Rule). B-Tree отсортирован сначала по `status`, затем по `created_at`. Без фильтрации по `status` оптимизатор выберет `Seq Scan`.
	
	**EXPLAIN vs ANALYZE:** `EXPLAIN` строит план на основе статистики таблиц. `EXPLAIN ANALYZE` реально выполняет запрос, замеряет время и показывает фактическое число строк. Планы могут отличаться, если статистика устарела, включен параллельный план или параметры сессии (`work_mem`) отличаются.
	
	**Поиск блокировок/медленных запросов:** Для этого используется системное представление `pg_stat_activity`. Запрос `SELECT * FROM pg_stat_activity WHERE state != 'idle' AND wait_event IS NOT NULL;` покажет все сессии, которые ждут блокировок или IO. Дополнительно `pg_locks` помогает найти цепочки блокировок.»


##### Разработал Телеграм-бота для мгновенных уведомлений селлерам по критическим параметрам: «остаток товара < 10 шт», «конкурент снизил цену на 15%», «рекламный бюджет израсходован на 90%». Результат: 70% активных пользователей подключили бота, время реакции на критичные события сократилось с часов до секунд.

> «**Архитектура:** Бот построен на `aiogram` (async), работает через Webhook на FastAPI-эндпоинте. События генерируются в основном сервисе (например, `PriceDropEvent`), публикуются в Kafka-топик `notifications.telegram`. Отдельный Celery-воркер подписывается на топик, маппит `seller_id` → `telegram_chat_id` (из PostgreSQL), и отправляет сообщение через Telegram Bot API с ретраями и экспоненциальным бэкоффом при 429.
> 
> **Адаптивный мониторинг:** Для рекламного бюджета настроена динамическая периодичность проверок: в Celery Beat задача `check_ad_budget` имеет `crontab(minute='*/30')`, но если `spent > 70%`, задача через `add_periodic_task` пересоздаётся с интервалом 10 минут. Это снижает нагрузку на внешние API, не теряя точности у критических порогов.
> 
> **Замер метрики «часы → секунды»:** Мы не сравнивали «исторические данные». Мы внедрили логирование в ClickHouse: `event_id`, `event_ts`, `notification_sent_ts`, `user_click_ts` (если пользователь нажал на inline-кнопку в боте). Затем аналитики считали `median(user_click_ts - event_ts)` для когорты с ботом и без. Разница: 4.2 часа → 47 секунд.
> 
> **Безопасность и надёжность:** Уведомления идемпотентны (хэш `seller_id + event_type + timestamp`), при недоступности Telegram события буферизуются в Redis с TTL 24 часа. Маппинг `seller_id → chat_id` кэшируется в Redis, инвалидируется при `/stop` бота.»

	**1. 403 Forbidden:** При получении 403/401 от Telegram API мы не делаем retry. Воркер ловит исключение, помечает `seller.telegram_chat_id` как `inactive` в PostgreSQL, инвалидирует ключ в Redis и переключает последующие уведомления на email-канал. В логи пишется 1 запись уровня `WARN`. Это экономит ресурсы и защищает от spam-loops.
	
	**2. Тестирование адаптивного поллинга:**
	
	- _Unit:_ Мокаем `fetch_ad_spent`, передаём `spent=75%`, проверяем, что вызван `add_periodic_task` с правильным `crontab(minute='*/10')` и `queue='high_priority'`.
	- _Integration:_ Поднимаем тестовый Celery + RabbitMQ в Docker. Запускаем задачу, эмулируем изменение порога, верифицируем через `celery.control.inspect()` или query к БД расписаний, что новая периодичность применена.
	
	**3. Redis-кэш и риски:** Используем паттерн `Cache-Aside`. Риск: рассинхрон (кэш жив, а в БД уже `/stop`). Смягчение: при `/stop` бот-сервис пишет `is_active=False` в БД и делает `DEL` в Redis. При промахе кэша воркер идёт в БД и заполняет Redis заново. TTL 24ч — это fallback, а не основной механизм консистентности.
	
	**4. Обработка `/stop` и «мёртвых чатов»: Перед отправкой воркер проверяет `is_active` в кэше/БД. Если `False` или приходит 403 — событие маршрутизируется в DLQ или на email. Магия `aiogram` не распространяется на другие сервисы, нужна явная синхронизация состояния через общий источник истины (PostgreSQL + Redis pub/sub).»

> «Потеря данных возникала из-за трёх факторов при пиковых нагрузках:
> 
> 1. Консьюмеры использовали `auto_ack=True`. При падении процесса или перезапуске неподтверждённые сообщения исчезали.
> 2. Очереди и сообщения не были помечены как `durable` и `persistent`. При рестарте брокера RabbitMQ сбрасывал всё в память по умолчанию.
> 3. Бездумный `prefetch` забивал RAM воркеров. RabbitMQ включал `flow control`, а при превышении лимитов начинал отбрасывать сообщения.
>

Время обработки выросло на 15-20% из-за ручных подтверждений и экспоненциальных ретраев, но система стала детерминированной и отказоустойчивой.»
