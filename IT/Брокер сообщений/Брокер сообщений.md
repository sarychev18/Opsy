#flashcards/брокер_сообщений
что такое брокер сообщений ?  :: это промежуточное ПО (middleware) - сервер, который обеспечивает асинхронную связь между сервисами. Он принимает сообщения от отправителей (producers) и доставляет их получателям (consumers). <br><br>Нужен для:<br>• **Слабой связанности** сервисов (decoupling)<br>• **Асинхронности** (сервисы не ждут ответа)<br>• **Балансировки нагрузки**<br>• **Надежности** (сообщения не теряются)<br>• **Масштабирования**<br><br>**Точно НЕ**
- ❌ НЕ работает напрямую с базой данных - брокер работает с сервисом, который работает с БД, например с докером

### 1. Архитектурная философия

|            | **RabbitMQ**                                               | **Kafka**                                                                |
| ---------- | ---------------------------------------------------------- | ------------------------------------------------------------------------ |
| **Модель** | Умный брокер, глупый клиент                                | Глупый брокер, умный клиент                                              |
| **Суть**   | Брокер берёт на себя маршрутизацию, подтверждения, очереди | Брокер только хранит поток сообщений, клиенты управляют своим положением |

### 2. Технические характеристики

| **Критерий**           | **RabbitMQ**                               | **Kafka**                                  |
| ---------------------- | ------------------------------------------ | ------------------------------------------ |
| **Тип**                | Message Broker (брокер сообщений)          | Distributed Event Streaming Platform       |
| **Протоколы**          | AMQP, MQTT, STOMP, HTTP                    | Собственный протокол (binary over TCP)     |
| **Производительность** | 10 000 - 10 000 сообщений/сек              | 1 000 000+ сообщений/сек                   |
| **Задержка (latency)** | Минимальная (microseconds)                 | Чуть выше (из-за batch)                    |
| **Хранение**           | Удаляет после ack                          | Хранит в логе (настраиваемое время/размер) |
| **Порядок сообщений**  | В пределах очереди                         | В пределах партиции                        |
| **Маршрутизация**      | Exchanges (Direct, Topic, Fanout, Headers) | Topics + Partitions                        |
| **Подтверждение**      | Индивидуальное (ack)                       | Автоматическое/ручное commit               |

### **2. Гарантии доставки (delivery guarantees)?**

#flashcards/брокер_сообщений
что такое Гарантии доставки (delivery guarantees) ? ::<br>• **Уровни:**<br>• **At most once** (не более раза) — сообщение может потеряться, но не будет дубликатов(RQ, kafka)<br>• **At least once** (не менее раза) — сообщение точно доставлено, но возможны дубликаты(RQ, kafka)<br>• **Exactly once** (ровно один раз) — идеально, сложно реализовать(kafka)
    

```python

# RabbitMQ: по умолчанию at least once
channel.basic_consume(queue='task', auto_ack=False)  # Ручное подтверждение
channel.basic_ack(delivery_tag=method.delivery_tag)  # Подтверждение
```

### **3. Что такое dead letter queue (DLQ)?**

#flashcards/брокер_сообщений
что такое dead letter queue (DLQ)? :: Очередь мертвых писем — специальная очередь для сообщений, которые не удалось обработать (превышен retry, истек TTL, очередь переполнена). Позволяет не терять "плохие" сообщения, а анализировать их позже.

```python
# Настройка DLQ
arguments = {
    'x-dead-letter-exchange': 'dlx',
    'x-dead-letter-routing-key': 'errors'
}
channel.queue_declare('main_queue', arguments=arguments)
```

### **4. Что такое подтверждение (acknowledgment) и зачем оно нужно?**

#flashcards/брокер_сообщений
что такое подтверждение (acknowledgment) и зачем оно нужно? :: Подтверждение от consumer, что сообщение обработано успешно. Если consumer упал без ack, сообщение возвращается в очередь и отдается другому consumer. Предотвращает потерю сообщений.

```python

def callback(ch, method, properties, body):
    try:
        process(body)
        ch.basic_ack(delivery_tag=method.delivery_tag)  # ✅ Успех
    except:
        ch.basic_nack(delivery_tag=method.delivery_tag, requeue=True)  # Повторить
```

### **5. Что такое prefetch count (QoS)?**

#flashcards/брокер_сообщений
что такое prefetch count (QoS)? :: Ограничение количества неподтвержденных сообщений, которые consumer может получить одновременно. Защищает consumer от перегрузки и помогает балансировать нагрузку.

```python

channel.basic_qos(prefetch_count=1)  # Брать по одному сообщению
```

### **6. Как обрабатывать дубликаты сообщений?**

#flashcards/брокер_сообщений
**Как обрабатывать дубликаты сообщений?** ::<br>**Подходы:**<br>• **Идемпотентность** — операцию можно выполнить много раз с тем же результатом<br>• **Дедупликация** — хранить ID обработанных сообщений в Redis/БД<br>• **Транзакции** — проверять уникальность на уровне БД

```python

def process_with_deduplication(message):
    msg_id = message.headers['id']
    if redis.sismember('processed_ids', msg_id):
        return  # Уже обработано
    # Обработка
    redis.sadd('processed_ids', msg_id)
```

### **7. Что такое backpressure (обратное давление)?**

#flashcards/брокер_сообщений
что такое backpressure (обратное давление)? :: Ситуация, когда producer шлет быстрее, чем consumer успевает обрабатывать. <br>**Решения:**<br>• Ограничить prefetch<br>• Использовать очереди с ограничением длины<br>• Реализовать механизм отказа (throttling)

```python

# Очередь с ограничением длины
channel.queue_declare('tasks', arguments={'x-max-length': 1000})
```

### **9. Что такое RPC (Remote Procedure Call) через брокер?**

#flashcards/брокер_сообщений
что такое RPC (Remote Procedure Call) через брокер?** :: Паттерн, где клиент отправляет сообщение и ждет ответа. Использует correlation ID для связывания запроса и ответа.

```python

# Клиент
result = channel.queue_declare('', exclusive=True)
callback_queue = result.method.queue
corr_id = str(uuid.uuid4())
channel.basic_publish(
    exchange='',
    routing_key='rpc_queue',
    properties=pika.BasicProperties(
        reply_to=callback_queue,
        correlation_id=corr_id
    ),
    body=request
)
# Сервер отвечает в reply_to с тем же correlation_id
```

## Способы асинхронного взаимодействия

Брокеры сообщений предлагают два основных паттерна обмена данными: [getanalyst.ru](https://getanalyst.ru/database/broker-and-message-queues)

1. **Point-to-Point («точка-точка»)**. Используется, когда продюсеры имеют отношения «один на один» с потребителями. То есть сообщение должно быть обработано только один раз и только одним получателем. Пример — обработка финансовых транзакций.
2. **Publish/Subscribe («публикация-подписка»)**. Используется, когда продюсеры (издатели) имеют отношения «один-ко-многим» с потребителями. Один или несколько издателей могут публиковать сообщения по определённой теме (топику), а заинтересованные потребители могут подписаться на неё, чтобы получать эти данные. ### **12. Типичные проблемы и их решение**

#flashcards/брокер_сообщений
Как масштабировать consumers? ::<br>**Проблема - Решение**<br>• **Сообщения накапливаются** - Увеличить consumers, проверить prefetch<br>• **Дубликаты сообщений** - Сделать обработку идемпотентной<br>• **Потеря сообщений** - Включить persistent, publisher confirms<br>• **Медленная обработка** - Оптимизировать consumer, увеличить prefetch<br>• **Deadlock consumers** - Проверить ack, добавить таймауты
