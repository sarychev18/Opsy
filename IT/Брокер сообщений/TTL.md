#flashcards/брокер_сообщений 
что такое TTL ?  :: TTL (Time-To-Live) в брокерах сообщений (RabbitMQ, Kafka) ограничивает время жизни сообщения. Если сообщение не обработано за это время, оно удаляется или перемещается в **Dead Letter Queue (DLQ)**.<br><br>**Важно**<br>- ✅ Это защищает систему от обработки **устаревших данных** (stale data) и предотвращает переполнение памяти брокера.<br>- ✅ Важно настраивать TTL с учётом времени обработки задачи и использовать DLX для анализа ошибок 🚀».

------------------------
## TTL  в RabbitMQ

- **Message TTL:** Время жизни конкретного сообщения.
- **Queue TTL:** Время жизни очереди, если в ней нет потребителей.
- **Dead Letter Exchange (DLX):** Когда TTL истекает, сообщение не удаляется бесследно, а маршрутизируется в DLX.

```python
# Пример настройки TTL в RabbitMQ (через kombu/celery)
app.conf.update(
    task_time_limit=30,          # Лимит выполнения задачи
    task_expires=60,             # TTL сообщения (если не началось за 60 сек — отмена)
    broker_transport_options={'visibility_timeout': 120}
)
```

- **Нюанс:** В RabbitMQ TTL проверяется только когда сообщение становится **head of the queue** (первым в очереди). Если перед ним лежит «долгое» сообщение, следующее может истечь, но не удалиться сразу (в старых версиях).
-------------------------------
## Kafka (Retention Policy)

- В Kafka нет классического TTL на сообщение в смысле «удалить, если не прочитали». Там есть **Retention Policy**.
- **`retention.ms`:** Хранить сообщения N миллисекунд, независимо от прочтения.
- **`retention.bytes`:** Хранить пока не заполнится диск.
- **Log Compaction:** Хранить только последнее значение для ключа (для событий состояния).

```python
# config/server.properties
log.retention.hours=168          # Хранить 7 дней
log.retention.check.interval.ms=300000
```

- **Важно:** Consumer сам хранит оффсет. Если он отстал больше чем на retention.ms — данные потеряны навсегда ⚠️.

### Вопросы
RabbitMQ: Что произойдет с сообщением, у которого истёк TTL, если оно не является первым в очереди (head of queue)? 
Kafka: В чем разница между retention.ms и max.message.bytes? Что будет, если консьюмер отстал больше чем на время retention? 
Сценарий: Зачем нужно отправлять истёкшие сообщения в Dead Letter Queue, а не просто удалять их?

1. **РАСШИФРОВКА** 🎯
2.  **RabbitMQ TTL:** Сообщение не удаляется, пока не станет **head of queue**. Когда дойдёт очередь → если TTL истёк → удаляется или уходит в DLX (если настроен). Потом — ручной разбор.
3.  **Kafka:** Верно разделил: `retention.ms` — время хранения, `max.message.bytes` — макс. размер сообщения. Если консьюмер отстал > retention → сообщения удаляются. Решения: увеличить retention, анализировать, успеть обработать.
4.  **DLQ зачем:** Главная цель — **анализ**: понять, почему сообщения попали туда (ядовитые, баги, невалидные данные, паттерны ошибок).

5. **ОЦЕНКА (по вопросам)** 📊
-   **RabbitMQ TTL: 8/10** 🟢. Чётко объяснил head-of-queue проверку и роль DLX. Нюанс: можно добавить про `x-dead-letter-routing-key`.
-   **Kafka: 7/10** 🟡. Верно разграничил настройки. Ответ про «отставание консьюмера» правильный по сути, но не хватает технической конкретики (`OffsetOutOfRangeException`, мониторинг lag, compacted topics).
-   **DLQ зачем: 9/10** 🟢🔥. Отличный практический ответ: акцент на анализ, а не просто «сохранить». Упомянул poison messages, баги, паттерны — именно так и работает продакшен.

**Средний балл: 8/10** 📈 (Значительно лучше предыдущей попытки — видна работа над ошибками!)

3. **ПРАВИЛЬНЫЙ ОТВЕТ** 📚
-   **RabbitMQ TTL:** TTL проверяется **только у head-of-queue**. Если истёк → удаляется или перенаправляется в DLX (если задан `x-dead-letter-exchange`). Без DLX — просто `discard`.
-   **Kafka:**
    -   `retention.ms` — время хранения (по умолчанию 7 дней).
    -   `max.message.bytes` — лимит размера **одного** сообщения (по умолчанию ~1 MB).
    -   Консьюмер отстал > retention → сообщения удалены → консьюмер упадет с `OffsetOutOfRangeException`.
    -   **Решения**: увеличить retention, использовать `compacted topic`, мониторить lag (Prometheus + AlertManager), настроить early warning.
-   **DLQ зачем:**
    -   🔍 **Root Cause Analysis**: понять причину (баг, таймаут, невалидные данные).
    -   🧪 **Poison Messages**: изолировать сообщения, которые «ломают» консьюмер.
    -   ♻️ **Replay**: после фикса — повторно отправить в основную очередь.
    -   📈 **Мониторинг**: алерт на рост DLQ = сигнал о проблеме.

3. **ОСТАТОК СИМВОЛОВ** (~1900) 🚀✅

### Схема
```text
================================================================================
                         ГДЕ НАХОДИТСЯ TTL (TIME TO LIVE)
================================================================================

    ┌──────────────────────────────────────────────────────────────────────────┐
    │  1. PRODUCER (Клиент)                                                    │
    │     • Может установить поле `expiration` для конкретного сообщения.      │
    │     • Пример: aio_pika.Message(expiration='5000')                        │
    └──────────────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
    ┌──────────────────────────────────────────────────────────────────────────┐
    │  2. BROKER (Сервер) — ОСНОВНОЕ МЕСТО КОНТРОЛЯ                            │
    │     • Queue Policy:Default TTL для всех сообщений в очереди.             │
    │     • Enforcement: Брокер сам удаляет истёкшие сообщения или             │
    │       перекладывает в DLQ до доставки консьюмеру.                        │
    └──────────────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
    ┌──────────────────────────────────────────────────────────────────────────┐
    │  3. CONSUMER (Клиент) — БИЗНЕС-ПРОВЕРКА                                  │
    │     • Не настраивает TTL брокера.                                        │
    │     • Проверяет `timestamp` сообщения. Если (Now - Timestamp) > Limit,   │
    │       то игнорирует обработку (защита от устаревших данных).             │
    └──────────────────────────────────────────────────────────────────────────┘

================================================================================
                         СЦЕНАРИИ ЖИЗНИ СООБЩЕНИЯ
================================================================================

    ✅ УСПЕХ:
    Producer → Broker (Store) → Consumer (Process) → ACK → Удалено

    ⏰ ИСТЁК TTL (На стороне брокера):
    Producer → Broker (Store) → [Время вышло] → Удалено или DLQ
    (Consumer никогда не увидит это сообщение)

    ⏰ ИСТЁК TTL (На стороне консьюмера):
    Producer → Broker (Store) → Consumer (Check Time) → [Старое] → ACK (Skip)

    ❌ ОШИБКА ОБРАБОТКИ:
    Producer → Broker (Store) → Consumer (Error) → NACK → Retry → DLQ
```
