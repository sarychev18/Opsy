## 1. Каталог на FastAPI + Elasticsearch: время ответа 1.5с → 400мс

### Контекст

Основной сайт работал на Django — и продолжал работать. Каталог товаров был узким местом: менеджеры постоянно просили новые фильтры — "добавь по возрасту", "добавь по механикам". Каждый новый фильтр в Django требовал изменений в нескольких местах.
Поиск работал через ILIKE в PostgreSQL — медленно и без релевантности. Решили не переписывать весь Django, а вынести каталог отдельным микросервисом на FastAPI.

### Проблема

Каталог отвечал за **1.5 секунды** при любом запросе.
Поиск через `ILIKE '%каркасс%'` не умел в опечатки, синонимы, морфологию.
Пользователи писали "каркасон" — получали пустой результат вместо "Каркассон".

### Решение

Разделил ответственность между двумя хранилищами:

**PostgreSQL + SQLAlchemy** — для структурированных фильтров:
цена, количество игроков, время партии, возраст, издатель, наличие.
Эти поля числовые или enum — работают через индексы, Elasticsearch здесь избыточен.

**Elasticsearch** — для полнотекстового поиска по названию и описанию:
морфология, нечёткий поиск (fuzziness), релевантность результатов.
"каркасон" → находит "Каркассон".

**Redis** — кэш ответов каталога. Ключ = MD5-хэш всех параметров запроса.
TTL = 5 минут, инвалидация при обновлении данных через ETL.

**FastAPI** вместо Django для API: нативный async, автоматическая валидация
Query-параметров через Pydantic, добавление нового фильтра — 3 строки.

```python
# Elasticsearch для поиска по тексту
hits = await es.search(index="games", body={
    "query": {"fuzzy": {"title": {"value": search, "fuzziness": "AUTO"}}}
})

# PostgreSQL для структурированных фильтров
stmt = select(Game).where(Game.price >= min_price, Game.in_stock == True)
```

### Результат

- Время ответа: **1.5с → 400мс** (при кэш-попадании — 5–10мс)
- Поиск с опечатками и морфологией — пользователи находят то, что ищут
- **30+ фильтров** поддерживаются единообразно

### Почему именно так

Можно было перенести все фильтры в Elasticsearch — отвергли:
числовые диапазонные фильтры (цена, игроки) в PostgreSQL с индексами работают
так же быстро, а поддерживать синхронизацию схем в ES сложнее.
Разделение: ES для поиска, PG для фильтров — каждый инструмент делает то, в чём силён.

## 2. ETL-перестройка: курсорная пагинация и Celery

### Контекст

Мир Хобби продаёт через Ozon, Wildberries и Яндекс.Маркет. Каждый маркетплейс —отдельная система со своим REST API (авторизация через JWT-токен). Через API мы забирали: остатки на складах маркетплейсов, новые заказы, данные о продажах, актуальные цены и акции — всё это нужно для внутренней аналитики и системы отчётности. Существующий скрипт делал это синхронно и запускался по крону раз в несколько часов.

На маркетплейсах у них своя витрина, свои склады (FBO — fulfillment by operator), свои заказы. И маркетплейс не присылает данные сам — нужно их забирать через API.

---
Что именно тянули через ETL — скорее всего три вещи:

1. Заказы — кто что купил, когда, по какой цене. Нужно для учёта, аналитики, отчётности руководству
2. Остатки на складах маркетплейсов — у Ozon свой склад, у WB свой. Фактический остаток там может расходиться с тем, что знает внутренняя система
3. Аналитика продаж — какие игры продаются лучше на каком маркетплейсе, динамика по дням

Почему 1 млн записей/день — это не 1 млн товаров. Это транзакции: заказы, движения по складу, изменения статусов. Три маркетплейса × тысячи SKU × статусы заказов за день — цифра реалистичная.

---
На собеседовании можно сказать так:

▎ "Мир Хобби продаёт через Ozon, Wildberries и Яндекс.Маркет. Каждый маркетплейс — отдельная система с отдельным API. Нам нужно было собирать оттуда данные о продажах и остатках для внутренней аналитики и системы отчётности — в реальном времени видеть что продаётся, где, по какой цене и сколько осталось на складе маркетплейса."

ETL = Extract, Transform, Load

Три шага:

- Extract — забираем данные из внешнего источника (в нашем случае API маркетплейса)
- Transform — приводим к нужному формату (переименовываем поля, меняем типы, чистим мусор)
- Load — загружаем в нашу БД (PostgreSQL)

### Проблема

Скрипт работал синхронно и использовал **offset-пагинацию**.
Три конкретные проблемы:

1. **Rate-limit 429**: маркетплейсы ограничивают количество запросов в минуту.  Скрипт делал запросы без пауз, получал 429 и падал целиком —   следующий запуск начинался с нуля, данные за период терялись

2. **Дубли в БД**: при повторном запуске после падения скрипт создавал   дублирующие записи вместо обновления существующих 

3. **Всё в одном процессе**: падение синхронизации с WB блокировало Ozon и YM — три маркетплейса висели друг на друге

### Решение

**Celery-воркеры**: каждый маркетплейс — отдельная задача в очереди. При 429 задача уходит на retry через 60 секунд, остальные маркетплейсы продолжают работать независимо.

**Курсорная пагинация**: API маркетплейсов отдают данные постранично (например, заказы за период — по 100 штук за запрос). WB и Ozon поддерживают курсорную пагинацию — каждый ответ содержит токен следующей страницы. В отличие от offset, курсор не сбивается при добавлении новых записей во время обхода.  

**Батчинг**: коммитим в БД каждые 100 записей, не держим весь ответ в памяти.  

**Upsert по external_id**: `ON CONFLICT (external_id) DO UPDATE` —повторный запуск обновляет записи, не создаёт дубли.

**MongoDB** как промежуточное хранилище сырых ответов API:схемы ответов маркетплейсов периодически менялись без предупреждения —с MongoDB не нужна миграция при каждом изменении, сохраняем JSON как есть, нормализованные данные пишем в PostgreSQL отдельным шагом.

```python
# При rate-limit — retry, не падение
if e.response.status_code == 429:
    raise task.retry(countdown=60, exc=e)

# Идемпотентная запись в PostgreSQL
stmt = insert(Game).values(items)
stmt = stmt.on_conflict_do_update(
    index_elements=["external_id"],
    set_={col: stmt.excluded[col] for col in ("title", "price", "in_stock", "quantity")},
)
```

### Результат

- Стабильная обработка **до 1 млн записей/день**
- **0 потерь данных** при rate-limit
- **0 дублей** в PostgreSQL
- Сырые данные с маркетплейсов сохранены в MongoDB — можно перепарсить без повторного запроса к API

### Почему именно так

Apache Airflow смотрели — избыточен для 3 источников и нашей команды, Celery уже в стеке.
MongoDB для сырых данных вместо сразу PostgreSQL: схема ответов маркетплейсов
меняется без предупреждения — с Mongo не нужна миграция при каждом изменении API.

## 5. RabbitMQ: гарантированная доставка, 0 потерь сообщений

### Контекст

Между сервисами (каталог, аналитика, уведомления) были прямые синхронные HTTP-вызовы. Система аналитики продаж хранила агрегаты в **MongoDB** (гибкая схема под разные типы отчётов), и каталог дёргал её синхронно после каждого обновления.

### Проблема

Три проблемы от синхронных вызовов:

1. **Потеря событий**: сервис аналитики недоступен — событие теряется навсегда
2. **Каскадный отказ**: медленная аналитика тормозит каталог, хотя они независимы
3. **Нет retry**: при сетевом разрыве — ручной перезапуск

Конкретный случай: во время деплоя сервиса аналитики 15-минутное окно недоступности
привело к потере данных о продажах за этот период.

### Решение

Перевёл взаимодействие на **RabbitMQ** с тремя гарантиями:

**Manual ack**: сообщение подтверждается только после успешной обработки.
Упала обработка → `nack(requeue=False)` → сообщение в DLQ, не потеряно.

**Persistent messages** (`DeliveryMode.PERSISTENT`): RabbitMQ сохраняет на диск.
Рестарт брокера — сообщения остаются.

**Dead Letter Queue**: необработанные сообщения хранятся 24 часа в `catalog.dlq`.
Можно разобрать после фикса, не потерять данные.

**Fanout exchange**: одно событие получают несколько потребителей независимо
(аналитика пишет в MongoDB, уведомления шлют email, ETL-сервис обновляет Elasticsearch).
Добавить нового потребителя — не трогаем продюсера.

```python
async for message in queue_iter:
    try:
        await on_message(json.loads(message.body))
        await message.ack()         # подтверждаем только после успеха
    except Exception:
        await message.nack(requeue=False)  # → DLQ, не потеря
```

### Результат

- **0 потерь сообщений** при рестартах и сетевых разрывах
- Каталог, аналитика (MongoDB) и Elasticsearch независимы друг от друга
- Мониторинг через RabbitMQ Management UI: глубина очередей, DLQ, скорость обработки

### Почему именно так

Kafka рассматривали — отвергли: требует отдельной инфраструктуры, избыточна для наших объёмов.
Kafka подходит для event streaming с replay и retention,
RabbitMQ — для task queue с гарантированной доставкой. У нас второй случай.
