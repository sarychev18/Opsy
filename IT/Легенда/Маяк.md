### 1. 🔹 Сервис рекомендаций по ценообразованию (FastAPI)
#### 🔧 Как это делалось (пошагово)
1. **Сбор данных:** Написал Celery-задачу, которая раз в час забирает историю продаж, цены конкурентов (через API/парсинг) и сезонные коэффициенты.
2. **Фичи и расчёт:** В памяти pandas/numpy строил матрицу `SKU × регион × время`, считал маржу, эластичность спроса и дельту к конкурентам.
3. **Кэширование:** Готовые рекомендации складывал в Redis с ключом `price_rec:{sku}:{region}` и TTL 1ч.
4. **API-слой:** FastAPI-эндпоинт только валидировал вход через Pydantic, бил Redis, отдавал JSON. При cache-miss → фоллбэк на лёгкую эвристику, пока Celery не обновит кэш.
5. **Мониторинг:** Prometheus-метрики `cache_hit_ratio`, `rec_calc_latency`, алерты при падении hit_ratio < 60%.

#### 💡 Архитектурные решения
- Вынес тяжёлую математику из sync-пути → не блокируем Uvicorn-воркеры.
- Redis как слой кэша + fallback → гарантия ответа < 50мс даже при задержке Celery.
- Pydantic валидация на входе → защита от кривых запросов, автогенерация OpenAPI.

#### 🎤 Готовый ответ для интервью
> «Спроектировал FastAPI-сервис рекомендаций цен. Изначально расчёт маржи и сезонности выполнялся синхронно и съедал до 1.2с на запрос. Я вынес тяжёлую логику в Celery-задачу, которая раз в час агрегирует историю продаж, цены конкурентов и сезонность, а результат складывает в Redis. Эндпоинт теперь только валидирует параметры через Pydantic, бьёт кэш и отдаёт готовый JSON. При cache-miss срабатывает лёгкий эвристический фоллбэк, чтобы клиент не ждал пересчёта. В итоге p95 endpoint'а упал до 80мс, а селлеры получают актуальные цены в реальном времени без ручной аналитики».

#### ❓ Что спросят + как отвечать
| Вопрос | Ответ |
|--------|-------|
| `Как боролся с cache stampede?` | Добавил `lock=True` в Redis (SETNX) + jitter в Celery-расписании. Пока один воркер пересчитывает, остальные отдают stale-данные. |
| `Почему не ML-модель сразу?` | Начинал с rule-based эвристики для быстрого time-to-market. Когда набрали 3+ месяца чистых данных, подключили lightgbm-модель, но оставили fallback на правила для cold-start SKU. |

### 2. 🔹 Kafka: ~500K событий/час, реплей потока
#### 🔧 Как это делалось (пошагово)
1. **Топология топиков:** `raw-events` → `enriched-events` → `analytics-ready` → `dashboard-feed`.
2. **Продюсер:** `aiokafka`, батчинг (`batch.size=16KB`, `linger.ms=5ms`), сжатие `lz4`, идемпотентный `producer` (enable_idempotence=true).
3. **Консьюмеры:** Consumer groups по сервисам. Ручное управление офсетами: `auto_offset_reset='earliest'`, коммит только после успешной транзакции в БД.
4. **Идемпотентность:** В payload добавлен `event_id`. В PG: `INSERT ... ON CONFLICT (event_id) DO NOTHING`.
5. **Реплей:** Скрипт `kafka-replay` сбрасывает offset группы на нужный timestamp через AdminClient, консьюмеры перечитывают поток без потери сырых данных.

#### 💡 Архитектурные решения
- Ручные коммиты → гарантия «не потеряем, пока не сохраним в БД».
- Разделение топиков по этапам ETL → независимое масштабирование обогащения и дашбордов.
- Idempotent Producer + `event_id` → защита от дублей при рестарте продюсера.

#### 🎤 Готовый ответ для интервью
> «Настроил событийную шину на Kafka для асинхронной синхронизации микросервисов. Разделил поток на этапы: raw → enriched → analytics → dashboard. Продюсер писал пачками с lz4-сжатием для пропускной способности. Консьюмеры работали в consumer groups и коммитили офсет только после успешной транзакции в PostgreSQL. Чтобы избежать дублей при рестарте, добавил в каждое сообщение `event_id` и делал `ON CONFLICT DO NOTHING` в БД. Главный плюс архитектуры — возможность реплея: когда бизнес менял формулы расчёта маржи, мы просто сбрасывали офсеты группы на нужный timestamp и пересчитывали аналитику заново без потери исходных событий. Стабильно держали ~500K событий/час».

#### ❓ Что спросят + как отвечать
| Вопрос | Ответ |
|--------|-------|
| `Как обрабатывал poison messages?` | DLQ-топик. После 3 retry консьюмер шлёт `ack` и пишет событие в `failed-events` с меткой ошибки. Отдельный воркер разбирает DLQ раз в сутки. |
| `Почему не RabbitMQ?` | RabbitMQ даёт гарантии доставки, но не поддерживает retention и rewind. Нам нужен был replay для пересчёта аналитики → Kafka архитектурно точнее под задачу. |

### 3. 🔹 Celery + RabbitMQ: приоритетные очереди
#### 🔧 Как это делалось (пошагово)
1. **Анализ блокировок:** APM показал, что 70% времени API съедали опросы WB/Ozon и генерация PDF-отчётов.
2. **Разделение очередей:** В RabbitMQ создал `critical` (x-max-priority=10) и `default` (x-max-priority=5).
3. **Роутинг в Celery:** `task_routes = {"tasks.alerts.*": {"queue": "critical", "priority": 9}, "tasks.reports.*": {"queue": "default", "priority": 3}}`.
4. **Retry-логика:** При 429/5xx от API маркетплейсов → `self.retry(exc=e, countdown=60 * 2**self.request.retries, max_retries=5)`.
5. **Мониторинг:** Grafana дашборд: `queue_depth`, `task_latency`, `retry_rate`. Алерт при глубине `critical` > 500.

#### 💡 Архитектурные решения
- Приоритеты через `x-max-priority` → бизнес-критичные задачи не ждут в хвосте.
- Exponential backoff → не DDOSим API маркетплейсов при лимитах.
- Разделение воркеров по очередям → изоляция нагрузки.

#### 🎤 Готовый ответ для интервью
> «Вынес тяжёлые операции — опрос маркетплейсов, генерацию отчётов, пересчёт метрик — в Celery. Настроил RabbitMQ с двумя очередями: `critical` (приоритет 9-10 для уведомлений об остатках) и `default` (приоритет 1-5 для отчётов). В Celery прописал роутинг по имени таска. При 429-ответах от API WB добавил retry с exponential backoff, чтобы не ронять лимиты. Мониторил глубину очередей и latency задач через Prometheus+Grafana. Это разгрузило Uvicorn-воркеры, p95 API упал с 2.5с до 400мс, а критичные уведомления обрабатывались вне общей очереди даже под пиковой нагрузкой».

#### ❓ Что спросят + как отвечать
| Вопрос | Ответ |
|--------|-------|
| `Как масштабировал консьюмеров?` | Запускал отдельных воркеров с `-Q critical,default -c 8` и `-Q critical -c 16`. При росте глубины очереди autoscaling через Docker Swarm/K8s HPA по метрике `rabbitmq_queue_messages`. |
| `Почему не Celery rate limiting?` | Rate limit на уровне брокера (`x-max-priority`) гибче: не режет задачи, а просто поднимает приоритет. Rate limit в Celery глобальный и не учитывает бизнес-контекст. |

### 4. 🔹 Оптимизация PostgreSQL (аналитика)
#### 🔧 Как это делалось (пошагово)
1. **Профилирование:** Включил `pg_stat_statements`, нашёл топ-3 запросов с `mean_time > 3s`.
2. **N+1 фикс:** Заменил циклы `for sale in sales: sale.sku.name` на `selectinload(Sale.sku)`.
3. **CTE + оконные функции:** Заменил подзапросы на `WITH daily AS (...) SELECT sku, SUM(qty) OVER(PARTITION BY sku ORDER BY date)`.
4. **Индексы:** Создал составной `CREATE INDEX CONCURRENTLY idx_sales_filters ON sales(marketplace, category, date)`.
5. **Валидация:** `EXPLAIN ANALYZE` → seq scan ушёл, буферы ↓ на 60%.

#### 💡 Архитектурные решения
- `pg_stat_statements` → объективный поиск bottleneck'ов, не гадание.
- `selectinload` → 2 запроса вместо N+1, нагрузка на сеть ↓ в разы.
- Raw SQL для агрегаций → полный контроль над планом выполнения, ORM не мешает оптимизатору PG.

#### 🎤 Готовый ответ для интервью
> «Оптимизировал слой аналитики через PostgreSQL. Включил `pg_stat_statements` — нашёл запросы с mean_time > 3с. ORM тянул связанные таблицы циклом N+1, переписал на `selectinload`. Тяжёлые агрегации по SKU и датам шли через подзапросы, заменил на CTE с оконными функциями `SUM() OVER(PARTITION BY)`. Под частые фильтры (маркетплейс + категория + дата) добавил составной B-Tree индекс. Прогнал `EXPLAIN ANALYZE` — seq scan ушёл, время отчёта «Продажи по SKU» сократилось с 6.5с до 800мс, нагрузка на БД в пике упала на 40%».

#### ❓ Что спросят + как отвечать
| Вопрос | Ответ |
|--------|-------|
| `Почему не оставил ORM для агрегаций?` | ORM абстрагирует, но для сложных оконных функций генерирует неоптимальный SQL. Raw SQL даёт контроль над планом выполнения, а `EXPLAIN` сразу показывает hotspots. |
| `Как индекс не тормозил writes?` | Создал через `CREATE INDEX CONCURRENTLY` (не блокирует таблицу). Индекс покрывает 80% read-запросов, write-overhead ~5%, что приемлемо для аналитики. |

### 5. 🔹 Telegram-бот для мгновенных уведомлений
#### 🔧 Как это делалось (пошагово)
1. **Фреймворк:** `aiogram` 3.x (нативный async, поддержка вебхуков).
2. **Интеграция:** Подписка на Kafka/RabbitMQ топики `alerts.stock`, `alerts.price`, `alerts.budget`.
3. **Debounce & Rate-limit:** Redis-ключ `notify:{sku}:{type}` с TTL 15 мин. Если событие приходит чаще → подавляю.
4. **Шаблонизация:** Jinja2-рендер сообщений, поддержка markdown, кнопки «Отключить алерты по SKU».
5. **Деплой:** Отдельный сервис в Docker, healthcheck, graceful shutdown при SIGTERM.

#### 💡 Архитектурные решения
- Async-фреймворк → не блокируем event loop, держим 10K+ concurrent подключений.
- Redis debounce → защита от спама при burst-событиях.
- Вебхуки > polling → меньше холостых запросов к API Telegram, ниже latency.

#### 🎤 Готовый ответ для интервью
> «Разработал async-бота на aiogram для мгновенных алертов селлерам. Подписался на Kafka/RabbitMQ топики с событиями: падение остатков, изменение цен конкурентов, расход бюджета. Чтобы не спамить, добавил debounce в Redis: если событие по одному SKU приходит чаще чем раз в 15 минут — подавляю и шлю только итоговый дайджест. Сообщения рендерю через Jinja2-шаблоны, добавил inline-кнопки для управления подписками. Бот быстро набрал 70% adoption, потому что селлеры перестали ждать email-рассылок и видели критичные изменения в секунды. Время реакции на out-of-stock сократилось с часов до секунд».

#### ❓ Что спросят + как отвечать
| Вопрос | Ответ |
|--------|-------|
| `Как обрабатывал отвал Telegram API?` | retry с exponential backoff + DLQ в брокере. Если webhook не доставил 3 раза → сообщение уходит в `failed_alerts` воркер, который шлёт через backup-канал (email/SMS). |
| `Почему не Firebase/OneSignal?` | Telegram уже был основным каналом коммуникации селлеров. Интеграция с ним через бота дала мгновенный onboarding без установки приложений, что ускорило adoption до 70% за 2 недели. |
